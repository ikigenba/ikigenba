# Decision 11 — Telemetry store: in-memory ring-buffer series + snapshot

**Decision.** A new package `dashboard/internal/telemetry` owns the in-memory
history. It holds one fixed-size ring buffer per **series**, plus the current
total-capacity figures the hero charts scale against. Nothing is persisted — the
whole store is RAM and a restart starts it empty (the product's "most recent day,
begins empty after a restart" promise).

```go
package telemetry

const MaxSamples = 1440 // 24h at one sample/minute

// Sample is one reading. Value is bytes; 0 means "unavailable" (a source that
// was absent or errored — see D12/D13), never a sentinel like -1.
type Sample struct {
	At    time.Time
	Value int64
}

// Store is the whole telemetry history: a ring buffer per series key, guarded by
// one mutex because the collector writes while HTTP handlers read. Series keys
// are fixed strings: "mem.free", "disk.free", and per service
// "svc.mem:<name>" / "svc.disk:<name>".
type Store struct {
	mu        sync.Mutex
	series    map[string]*ring
	totalMem  int64 // for the free-memory hero y-axis (0 → totalMem)
	totalDisk int64 // for the free-disk hero y-axis (0 → totalDisk)
}

func NewStore() *Store

// Append records one sample on a series, evicting the oldest when the ring is
// full (MaxSamples). SetCapacities records the current totals.
func (s *Store) Append(key string, sample Sample)
func (s *Store) SetCapacities(totalMem, totalDisk int64)

// Snapshot returns an independent, ordered (oldest→newest) copy of every series
// plus the capacities, so a renderer reads a stable view while the collector
// keeps writing. Mutating the store after Snapshot never changes a prior result.
type Snapshot struct {
	Series    map[string][]Sample
	TotalMem  int64
	TotalDisk int64
}

func (s *Store) Snapshot() Snapshot
```

The ring is a small `[]Sample` + head/len, not a growing slice, so memory is
bounded regardless of uptime (≈28 series × 1440 × 16 B ≈ 640 KB worst case).

**Rejected.**
- **SQLite-backed history.** The product explicitly scopes telemetry to in-memory,
  last-24h, non-persisted; a table + migration is breadth we were told not to buy
  and would change the restart-wipes-history promise.
- **A goroutine-per-series channel fan-in.** One mutex over a map of rings is
  simpler and correct for a once-a-minute writer and an occasional reader; channels
  add machinery with no benefit at this cadence.
- **Returning the live rings from Snapshot.** Handing out internal buffers would let
  a slow render observe a torn half-written series; a copy under lock is cheap at
  this size and removes the data race by construction.
- **`-1`/NaN sentinels for "unavailable".** `0` renders as a flat baseline (the
  agreed behavior so graphs always draw); a magic sentinel would need special-casing
  in every chart builder.

**Verification.**
- R-EZVQ-IQOL — Appending more than `MaxSamples` (1440) samples to one series
  retains exactly the last 1440, in insertion order, oldest evicted first (unit
  test appends 1441+ and asserts length 1440 and that the first retained sample is
  the 2nd appended).
- R-F13M-WIFA — `Snapshot` is independent of the live store: appending to the store
  after taking a snapshot does not change the already-returned snapshot's series
  (unit test snapshots, appends, and asserts the snapshot's lengths/values are
  unchanged).
- R-F2BJ-AA5Z — `Snapshot` carries the capacities set by `SetCapacities`
  (`TotalMem`/`TotalDisk` equal the values recorded), so the hero charts can scale
  `0 → total` (unit test).
