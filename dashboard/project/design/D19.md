# Decision 19 ŌĆö Introspection emits the identity headers (`X-Owner-Id`, `X-Owner-Name`, `X-Owner-Picture`), additively and injection-safe

## Decision

The three loopback `auth_request` introspection endpoints gain three new
response headers on their **allow** path, read from the stamped handle (D18).
Every existing header is emitted **byte-for-byte as today** ŌĆö this is purely
additive.

**Scope boundary.** This decision covers only the dashboard **emitting** the
headers. nginx *forwarding* them (`*/etc/nginx.conf`, `nginx/locations/*.conf`)
and appkit/services *consuming* them live in other trees and are **out of scope**
for this `project/`. Emitted-but-unforwarded headers are inert: nginx forwards
only the headers each service fragment explicitly maps, so nothing downstream
changes until a service opts in later. The dashboard's own tests assert the
introspection **response** headers directly, which is fully verifiable in
isolation.

**The three new headers**, on each endpoint's existing 200/allow branch:

| header | value |
|---|---|
| `X-Owner-Id` | the stamped identity handle (opaque `ids.New()` value) |
| `X-Owner-Name` | percent-encoded identity `name` (empty when absent) |
| `X-Owner-Picture` | percent-encoded identity `picture` URL (empty when absent) |

**Per endpoint** (`internal/server/`):

- `handleAuthn` (bearer) ŌĆö the chain is already loaded; read `chain.owner_id`,
  `a.identity.Lookup(ownerID)`, set the three headers. Existing `X-Owner-Email`,
  `X-Client-Id`, `X-Chain-Id`, `X-Token-Id` unchanged.
- `handleAuthnPAT` (PAT) ŌĆö read the PAT row's `owner_id`, `Lookup`, set the three
  headers. Existing `X-Owner-Email`, `X-Client-Id` (`pat:<public_id>`) unchanged.
- `handleSessionAuthn` (cookie) ŌĆö read the session's `owner_id`, `Lookup`, set
  the three headers **in addition to** the existing `X-Owner-Email` (today this
  endpoint emits only `X-Owner-Email`).

**Encoding seam.** A small helper percent-encodes a UTF-8 attribute for safe
header transport:

```go
// headerEncode returns a US-ASCII, CR/LF-free encoding of an identity
// attribute for an HTTP header value; empty in ŌåÆ empty out.
func headerEncode(s string) string // url.QueryEscape / PathEscape-style %-encoding
```

`X-Owner-Id` is already ASCII (`ids.New()` output) and is set verbatim.
`X-Owner-Name`/`X-Owner-Picture` pass through `headerEncode`. An empty attribute
yields an empty header value (the header is still set, but carries nothing to
decode ŌĆö never a sentinel like `%00`).

If `Lookup` returns not-found (a token stamped before D18 existed, i.e. never on
a fresh box), the endpoint still allows exactly as today and simply omits the
three new headers ŌĆö identity absence never turns an allow into a deny.

## Rejected

- **Emit raw (un-encoded) name/picture.** A Unicode `name` or a stray CR/LF
  would be non-ASCII or could inject/split a header. Percent-encoding is
  mandatory for `name`; applying it uniformly to `picture` too keeps one code
  path.
- **Emit name/picture as separate base64 vs percent-encoding.** Either is
  ASCII-safe; percent-encoding is human-readable in logs and matches the URL
  nature of `picture`. Chosen for legibility.
- **Also change nginx forwarding / appkit consumption here.** Out of this
  `project/`'s scope (other trees); folding them in would violate the scope
  boundary and isn't needed for the emission to be correct and tested.
- **Make identity lookup failure a deny.** Identity is display metadata, not an
  authorization input; a missing profile must never revoke access.

## Verification

- R-VULR-MABI ŌĆö a `handleAuthn` **allow** response carries `X-Owner-Id` equal to
  the chain's stamped `owner_id`, plus `X-Owner-Name`/`X-Owner-Picture` equal to
  the percent-encoded identity `name`/`picture`; and `X-Owner-Email`,
  `X-Client-Id`, `X-Chain-Id`, `X-Token-Id` are still present and unchanged from
  their pre-change values.
- R-VX1K-DTSW ŌĆö a `handleAuthnPAT` **allow** response carries `X-Owner-Id`/
  `X-Owner-Name`/`X-Owner-Picture` for the PAT's stamped identity, with
  `X-Owner-Email` and `X-Client-Id` (`pat:<public_id>`) unchanged.
- R-VY9G-RLJL ŌĆö a `handleSessionAuthn` **allow** response now carries
  `X-Owner-Id`/`X-Owner-Name`/`X-Owner-Picture` **in addition to** the existing
  `X-Owner-Email` (previously the only header), all for the session's identity.
- R-VZHD-5DAA ŌĆö an identity whose `name`/`picture` contain a non-ASCII rune
  and/or a CR/LF character is emitted such that the raw `X-Owner-Name`/
  `X-Owner-Picture` header values are pure US-ASCII and contain no CR or LF, and
  percent-decoding a value recovers the original attribute exactly (round-trip).
- R-W0P9-J50Z ŌĆö an identity with empty `name`/`picture` yields empty
  `X-Owner-Name`/`X-Owner-Picture` header values (no `%00` or other sentinel),
  and a stamped token whose identity `Lookup` is not-found still returns the
  allow status with the existing headers intact and no panic.
