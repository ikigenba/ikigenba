# Decision 19 — Introspection always emits the four identity headers from the identities row, and fails closed

## Decision

The three loopback `auth_request` introspection endpoints
(`internal/server/`: `handleAuthn` bearer, `handleAuthnPAT`,
`handleSessionAuthn` cookie) emit the full identity header set on **every**
allow, or deny the request — there is no allow that carries a partial set.

**Scope boundary.** This decision covers only the dashboard **emitting** the
headers. nginx *forwarding* them (`*/etc/nginx.conf`, `nginx/locations/*.conf`)
and appkit/services *consuming* them live in other trees and are **out of
scope** for this `project/`. The dashboard's own tests assert the introspection
**response** headers directly, which is fully verifiable in isolation.

**The header set**, on each endpoint's allow branch:

| header | value | emptiness |
|---|---|---|
| `X-Owner-Id` | the stamped identity handle (opaque `ids.New()` value) | never empty |
| `X-Owner-Email` | the identity's email | never empty |
| `X-Owner-Name` | percent-encoded identity `name` | may be empty |
| `X-Owner-Picture` | percent-encoded identity `picture` URL | may be empty |

`handleAuthn` additionally keeps `X-Client-Id`, `X-Chain-Id`, `X-Token-Id`;
`handleAuthnPAT` keeps `X-Client-Id` (`pat:<public_id>`) and `X-Token-Id`
exactly as before.

**Single source: the identities row.** On every allow the endpoint resolves the
artifact's stamped `owner_id` via `a.identity.Lookup(ownerID)` and reads **all
four values from the returned identity** — including the email. The
`owner_email` columns on the artifact rows (`web_sessions`, `oauth_chains`,
`personal_tokens`, `oauth_authcodes`) are mint-time copies kept for audit and
owner-scoped queries; they no longer feed headers, so a stale copy can never
disagree with what services see. (Those columns are vestigial pending the
suite-wide owner-id re-keying — see D24 — at which point they can be dropped.)

**Fail closed.** A `Lookup` failure of any kind — `sql.ErrNoRows` or a real DB
error — is a **500** with no identity headers, logged loudly
(`authn.identity_lookup` / `session_authn.identity_lookup`). It is never a
degraded 200. D24's foreign key makes not-found structurally impossible for a
stored `owner_id`, so this branch firing at all means the system is broken and
must surface, not limp. This supersedes the earlier stance that identity
absence stays an allow: identity is no longer optional display metadata — the
handle is the owner key the platform is converging on, and an allow without it
would poison downstream trust.

**Encoding seam** (unchanged): `headerEncode` percent-encodes `name`/`picture`
into US-ASCII, CR/LF-free header values; empty in → empty out (the header is
still set, never a sentinel). `X-Owner-Id` and `X-Owner-Email` are set
verbatim; both are non-empty by construction (`owner_id` is `NOT NULL` +
FK-backed per D23/D24, `identities.email` is `NOT NULL` per D17).

```go
// headerEncode returns a US-ASCII, CR/LF-free encoding of an identity
// attribute for an HTTP header value; empty in → empty out.
func headerEncode(s string) string
```

## Rejected

- **Best-effort emission (allow with headers omitted when `Lookup` fails).**
  The prior design. It hid a broken identity graph inside a 200 and made the
  header contract "usually": services could never rely on `X-Owner-Id` being
  present, which blocks keying anything on it. Against *fail loudly* — an
  unknown state is unacceptable.
- **Keep `X-Owner-Email` sourced from the artifact row.** Two sources of the
  same fact that can drift (`ResolveOrCreate` refreshes `identities.email` on
  every login; the artifact copy is frozen at mint). With the identity row
  guaranteed to exist there is no reason to keep the second source live.
- **Fail closed on empty `name`/`picture` too.** Google does not guarantee the
  claims and the schema stores them nullable; denying access over missing
  display fluff would block logins for no integrity gain.
- **Emit raw (un-encoded) name/picture.** A Unicode `name` or a stray CR/LF
  would be non-ASCII or could inject/split a header; percent-encoding is
  mandatory and uniform.

## Verification

- R-VULR-MABI — a `handleAuthn` **allow** response carries all four identity
  headers — `X-Owner-Id` equal to the chain's stamped `owner_id`,
  `X-Owner-Email` equal to the identity's email, `X-Owner-Name`/
  `X-Owner-Picture` equal to the percent-encoded identity `name`/`picture` —
  plus `X-Client-Id`, `X-Chain-Id`, `X-Token-Id` as before.
- R-VX1K-DTSW — a `handleAuthnPAT` **allow** response carries the same four
  identity headers for the PAT's stamped identity, with `X-Client-Id`
  (`pat:<public_id>`) unchanged.
- R-VY9G-RLJL — a `handleSessionAuthn` **allow** response carries the same four
  identity headers for the session's stamped identity.
- R-HW5G-3JJF — when an artifact row's stored `owner_email` **differs** from
  the identity row's current email, every endpoint's `X-Owner-Email` carries
  the **identities-row** value, proving the artifact copy no longer feeds
  headers.
- R-HSHQ-Y8BC — a `handleAuthn` request whose otherwise-valid chain has an
  `owner_id` the identity store cannot resolve (not-found or store error)
  returns **500** with **no** `X-Owner-*` headers — never a 200.
- R-HTPN-C021 — the same unresolvable-identity condition on `handleAuthnPAT`
  returns **500** with no `X-Owner-*` headers.
- R-HUXJ-PRSQ — the same unresolvable-identity condition on
  `handleSessionAuthn` returns **500** with no `X-Owner-*` headers.
- R-VZHD-5DAA — an identity whose `name`/`picture` contain a non-ASCII rune
  and/or a CR/LF character is emitted such that the raw `X-Owner-Name`/
  `X-Owner-Picture` header values are pure US-ASCII and contain no CR or LF,
  and percent-decoding a value recovers the original attribute exactly
  (round-trip).
- R-HXDC-HBA4 — an identity with empty `name`/`picture` yields an **allow**
  whose `X-Owner-Name`/`X-Owner-Picture` headers are present with empty values
  (no sentinel), while `X-Owner-Id`/`X-Owner-Email` are non-empty.
