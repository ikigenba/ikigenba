# Decision 17 — Identity model: `(iss, sub)` is the identity, an opaque handle is its durable key, in a new `identities` table

## Decision

The dashboard's concept of "who a user is" moves from **email** to the OIDC
subject pair **`(iss, sub)`**. Email, name, and picture become **attributes** of
that identity, not its key. A new store owns this.

**Schema — the `identities` table.** A new forward-only migration, created with
`bin/create-migration dashboard add_identities` (timestamped, immutable per the
suite migration rule), adds:

```sql
CREATE TABLE identities (
    id         TEXT PRIMARY KEY,          -- opaque handle: ids.New()
    iss        TEXT NOT NULL,             -- OIDC issuer, e.g. https://accounts.google.com
    sub        TEXT NOT NULL,             -- OIDC subject (stable per issuer)
    email      TEXT NOT NULL,             -- mutable attribute (last login wins)
    name       TEXT,                      -- mutable attribute, nullable (may be absent)
    picture    TEXT,                      -- mutable attribute, nullable (public URL, may be absent)
    created_at TEXT NOT NULL,
    updated_at TEXT NOT NULL,
    UNIQUE (iss, sub)
);
```

The primary key `id` is the **durable identity handle** — an opaque
`ids.New()` value (see D-conventions / research), *not* the email and *not* a
sequential integer. It is the value later stamped onto auth artifacts (D18) and
emitted to services (D19). `(iss, sub)` is the natural key used only at login to
find-or-create the row.

**Store — `internal/identity`.** A new package with a `Store` over `*sql.DB`,
mirroring the construction shape of `internal/pat` / `internal/session` (a
`Store` struct with an injectable `Now func() time.Time`). Illustrative shape:

```go
package identity

// Claims are the identity-bearing id_token fields captured at login.
type Claims struct {
    Iss, Sub, Email, Name, Picture string
}

// Identity is a stored identities row.
type Identity struct {
    ID, Iss, Sub, Email, Name, Picture string
}

type Store struct {
    DB  *sql.DB
    Now func() time.Time
    New func() string // ids.New; injectable so a test can assert the handle source
}

// ResolveOrCreate upserts by (iss, sub): inserts a new row with a fresh
// handle when the pair is unseen, otherwise refreshes email/name/picture on the
// existing row. Returns the durable handle either way.
func (s *Store) ResolveOrCreate(ctx context.Context, c Claims) (id string, err error)

// Lookup returns the stored identity for a handle; a distinguishable
// not-found (sql.ErrNoRows-shaped) for an unknown handle.
func (s *Store) Lookup(ctx context.Context, id string) (Identity, error)
```

`ResolveOrCreate` is a single upsert against the `UNIQUE (iss, sub)` constraint
(`INSERT … ON CONFLICT (iss, sub) DO UPDATE SET email=…, name=…, picture=…,
updated_at=…`), returning the surviving row's `id`. Because it is keyed on
`(iss, sub)`, an email change re-uses the same handle; a repeat login is
idempotent on identity and last-write-wins on attributes.

The store is constructed once at the composition root
(`cmd/dashboard/main.go` / `server.Register`) and shared with the callback and
introspection handlers.

## Rejected

- **Keep email as the identity key.** It is what everything used before, but
  email is mutable by OIDC's own contract; keying identity on it is the bug
  we're correcting. Nothing depends on the old behavior (no data to preserve),
  so this is the cheap moment to fix it.
- **Use the Google `sub` directly as the cross-service handle.** Opaque enough,
  but it couples every downstream consumer to Google and to `sub`'s per-issuer /
  pairwise quirks; a provider change would strand consumers. A local opaque
  handle decouples them at the cost of one column.
- **Introduce a ULID/UUID dependency for the handle.** `internal/ids` already
  provides an opaque, collision-free, timestamp-free random id that PATs and
  OAuth tokens use. Reusing it is consistent and dependency-free; ULID's only
  extra property (lexical time-sort) is not needed for an identity handle.
- **Emit `(iss, sub)` as two headers instead of one handle.** Forces every
  consumer to always pair two values and never split them, and re-exposes the
  provider coupling above. One opaque handle is cleaner.

## Verification

- R-VJMO-6CN9 — `ResolveOrCreate` with a `(iss, sub)` not present inserts exactly
  one new `identities` row and returns that row's `id`; a subsequent `Lookup` of
  the returned id yields the same `(iss, sub)`/email/name/picture. (Real temp
  `modernc.org/sqlite` migrated by the appkit runner — the substrate that
  enforces the schema and the `UNIQUE (iss, sub)` upsert.)
- R-VKUK-K4DY — the returned handle is an opaque `ids.New()`-sourced value, not
  the email and not a sequential integer: injecting a stub `New` returns that
  exact stub value as the row `id`, and two identities created with the real
  generator receive different, non-numeric ids.
- R-VM2G-XW4N — `ResolveOrCreate` called twice with the **same** `(iss, sub)`
  returns the **same** `id` both times and leaves exactly **one** row for that
  pair (no second row inserted), even when the second call carries a different
  email/name/picture.
- R-VNAD-BNVC — `ResolveOrCreate` on a repeat `(iss, sub)` whose email/name/
  picture differ **updates** the stored attributes to the new values (a
  following `Lookup` returns the second call's email/name/picture, last-login-
  wins), while `id` and `created_at` are unchanged.
- R-VOI9-PFM1 — `Lookup` of a handle that was never created returns the
  distinguishable not-found result (not a zero-value identity presented as
  found), so callers can tell "absent" from "empty".
