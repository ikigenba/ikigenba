# Decision 14 — HTTP surface: `/telemetry` page, `/telemetry/fragment`, the 60s poll

**Decision.** Two new session-gated routes on the apex table, plus one `*app`
field and a poll block in `app.js`. The page and its refresh fragment reuse the
exact in-process session seams the profile/grants pages already use — no nginx
`auth_request`, no new gate.

```go
// server.Options gains:
Telemetry *telemetry.Store   // the shared store the collector writes (D13)

// *app gains the field; register() mounts:
mux.Handle("GET /telemetry",          a.handleTelemetry())
mux.Handle("GET /telemetry/fragment", a.handleTelemetryFragment())
```

- **`handleTelemetry`** — full page. Resolves the session via `a.sessionOwner`; a
  **missing/invalid session redirects `303 → /`** (the profile-page pattern:
  telemetry is a place you navigate to, so send a signed-out visitor to the login
  landing, not a bare 401). A valid session renders `ui/html/telemetry.html` from
  the store's `Snapshot()` (charts built by D15), buffered-then-written so a
  template error is a clean 500.
- **`handleTelemetryFragment`** — just the charts block (the `telemetry_charts`
  partial) rendered from a fresh `Snapshot()`. It is the poll target, so a
  **missing session returns `401`** (the grants-fragment pattern: an XHR gets a
  status, not a redirect body).
- **The poll (`app.js`).** A block mirroring the live-grants client, but a plain
  `setInterval` rather than SSE (the data only moves on the 60s cadence, so a poll
  matches it and needs no long-lived connection): find `#telemetry-block`, read its
  `data-fragment` URL, and every **60000 ms** `fetch` it (`credentials:
  "same-origin"`) and swap `innerHTML`; on a non-OK response leave the stale block.
- **Templates.** `ui/html/telemetry.html` and `ui/html/partials/telemetry_charts.tmpl`
  are added to the `template.ParseFS(ui.Files, …)` list in `server.go`, so a broken
  telemetry template fails startup, not a request.

**Rejected.**
- **SSE for the refresh.** The live-grants block uses SSE because grants change on
  arbitrary events; telemetry changes only every 60s, so a poll is the simpler,
  cadence-matched fit — no write-deadline clearing, keepalives, or subscription bus.
- **A JSON `/telemetry/data` endpoint + client-side charting.** The chosen Option A
  keeps all chart geometry in Go (testable) and renders real SVG on first paint;
  a JSON+JS approach moves untested drawing into the browser and paints blank until
  JS runs.
- **Redirecting the fragment too.** A `303` in an XHR poll response is noise; the
  fragment returns a status the client can just ignore.
- **A shared page+fragment handler.** They differ in signed-out behavior (redirect
  vs 401) and in whether they emit the page chrome; two small handlers over one
  render helper is clearer than one branching on a query flag.

**Verification.**
- R-FI68-9AT0 — `GET /telemetry` with a valid session returns `200` and a body
  containing the telemetry charts container (http test with a minted session).
- R-FJE4-N2JP — `GET /telemetry` with no/invalid session returns `303` with
  `Location: /` and no charts (http test).
- R-FKM1-0UAE — `GET /telemetry/fragment` with a valid session returns `200` and
  the charts-block markup (http test).
- R-FLTX-EM13 — `GET /telemetry/fragment` with no session returns `401` and does
  not render the charts (http test).
- R-FN1T-SDRS — the served `app.js` wires the telemetry poll: it references the
  telemetry block, fetches its fragment URL, and uses a `60000` ms interval
  (served-asset text assertion).
