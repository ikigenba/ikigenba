# Decision 16 — Landing entry: a tile linking to the telemetry page

**Decision.** The signed-in landing (`GET /` with a session) gains a **tile/link to
`/telemetry`**, placed near the service list. It is the only entry point to the
telemetry page (the operator chose a landing tile over a banner link). The
logged-out login page carries **no** such link — telemetry is owner-only.

The link is added to `ui/html/index.html` inside the signed-in (`{{if .Owner}}`)
branch, styled with the existing landing chrome (a `.tile`/`.list` idiom already
present) — no new component, no change to the banner (D10) or the service rows
(D5). The href is the literal `/telemetry`.

```html
{{if .Owner}}
  …
  <a class="telemetry-tile" href="/telemetry">Telemetry</a>
  …
{{end}}
```

**Rejected.**
- **A banner link (shared chrome).** The operator chose a landing tile; the banner
  stays the two-control identity nav D10 settled.
- **Showing the tile when logged out.** Telemetry is session-gated; advertising it
  on the login page would offer a link that only redirects back to login.
- **A new tile component/CSS system.** The landing already has list/tile chrome;
  reuse it rather than introduce a second visual language for one link.

**Verification.**
- R-FWT0-UJPC — the signed-in landing (`GET /` with a valid session) renders a link
  whose href is `/telemetry` (http test asserts the anchor is present).
- R-FY0X-8BG1 — the logged-out login page (`GET /` with no session) does **not**
  contain a `/telemetry` link (http test asserts its absence).
