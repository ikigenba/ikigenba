# Decision 25 — GitHub identity provider: `internal/githubidp`, a sibling seam with REST-read identity and membership

**Decision.** GitHub sign-in gets its own provider package,
`dashboard/internal/githubidp`, a **sibling** of `internal/googleidp` — not a
generalization of it. The two providers share a *shape* (an `AuthorizeURL` /
`ExchangeCode` seam behind an interface, a live impl, a stub double) but not
code: Google's identity arrives in a signature-verified OIDC id_token, GitHub's
arrives by REST reads with a short-lived user token (research: GitHub issues no
id_token; authenticity comes from the token exchange being performed directly
against `github.com` with the client secret). Forcing one abstraction over both
would bury that real difference.

**The issuer constant.** GitHub identities are stored with
`Iss = "https://github.com"` and `Sub` = the **numeric GitHub user id** rendered
as a decimal string (the immutable id; `login` is user-mutable and is never the
subject). With D17's `(iss, sub)` keying this makes every GitHub identity a
distinct `identities` row that can never collide with a Google row — the
"never unified" product promise falls out of the existing schema.

**Package shape** (mirroring `googleidp`'s layout: `Credentials`, a `Provider`
interface, a live impl, `NewStub`):

```go
package githubidp

// Credentials configures the live provider.
type Credentials struct {
    ClientID     string // GITHUB_CLIENT_ID
    ClientSecret string // GITHUB_CLIENT_SECRET
    Org          string // GITHUB_ORG — the org whose active members are admitted
}

// Identity is what a completed GitHub login knows about the human.
type Identity struct {
    Iss           string // always "https://github.com"
    Sub           string // numeric user id, decimal string
    Login         string // GitHub username (mutable; display/fallback only)
    Email         string // primary email from /user/emails
    EmailVerified bool   // that entry's verified flag
    Name          string // profile name; empty when GitHub returns null
    Picture       string // avatar_url
    OrgMembership string // membership state in Credentials.Org: "active", "pending", or "" (none/404)
}

type Provider interface {
    AuthorizeURL(state, redirectURI string) string
    ExchangeCode(ctx context.Context, code, redirectURI string) (Identity, error)
}
```

`AuthorizeURL` builds `https://github.com/login/oauth/authorize` with
`client_id`, `redirect_uri`, and `state` (no `scope` — a GitHub App's
user-to-server abilities come from the App's permissions; research). No
`prompt=login` equivalent exists; silent re-auth is accepted.

`ExchangeCode` performs, with one retried attempt on 5xx (same policy as
`googleidp`):

1. `POST https://github.com/login/oauth/access_token` (form-encoded, `Accept:
   application/json`) with `client_id`, `client_secret`, `code`,
   `redirect_uri`. Per research, GitHub reports exchange failure as **HTTP 200
   with an `error` field** — the impl fails on a present `error` or an empty
   `access_token`, never trusting status code alone.
2. `GET https://api.github.com/user` — `id`, `login`, `name` (nullable →
   empty), `avatar_url`.
3. `GET https://api.github.com/user/emails` — selects the `primary: true`
   entry for `Email`/`EmailVerified`. No primary entry → error (fail loudly;
   the App's Email-addresses permission makes this an exceptional state).
4. `GET https://api.github.com/user/memberships/orgs/{Org}` — `OrgMembership`
   is the returned `state`; a **404 maps to ""** (not an error — "not a
   member" is a normal answer the callback's gate acts on). Any other non-2xx
   is an error.

The user token lives only inside `ExchangeCode` and is discarded on return —
never stored, never logged. All four hops go through an injectable base-URL
pair (`webBase`, `apiBase`) so tests run against `httptest` fakes; the
**policy decision** (is `OrgMembership` good enough?) is *not* made here — the
provider reports facts, the callback (D28) enforces admission, mirroring how
`googleidp` reports `HostedDomain` and the handler enforces it.

**Stub.** `NewStub()` returns a canned-identity fake with the same shape as
`googleidp.NewStub` — settable `Identity`, settable error — so `server` tests
drive the callback without any network.

**Composition root.** `cmd/dashboard/main.go` requires `GITHUB_CLIENT_ID`,
`GITHUB_CLIENT_SECRET`, and `GITHUB_ORG` at the boundary exactly as it requires
the `GOOGLE_*` triple (presence-checked via `requireEnv`, values never logged),
builds the one live `githubidp` provider, and hands it to `server.Register`
alongside the Google one. The `(*app)` struct gains a second provider field
(`githubProvider githubidp.Provider`) next to the existing `idpProvider`.

**Rejected.**
- **Generalizing `googleidp` into one multi-provider `idp` package.** The two
  verification models (JWKS-verified id_token vs authenticated REST reads)
  share almost no code; a common abstraction would be an interface with two
  unrelated implementations plus a lowest-common-denominator `Identity` that
  hides `HostedDomain`/`OrgMembership` behind vague fields. Sibling packages
  keep each contract explicit.
- **Checking org membership with the App's installation token**
  (`GET /orgs/{org}/members/{username}`). Works, but requires the App private
  key in the dashboard's environment; the key currently lives only in the
  `github` service and should stay confined there. The user's own token can
  answer the same question (research).
- **Trusting `GET /user`'s `email` field.** It is null unless the user opted
  into a public email — unreliable by design. `/user/emails` with the App's
  Email-addresses permission is the reliable source.
- **Keying on `login` instead of the numeric id.** `login` is user-mutable; a
  rename would orphan the identity. The numeric `id` is immutable.
- **Storing or refreshing the user token.** Nothing after the callback needs
  GitHub; storing a live credential would add a secret-at-rest for zero
  capability.

**Verification.**
- R-I4AP-U132 — `AuthorizeURL(state, redirectURI)` returns a URL on
  `https://github.com/login/oauth/authorize` whose query carries exactly the
  configured `client_id`, the given `redirect_uri`, and the given `state`, and
  no `scope` parameter.
- R-I5IM-7STR — `ExchangeCode` against a fake GitHub (httptest `webBase`/
  `apiBase`) that returns a valid token then `/user` `{id: 583231, login:
  "octocat", name: "The Octocat", avatar_url: …}`, `/user/emails` with a
  `primary: true, verified: true` entry, and an `active` org membership, yields
  `Identity{Iss: "https://github.com", Sub: "583231", …}` with the email,
  `EmailVerified: true`, name, picture, and `OrgMembership: "active"` populated
  from those responses — proving the numeric id becomes the decimal-string
  subject and the primary email (not `/user`'s null `email`) is used.
- R-I6QI-LKKG — `ExchangeCode` fails with an error when the token endpoint
  answers **HTTP 200 with an `error` field** (e.g. `bad_verification_code`),
  and the impl sent `Accept: application/json` plus the configured
  `client_id`/`client_secret`/`code`/`redirect_uri` in the exchange request.
- R-I7YE-ZCB5 — a `/user` `name: null` yields `Identity.Name == ""` (no
  panic, no `"null"` string), and the membership endpoint answering **404**
  yields `OrgMembership == ""` with no error — while a membership endpoint
  answering 500 (after the one retry) is an error.
- R-I96B-D41U — a `/user/emails` response with **no** `primary: true` entry
  causes `ExchangeCode` to return an error (fail loudly), not an Identity with
  an empty email.
- R-IAE7-QVSJ — `dashboard` fails startup with a loud missing-env error when
  any of `GITHUB_CLIENT_ID`, `GITHUB_CLIENT_SECRET`, `GITHUB_ORG` is absent
  (same `requireEnv` gate the `GOOGLE_*` triple has), proving the credentials
  are boundary-validated env, never flags.
