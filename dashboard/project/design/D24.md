# Decision 24 — `owner_id` becomes a real foreign key into `identities`

## Decision

D23 made `owner_id` `NOT NULL` on the four auth-artifact carriers but declared
no referential constraint, so the database guaranteed the column *has a value*
without guaranteeing the value *points at an existing identity*. The appkit
connection already opens every handle with `_pragma=foreign_keys(ON)`
(`appkit/db/db.go`); the schema just gives it nothing to enforce. This decision
declares the constraint, making "every auth artifact belongs to a real
identity" a fact the storage engine enforces rather than a convention the
writers uphold.

### One forward-only migration, rebuild with row copy

A single new migration (`bin/create-migration dashboard
add_owner_id_foreign_keys`) rebuilds the four carrier tables —
`web_sessions`, `oauth_authcodes`, `oauth_chains`, `personal_tokens` — so each
declares:

```sql
owner_id TEXT NOT NULL REFERENCES identities(id)
```

No `ON DELETE` clause: the default `NO ACTION` makes deleting an identity that
still owns live artifacts a constraint error. Nothing deletes identities today;
a future deletion feature must decide the fate of live credentials explicitly
rather than inherit a silent cascade.

SQLite has no `ALTER COLUMN`, so this is the standard rebuild — `CREATE` the
new-shape table, **copy every existing row** (`INSERT INTO … SELECT …`),
`DROP` the old, `ALTER … RENAME` into place, recreate the indexes
(`idx_web_sessions_owner_email`, `idx_oauth_chains_owner_email`,
`idx_oauth_chains_client_id`, `idx_personal_tokens_owner`). The row copy is
safe by construction: since D23's purge, every row was written by a stamping
path that resolves a real identity first, so every `owner_id` present
references a live `identities` row and the copy cannot violate the new
constraint. No purge, no forced re-login, no re-minted PATs.

Each table's rebuild follows the migration runner's foreign-key handling
contract, and the copy lists columns explicitly (never `SELECT *`) so the
migration stays correct against the exact D23 shapes.

### No Go changes

Writers already stamp a resolved, non-empty `owner_id` on every creation path
and fail loud when resolution fails (D18/D23); reads scan a `NOT NULL` column.
The FK adds enforcement beneath code that already behaves. (The read side's
consumption of the now-guaranteed identity row is D19's decision, built as its
own phase.)

The artifact `owner_email` columns are untouched here: they remain for audit
and the owner-scoped list queries (`pat.ListByOwner`, the grants listing).
They are vestigial pending the suite-wide owner-id re-keying, which is the
natural point to swap those queries to `owner_id` and drop the columns — out
of scope for this design.

## Rejected

- **`ON DELETE CASCADE`.** Deleting an identity would silently destroy its
  sessions, chains, and PATs. Credential destruction should never be an
  implicit side effect; a future deletion feature must own that decision.
- **Purge-again (D23-style clean slate) instead of row copy.** Simpler
  migration body, but logs every user out and invalidates every PAT and MCP
  client on the live box for zero integrity gain — post-D23 there are no dirty
  rows to flush.
- **Enforce in code only (keep the schema FK-less).** The invariant stays a
  convention; a future writer bug reintroduces dangling references silently.
  *Enforce with systems, not willpower.*
- **Also add `REFERENCES` on `oauth_tokens.chain_id` etc.** Already present
  (`004_oauth_as.sql`); this migration touches only the four `owner_id`
  carriers.

## Verification

- R-HYL8-V30T — applying the migration to a **real SQLite DB** pre-seeded with
  identities and rows in all four carrier tables (each stamped with a valid
  `owner_id`) completes without error and leaves **every pre-seeded row intact
  and byte-equal** (same counts, same column values) in the rebuilt tables.
  Proves rebuild-with-copy preserves live state.
- R-HZT5-8URI — after the migration, each of `web_sessions`,
  `oauth_authcodes`, `oauth_chains`, `personal_tokens` **rejects an `INSERT`
  whose `owner_id` references no `identities` row**, the real SQLite engine
  returning a foreign-key constraint error (a claim only the real storage
  engine can falsify).
- R-I111-MMI7 — after the migration, deleting an `identities` row that still
  has a dependent artifact row fails with a foreign-key constraint error
  (`NO ACTION`), and the identity and artifact both survive; deleting an
  identity with **no** dependents succeeds.
