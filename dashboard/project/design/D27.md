# Decision 27 — `/login` becomes the provider chooser; `/login/google` and `/login/github` start the flows

**Decision.** With two IdPs, "start signing in" splits from "start signing in
*with X*". The route map becomes:

| Route | Behavior |
|---|---|
| `GET /login` | renders the **chooser**: the sign-in composition with both provider buttons. Mints nothing. |
| `GET /login/google` | mints the web handshake (`CreateWeb(ctx, ProviderGoogle, safeReturnTo(...))`), sets the binding cookie, 302 to Google's authorize URL — exactly what `GET /login` did before, one path deeper. |
| `GET /login/github` | the same, with `ProviderGitHub` and `githubidp.AuthorizeURL` against the `/oauth/github/callback` redirect URI. |

This keeps D20 byte-for-byte intact: the nginx `@login_bounce` primitive still
redirects to `/login?return_to=$request_uri`, and the bounced human now lands on
a chooser instead of being forced into Google.

- **The chooser is the existing logged-out composition, rendered on a second
  route.** `handleLoginChooser` renders the same `signin-wall` markup D7
  defines (title, tagline, lede, rule, the two CTA anchors, rule, etymology
  table) — one template, two routes (`GET /` logged-out and `GET /login`), so
  the composition can never drift between them. The handler passes the
  validated `return_to` into the view; when present, each provider anchor's
  `href` carries it as a query parameter
  (`/login/github?return_to=<url-escaped>`), threading the destination through
  the extra hop. An invalid `return_to` renders plain anchors (D21's
  `safeReturnTo` contract: hostile values die at the boundary).
- **`safeReturnTo` runs at both ends.** The chooser validates before embedding
  in the anchors; each start route validates again before persisting on the
  handshake (`/login/google` and `/login/github` are directly linkable — the
  chooser hop is not the only way in).
- **The start routes are the only handshake minters for the web origin.**
  `handleLogin`'s current body moves to `handleLoginGoogle`; `handleLoginGitHub`
  mirrors it with the GitHub provider and the
  `publicBaseURL + "/oauth/github/callback"` redirect URI. Both register in
  `routes.go` beside the existing entries.

**Rejected.**
- **Keeping `/login` as "start Google" and adding only `/login/github`.** A
  D20 bounce would then force every logged-out visitor into Google regardless
  of which identity they own — the GitHub half of the product promise would be
  unreachable from a bounce.
- **Bouncing to `/` instead and threading `return_to` through the index.**
  Requires changing the deployed nginx primitive (D20) and overloading the
  landing route with login-flow state; the chooser route keeps nginx frozen.
- **A separate chooser template.** Two near-identical sign-in compositions
  would drift; one template rendered by two handlers cannot.
- **Carrying `return_to` in a cookie across the chooser hop.** Rejected for
  the same reasons D21 rejected the return_to cookie: a second lifetime to
  manage, redundant with a query parameter that is validated at every entry.

**Verification.**
- R-IGHP-NQI0 — `GET /login` returns **200** with the sign-in composition
  containing exactly the two provider anchors of D7 (hrefs `/login/google` and
  `/login/github`), mints **no** handshake row and sets **no** binding cookie;
  and `GET /login?return_to=/srv/sites/private/test07/` renders both anchors
  with that value url-encoded in their `href` query strings, while a hostile
  `return_to` (`//evil.com`) renders both anchors with **no** `return_to`
  parameter.
- R-IHPM-1I8P — `GET /login/google?return_to=/srv/wiki/` mints one web
  handshake with `Provider == "google"` and `ReturnTo == "/srv/wiki/"`, sets
  the binding cookie, and 302-redirects to an
  `accounts.google.com/o/oauth2/v2/auth` URL whose `state` is the minted
  handshake id — the exact pre-existing `/login` behavior relocated.
- R-IIXI-F9ZE — `GET /login/github` mints one web handshake with
  `Provider == "github"`, sets the binding cookie, and 302-redirects to a
  `github.com/login/oauth/authorize` URL whose `state` is the minted handshake
  id and whose `redirect_uri` ends `/oauth/github/callback`.
- R-IK5E-T1Q3 — `GET /login/github?return_to=//evil.com` (and
  `https://evil.com/x`) persists **empty** `ReturnTo` on the minted handshake —
  `safeReturnTo` guards the GitHub start route exactly as D21 proved for the
  Google one.
