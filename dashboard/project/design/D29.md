# Decision 29 — MCP authorize keeps one URL and grows a chooser: `provider` selects the IdP, absent renders the choice

**Decision.** The AS metadata document advertises exactly one
`authorization_endpoint`, and MCP clients follow it non-interactively
(research) — so the provider choice must happen *inside* `GET /oauth/authorize`,
made by the human in the browser. The endpoint becomes two-mode on an optional
**`provider`** query parameter:

- **`provider` absent** — after all the existing request validation passes
  (response_type, PKCE S256, resource, client, redirect_uri — unchanged and
  still first, so a malformed request errors identically to today), the
  handler renders a minimal **chooser page** instead of redirecting: the
  `signin-wall` styling with the same two provider buttons, where each
  button's `href` is **this same authorize URL with every original query
  parameter preserved plus `provider=google` / `provider=github`**. Nothing is
  minted — no handshake, no cookie; the chooser is stateless and refreshable.
- **`provider=google`** — exactly today's behavior: validate, `CreateMCP(ctx,
  ProviderGoogle, mcp)`, binding cookie, 303 to Google's authorize URL.
- **`provider=github`** — the mirror: `CreateMCP(ctx, ProviderGitHub, mcp)`,
  binding cookie, 303 to GitHub's authorize URL with the
  `/oauth/github/callback` redirect URI.
- **any other `provider` value** — `400 invalid_request` via `writeOAuthError`,
  like every other malformed-parameter case on this endpoint.

The handshake is minted only once a provider is chosen, so the chooser adds no
state, no TTL concern, and no consumed-handshake edge case; a human who walks
away from the chooser leaves nothing behind.

**Rejected.**
- **A second authorize endpoint per provider.** RFC 8414 metadata names one
  `authorization_endpoint` and MCP clients have no UI to pick among several
  (research; settled during grilling).
- **Minting the handshake before the choice and carrying its id through the
  chooser links.** Adds a persisted row + TTL for a page the human may abandon,
  and a handshake that exists before its provider is known — the exact
  ambiguity D26 removes.
- **Defaulting `provider` to Google when absent.** Re-creates the forced-Google
  path for every MCP connection; the whole point is that the human chooses.
- **Client-supplied provider via OAuth extension parameters.** MCP clients
  don't send it, and trusting a client to pick the human's IdP inverts who owns
  the choice.

**Verification.**
- R-IWCE-MR51 — a fully valid `GET /oauth/authorize` request **without**
  `provider` returns **200** HTML containing exactly two provider anchors
  whose `href`s are the same `/oauth/authorize` path with the original
  `client_id`, `redirect_uri`, `code_challenge`, `code_challenge_method`,
  `state`, `resource`, and `response_type` values preserved plus
  `provider=google` / `provider=github` respectively — and **no** handshake
  row is minted and no binding cookie set.
- R-IXKB-0IVQ — the same request with `provider=google` behaves exactly as the
  endpoint did before this Decision: one MCP handshake minted with
  `Provider == "google"`, binding cookie set, **303** to an
  `accounts.google.com` authorize URL carrying the handshake id as `state`.
- R-IYS7-EAMF — the same request with `provider=github` mints one MCP
  handshake with `Provider == "github"`, sets the binding cookie, and
  303-redirects to a `github.com/login/oauth/authorize` URL whose `state` is
  the handshake id and whose `redirect_uri` ends `/oauth/github/callback`.
- R-J003-S2D4 — `provider=gitlab` (any unknown value) yields **400**
  `invalid_request`, and an otherwise-invalid request (e.g. missing
  `code_challenge`) **without** `provider` still yields its usual OAuth error,
  not the chooser — validation stays ahead of the choice.
