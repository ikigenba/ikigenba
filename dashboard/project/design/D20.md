# Decision 20 — Apex login-bounce nginx primitive: a shared `@login_bounce` that redirects logged-out *navigations* to `/login`, and 401s everything else

**Decision.** The dashboard owns the apex nginx `server` block
(`dashboard/etc/nginx.conf`). This change adds **one dashboard-owned, box-global
named location** — `@login_bounce` — that any session-gated `/srv/<svc>/`
location in the suite can opt a logged-out request into via a single
`error_page 401 = @login_bounce;` line. It turns the bare nginx **401** a
logged-out human currently sees when navigating to a session-gated page into a
**302 to `/login`** carrying a `return_to` back to where they were headed, while
leaving programmatic requests (browser `fetch`/XHR) with the clean **401** they
can handle.

The discriminator is the request's **`Sec-Fetch-Mode`** header, which every
evergreen browser sends: a top-level navigation carries `navigate`; a scripted
`fetch`/XHR carries `cors` / `same-origin` / `no-cors`. The rule is
**default-to-bounce**: a `navigate` request *or* a request with **no**
`Sec-Fetch-Mode` at all (older/edge clients) is bounced to login; only the three
explicit scripted modes get the 401.

**Why a named location with an `if`-chain, not a `map`.** A `map` is the natural
shape for "header value → decision string", but `map` is only legal in the
`http{}` context. `dashboard/etc/nginx.conf` is a **server-block fragment** (two
`server{}` blocks, no enclosing `http{}`) that `opsctl init-box` drops into the
box's nginx; giving it a `map` would force a second, http-scope provisioning file
and an `opsctl` change. Keeping the whole primitive inside the server block as a
named location avoids that entirely. The location uses the **return-only `if`
form** — the one use of `if` in `location` that nginx documents as safe (each
branch is a bare `return`, no side effects):

```
# Dashboard-owned login-bounce primitive. A session-gated /srv/<svc>/ location
# opts in with `error_page 401 = @login_bounce;`. A logged-out NAVIGATION (or a
# client that sends no Sec-Fetch-Mode) is sent to /login carrying where it was
# headed; a scripted fetch/XHR (cors/same-origin/no-cors) keeps the clean 401 so
# the calling JS can handle it. return_to is validated same-site by /login before
# it is ever honored (D21), so embedding $request_uri raw here is safe.
location @login_bounce {
    if ($http_sec_fetch_mode = cors)        { return 401; }
    if ($http_sec_fetch_mode = same-origin) { return 401; }
    if ($http_sec_fetch_mode = no-cors)     { return 401; }
    return 302 /login?return_to=$request_uri;
}
```

**Why this cannot loop and does not leak.** `error_page 401 = @login_bounce;` is
set only on the individual session-gated *service* locations, so it is **not**
inherited by the `@login_bounce` named location itself (named locations inherit
from `server`/`http`, not from the location that jumped to them). No server-level
`error_page 401` exists, so `@login_bounce`'s own `return 401` is a final
response, never re-intercepted. `$request_uri` and `$http_sec_fetch_mode` are the
**original** request's values across the internal `error_page` redirect. The
primitive is defined **once** in the apex block; services never author it — they
only reference it, exactly as they already reference the apex-owned
`/_session-authn` internal location.

**Scope of this Decision.** It adds the `@login_bounce` named location to
`dashboard/etc/nginx.conf` and nothing else in that file changes. It does **not**
add the `error_page` opt-in to any service fragment (each service owns that in
its own `project/`; sites does so against this primitive). The dashboard's *own*
pages (`/profile`, `/telemetry`) keep their existing in-process
`303 → /` sign-out redirect (D2/D14) and are unaffected — they are not
nginx-gated. The **dev front door** `nginx/nginx.conf` (repo root) must carry a
mirror of this primitive to exercise it locally, but that file is suite
infrastructure **outside** this `project/` tree and outside the scope boundary;
it is maintained as root infra, not by this spec.

**Verification.** (Proven by a Go test that reads `dashboard/etc/nginx.conf` from
disk and asserts over its content — the content-assertion pattern this repo
already uses for nginx fragments. nginx is not run by the suite; the assertion
proves the config *expresses* the contract, and the live 302/401 routing is a
deploy-time smoke, not a unit test.)

- R-XJBT-7YIF — `dashboard/etc/nginx.conf` defines a `location @login_bounce`
  block whose **fallthrough** is `return 302 /login?return_to=$request_uri;` —
  i.e. a `navigate` request and a request with an empty/absent `Sec-Fetch-Mode`
  both reach the 302 (default-to-bounce), and the redirect target is `/login`
  carrying `return_to=$request_uri`.
- R-XKJP-LQ94 — within that same block, each of the three scripted fetch modes
  `cors`, `same-origin`, and `no-cors` is matched on `$http_sec_fetch_mode` and
  answered with `return 401;` (a scripted request is **not** bounced) — all three
  present, not a subset.
