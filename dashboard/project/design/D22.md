# Decision 22 — The web callback returns you to `return_to`, or `/` by default

**Decision.** `callbackWeb` (the web-origin completion of a sign-in
round-trip, shared by the Google and GitHub callbacks) redirects to the consumed handshake's validated **`return_to`** when
it is non-empty, and to the apex home `/` otherwise — preserving today's exact
behavior for every login that carries no destination. The value was already
same-site-validated at capture (D21, `safeReturnTo`), so the callback replays it
as-is; it performs **no** second validation and constructs no new target beyond
reading `handshake.ReturnTo`.

Only the final redirect line changes:

```go
// callbackWeb: mint the session, set the cookie, then return the human to where
// they were headed. ReturnTo is "" for a plain sign-in (index button, direct
// /login) and for any rejected destination — both fall to the apex home.
dest := "/"
if handshake.ReturnTo != "" {
    dest = handshake.ReturnTo
}
setSessionCookie(w, r, issued.CookieValue)
http.Redirect(w, r, dest, http.StatusFound)
```

`callbackWeb`'s signature widens to receive the consumed `handshake` (it already
has it in `handleCallback`) so it can read `ReturnTo`; session minting, the
audit events, and the cookie are unchanged. The **MCP-origin** branch
(`callbackMCP`) is untouched — it resumes via its own `MCPRedirectURI` and never
consults `return_to`.

**Rejected.**
- **Re-validating `return_to` in the callback** — it was validated at capture and
  stored server-side on the row; nothing between capture and replay can alter it,
  so a second check is dead code. (If D21's validator is wrong, the fix is one
  place, not two.)
- **Defaulting a missing `return_to` to the login page or the profile page** —
  the established behavior is "web sign-in lands on the apex home `/`"; this
  change adds a *destination when one was requested*, it does not change the
  no-destination default.
- **Honoring `return_to` on the MCP-origin path** — an MCP login already has an
  authoritative resume target (`MCPRedirectURI`); layering a browser `return_to`
  onto it is meaningless and would cross the two origins' contracts.

**Verification.**
- R-XQN7-IKYL — a web sign-in whose handshake carries
  `return_to=/srv/sites/private/test07/` completes with a `302`/`Location`
  redirect to **that path** (driven through the real `/login` → callback route
  against a temp DB, asserting the `Location` header), proving the destination is
  replayed.
- R-XRV3-WCPA — a web sign-in whose handshake carries an **empty** `return_to`
  redirects to **`/`** (default preserved), and an **MCP-origin** callback still
  redirects to its `MCPRedirectURI` — proving the default is intact and the MCP
  path is unaffected.
