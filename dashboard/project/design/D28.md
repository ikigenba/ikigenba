# Decision 28 — The GitHub callback: `/oauth/github/callback`, gated on verified primary email + active ikigenba-org membership

**Decision.** `GET /oauth/github/callback` (`handleGitHubCallback`, in
`internal/server/callback.go` beside `handleCallback`) completes the GitHub
sign-in for both origins. It mirrors the Google callback stage for stage — the
differences are exactly the provider seam and the federation policy:

1. **Bind + consume.** Same as Google: require the binding cookie, consume the
   handshake, reject (`400`, audit) on failure. Then the D26 check: the
   handshake's `Provider` must be `ProviderGitHub`.
2. **Exchange.** `a.githubProvider.ExchangeCode(ctx, code,
   publicBaseURL+"/oauth/github/callback")` (D25).
3. **Federation policy — inline in the handler, like Google's.** Two gates, in
   order:
   - `EmailVerified` must be true → else warn log,
     `writeFederationReject(r, email, "email_not_verified")`, `403` (the same
     reject reason the Google gate uses — same meaning, same audit vocabulary).
   - `OrgMembership` must be exactly `"active"` → else warn log (including the
     login and the observed state), `writeFederationReject(r, email,
     "org_membership")`, `403`. This is the GitHub analog of Google's
     workspace-domain gate: `pending` invitees, ex-members, and strangers
     (`""`) are all refused identically. Bare-`http.Error` responses, matching
     the established 403 pattern — no styled error page.
4. **Resolve identity.** `a.identity.ResolveOrCreate` with
   `Claims{Iss: identity.Iss (== "https://github.com"), Sub, Email, Name,
   Picture}` where **`Name` falls back to `Login`** when the profile name is
   empty (every identity gets a display name; the GitHub username is the
   natural one). D17's `(iss, sub)` upsert does the rest — a GitHub row can
   never merge with a Google row.
5. **Branch on origin.** The existing `callbackWeb` / `callbackMCP` helpers are
   called unchanged — session mint + `return_to` redirect (D22) for web,
   authorization-code issuance for MCP. Nothing downstream of `ownerID` knows
   or cares which IdP authenticated the human; this is what makes GitHub
   identities full citizens of the OAuth AS and PAT machinery with zero changes
   there.

**Rejected.**
- **A shared parameterized callback handler for both providers.** The two
  federation policies gate on different facts (`HostedDomain` vs
  `OrgMembership`) carried on different Identity types; parameterizing over
  that difference obscures the two policies without deduplicating anything
  that matters. The shared halves (`callbackWeb`/`callbackMCP`) are already
  factored.
- **Enforcing membership inside `githubidp.ExchangeCode`.** The provider
  reports facts, the handler decides admission — the split the Google pair
  already uses (`ExchangeCode` proves authenticity; the handler owns policy).
  Moving policy into the provider would also make the stub lie about it.
- **A styled "you're not a member" page.** The Google gate's rejects are bare
  403s; a stranger or a misconfigured member gets the same. Consistency over a
  new UI surface (settled during grilling).
- **Admitting `pending` members.** A pending invitation is not membership;
  admitting it would let a revoked-before-accepting invitee in.

**Verification.**
- R-INT3-YCY6 — a web-origin GitHub callback whose stub identity is an active
  org member with a verified email mints a `dashboard_session` cookie and
  302-redirects to the handshake's `ReturnTo` (or `/` when empty) — the D22
  behavior, driven through the GitHub door.
- R-IP10-C4OV — the created `identities` row carries
  `iss == "https://github.com"` and `sub` equal to the stub's numeric-id
  string, read back from the temp DB — and a Google identity with the **same
  email** resolves to a **different** `identities` row and a different
  `owner_id` (never unified, proven at the substrate that enforces the
  `(iss, sub)` key).
- R-IQ8W-PWFK — a callback whose stub identity has `OrgMembership: ""` (and
  another with `"pending"`) is refused with **403**, an `org_membership`
  federation-reject audit record, and **no** session cookie and **no**
  `identities` row created.
- R-IRGT-3O69 — a callback whose stub identity has `EmailVerified: false` is
  refused with **403**, an `email_not_verified` federation-reject audit
  record, and no session.
- R-ISOP-HFWY — an MCP-origin GitHub callback (handshake minted with
  `CreateMCP(ctx, ProviderGitHub, mcp)`) issues an authorization code bound to
  the resolved GitHub `owner_id` and 302-redirects to the MCP client's
  registered redirect URI with the client's state — completing the token flow
  yields tokens whose introspection reports that GitHub identity's
  `owner_id`.
- R-ITWL-V7NN — a stub identity with an empty `Name` resolves to an
  `identities` row whose `name` equals the stub's `Login` (the username
  fallback), while a non-empty profile `Name` is stored as-is.
