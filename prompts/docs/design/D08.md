# Decision 8 — DB migration

## Decision

A new timestamped migration (created with `bin/new-migration prompts backfill-provider-and-model-aliases`) handles two backfills in a single forward-only SQL file.

**Backfill 1 — provider**: every prompt row whose `config_json` has a `null` or empty `$.provider` value is updated to `"anthropic"`. This restores the invariant that all stored prompts have an explicit provider.

```sql
UPDATE prompts
   SET config_json = json_set(config_json, '$.provider', 'anthropic')
 WHERE json_extract(config_json, '$.provider') IS NULL
    OR json_extract(config_json, '$.provider') = '';
```

**Backfill 2 — model aliases**: the local `agentkit/model` package accepted short aliases (`opus`, `sonnet`, `haiku`, `pro`) and stored them verbatim. With aliases dropped (product decision), existing rows that stored an alias must be updated to the canonical bare ID. The alias-to-bare-ID map is hard-coded in the migration:

```sql
UPDATE prompts SET config_json = json_set(config_json, '$.model', 'claude-opus-4-7')
 WHERE json_extract(config_json, '$.model') = 'opus';

UPDATE prompts SET config_json = json_set(config_json, '$.model', 'claude-sonnet-4-6')
 WHERE json_extract(config_json, '$.model') = 'sonnet';

UPDATE prompts SET config_json = json_set(config_json, '$.model', 'claude-haiku-4-5')
 WHERE json_extract(config_json, '$.model') = 'haiku';

UPDATE prompts SET config_json = json_set(config_json, '$.model', 'gemini-3.1-pro-preview')
 WHERE json_extract(config_json, '$.model') = 'pro';
```

Both backfills are idempotent: re-running on an already-migrated database is a no-op (the WHERE conditions simply match no rows).

**Ordering**: the two backfills are in one SQL file; the migration runner applies them in a single transaction. No new columns or table reshaping — only JSON column value updates.

## Rejected

**Two separate migrations** — splitting provider backfill and alias backfill into two files adds version ordering complexity for no benefit; they are logically one backfill operation.

**Runtime fallback in `validateConfig`** — having the service silently remap stored aliases to bare IDs at run time was considered; rejected because it masks stale data in the database. The migration fixes the source of truth; the runtime should see only valid data.

## Verification

- R-KBLR-VBHQ — after the migration runs against a database containing a prompt with `config_json` where `$.provider` is `null` or `""`, querying that prompt returns a Config with `Provider == "anthropic"`.
- R-KCTO-938F — after the migration runs against a database containing a prompt with `config_json` where `$.model` is `"sonnet"`, querying that prompt returns a Config with `Model == "claude-sonnet-4-6"`.
