# Decision 3 — Validation

## Decision

`validateConfig` in `prompts/internal/prompt/service.go` is rewritten to accept all four providers and validate the model against each provider's registry via the published agentkit.

**Provider env var map** (contractual — must match exactly):

| provider   | env var             |
|------------|---------------------|
| anthropic  | `ANTHROPIC_API_KEY` |
| openai     | `OPENAI_API_KEY`    |
| google     | `GEMINI_API_KEY`    |
| zai        | `ZAI_API_KEY`       |

**Validation steps** (in order — fail fast):

1. `cfg.Provider` must be one of the four known strings; anything else returns a `*ValidationError`.
2. A minimal provider handle is constructed with an empty API key (sufficient for `Pricing` lookup, which does not use the key). `provider.Pricing(cfg.Model)` is called; `!ok` returns a `*ValidationError` naming the unknown model.
3. The provider's env var is checked via `os.Getenv`; empty returns a `*ValidationError`.

The existing `clientFactory` seam in `runner.go` is replaced by a new `buildProvider(cfg Config, getenv func(string) string) (agentkit.Provider, error)` function (D4), which also validates the API key presence — but `validateConfig` remains the create/update gate that rejects bad config before a prompt row is stored.

**Import changes**: `service.go` imports `github.com/ikigenba/agentkit/anthropic`, `.../openai`, `.../google`, `.../zai` to call each provider's `New("").Pricing(model)`. The import of `agentkit/model` is removed.

The hardcoded `cfg.Provider = string(model.ProviderAnthropic)` normalization in `Create` and `Update` is removed; the provider is stored as supplied (already validated).

**Test seam**: `validateConfig` accepts a `getenv func(string) string` parameter rather than calling `os.Getenv` directly, so tests can inject a fake environment without setting real process env vars.

```go
func validateConfig(c Config, getenv func(string) string) error
```

Existing callers pass `os.Getenv`.

## Rejected

**Validate model at run time** (rejected at product level) — not reconsidered here.

**Single combined validate+build step** — validate and build are separate because validate runs at create/update (no config pinned to disk yet) while build runs at spawn (reading pinned config from disk). Merging them would couple the service layer to the runner.

**Use `agentkit/model.Resolve` for model validation** — the local `model` package is being removed (D1). Model validation goes through the provider's own `Pricing` registry, which is the published agentkit's canonical truth.

## Verification

- R-JVR2-WAUP — `validateConfig` with an unknown provider string (e.g. `"badprovider"`) returns a non-nil `*ValidationError`.
- R-JWYZ-A2LE — `validateConfig` with a valid provider and an unrecognised model string (not in that provider's `Pricing` registry) returns a non-nil `*ValidationError`.
- R-JY6V-NUC3 — `validateConfig` with a valid provider and known model, but the provider's API key env var empty, returns a non-nil `*ValidationError`.
- R-JZES-1M2S — `validateConfig` with a valid provider, known model, and the API key env var set returns `nil`.
