# Decision 32 — `calls` and `usage`: the inspection and reporting MCP tools

## Decision

Two new domain tools join the `internal/mcp` tool table (D17 registration, D27 structured-result conventions: `StructuredResult`, typed error codes, output schemas), making the `calls` table (D28) the owner's one-stop inference surface.

**`calls` — inspection.** Input:

```json
{ "class": "completion", "origin": "service:wiki", "name": "wiki.compile",
  "group_id": "…", "errors_only": true, "since": "2026-07-01T00:00:00Z",
  "until": "…", "limit": 50, "offset": 0, "call_id": "01J…" }
```

All fields optional. Without `call_id`: a filtered, paginated list (D28 `Store.List`) of metric rows — every column **except** the bodies (list responses stay small). With `call_id`: the single-row detail (D28 `Store.Get`) **including** `request_body`/`response_body` when retention still holds them; after pruning, the detail returns the metrics with the bodies absent and a `bodies_pruned: true` marker. Unknown `call_id` → the D27 `not_found` code.

**`usage` — reporting.** Input:

```json
{ "group_by": "name", "class": "session", "since": "…", "until": "…" }
```

`group_by` required, ∈ {`name`, `origin`, `model`, `day`}; the rest optional. Output: one bucket per key (D28 `Store.Aggregate`) with `calls`, `input_tokens`, `output_tokens`, `total_tokens`, `cost_usd`, `errors`. This answers "what did wiki ingest cost this month" and "total spend by model" in one call.

**Scoping: identity-gated, not row-filtered.** Both tools ride the chassis identity gate like every domain tool, but return rows regardless of the row's `owner_email` or origin class. Attribution is the answer to "whose is this" (every row carries `origin`, plus `owner_email` when a user identity applies); hiding co-users' or services' rows would falsify exactly the whole-box spend picture these tools exist to give. Per-user privacy, if ever needed, is a dashboard grant question, not row filtering here.

wiki's own `llm_calls` MCP tool retires when wiki converts (wiki's spec, not this one); its `job_id`/`stage` filters map to `group_id`/`name` here.

## Rejected

**One combined tool** — inspection (rows) and accounting (sums) have different shapes and different questions; two thin tools over one store beat a mode-switched envelope.

**Owner-scoped rows** — see above; defeats the accountability purpose.

**A cost-budget/alerting surface** — reporting first; enforcement is future work informed by what `usage` shows.

## Verification

- R-6DJD-SKVZ — the `calls` tool with a `name` filter and a time window returns exactly the seeded rows matching both, and no list row carries a `request_body` or `response_body` field.
- R-6ERA-6CMO — the `calls` tool with a `call_id` returns the row's bodies while they are retained; after `PruneBodies` covers that row, the same call returns the metrics with bodies absent and `bodies_pruned: true`.
- R-6FZ6-K4DD — the `usage` tool grouped by `name` over seeded rows of mixed names returns per-name buckets whose call counts, token sums, and cost sums equal the seeded totals.
- R-6H72-XW42 — the `usage` tool grouped by `origin` distinguishes `user:…`, `trigger:…`, and `service:…` buckets from a seeded mix.
- R-6IEZ-BNUR — a `calls` query made under one identity returns rows whose `owner_email` is a different user and rows with no owner at all — the tools are not row-filtered by caller.
