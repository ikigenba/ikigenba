# Decision 29 — `POST /complete`: the synchronous one-shot completion endpoint

## Decision

prompts grows a second workload class beside agent sessions: a **stateless, synchronous, tool-less completion**, exposed to sibling daemons as a loopback-only plumbing endpoint. This is the seam a service like wiki calls for its pipeline inference instead of holding provider keys itself. It is deliberately **not** an MCP tool: `/mcp` is the product surface for agents and owners; `/complete` is infrastructure, in the same tier as `/feed` and `/run-content` — mounted via the chassis loopback guard (`rt.HandleLoopback`), no identity headers, trusted because only loopback can reach it and nginx never routes it.

**Request** (`POST /complete`, JSON):

```json
{
  "origin":   "service:wiki",            // required, D28 origin grammar
  "name":     "wiki.extract",            // required, D28 name grammar for service calls
  "group_id": "job-01H…",                // optional correlation id
  "attempt":  2,                          // optional, default 1
  "model":    "claude-sonnet-4-6",       // required, catalog name
  "provider": "anthropic",               // optional, catalog default when omitted
  "config":   { "temperature": 0, "max_tokens": 16384 },  // optional, D2 config keys
  "system":   "…",                        // optional system prompt
  "messages": [                           // required, non-empty, last role must be "user"
    { "role": "user",      "text": "…" },
    { "role": "assistant", "text": "…" },
    { "role": "user",      "text": "…" }
  ]
}
```

`messages` roles are exactly `user` | `assistant`. The array form exists so a caller can replay a corrective-retry conversation statelessly (wiki's parse-retry appends the model's bad reply plus a corrective user turn); a plain one-shot is a one-element array. `config` accepts the same keys as a prompt's config (the product's contractual constants) with two caveats: `tool_loop_limit` is meaningless here (no tools) and is silently ignored; all other keys map exactly as in a run.

**Validation** reuses the create/update gate: D3's `validateConfig` is exported as `prompt.ValidateConfig` and applied to the assembled `{provider, model, config}` — catalog membership, provider routability, reasoning vocabulary, key presence — plus envelope checks: origin and name grammar (D28), messages non-empty with a final `user` turn, body size ≤ 10 MiB. Any failure → `400` with a JSON `{"error": "…"}` naming the problem, and **no `calls` row** (nothing executed, nothing to account).

**Execution** (`internal/inference`, the new package owning both this decision and D30):

```go
routeProvider, wireModel, entry, _ := catalog.Resolve(req.Provider, req.Model)
prov, err := provider.Build(cfg, os.Getenv)        // D4's relocated factory
release, err := gate.AcquireCall(ctx, routeProvider) // D31; blocks, ctx-aware
defer release()

conv := &agentkit.Conversation{
    Provider: prov,
    Model:    wireModel,
    Pricing:  entry.Pricing,
    System:   req.System,
    Gen:      buildGenSettings(cfg),   // shared with the runner
    Retry:    buildRetryPolicy(cfg),
    History:  history,                 // messages[:last] as agentkit.Message values
}
stream := conv.Send(ctx, lastUserText)
// drain; final text = last MessageDone; usage = stream.Usage(); cost = stream.Cost()
```

No `Tools`, no `DeferredTools`, no sandbox, no run row, no outbox event, no log file.

**Recording**: after the drain (success or provider error), one `calls` row is inserted — class `completion`, catalog model name, `request_body` = the marshaled request envelope (system + messages + config), `response_body` = the final text, `error` set on failure. An insert failure fails the whole call with `500` (D28's fail-loud rule).

**Response**: `200` with

```json
{ "call_id": "01J…", "text": "…", "usage": { …full agentkit Usage… }, "cost_usd": 0.0123 }
```

Provider/stream error → `502` with `{"error": "…"}` (the row still records the attempt and its error). Status taxonomy: `400` envelope/validation, `405` non-POST, `500` recording or internal failure, `502` provider failure.

**Test seam**: the executor takes the same injectable provider-factory type as the runner (`func(prompt.Config, func(string) string) (agentkit.Provider, error)`), so handler-level tests drive the real HTTP handler with a fake provider that records the incoming `*agentkit.Request` and returns canned round-trips. The live-provider contract (a real API accepting a tool-less conversation) is agentkit's proven ground, not re-proven here.

## Rejected

**An MCP tool instead of a plain endpoint** — forces every inference-consuming daemon to speak MCP handshake and envelope on a hot path, and puts plumbing on the product surface. Services that never do inference should never need `/mcp` at all.

**Request/response over the event plane** — the plane is one-directional fact publication with no reply channel; synchronous capability calls are the documented suite exception (nginx→dashboard `/internal/authn` precedent).

**Server-side structured-output enforcement (schema in, validated JSON out)** — the caller's validators and corrective-retry loops are domain logic coupled to its own types (wiki's live line-by-line with its prompts); centralizing them splits one artifact across two services. The caller keeps parsing/validation; each retry is simply another `/complete` call with its `attempt` number.

**A stored prompt-template entity for completions** — same coupling argument; the caller sends full text, and the `calls` row preserves exactly what ran.

**Streaming responses** — every known caller drains to completion; a streaming surface adds protocol for no consumer. Revisit only with a real latency-sensitive consumer that streams to an end user.

## Verification

- R-5P5E-5623 — a valid request POSTed to the real handler with an injected fake provider returns `200` whose `text` is the fake's reply and whose `usage`/`cost_usd` reflect the fake's reported usage priced by the model's catalog rates.
- R-5QDA-IXSS — that same call leaves exactly one `calls` row: class `completion`, the request's origin/name/group_id/attempt, catalog model name, non-NULL `request_body` containing the messages and system text, and `response_body` equal to the returned text.
- R-5ST3-AHA6 — a request with a model outside the catalog, a provider that cannot route the model, or a reasoning setting the model rejects returns `400` naming the problem, and the `calls` table gains no row.
- R-5U0Z-O90V — a request with a malformed `origin`, a `name` violating the `<service>.<stage>` grammar, an empty `messages` array, or a final message whose role is not `user` returns `400`; no row.
- R-5V8W-20RK — a request whose `messages` carry a user/assistant/user history reaches the injected provider with that full history present in the `*agentkit.Request` (the corrective-retry replay works).
- R-5WGS-FSI9 — when the injected provider fails, the endpoint returns `502` and the `calls` row records the error string with the class/origin/name intact.
- R-5XOO-TK8Y — when the `calls` insert fails (injected store error), the endpoint returns `500` — a completion that cannot be recorded does not report success.
- R-5YWL-7BZN — the endpoint is mounted through the chassis loopback guard: a loopback request carrying no `X-Owner-Email` header succeeds, matching the `/run-content` pattern.
