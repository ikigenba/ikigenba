# Decision 9 — MCP schema

## Decision

`configSchema()` in `prompts/internal/mcp/tools.go` exposes all 11 optional config keys alongside `provider` (optional — catalog-derived when omitted, D3) and `model` (required). The schema is the shared input shape for both `create` and `update` tool descriptors.

```go
func configSchema() map[string]any {
    return obj(map[string]any{
        // model is required; provider is optional (derived from the model's
        // catalog default when omitted, settable to pick an alternate route)
        "provider": typ("string"),
        "model":    typ("string"),
        // Generation controls
        "temperature":     typ("number"),
        "top_p":           typ("number"),
        "max_tokens":      typ("integer"),
        "effort":          typ("string"),
        "thinking_budget": typ("integer"),
        "thinking_level":  typ("string"),
        "thinking":        typ("boolean"),
        // Retry policy
        "max_attempts":       typ("integer"),
        "base_delay":         typ("string"),
        "max_delay":          typ("string"),
        "max_elapsed":        typ("string"),
        "ignore_retry_after": typ("boolean"),
        // Loop and provider tuning
        "tool_loop_limit": typ("integer"),
        "base_url":        typ("string"),
    }, "model")
}
```

`model` is the only required field. All other keys are present in the schema object but not in the required array.

**`describe` tool — the model inventory is generated from the catalog, not hand-written.** The `describe` implementation renders, at call time, a per-provider inventory built from `catalog.ListByProvider` over the five provider names (chat entries only — entries with non-nil `Pricing`; embedding-only entries are skipped): each model's catalog name, its default provider, any alternate routes, its context size, and its reasoning vocabulary rendered from the `ReasoningSpec` (enum → the level list and default; range → min/max, sentinels, default; toggle → on/off; nil → "no reasoning control"). Because the inventory is computed from the same catalog that validation (D3) consults, the doc can never drift from what validation accepts, and a catalog bump updates the doc for free. The surrounding prose covers: provider optionality and override semantics, the reasoning-control precedence and per-model validation, and the run output format (LogRecord JSONL).

## Rejected

**Separate schemas for create vs update** — both operations accept the same config shape; a separate schema would diverge over time for no benefit.

**Enum constraints on `provider`/`model` in the JSON schema** — the validation layer (D3) is the authoritative gate and its error messages name the accepted vocabulary; baking the catalog into the schema would duplicate the same truth in a second format.

**A dedicated `models` tool** — the inventory is reference documentation; folding it into `describe` keeps the tool surface flat and puts the models beside the config semantics an agent reads anyway.

- R-20UM-JESS — the `config` property schema in both `create` and `update` tool descriptors lists exactly `["model"]` as its `required` array; `provider` is present as a property but not required.
