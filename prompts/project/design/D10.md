# Decision 10 — The landing page: a session-gated human web surface (`GET /{$}`)

## Decision

prompts gains its **first HTML web surface**: a single **landing page** served at
the bare mount root, `GET /{$}`, rendering the service **name** and **version**,
styled with the Carbon design system. This is a deliberate, suite-wide
introduction — every deployable app gains the same v1 landing page — not a
contradiction of the loopback-only posture. prompts keeps everything it already
is (the `prompts_*` MCP tool surface, `/health`, the RFC 9728 PRM doc, the
loopback `/feed` producer, the consumer loops); the landing page is **added
beside** them, never in place of them. v1 is intentionally the uniform
name+version starting point; the page will diverge later to serve prompts' own
purpose (surfacing prompt/run state to a logged-in human), which is why prompts
**owns its own landing content** (its `share/www` template + assets) rather than
importing a shared page — only the rendering *mechanism* (template loading,
static serving) is chassis-owned. (D16 moved this surface from an embedded
`internal/web` package to the on-disk `share/www` tree served through the chassis
`Spec.WWW`; this Decision is edited to that current shape.)

## Two front doors, two audiences (the gate is the spine of this Decision)

Under `/srv/prompts/` there are now two kinds of caller, each authenticated
differently — and the difference is the whole design:

- **Agents / MCP clients → bearer token.** `POST /mcp` stays gated by
  `auth_request /_authn` (opaque bearer). The **gate** is unchanged by this
  Decision — the existing prefix `location /srv/prompts/` fragment keeps gating
  it (and the exact-match PRM and `= /srv/prompts/feed → 404` locations are
  likewise untouched). Its **identity forwarding** is now the four-owner-header
  contract in *Identity-header forwarding* below.
- **Humans in a browser → login-session cookie.** The landing page is gated by
  `auth_request /_session-authn` — the dashboard-owned, loopback-only endpoint
  that validates the `dashboard_session` cookie against the web-session store.
  This is the **same** mechanism `sites` already uses for its private static
  tier. It is deliberately **coarse**: any logged-in dashboard user may view any
  app's landing page (no per-resource check). That is correct for a name+version
  page.

A browser cannot satisfy the bearer gate (it has no token), and the bearer gate
is the wrong audience for a human page; so the landing page **must** be
cookie-gated, not bearer-gated — and not fully public, since the suite exposes
nothing un-gated except the PRM bootstrap doc.

## The seam — Go side (`cmd/prompts/main.go`, `registerRoutes`)

The landing route is mounted **ungated in-process** — nginx is the sole gate,
exactly as the existing `POST /mcp` route trusts nginx for identity. It is
mounted in `registerRoutes` beside the existing MCP wiring, rendering through the
chassis-loaded site (D16); the service-side `GET /static/` mount is gone (the
chassis auto-mounts it from `Spec.WWW`):

```go
rt.HandleFunc("GET /{$}", func(w http.ResponseWriter, r *http.Request) {
    _ = rt.WWW().Render(w, "landing.html",
        struct{ Service, Version string }{rt.Service(), rt.Version()})
})
```

- `{$}` (Go 1.22+ `ServeMux`) matches **only** the exact root path `/`, so it
  cannot shadow `/mcp`, `/health`, `/feed`, or
  `/.well-known/oauth-protected-resource`. Those routes are untouched.
- It is **not** wrapped in `rt.RequireIdentity(...)` (that wrapper is the bearer
  gate, used for `POST /mcp`). The cookie gate lives entirely in nginx;
  in-process the route is open and trusts the front door — the established suite
  posture. The handler reads no `X-Owner-Email` / `X-Client-Id` header.
- `rt.Service()` returns `"prompts"` and `rt.Version()` returns the build version
  (`VERSION`, e.g. `0.12.0`); together they are the entire v1 page content. The
  chassis renders the on-disk `landing.html` `html/template`. No DB, no LLM, no
  runner, no identity header is read — the page is a pure function of
  name+version.

## The seam — assets (the on-disk `prompts/share/www` tree)

prompts ships its **own** copy of the Carbon assets and template on disk in the
release `share` tier (D16 moved them here from the old embedded `internal/web`
package):

- `prompts/share/www/` holds `landing.html` (Go `html/template`) and
  `static/tokens.css` + the woff2 font files, **copied** from the repo-root
  Carbon source (rules in `design/carbon.md`). `bin/ship` bundles
  `prompts/share/` into the release `share/<version>` tier; the chassis resolves
  the root to `share/current/www` on the box, `./share/www` or the
  `PROMPTS_WWW_PATH` override in dev — the binary stays one static file, with its
  page assets shipped beside it as diffable release artifacts.
- `Spec.WWW: true` makes the chassis load the site at startup (`appkit/web.Load`
  over the root), auto-mount `GET /static/` (serving `tokens.css` + fonts with
  correct content-types), and expose the site as `rt.WWW()`. The composition
  root renders `landing.html` through `rt.WWW().Render(...)` with the injected
  name+version, writing `Content-Type: text/html; charset=utf-8` and 200. The
  static route, like the page, is ungated-in-process and session-gated by the
  same nginx exact-match root — acceptable because the page and its CSS/fonts are
  one coherent human surface.
- Visual form (Carbon): a single centered card — service **name** in display
  type (Space Grotesk), **version** as a mono label (IBM Plex Mono); monochrome
  neutrals, blue `#2563EB` the only signal color, 4px spacing grid. Minimal and
  on-system; no JS required.

## The seam — nginx (`prompts/etc/nginx.conf`)

Add an **exact-match** location for the bare mount root that session-gates and
proxies to the app's `GET /{$}`, beside the existing gated prefix:

```
# Landing page — human web UI, gated by the dashboard browser SESSION (cookie),
# validated by the dashboard-owned /_session-authn internal location (sibling of
# /_authn). Exact-match `=` so it never shadows the gated prefix below or the MCP
# endpoint. A failed session check yields 401.
location = /srv/prompts/ {
    auth_request /_session-authn;
    auth_request_set $prompts_owner         $upstream_http_x_owner_email;
    auth_request_set $prompts_owner_id      $upstream_http_x_owner_id;
    auth_request_set $prompts_owner_name    $upstream_http_x_owner_name;
    auth_request_set $prompts_owner_picture $upstream_http_x_owner_picture;
    proxy_set_header X-Owner-Email   $prompts_owner;
    proxy_set_header X-Owner-Id      $prompts_owner_id;
    proxy_set_header X-Owner-Name    $prompts_owner_name;
    proxy_set_header X-Owner-Picture $prompts_owner_picture;
    proxy_pass http://127.0.0.1:3002/;   # trailing slash → upstream sees "/"
    proxy_set_header Host $host;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_http_version 1.1;
}
```

The existing `location /srv/prompts/ { auth_request /_authn; ... }` **prefix**
stays for everything else (the MCP endpoint, the 429 re-emit handling); the
exact-match `=` root wins for the bare root only. The existing exact-match
locations — the PRM doc (`= /srv/prompts/.well-known/oauth-protected-resource`)
and the loopback feed guard (`= /srv/prompts/feed { return 404; }`) — are
unaffected, since exact matches over distinct paths coexist. The loopback port is
a literal in the fragment (`PORT=3002`, prompts's fixed registry port) — nothing
substitutes it at install.

## Identity-header forwarding — all four owner headers

The dashboard's introspection hooks (`/_authn` bearer and `/_session-authn`
session) emit **four** owner headers on every allow, fail-closed: `X-Owner-Id`
(durable owner key, never empty), `X-Owner-Email` (display/audit, never empty),
`X-Owner-Name` and `X-Owner-Picture` (percent-encoded, MAY be empty). This
fragment captures and forwards **all four** on **every** location that forwards
identity today — replacing the old email-only forwarding — so the durable
`X-Owner-Id` reaches the upstream instead of dying at nginx:

- **Bearer prefix `location /srv/prompts/`** (the `auth_request /_authn` tier):
  `auth_request_set` all four owner headers into `$prompts_owner`,
  `$prompts_owner_id`, `$prompts_owner_name`, `$prompts_owner_picture` and
  `proxy_set_header` each of `X-Owner-Email`, `X-Owner-Id`, `X-Owner-Name`,
  `X-Owner-Picture` from those variables — **plus** the existing
  `X-Client-Id` from `$prompts_client` (unchanged). The 429 re-emit capture
  (`$authn_status`/`$authn_retry`) and `@prompts_authn_500` handling are
  untouched.
- **Session landing `location = /srv/prompts/`** (the `auth_request
  /_session-authn` tier): the same four `auth_request_set` / `proxy_set_header`
  pairs, into the same `$prompts_owner*` variables (today it forwards only
  `X-Owner-Email`). It carries **no** `X-Client-Id` (the session tier has no
  client identity). Its `error_page 401 = @login_bounce;` opt-in (D20) is
  untouched.

**Variable naming.** The fragment already reuses one `$prompts_owner` variable
across both identity-forwarding locations (nginx values `auth_request_set`
variables per request, so a single name is safe); the three new variables follow
that convention — `$prompts_owner_id`, `$prompts_owner_name`,
`$prompts_owner_picture`, each `prompts`-prefixed so it stays unique when every
service fragment is included into the one apex `server` block.

**Identity hygiene extends to the new headers.** As with `X-Owner-Email` today,
nginx does not pass an inbound client header as a proxy header unless explicitly
set, and the last `proxy_set_header` for a name wins; a single `proxy_set_header`
per owner-header name, sourced from the auth-subrequest variable, therefore
**replaces** any client-smuggled `X-Owner-Id` / `X-Owner-Name` / `X-Owner-Picture`
entirely — a client cannot smuggle identity through.

**Untouched locations.** The unauthenticated PRM bootstrap
(`= /srv/prompts/.well-known/oauth-protected-resource`), the `= /srv/prompts/feed`
404 guard, and the session-gated static assets (`location /srv/prompts/static/`,
which forwards **no** identity today) forward no owner headers and stay exactly as
they are. Downstream consumption of `X-Owner-Id` by the prompts binary is out of
scope here — this pass is fragment forwarding only.

## Stale-doc consequence

prompts has **no** service-level `AGENTS.md`/`CLAUDE.md`; its authoritative
surface-posture doctrine is the **package doc comment of `cmd/prompts/main.go`**,
which enumerates the domain surface it wires through the Handlers hook as ending
in "**the bare MCP surface**." With the landing route mounted that enumeration is
**incomplete/false** — prompts now also serves a human landing page. Per the
suite doctrine (docs state current truth; purge what the change falsified,
rewrite in place), that clause must be corrected to name the landing page beside
the MCP surface. It is **not** edited here — it is a dedicated **plan phase**
(Phase 13) so code and docs evolve together through the build loop. (The "performs
no token logic of its own" clause stays true: the landing page reads no token and
runs no token logic — nginx is still the trust boundary.)

## Rejected

- **A shared landing handler in appkit.** Each app's landing page will diverge to
  serve that app's purpose; v1 uniformity is a starting point, not a contract.
  prompts owns its handler + template + assets so divergence is a local edit, not
  a shared-library change. (The Carbon *tokens* are copied, not shared code.)
- **Bearer-gating the landing page (`auth_request /_authn`).** A browser carries
  no opaque bearer token, so the human page would be permanently unreachable. The
  audience is a logged-in human; the cookie session is the right credential.
- **Serving the landing page fully public (no auth).** The suite exposes nothing
  un-gated except the PRM bootstrap doc; a name+version page is innocuous but
  still inside the tenant boundary. Coarse session-gating matches `sites`' private
  tier and keeps one consistent rule.
- **Wrapping `GET /{$}` in `RequireIdentity` in-process.** That is the bearer
  gate; it would reject browsers and duplicate the trust boundary nginx owns. The
  route is ungated in-process by design — nginx is the sole gate, as for `/mcp`
  and PRM.
- **A catch-all `GET /` instead of `GET /{$}`.** A bare `/` prefix would shadow
  `/mcp`, `/health`, `/feed`, and PRM. `{$}` matches the exact root only, so the
  landing route is collision-free by construction.
- **Reusing a shared `ui` package.** prompts is a separate Go module and a
  separate release; it ships its own copy of the Carbon assets on disk in
  `share/www` (the *pattern* is copied, not a dependency imported).
- **Keeping the assets embedded in the binary (`//go:embed`).** The suite
  reverted embedding (D16): embedded assets are invisible on the box and cost a
  rebuild for any dev tweak; the shipped `share` tier plus the chassis static
  mount serve them uniformly.

## Verification

- R-LAND-PG01 — `GET /{$}` (exact root `/`) returns **200** with
  `Content-Type: text/html; charset=utf-8` — driven through the handler the
  composition root mounts (`net/http/httptest`), no DB/LLM/runner/identity
  required.
- R-LAND-NMVR — the rendered body contains the **service name** (`rt.Service()`,
  `"prompts"`) and the **version** (`rt.Version()`); built with a non-default
  injected version, the page shows that exact version (proving it renders
  `Version()`, not a hard-coded literal).
- R-LAND-CARB — the page is Carbon-styled from **on-disk** assets: the
  `prompts/share/www` tree contains `static/tokens.css` and the woff2 font(s),
  the served page references `tokens.css`, and a request through the chassis
  static mount for the `tokens.css` asset returns 200 with a CSS content-type —
  proving the assets ship on disk in the `share` tier (and are served by the
  chassis), not embedded in the binary.
- R-LAND-ROOT — `{$}` matches **only** the exact root: a request to a non-root
  path the landing handler does not own (e.g. `/health`, `/mcp`, `/nope`) is
  **not** served the landing page by this route — the landing mount does not
  shadow the existing routes.
- R-LAND-UNGT — the landing route is mounted **ungated in-process** (not wrapped
  in `RequireIdentity`): an httptest request with **no** identity header / no
  bearer still gets the 200 page, proving the in-process route trusts nginx as
  the sole gate (the cookie gate is asserted in nginx, Phase 12).
- R-7NY0-UIO6 — in `etc/nginx.conf`, the **bearer prefix** `location
  /srv/prompts/` (the `auth_request /_authn` tier) captures **all four** owner
  headers with `auth_request_set $prompts_owner $upstream_http_x_owner_email;`,
  `auth_request_set $prompts_owner_id $upstream_http_x_owner_id;`,
  `auth_request_set $prompts_owner_name $upstream_http_x_owner_name;`, and
  `auth_request_set $prompts_owner_picture $upstream_http_x_owner_picture;`, and
  forwards each with `proxy_set_header X-Owner-Email $prompts_owner;`,
  `proxy_set_header X-Owner-Id $prompts_owner_id;`, `proxy_set_header X-Owner-Name
  $prompts_owner_name;`, and `proxy_set_header X-Owner-Picture
  $prompts_owner_picture;` — **and** retains `proxy_set_header X-Client-Id
  $prompts_client;`. A block missing any owner header, or forwarding email only,
  fails. *(string assertions over the `location /srv/prompts/` block of
  `etc/nginx.conf` in `cmd/prompts/web_test.go`)*
- R-7P5X-8AEV — in `etc/nginx.conf`, the **session landing** exact location
