# Decision 10 — Two front doors: the session-gated human root and the bearer-gated agent surface

## Decision

Under `/srv/prompts/` there are two kinds of caller, each authenticated
differently — and the difference is the spine of this Decision:

- **Agents / MCP clients → bearer token.** `POST /mcp` is gated by
  `auth_request /_authn` (opaque bearer) via the prefix
  `location /srv/prompts/` fragment; the exact-match PRM bootstrap and the
  `= /srv/prompts/feed → 404` guard sit beside it. Identity forwarding is the
  four-owner-header contract below.
- **Humans in a browser → login-session cookie.** The human surface is gated
  by `auth_request /_session-authn` — the dashboard-owned, loopback-only
  endpoint that validates the `dashboard_session` cookie against the
  web-session store; failed checks bounce to login (`@login_bounce`, D20).
  This is the same mechanism `sites` uses for its private tier, and it is
  deliberately **coarse**: any logged-in dashboard user may view prompts'
  human pages (no per-resource check, no owner scoping).

A browser cannot satisfy the bearer gate (it has no token), and the bearer
gate is the wrong audience for a human page; so the human surface **must** be
cookie-gated — and never fully public, since the suite exposes nothing
un-gated except the PRM bootstrap doc.

**The mount root is the human front door, and it redirects.** The bare mount
root `GET /{$}` — session-gated by the exact-match nginx location below — no
longer renders a page of its own: it issues a relative redirect into the
browse UI (`ui/` — mechanism and id in D34; the pages themselves in D35). The
original v1 landing card (name+version, the seed this surface grew from) is
retired; its name+version and Carbon-asset behaviors live on in the UI chrome
(ids moved to D35), and `share/www/landing.html` is deleted with it.

## The seam — Go side (`cmd/prompts/main.go`, `registerRoutes`)

The root route is mounted **ungated in-process** — nginx is the sole gate,
exactly as `POST /mcp` trusts nginx for identity:

```go
rt.HandleFunc("GET /{$}", func(w http.ResponseWriter, r *http.Request) {
    http.Redirect(w, r, "ui/", http.StatusSeeOther)
})
```

- `{$}` (Go 1.22+ `ServeMux`) matches **only** the exact root path `/`, so it
  cannot shadow `/mcp`, `/health`, `/feed`, or
  `/.well-known/oauth-protected-resource`.
- It is **not** wrapped in `rt.RequireIdentity(...)` (the bearer gate). The
  cookie gate lives entirely in nginx; in-process the route is open and reads
  no identity header.

## The seam — nginx (`prompts/etc/nginx.conf`)

The exact-match root location session-gates and proxies to the app's
`GET /{$}`, beside the bearer-gated prefix:

```
# Human front door — gated by the dashboard browser SESSION (cookie),
# validated by the dashboard-owned /_session-authn internal location. Exact-
# match `=` so it never shadows the gated prefix below or the MCP endpoint.
location = /srv/prompts/ {
    auth_request /_session-authn;
    error_page 401 = @login_bounce;
    auth_request_set $prompts_owner         $upstream_http_x_owner_email;
    auth_request_set $prompts_owner_id      $upstream_http_x_owner_id;
    auth_request_set $prompts_owner_name    $upstream_http_x_owner_name;
    auth_request_set $prompts_owner_picture $upstream_http_x_owner_picture;
    proxy_set_header X-Owner-Email   $prompts_owner;
    proxy_set_header X-Owner-Id      $prompts_owner_id;
    proxy_set_header X-Owner-Name    $prompts_owner_name;
    proxy_set_header X-Owner-Picture $prompts_owner_picture;
    proxy_pass http://127.0.0.1:3002/;   # trailing slash → upstream sees "/"
    proxy_set_header Host $host;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_http_version 1.1;
}
```

The existing `location /srv/prompts/ { auth_request /_authn; ... }` **prefix**
stays for everything else (the MCP endpoint, the 429 re-emit handling); the
exact-match `=` root wins for the bare root only, and the session-gated
`/srv/prompts/ui/` prefix (D34) and `/srv/prompts/static/` (D13) sit beside
it. The PRM doc and the `/feed` 404 guard are unaffected. The loopback port is
a literal in the fragment (`PORT=3002`) — nothing substitutes it at install.

## Identity-header forwarding — all four owner headers

The dashboard's introspection hooks (`/_authn` bearer and `/_session-authn`
session) emit **four** owner headers on every allow, fail-closed: `X-Owner-Id`
(durable owner key, never empty), `X-Owner-Email` (display/audit, never
empty), `X-Owner-Name` and `X-Owner-Picture` (percent-encoded, MAY be empty).
The fragment captures and forwards **all four** on every location that
forwards identity:

- **Bearer prefix `location /srv/prompts/`** (the `auth_request /_authn`
  tier): `auth_request_set` all four owner headers into `$prompts_owner`,
  `$prompts_owner_id`, `$prompts_owner_name`, `$prompts_owner_picture` and
  `proxy_set_header` each of `X-Owner-Email`, `X-Owner-Id`, `X-Owner-Name`,
  `X-Owner-Picture` from those variables — **plus** the existing
  `X-Client-Id` from `$prompts_client`. The 429 re-emit capture
  (`$authn_status`/`$authn_retry`) and `@prompts_authn_500` handling are
  untouched.
- **Session root `location = /srv/prompts/`** (and the `ui/` prefix, D34): the
  same four pairs, into the same `$prompts_owner*` variables. No
  `X-Client-Id` (the session tier has no client identity). The
  `error_page 401 = @login_bounce;` opt-in (D20) is untouched.

**Variable naming.** One `$prompts_owner*` variable set is reused across the
identity-forwarding locations (nginx values `auth_request_set` variables per
request, so a single name is safe), each `prompts`-prefixed so it stays unique
when every service fragment is included into the one apex `server` block.

**Identity hygiene.** nginx does not pass an inbound client header as a proxy
header unless explicitly set, and the last `proxy_set_header` for a name wins;
a single `proxy_set_header` per owner-header name, sourced from the
auth-subrequest variable, **replaces** any client-smuggled identity header
entirely.

**Untouched locations.** The unauthenticated PRM bootstrap
(`= /srv/prompts/.well-known/oauth-protected-resource`), the
`= /srv/prompts/feed` 404 guard, and the session-gated static assets
(`location /srv/prompts/static/`, which forwards **no** identity) forward no
owner headers and stay exactly as they are.

## Rejected

- **Bearer-gating the human surface (`auth_request /_authn`).** A browser
  carries no opaque bearer token, so the human pages would be permanently
  unreachable. The audience is a logged-in human; the cookie session is the
  right credential.
- **Serving the human surface fully public (no auth).** The suite exposes
  nothing un-gated except the PRM bootstrap doc; the browse UI shows prompt
  text and inference logs — squarely inside the tenant boundary. Coarse
  session-gating matches `sites`' private tier and keeps one consistent rule.
- **Wrapping the human routes in `RequireIdentity` in-process.** That is the
  bearer gate; it would reject browsers and duplicate the trust boundary
  nginx owns. The routes are ungated in-process by design.
- **A catch-all `GET /` instead of `GET /{$}`.** A bare `/` prefix would
  shadow `/mcp`, `/health`, `/feed`, and PRM. `{$}` matches the exact root
  only, so the root route is collision-free by construction.
- **Keeping the name+version landing card as the root page.** The human
  surface's job is now browsing prompts and runs (D35); a card in front of it
  is a click that conveys nothing the UI chrome doesn't. The root redirects
  (D34) and the card retires.

## Verification

- R-LAND-ROOT — the root route matches **only** the exact root: a request to
  a non-root path the root handler does not own (e.g. `/health`, `/mcp`,
  `/nope`) is **not** answered by the root redirect — the mount does not
  shadow the existing routes.
- R-LAND-UNGT — the root route is mounted **ungated in-process** (not wrapped
  in `RequireIdentity`): an httptest request with **no** identity header / no
  bearer still gets the root's redirect response, proving the in-process
  route trusts nginx as the sole gate.
- R-7NY0-UIO6 — in `etc/nginx.conf`, the **bearer prefix** `location
  /srv/prompts/` (the `auth_request /_authn` tier) captures **all four** owner
  headers with `auth_request_set $prompts_owner $upstream_http_x_owner_email;`,
  `auth_request_set $prompts_owner_id $upstream_http_x_owner_id;`,
  `auth_request_set $prompts_owner_name $upstream_http_x_owner_name;`, and
  `auth_request_set $prompts_owner_picture $upstream_http_x_owner_picture;`,
  and forwards each with `proxy_set_header X-Owner-Email $prompts_owner;`,
  `proxy_set_header X-Owner-Id $prompts_owner_id;`, `proxy_set_header
  X-Owner-Name $prompts_owner_name;`, and `proxy_set_header X-Owner-Picture
  $prompts_owner_picture;` — **and** retains `proxy_set_header X-Client-Id
  $prompts_client;`. A block missing any owner header, or forwarding email
  only, fails. *(string assertions over the `location /srv/prompts/` block of
  `etc/nginx.conf` in `cmd/prompts/web_test.go`)*
- R-7P5X-8AEV — in `etc/nginx.conf`, the **session root** exact location
  `location = /srv/prompts/` (the `auth_request /_session-authn` tier)
  captures **all four** owner headers into `$prompts_owner`,
  `$prompts_owner_id`, `$prompts_owner_name`, `$prompts_owner_picture` and
  forwards each with `proxy_set_header X-Owner-Email $prompts_owner;`,
  `proxy_set_header X-Owner-Id $prompts_owner_id;`, `proxy_set_header
  X-Owner-Name $prompts_owner_name;`, and `proxy_set_header X-Owner-Picture
  $prompts_owner_picture;` — not email-only forwarding. A block forwarding
  only `X-Owner-Email` fails. *(string assertions over the
  `location = /srv/prompts/` block of `etc/nginx.conf` in
  `cmd/prompts/web_test.go`)*
