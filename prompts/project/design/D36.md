# Decision 36 ŌĆö Owner-id keying: rebuild `prompts`/`runs`, rekey the store on `owner_id`

## Decision

The suite is converting from email-keyed to id-keyed ownership
(`docs/owner-id-design.md`): the appkit chassis now carries the stable owner id
on `server.Identity.OwnerID` (from nginx's `X-Owner-Id`) and its gate refuses on
a missing `X-Owner-Id`, not `X-Owner-Email` (appkit D13, the shipped external
contract). Email is becoming user-mutable, so it can no longer key what a caller
owns. This Decision converts prompts' two owner-scoped tables ŌĆö `prompts` and
`runs` ŌĆö and the store/service that reads them, so every scoping and lookup path
keys on `owner_id` and only `owner_id`; email is reduced to a write-once display
snapshot.

**Schema rebuild.** The legacy `prompts` and `runs` tables are email-keyed
(frozen migrations `006_prompt_redesign.sql`, `20260609135507_add_source_path.sql`,
`20260712192403_trigger_filters.sql`; a service cannot map stored emails to owner
ids). One NEW timestamped migration ŌĆö created with
`bin/create-migration prompts <name>`, never hand-numbered, with the frozen
migrations untouched ŌĆö **drops and recreates both tables** with the shapes below,
adding `owner_id` as the scoping key beside a write-once `owner_email`. Rows are
dropped (suite-settled: only ledger holds live data during the migration window;
prompts state is disposable and the rebuild is deterministic on any database).

```sql
CREATE TABLE prompts (
    id            TEXT PRIMARY KEY,
    owner_id      TEXT NOT NULL,   -- stable owner key; the ONLY identity read for scoping
    owner_email   TEXT NOT NULL,   -- write-once snapshot at create; display only, never updated, never read for logic
    name          TEXT,
    user_prompt   TEXT NOT NULL,
    system_prompt TEXT,
    config_json   TEXT NOT NULL,
    source_path   TEXT,
    created_at    TEXT NOT NULL,
    updated_at    TEXT NOT NULL
);
CREATE INDEX idx_prompts_owner ON prompts(owner_id);
-- the import-upsert invariant moves to owner_id: re-importing the same
-- (owner_id, source_path) updates the same row; NULL source_path rows are exempt.
CREATE UNIQUE INDEX idx_prompts_source ON prompts(owner_id, source_path) WHERE source_path IS NOT NULL;

CREATE TABLE runs (
    id               TEXT PRIMARY KEY,
    prompt_id        TEXT NOT NULL,
    owner_id         TEXT NOT NULL,   -- denormalized from the prompt at run start; the run's scoping key
    owner_email      TEXT NOT NULL,   -- denormalized display snapshot; never read for logic
    prompt_name      TEXT,
    status           TEXT NOT NULL,
    started_at       TEXT NOT NULL,
    ended_at         TEXT,
    usage_json       TEXT,
    error            TEXT,
    trigger_source   TEXT,
    trigger_kind     TEXT,
    trigger_subject  TEXT,
    trigger_event_id TEXT,
    log_path         TEXT NOT NULL
);
CREATE INDEX idx_runs_owner  ON runs(owner_id);
CREATE INDEX idx_runs_prompt ON runs(prompt_id, started_at);
CREATE INDEX idx_runs_status ON runs(status);
```

`owner_id` is the nginx-forwarded `X-Owner-Id` (appkit `Identity.OwnerID`).
`owner_email` is captured from `Identity.OwnerEmail` at row creation and never
touched again. Two callers presenting the same email but different owner ids are
**different owners**; two callers presenting the same owner id are the same owner
regardless of email.

**Store/service rekey.** The domain structs carry both fields; every scoping
parameter is an **owner id**, never an email:

```go
type Prompt struct {
    ID         string
    OwnerID    string // scoping key
    OwnerEmail string // display snapshot
    // ŌĆ”Name, UserPrompt, SystemPrompt, Config, SourcePath, CreatedAt, UpdatedAt
}

type Run struct {
    ID         string
    PromptID   string
    OwnerID    string // denormalized from the prompt at run start; scoping key
    OwnerEmail string // denormalized display snapshot
    // ŌĆ”PromptName, Status, timestamps, trigger context, LogPath
}
```

- `Store` write/read/scope methods key on `owner_id`: `InsertPrompt` /
  `ImportUpsert` persist `owner_id` + `owner_email` (the upsert `ON CONFLICT`
  target becomes `(owner_id, source_path)`); `GetPrompt`, `ListByOwner`,
  `UpdatePrompt`, `DeletePrompt` filter `WHERE ŌĆ” owner_id = ?`; `InsertRun`
  persists the run's denormalized `owner_id` + `owner_email`; the run scoping
  reads compare the run row's `owner_id`.
- `Service` methods that scope take an **owner id** (`Get`, `List`, `Update`,
  `Delete`, `SetTrigger`, `ClearTrigger`, `Run`, `RunList`, `RunGet`,
  `RunOutput`, `RunCancel`, `RunFsList`, `RunFsRead`). The two create paths take
  **both** ŌĆö `Create(ctx, ownerID, ownerEmail, in)` and
  `Import(ctx, ownerID, ownerEmail, sourcePath, name)` ŌĆö snapshotting the email
  onto the new row while keying ownership on the id.
- `SpawnRun` / `SpawnTriggeredRun` denormalize the run's `owner_id` **and**
  `owner_email` from the prompt row (as `prompt_name` is denormalized today), so
  a run stays owner-addressable by id and survives a tombstoned prompt. A
  triggered run (fired from an event, no request identity) inherits the stored
  prompt's `owner_id`.

No store or service method accepts or filters on `owner_email` for logic; the
column exists only to be written once and displayed (D37 renders it).

## Rejected

- **Backfilling email ŌåÆ id instead of dropping rows.** Only the dashboard knows
  the mapping; the service cannot. Suite-settled: rebuild empty, deterministic on
  any database.
- **Keying on `(owner_id, owner_email)` or an email fallback.** Re-anchors logic
  to the mutable display field; `owner_id` alone is the contract (mirrors appkit
  D13's gate rationale).
- **Two migrations (prompts, then runs).** Both tables convert together as one
  forward-only file; splitting adds ordering complexity for no benefit.
- **Adding `owner_id` to the `calls` table (D28) in this Decision.** `calls` is
  the inference **accounting** record, deliberately not owner-scoped or
  owner-filtered (D32): its `owner_email` is attribution that already conforms to
  the write-once/display-only email rule, and the suite design's prompts
  inventory scopes this conversion to `prompts`/`runs`. Left unchanged here; a
  display-parity `owner_id` on `calls`/`usage` is a separable follow-up.

## Verification

(Real substrate: a temp-file SQLite database with the appkit migration runner and
the concrete `Store`/`Service` ŌĆö the DB constraints and the SQL scoping are what
falsify these, not a mock.)

- R-E59O-RJC7 ŌĆö Applying the full migration set to a database that already holds
  pre-conversion `prompts` and `runs` rows leaves both tables with `owner_id` and
  `owner_email` as `NOT NULL` columns and **zero rows** ŌĆö the rebuild drops
  pre-conversion data rather than carrying rows with no owner id. (Seed the
  pre-conversion shapes with a row each, run the runner over the full migration
  dir, inspect `PRAGMA table_info` and both row counts.)
- R-E6HL-5B2W ŌĆö With two prompts seeded under the **same `owner_email` but
  distinct `owner_id`s** (idA, idB), `List(idA)` returns exactly idA's prompt and
  never idB's; `Get(idA, idBsPrompt)` returns `ErrNotFound`; `Delete(idA,
  idBsPrompt)` removes nothing and idB's prompt still reads back ŌĆö an
  implementation still filtering on email admits the cross-access and fails.
- R-E7PH-J2TL ŌĆö With two runs seeded (via their prompts) under the same
  `owner_email` but distinct `owner_id`s, the run-scoped reads
  (`RunGet`/`RunList`/`RunCancel`/`RunOutput`/`RunFsList`/`RunFsRead`) scope on
  the **run row's `owner_id`**: idA sees only idA's run and idB's run id returns
  `ErrNotFound` under idA, even though both share one email.
- R-E8XD-WUKA ŌĆö `Create(ctx, ownerID, ownerEmail, ŌĆ”)` persists the row with
  `owner_id == ownerID` and `owner_email == ownerEmail` **verbatim**; a
  subsequent `Update` of other fields leaves `owner_email` unchanged (write-once
  snapshot); and a run spawned from that prompt reads back with `owner_id` equal
  to the prompt's `owner_id` (the denormalization carries the id, not the email,
  as the scoping key).
