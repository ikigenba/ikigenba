# Decision 31 — Admission control: bounded inference concurrency

## Decision

prompts today spawns an unbounded goroutine per run and per event match; once it owns all suite inference, a pathological burst (a trigger matching hundreds of events) must degrade into slow-but-orderly progress, not a provider-quota incident that also takes down sibling services' synchronous calls. v1 is semaphores only — no queues, no priorities, no budgets.

A new `internal/admit` package:

```go
type Gate struct { /* per-key weighted semaphores, lazily created */ }

func New(callCap, runCap int) *Gate

// AcquireCall bounds in-flight /complete + /embed provider calls, per provider.
// Blocks until a slot frees; returns ctx.Err() if ctx is done first.
func (g *Gate) AcquireCall(ctx context.Context, provider string) (release func(), err error)

// AcquireRun bounds concurrently executing agent-session runs, globally.
func (g *Gate) AcquireRun(ctx context.Context) (release func(), err error)
```

**Two independent pools, deliberately:**

- **Calls** — one semaphore per provider name, capacity `PROMPTS_MAX_INFLIGHT_CALLS` (default `8`). Acquired around the provider round-trip of every completion and embedding. Distinct providers never contend.
- **Runs** — one global semaphore, capacity `PROMPTS_MAX_CONCURRENT_RUNS` (default `8`). Acquired inside the run goroutine before `execute` begins (after the run row exists — a queued run is simply `running` and not yet talking to a provider), released at the terminal write. Wiring lands with D33's runner change.

Sessions do not draw from the call pool: a run holds its slot for minutes, and letting sessions and completions contend in one pool would let a run storm starve wiki's user-facing ask path. Splitting the pools is the cheap form of the "reserved completion capacity" refinement, chosen over a shared pool precisely because starvation across classes is the failure that matters.

Both caps surface in `ManifestExtras`. Retry/backoff against provider 429s stays agentkit's job (`RetryPolicy`, Retry-After aware); the gate only bounds how much prompts fires at once.

## Rejected

**One shared per-provider pool for everything** — a long-held session slot starves the synchronous class behind it; cross-class isolation is the actual requirement.

**A queueing/QoS layer (priorities, budgets, persistent queues)** — no evidence demands it on a one-owner box; the `calls` table makes any future backlog visible, and the semaphore converts bursts into ordered progress today. Complexity deferred until measured need.

**Rejecting over blocking (429 from `/complete`)** — pushes retry loops into every caller; blocking with ctx cancellation gives callers their own timeout control with zero added protocol.

## Verification

- R-67FV-VQ6I — with a call capacity of 1, two concurrent `AcquireCall`s for the same provider serialize: the second acquire does not return until the first's release runs.
- R-68NS-9HX7 — with a call capacity of 1, `AcquireCall` for two different providers proceeds concurrently — distinct providers never contend.
- R-6CBH-ET5A — an `AcquireCall` blocked on a full semaphore returns the context's error when the context is canceled, without ever holding a slot.
- R-6B3L-11EL — with a run capacity of 1, two spawned runs execute serially: the second run's provider call does not begin until the first run reaches its terminal state (runner-level; realized with D33's wiring).
