# Decision 35 — The browse UI: server-rendered prompts/runs pages with a per-run calls log

## Decision

The `/ui/` namespace (D34) serves a **server-rendered browse surface** over the
three tables prompts already owns: the `prompts` and `runs` tables
(`internal/prompt`) and the `calls` table (`internal/calls`, D28). Two tabs —
**Prompts** (the default) and **Runs** — each a filterable, paginated table;
each row clicks through to a detail page; a run's detail page **is the log**:
the run's calls render inline, in order, with their request/response bodies.
Everything is filtered and paginated **server-side** (the runs and calls tables
are too big to ship to the client — the deliberate inversion of sites' render-
all-then-filter-in-JS landing page); pages are plain GET-driven HTML — query
params in, one page of rows out, no JS required. Pages are static snapshots: a
running run's page updates on browser refresh, never by push.

**Routes** (Go 1.22 `ServeMux` patterns, mounted in `registerRoutes`, all
ungated in-process per D34):

```
GET /ui/{$}                prompts tab (default)
GET /ui/runs               runs tab
GET /ui/prompts/{id}       prompt detail
GET /ui/runs/{id}          run detail — the inline calls log
GET /ui/calls/{id}/raw     one call body, raw (?side=request|response)
```

**Tabs and chrome.** Every page shares one layout: the D12 Home link
(`href="/"`), a header carrying the service name and version (the same
`rt.Service()`/`rt.Version()` data the old landing card showed), and a two-tab
nav — Prompts → `/srv/prompts/ui/`, Runs → `/srv/prompts/ui/runs` (links are
mount-literal absolute, like the asset paths, D34) — with the active tab
marked. Carbon-styled from the shared on-disk assets via the mount-literal
paths (D34), visually in line with the `design/` reference and the sites
landing page. Templates are new flat files in `share/www/`
(`ui-prompts.html`, `ui-runs.html`, `ui-prompt.html`, `ui-run.html`), loaded by
the same chassis `Spec.WWW` site and rendered through `rt.WWW().Render`; the
retired `landing.html` is deleted (D10/D11 supersession).

**The lists — columns, order, filters.** Page size is **50** rows, newest
first; a `page` query param (1-based) drives `LIMIT/OFFSET`; the pager renders
"Page N of M" with Prev/Next links that **preserve the active filters**; the
filter form is a plain GET form that resets to page 1.

- **Prompts tab** (`?q=&page=`): columns name, owner_email, updated_at,
  created_at; ordered `updated_at DESC`. `q` matches case-insensitive substring
  against `name` **or** `owner_email`.
- **Runs tab** (`?q=&status=&prompt_id=&page=`): columns prompt_name (linking
  to the prompt detail), status, owner_email, started_at, duration
  (`ended_at − started_at`, blank while running), trigger (`trigger_source` +
  `trigger_type` collapsed, blank = manual); each row links to the run detail;
  ordered `started_at DESC`. `q` matches substring against `prompt_name` or
  `owner_email`; `status` matches exactly; `prompt_id` matches exactly (the
  entry point from a prompt's detail page).

**Store queries** — the domain stores grow paged, counted list methods (the
existing `ListPrompts`/`ListRunsByPrompt` are owner-scoped or unpaged MCP
shapes and stay untouched):

```go
// internal/prompt
type BrowseFilter struct{ Q, Status, PromptID string; Limit, Offset int }
func (s *Store) BrowsePrompts(ctx context.Context, f BrowseFilter) ([]Prompt, int, error) // rows, total
func (s *Store) BrowseRuns(ctx context.Context, f BrowseFilter) ([]Run, int, error)

// internal/calls
func (s *Store) ListByGroup(ctx context.Context, groupID string) ([]Row, error) // bodies included, started_at ASC
```

The `int` total backs the pager; `Q`/`Status`/`PromptID` are ignored where a
dimension does not apply (prompts ignore Status/PromptID). **No owner scoping**
— the queries take no owner argument (D34; the D32 whole-box posture).

**Prompt detail** (`/ui/prompts/{id}`) — the columns the list omits: the full
`user_prompt`, `system_prompt`, and pretty-printed `config_json`, plus the
prompt's triggers (`prompt_triggers` rows: source + event_filter) and a "runs
of this prompt" link to `/srv/prompts/ui/runs?prompt_id=<id>`. A missing id —
which, tombstone delete being a real row delete, is exactly what a deleted
prompt looks like — renders a styled **404** page saying the prompt does not
exist or was deleted (the graceful landing for a dangling `prompt_id` link from
an old run).

**Run detail — the log** (`/ui/runs/{id}`). Top: the columns the runs list
omits — id, prompt_id (linked), error, pretty-printed usage_json,
trigger_event_id, log_path. Below: the run's calls (`ListByGroup(runID)`)
inline in `started_at` order — reading top to bottom replays the run. Each call
renders a metadata line (class, name, attempt, provider, model, input/output/
total tokens, cost_usd, error if any, started/ended) and its `request_body` and
`response_body`, pretty-printed (`json.Indent`; shown verbatim when not valid
JSON) in scrollable `<pre>` blocks. Two degradations are explicit, never
silent:

- A **NULL body** (pruned by the D28 retention sweep, or never stored) renders
  a visible "body pruned by retention" / "no body recorded" note in place of
  the `<pre>` block.
- A body **over 64 KiB** (`uiBodyInlineLimit = 64 << 10`, bytes) renders only
  its first 64 KiB with a truncation note and a **"view full body"** link to
  the raw endpoint; under the threshold it renders whole with no link needed.

An unknown run id renders the styled 404 page.

**The raw-body endpoint** (`/ui/calls/{id}/raw?side=request|side=response`) —
the full, unformatted body as `text/plain; charset=utf-8`, streamed whole
(`Store.Get` already returns bodies). `side` other than `request`/`response` →
400; unknown call id → 404; a NULL (pruned/never-stored) body → 404 with a
body-pruned message, so the link a page rendered yesterday fails loudly, not
emptily. It lives under `/ui/` so the one session gate covers it.

**Out of the walk:** `completion`/`embedding`-class calls have no run, so this
UI never reaches them (accepted boundary — they remain inspectable via the
`calls`/`usage` MCP tools, D32); there is no flat calls browser, no date-range
filter, no auto-refresh, and no MCP change.

## Rejected

- **Client-side filtering like sites' landing page** — sites ships a bounded
  list of site names; runs and calls grow without bound. Dumping them to the
  client is the failure mode this design exists to avoid.
- **A third "Calls" tab** — a flat calls browser is a cost/usage view over
  rows that mostly belong to sibling services' one-shot work; the stated job
  is browsing prompts and replaying runs. Drill-down only.
- **A separate call detail page** — a run's calls are experienced as one
  story; forcing a click per call to reconstruct it inverts the reading. The
  run page is the log; the raw endpoint covers the oversized tail.
- **Inlining the run's on-disk output (`log_path`) on the run page** — the
  calls are the durable, queryable record with exact wire bodies; the run
  page shows `log_path` as data and stays a DB-backed page.
- **Owner-scoped rows** — falsifies the whole-box picture, diverges from the
  D32 tools, and adds an identity read the ungated-in-process posture avoids.
- **SSE/auto-refresh for running runs** — an inspection surface, not a
  monitor; browser refresh is enough for v1.

## Verification

*Page ids drive the mounted handlers via `net/http/httptest` over a seeded
SQLite DB and the repo-real `share/www` tree; store ids drive the store
directly over a seeded DB.*

- R-ZZVE-DK4O — `BrowsePrompts` orders by `updated_at DESC`, matches `Q` as a
  case-insensitive substring of name **or** owner_email, honors Limit/Offset,
  and returns the filtered total: seeded prompts of two owners and distinct
  names return exactly the matching rows and a total equal to all matches, not
  the page size.
- R-013A-RBVD — `BrowseRuns` orders by `started_at DESC` and filters by exact
  `Status`, exact `PromptID`, and `Q` over prompt_name/owner_email, each
  independently and combined, with Limit/Offset and a correct filtered total.
- R-03J3-IVCR — `ListByGroup` returns exactly the rows whose `group_id`
  matches, ordered `started_at ASC`, with `request_body`/`response_body`
  populated when present — while rows of other groups are absent.
- R-04QZ-WN3G — `GET /ui/` renders the prompts tab: every seeded prompt's
  name, owner_email, updated_at, and created_at appear, newest-updated first,
  each row linking to its `/srv/prompts/ui/prompts/<id>` detail, with the tab
  nav marking Prompts active and linking Runs.
- R-05YW-AEU5 — `GET /ui/?q=…` filters server-side: a seeded prompt matching
  the substring (by name, and separately by owner_email) appears while a
  non-matching prompt's name is **absent from the HTML** — not merely hidden.
- R-076S-O6KU — with 60 seeded rows, page 1 renders exactly 50 and page 2 the
  remaining 10; the pager shows the page count and its Prev/Next links carry
  the active `q`/filter params.
- R-08EP-1YBJ — `GET /ui/runs` renders the runs tab: prompt_name (linked to
  the prompt detail), status, owner_email, started_at, a computed duration for
  a finished run and a blank one for a running run, and the trigger column
  showing `trigger_source`/`trigger_type` for an event-fired run and blank for
  a manual run, ordered `started_at DESC`, each row linking to its run detail.
- R-09ML-FQ28 — `GET /ui/runs?status=failed` returns only failed runs, and
  `?q=` matches prompt_name and owner_email substrings, excluding non-matches
  from the HTML.
- R-0AUH-THSX — `GET /ui/runs?prompt_id=<id>` returns only that prompt's runs.
- R-0C2E-79JM — `GET /ui/prompts/{id}` renders the full user_prompt,
  system_prompt, pretty-printed config_json, every trigger row
  (source + event_filter), and a link to
  `/srv/prompts/ui/runs?prompt_id=<id>`.
- R-0DAA-L1AB — `GET /ui/prompts/{id}` for an id with no row returns **404**
  with the styled page stating the prompt does not exist or was deleted — not
  a bare error.
- R-0EI6-YT10 — `GET /ui/runs/{id}` renders the run's id, linked prompt_id,
  error, usage_json, trigger_event_id, and log_path, followed by that run's
  calls in `started_at` order — each with class, name, attempt, provider,
  model, token counts, cost, and its pretty-printed request and response
  bodies — while another run's calls are absent.
- R-0FQ3-CKRP — a call whose bodies are NULL renders an explicit
  pruned/no-body note in place of each missing body, and the note is absent
  for a call whose bodies are present.
- R-0GXZ-QCIE — a body larger than 64 KiB renders only its leading 64 KiB
  plus a truncation note and a link to that call's raw endpoint; a body under
  the threshold renders whole with no truncation note.
- R-0I5W-4493 — `GET /ui/calls/{id}/raw?side=request` (and `side=response`)
  returns the complete stored body verbatim as `text/plain; charset=utf-8` —
  including one larger than the inline threshold; an invalid `side` returns
  400; an unknown id, and a call whose requested body is NULL, return 404.
- R-0JDS-HVZS — `GET /ui/runs/{id}` for an unknown run id returns the styled
  404 page.
- R-LAND-NMVR — every UI page's shared header shows the **service name**
  (`rt.Service()`) and the **version** (`rt.Version()`); rendered with a
  non-default injected version, the pages show that exact version (id retained
  from the retired landing card — the behavior moved to the UI chrome).
- R-LAND-CARB — the UI pages are Carbon-styled from **on-disk** assets: the
  `share/www` tree contains `static/tokens.css` and the woff2 fonts, every UI
  page references `tokens.css` (via the mount-literal path, D34), and the
  chassis static mount serves it as CSS (id retained from the landing card;
  same proof, new pages).
