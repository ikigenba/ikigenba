# Decision 33 — Sessions on the record: runs write `calls` rows

## Decision

Agent-session runs join the `calls` table so the accounting surface covers all three workload classes. The runner records **one row per run** (D28's unit grain) at the terminal write, **in the same transaction** as `FinishRun`'s status/usage/outbox write — the run's terminal state and its spend record commit or fail together.

**Mechanics**: `prompt.Store.FinishRun` (which already writes the terminal status, `usage_json`, and the outcome outbox event in one tx) additionally calls `calls.Store.InsertTx` with:

- `class` = `session`
- `origin` = `trigger:<trigger_source>` for an event-fired run; `user:<owner_email>` for a manual run (empty `trigger_source` discriminates, as today)
- `name` = the run's denormalized `prompt_name`, falling back to `prompt_id` when unnamed
- `group_id` = the run id; `attempt` = 1; `owner_email` = the run's owner
- `provider`, `model` = the pinned config's values (catalog name)
- usage/token/cost columns from the drained stream's aggregate (`stream.Usage()`, `stream.Cost()` — the same values serialized into `usage_json` today)
- `error` = the run's terminal error, if any; `request_body`/`response_body` = NULL (the run's pinned inputs and `output.jsonl` are the session's body record; duplicating a full conversation per row would swamp the table)
- `started_at`/`ended_at` = the run's own timestamps

The runner threads usage/cost/provider/model into `FinishRun` (they are already in hand at the terminal write); cancelled runs record too — a cancelled run may still have burned tokens.

**Run-cap wiring (D31)**: the run goroutine wraps `execute` in `gate.AcquireRun(ctx)` — acquired after the run row exists (a queued run shows `running`), released at the terminal write. The run TTL clock starts at spawn as today, so a run stuck queuing still times out loudly.

## Rejected

**Recording from the runner outside the tx** — a crash between `FinishRun` and a separate insert silently loses spend; atomicity is the point of the outbox pattern already used for outcome events, and the calls row rides the same guarantee.

**Bodies for session rows** — the run already durably pins its inputs and (while it lives) logs its full conversation; copying them per-row duplicates megabytes for no new information.

**A calls row at run start updated at finish** — two writes and an in-flight state for a table whose grain is completed units; the run row already provides live in-flight visibility.

## Verification

- R-6JMV-PFLG — after a run completes against a fake provider reporting non-zero usage, exactly one `calls` row exists with class `session`, `group_id` equal to the run id, token and cost values matching the run's `usage_json`, and the pinned catalog model name.
- R-6KUS-37C5 — a manual run's row carries `origin` `user:<owner_email>`; an event-triggered run's row carries `origin` `trigger:<source>`.
- R-6M2O-GZ2U — when the calls insert fails (injected store error), `FinishRun`'s transaction rolls back: the run row does not reach a terminal state in that write and no calls row exists (the spend record and terminal state are atomic).
- R-6NAK-UQTJ — a failed run records a `calls` row whose `error` is non-empty and whose usage reflects the tokens consumed before the failure.
