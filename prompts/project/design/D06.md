# Decision 6 — Suite discovery

## Decision

`prompts/internal/suite/suite.go` returns the suite's peer tools as **deferred tool groups** — `[]agentkit.DeferredToolGroup`, one group per reachable peer — instead of a flat `[]agentkit.Tool`. The heavy per-tool descriptions and schemas still get fetched eagerly at spawn (they are needed the moment the in-run agent loads a tool), but they are no longer all surfaced to the model up front: agentkit's deferred-tools feature (research §2) shows the model a compact catalog and loads tools on demand. The best-effort, never-crash contract is unchanged: an unreachable or garbled peer is logged and skipped; `Discover` never returns an error and always returns a non-nil slice (possibly empty); self (`prompts`) is excluded so a run cannot spawn a run.

**Signature** (owner-id keyed, `docs/owner-id-design.md`): the run's owner is
carried as the **id** peers now gate on, plus the email for display:

```go
func Discover(ctx context.Context, manifestRoot, ownerID, ownerEmail, promptID string) []agentkit.DeferredToolGroup
```

**Peer identity headers.** Every outbound MCP call (the `tools/list` and
`initialize` during discovery, and each `tools/call` during dispatch) carries
the run's identity so the peer's chassis gate admits it. Since appkit D13 the
peer gate refuses on a missing `X-Owner-Id`, so the forwarded headers are
`X-Owner-Id: ownerID` (the gating key) and `X-Owner-Email: ownerEmail` (display
only), beside `X-Client-Id: prompts:<promptID>`. Forwarding only the email — as
before the conversion — would 401 against every id-gated peer, so the id header
is the load-bearing field.

**Group shape**: for each non-self peer whose `tools/list` succeeds, one group:

- `Name` — the service's manifest name (`svc.Name`), the catalog section heading.
- `Blurb` — the peer's MCP `initialize` `instructions` string (research §3), fetched concurrently alongside `tools/list` with the same identity headers and per-call timeout. Best-effort within best-effort: an `initialize` failure or an absent `instructions` field yields `Blurb: ""` and the group is still returned — a peer with tools but no blurb degrades to bare names in the catalog, never to exclusion.
- `Tools` — the same `agentkit.RawTool` wrapping as before, unchanged: qualified name `ikigenba_<svc>_<verb>` (the existing `qualify` logic), discovered description and schema, `Call` dispatching `tools/call` to the peer with transport errors and MCP `isError` both surfaced as non-terminal Go errors (agentkit feeds them back as `isError` tool results; `terminalToolError` deliberately unused). The within-service duplicate guard is unchanged.

**`mcpclient` gains `Initialize`**: one new method on the existing client, same JSON-RPC plumbing as `ListTools`/`CallTool`:

```go
// Initialize calls JSON-RPC initialize and returns the server-level
// instructions string ("" when the server publishes none).
func (c *Client) Initialize(ctx context.Context) (string, error)
```

It sends the standard `initialize` params (protocol version, client info) and reads only `result.instructions`; capabilities negotiation is not consumed (the suite protocol is fixed).

**Eager pre-flight preserved**: discovery still happens once, at spawn, inside the runner's injectable `discover` closure — not lazily at `Send`. `agentkit.MCPServer` remains unused for the same reason as before: its resolution fails the whole `Send` when a peer is down, violating best-effort. Deferral is agentkit's `DeferredTools` field consuming this function's output (D19), not agentkit's MCP path.

## Rejected

**Use `agentkit.MCPServer`** — unchanged rejection: the published agentkit's MCP resolution propagates a down peer as a `Send` error; a down peer must not fail the run. (agentkit's own spec additionally records `MCPServers` deferral as out of scope.)

**Keep returning a flat `[]agentkit.Tool` and let the runner group it** — the grouping facts (service name, blurb) are discovery facts, known here and only here; flattening them away and re-deriving them in the runner splits one concern across two packages.

**Skip a peer whose `initialize` fails** — conflates "no blurb" with "no service"; the tools are already listed and callable, and a catalog section with bare names is strictly better than a missing service.

**Fetch blurbs from each service's `describe` tool** — `describe` is a per-service domain verb that not all peers expose; `initialize.instructions` is the MCP-standard server-level description every peer already publishes (research §3).

## Verification

- R-K32H-6XAV — when `Discover` is called with an inventory that includes a peer whose `tools/list` call returns an error, the returned groups contain no group for that peer and `Discover` does not return an error.
- R-K4AD-KP1K — when `Discover` is called with a reachable peer that lists one tool, the returned groups contain a group for that peer whose `Tools` holds exactly one `agentkit.Tool` with the service-qualified name (`ikigenba_<svc>_<verb>`) and the discovered schema; calling that tool's `Call` method dispatches to the peer.
- R-9JNO-RZM2 — `mcpclient.Initialize` against a server whose `initialize` result carries an `instructions` string returns that string; against a result with no `instructions` field it returns `""` without error.
- R-9KVL-5RCR — a peer whose `tools/list` succeeds but whose `initialize` fails (or returns no instructions) still yields its group, with `Blurb == ""` — the group is not dropped and `Discover` does not return an error.
- R-9M3H-JJ3G — with two reachable non-self peers and a `prompts` self entry in the inventory, `Discover` returns exactly two groups, each `Name` equal to its peer's manifest service name and each `Blurb` equal to that peer's published instructions; no group exists for `prompts`.
- R-EF0V-TP9R — when `Discover` is called with `ownerID`/`ownerEmail`, the outbound requests a peer receives (the discovery `tools/list` and a dispatched `tools/call`) carry `X-Owner-Id` equal to `ownerID` and `X-Owner-Email` equal to `ownerEmail` — asserted by a recording peer that captures the inbound headers; a build forwarding only `X-Owner-Email` sets no `X-Owner-Id` and fails this (the discriminating property against the pre-conversion header set peers would 401).
