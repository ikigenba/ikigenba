# Decision 5 — Built-in sandbox tools

## Decision

The run's built-in sandbox tools are assembled by `prompts/internal/tools/` and
implement the published `agentkit.Tool` interface. The toolset is **thirteen**
tools in two provenances:

- the **six standard coding tools** — `Bash`, `Read`, `Write`, `Edit`, `Glob`,
  `Grep` — come from agentkit's `toolkit` subpackage (`toolkit.All(sandboxRoot)`,
  D1, research §6). prompts no longer implements them.
- the **seven prompts-specific tools** — `Fetch` (the content-plane acceptor,
  D21) and the six file-share tools `FileList`, `FileGet`, `FilePut`,
  `FileDelete`, `FileMove`, `FileMkdir` (D26 owns their contract) — remain
  implemented in `prompts/internal/tools/`.

**Package shape**:

```go
// prompts/internal/tools/tools.go
package tools

import (
    "github.com/ikigenba/agentkit"
    "github.com/ikigenba/agentkit/toolkit"
)

// ShareConfig locates the account file share's loopback filesystem API for the
// six file-share tools (D26). BaseURL is the share service's loopback base
// (registry-defaulted at the composition root, the existing DROPBOX_BASE_URL
// seam); ClientID is the caller identity sent as X-Client-Id on every mutating
// call ("prompts:" + the run's prompt id, the suite.go convention).
type ShareConfig struct {
    BaseURL  string
    ClientID string
}

// All returns the thirteen built-in tools confined to sandboxRoot: the six
// standard toolkit tools (toolkit.All(sandboxRoot)) followed by the seven
// prompts-specific tools. It remains the single constructor — the runner never
// assembles the toolset itself, so the wrong variant cannot be wired.
// sandboxRoot is threaded in at call time (not package-level global) so tests
// supply a temp directory and the runner supplies the per-run sandbox path.
// sourcePortAllowed is the Fetch tool's loopback-port confinement seam (D21).
// share locates the file share's loopback API for the File* tools (D26);
// tests point BaseURL at an httptest server.
func All(sandboxRoot string, sourcePortAllowed func(port int) bool, share ShareConfig) []agentkit.Tool
```

Each prompts-specific tool is constructed with
`agentkit.NewTool[In](name, description, fn)` over a typed input struct (the
JSON Schema is derived from the struct's `jsonschema` tags), where `fn` is a
closure over `sandboxRoot` (and, for `Fetch`, `sourcePortAllowed`; for the
`File*` tools, `share`) returning `(string, error)` per `agentkit.Tool.Call`.
The six toolkit tools are taken as-is; their names, descriptions, schemas, and
behavior are toolkit's (research §6).

**Tool names** (names are capitalized in the tool table; the framing prompt
refers to them in lowercase prose and is deliberately unchanged by the toolkit
adoption):

| tool  | provenance | description |
|-------|------------|-------------|
| Bash  | toolkit | Run a shell command. `timeout` param (default 120s), process-group kill on timeout/cancel, nonzero exit reported inline as `[exit status N]`. |
| Read  | toolkit | Read a file, optionally selecting a range of lines. |
| Write | toolkit | Write a file, creating its parent directories when needed. |
| Edit  | toolkit | Replace exact text in a file; refuses an ambiguous match without `replace_all`. |
| Glob  | toolkit | Find files matching a glob pattern (recursive `**` supported). |
| Grep  | toolkit | Search file contents for a pattern (skips `.git` and binary files). |
| Fetch | prompts | Fetch a suite content URL into the sandbox (D21 owns its full contract). |
| FileList | prompts | List entries in the account's file share (D26). |
| FileGet | prompts | Copy a file from the file share into the sandbox (D26). |
| FilePut | prompts | Copy a sandbox file into the file share (D26). |
| FileDelete | prompts | Delete a file or folder in the file share (D26). |
| FileMove | prompts | Move or rename an entry in the file share (D26). |
| FileMkdir | prompts | Create a folder in the file share (D26). |

**Accepted toolkit deltas** (settled at adoption; recorded so they are not
re-litigated): every toolkit result is capped at 30,000 characters with a
truncation marker (`Read` pages via `offset`/`limit`); Bash defaults to a 120s
timeout (the agent may pass a larger `timeout`; there is no ceiling) and
reports nonzero exit inline instead of as a tool error; toolkit input schemas
carry no per-field descriptions; and **Glob's non-`**` patterns are not
confined** — `pattern: "../*"` lists entry names outside the sandbox root
(empirically verified, research §6). The Glob gap is accepted as
Bash-equivalent (prompts' Bash is and always was unconfined, so it widens
nothing) and list-only (Read/Write/Edit still refuse escaped paths); fixing it
belongs to a future toolkit release, not to prompts.

**Confinement prompts still owns**: the sandbox path validation (join, Clean,
resolve the longest existing ancestor's symlinks, verify containment with
`filepath.Rel`) remains in `prompts/internal/tools/` for the sandbox-path-taking
arguments of the prompts-specific tools — `Fetch.dest_path`,
`FileGet.dest_path`, and `FilePut.source_path`. The six toolkit tools'
confinement is toolkit's, proven by toolkit's own tests, not re-proven here.

**No package-level state**: `tools.All(...)` is called once per run (in the
runner's `execute`), producing a fresh slice of tool values. The tool
implementations are stateless.

## Rejected

**Runner composes the toolset** (`toolkit.All` + a shrunken `tools.All` called
separately in the runner) — rejected: multiple assembly sites for one toolset
invite the wrong variant being wired by mistake; the single `tools.All`
constructor stays the only place the thirteen are put together, and the runner
seam is untouched.

**Wrap the toolkit tools to preserve prompts' old semantics** (error-on-nonzero
Bash exit, uncapped Read, described schemas) — rejected: wrappers are
reinvention with extra steps; the point of the adoption is that prompts stops
owning this behavior, and the toolkit variants are strictly better where they
differ (timeouts, caps, ambiguity refusal).

**Keep the local six implementations** — rejected: ~530 lines of tool code
duplicating what the pinned dependency now ships; drift is guaranteed and the
duplicated behavior is the worse of the two (no timeout, no output cap,
scanner-limited grep).

**An `All` variant without the confinement/share seams** — multiple
constructors for one toolset invite the wrong variant being wired by mistake;
the seams are required arguments, and a caller that wants "nothing allowed"
passes a `func(int) bool { return false }` (and an empty `ShareConfig` is a
composition-root error, not a silent degrade).

## Verification

- R-F5X1-XH6C — `tools.All(sandboxRoot, allowed, share)` returns exactly
  **13** tools named `Bash`, `Read`, `Write`, `Edit`, `Glob`, `Grep`, `Fetch`,
  `FileList`, `FileGet`, `FilePut`, `FileDelete`, `FileMove`, `FileMkdir`;
  each satisfies the `agentkit.Tool` interface.
- R-GNY2-Y47H — the six standard tools returned by `tools.All` are the toolkit
  variants, proven by discriminating behavior: the `Bash` tool called with a
  command that exits nonzero returns a **nil error** whose output ends with the
  `[exit status N]` marker (the retired local implementation returned a tool
  error and no marker, so it fails this test).
- R-K1UK-T5K6 — each prompts-owned sandbox-path argument — `Fetch.dest_path`,
  `FileGet.dest_path`, `FilePut.source_path` — rejects a path that escapes
  `sandboxRoot` (a traversal attempt returns an error without touching the
  filesystem outside the sandbox).
