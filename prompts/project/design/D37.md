# Decision 37 — Owner-id at the MCP tool surface: scope on `X-Owner-Id`, snapshot the email, expose both

## Decision

prompts' owner-facing MCP verbs run behind the chassis identity gate (D17
transport over `appkit/mcp`, `POST /mcp` wrapped in `RequireIdentity`). The gate
now fills `server.Identity` from `X-Owner-Id` and refuses when it is absent
(appkit D13). This Decision points the domain-tool owner keying at
`Identity.OwnerID` (replacing the legacy `Identity.OwnerEmail` scoping), keeps
`Identity.OwnerEmail` as the display value the create/import paths snapshot, and
gives every owner-exposing result and output schema an `owner_id` field beside
the existing `owner_email` (suite Decision 5).

**Scoping keys on the id.** Every domain verb that scopes to the caller threads
`id.OwnerID` into the `prompt.Service` hop (D36's id-keyed methods): `list`,
`get`, `update`, `delete`, `set_trigger`, `clear_trigger`, `run`, `run_list`,
`run_get`, `run_output`, `run_cancel`, `run_fs_list`, `run_fs_read`. The two
create paths pass **both** — `create` calls `svc.Create(ctx, id.OwnerID,
id.OwnerEmail, …)` and `import` calls `svc.Import(ctx, id.OwnerID,
id.OwnerEmail, …)` — so the email is read only to be snapshotted onto the new
row, never to decide ownership. A verb acting on a prompt or run the caller does
not own (by id) resolves to `prompt.ErrNotFound` exactly as an unknown id does
(the D27 `not_found` code mapping is unchanged); "foreign-owned" now means a
different `owner_id`.

```go
// representative handler bodies (internal/mcp/tools.go)
p, err := svc.Create(ctx, id.OwnerID, id.OwnerEmail, prompt.CreateInput{ … })
prompts, err := svc.List(ctx, id.OwnerID)
run, err := svc.RunGet(ctx, id.OwnerID, in.RunID)
```

**Results and schemas expose both.** The owner-exposing tool results keep
`owner_email` and **add** `owner_id`, both read from the stored row (never from
the request): the `Prompt` view (`list`, `get`, `update`), the `PromptDetail`
view (`get`), and the `Run` view (`run_get`, `run_list`). Their D27
`outputSchema` literals add `owner_id` (`typ("string")`) beside `owner_email` and
include it in the `required` array wherever `owner_email` is required. The tool
descriptors' declared output schemas (`promptSchema`, `detailSchema`,
`runSchema`) gain the `owner_id` property/required entry likewise.

The `describe` prose, the raw-content tools (`run_output`, `run_fs_read`), and the
chassis `health`/`reflection` are unaffected (they expose no owner value here;
the chassis health envelope's `owner_id` is appkit-owned, D13/D9).

## Rejected

- **Reading owner from tool arguments.** Spoofable; the authoritative owner is
  the authenticated `Identity` nginx established. Tool args never carry identity.
- **Dropping `owner_email` from results now that scoping is by id.** The email is
  the human-readable display value; suite Decision 5 keeps it and adds the id.
  Callers that showed the email keep working; the id is the new stable handle.
- **Keying any verb on `id.OwnerEmail` "as a fallback".** Re-anchors access
  control to the mutable field; a single same-email/different-id prober would
  cross owners. `owner_id` alone gates.

## Verification

(Real substrate: the assembled `appkit/mcp` handler built by `NewHandler` over a
real `prompt.Service` + temp SQLite, driven via `net/http/httptest` through the
`server.New`/`RequireIdentity` seam; the authenticated caller is supplied as
`X-Owner-Id` plus `X-Owner-Email`.)

- R-EBD6-OE1O — Two callers presenting the **same `X-Owner-Email` but different
  `X-Owner-Id`** are distinct owners at the tool layer: each caller's `list` (and
  `run_list`) shows only its own prompts/runs, and a `get`/`delete`/`run_cancel`
  of the other caller's prompt or run id returns the `not_found` error envelope
  and mutates nothing — an implementation still keying any tool path on the email
  admits the cross-access and fails.
- R-ECL3-25SD — `create` (and `import`) under `X-Owner-Id: idA` /
  `X-Owner-Email: a@x` persists the prompt with `owner_id == idA` and
  `owner_email == a@x`, and the prompt is thereafter visible to `idA` and not to
  a caller presenting a different `X-Owner-Id` with the same email — ownership
  came from the id header, the email was snapshotted.
- R-EDSZ-FXJ2 — the `get`/`list` prompt result and the `run_get`/`run_list` run
  result each carry both an `owner_id` (equal to the stored row's `owner_id`) and
  an `owner_email` field, and `tools/list` advertises `owner_id` in the
  `outputSchema` of each of those owner-exposing tools beside `owner_email`.
