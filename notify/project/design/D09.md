# Decision 9 — Adopt `registry`: resolve notify's own loopback port by name at the composition root

**Decision.** notify writes no loopback port literals in its Go source: it
references itself **by name** through the shared `registry` library
(`module registry`, the repo-root leaf module). Resolution happens **once, at
the composition root** (`cmd/notify/main.go`) — never per request — because the
table is compile-time data: if a port ever changes we rebuild the binaries
anyway. This is **behavior-preserving**: `registry`'s table carries notify's
current value (`notify=3201`), so the resolved value is byte-identical to the
literal it replaced.

**Why.** "Which port a service answers on" used to be duplicated across the
suite and drift silently — a renumber only surfaced at deploy, uncatchable by
the compiler. Making `registry` the single source of truth means notify asks
the table for the address and never repeats the number.

**The call site (in `cmd/notify/main.go`).** The appkit `Spec.Port` value is a
name lookup, the strict composition-root form that panics on an unknown name (a
programming error):

```go
appkit.Main(appkit.Spec{
    App:   "notify",
    Mount: "/srv/notify/",
    Port:  registry.MustPort("notify"),
    ...
})
```

**Peer feed addresses are not notify's concern.** notify's upstream feed URLs
(crm, prompts) are resolved by the **chassis** from the same `registry` table —
appkit's `Spec.Consumers` seam owns the `<APP>_<SRC>_FEED_URL` env override and
the `registry.BaseURL(src)+"/feed"` default (appkit design D10, consumed here
as a fixed external contract via D11). No notify code resolves, defaults, or
overrides a peer address.

**Module wiring (inside `notify/` only).** `notify/go.mod` carries
`require registry v0.0.0` and the committed `replace registry => ../registry`,
mirroring the `appkit` and `eventplane` in-repo replace-siblings.

> **External precondition (OUT OF SCOPE — not built by any notify phase).** The
> repo-root `go.work` must carry `use ./registry`, and the `registry` module
> itself must exist and be green. Both are owned at the repo root / by the
> `registry` project and are assumed satisfied. No notify Decision or phase edits
> `../go.work`, `../registry/`, `../bin/`, nginx, or any sibling module — the
> executor runs from `notify/` and cannot reach outside it.

**Resolve-at-startup, not per-request.** `Spec.Port` is read once by appkit at
boot. No handler or request path calls `registry`. This matches D3's contract
(the table is compile-time data) and adds no runtime cost.

**Rejected.**
- **Per-request resolution / a `registry` lookup in a handler.** The table is
  compile-time constant; looking it up per request buys nothing. Resolve at the
  composition root, store the value, reuse it.
- **`registry.Port("notify")` (the checked form) at the composition root.** An
  unknown own-name is a build-time programming error, not a recoverable runtime
  condition — `MustPort`'s panic is the correct fail-loud form here (D3).
- **notify-side peer-feed resolution (the pre-D11 form).** Kept the
  `CRM_FEED_URL`/`PROMPTS_FEED_URL` defaults in notify's composition root; the
  chassis now owns that resolution uniformly for every consumer, and notify
  keeping a parallel copy would be exactly the drift the seam removes.

**Verification.**
- R-RGSP-4A1K — **notify's own listen port is `registry.MustPort("notify")`**, not
  a `3201` integer literal, at the composition root. Falsifiable: the served/emitted
  port must equal `registry.MustPort("notify")`; a hardcoded `Port: 3201` in the
  Spec fails the intent (proven concretely by the manifest drift guard R-RGDR,
  which emits with `registry.MustPort("notify")` and matches the committed
  `PORT=3201`).
