# Decision 4 — nginx fragment: session-gated `= /srv/notify/` location and the identity headers every gated location forwards

**Decision.** notify's nginx fragment (`notify/etc/nginx.conf`) session-gates the
human landing page through an **exact-match** location for the bare mount root,
and — across **every** location that forwards owner identity — captures and
forwards the **full owner header set** the dashboard introspection hooks now emit
on each allow (`/_authn` for the bearer tier, `/_session-authn` for the session
tier):

- `X-Owner-Id` — the durable owner key (never empty),
- `X-Owner-Email` — display/audit (never empty),
- `X-Owner-Name` — percent-encoded, MAY be empty,
- `X-Owner-Picture` — percent-encoded, MAY be empty,

plus `X-Client-Id` on the bearer tier as today. Two of notify's locations forward
owner identity and therefore carry this full set; the rest (PRM bootstrap, the
session-gated static path, `@notify_authn_500`) forward **no** identity and are
untouched.

**The exact-match session-gated landing location.** The bare mount root
session-gates the human landing page and proxies it to the service root. It sits
alongside (does not replace) the existing locations, and forwards all four owner
headers captured from the `/_session-authn` subrequest with the fragment's
session-scoped variable convention (`$notify_session_owner*`):

```
# Landing page — human web UI, gated by the dashboard browser SESSION (cookie),
# validated by the dashboard-owned /_session-authn internal location (a sibling of
# /_authn). Exact-match `=` so it never shadows the gated /srv/notify/ prefix below
# or the MCP endpoint. A failed session check yields 401.
location = /srv/notify/ {
    auth_request /_session-authn;
    error_page 401 = @login_bounce;
    auth_request_set $notify_session_owner_id      $upstream_http_x_owner_id;
    auth_request_set $notify_session_owner          $upstream_http_x_owner_email;
    auth_request_set $notify_session_owner_name     $upstream_http_x_owner_name;
    auth_request_set $notify_session_owner_picture  $upstream_http_x_owner_picture;
    proxy_set_header X-Owner-Id      $notify_session_owner_id;
    proxy_set_header X-Owner-Email   $notify_session_owner;
    proxy_set_header X-Owner-Name    $notify_session_owner_name;
    proxy_set_header X-Owner-Picture $notify_session_owner_picture;
    proxy_pass http://127.0.0.1:3201/;   # trailing slash → upstream sees "/"
    proxy_set_header Host $host;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_http_version 1.1;
}
```

**The bearer-gated prefix location.** The pre-existing bearer prefix
(`location /srv/notify/ { auth_request /_authn; … }`) serves `/srv/notify/mcp`
and every other sub-path under the mount. It forwards owner identity too, so it
carries the same four owner headers — captured from the `/_authn` subrequest with
the fragment's bearer-scoped variable convention (`$notify_owner*`) — and keeps
`X-Client-Id` (`$notify_client`) unchanged, alongside its unchanged
`@notify_authn_500` rate-limit re-emit machinery:

```
location /srv/notify/ {
    auth_request /_authn;
    auth_request_set $notify_owner_id       $upstream_http_x_owner_id;
    auth_request_set $notify_owner           $upstream_http_x_owner_email;
    auth_request_set $notify_owner_name      $upstream_http_x_owner_name;
    auth_request_set $notify_owner_picture   $upstream_http_x_owner_picture;
    auth_request_set $notify_client          $upstream_http_x_client_id;
    # … $authn_status / $authn_retry capture and error_page 500 = @notify_authn_500 (unchanged) …
    proxy_set_header X-Owner-Id      $notify_owner_id;
    proxy_set_header X-Owner-Email   $notify_owner;
    proxy_set_header X-Owner-Name    $notify_owner_name;
    proxy_set_header X-Owner-Picture $notify_owner_picture;
    proxy_set_header X-Client-Id     $notify_client;
    proxy_pass http://127.0.0.1:3201/;   # trailing slash strips the /srv/notify/ mount
    proxy_set_header Host $host;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_http_version 1.1;
}
```

**Identity hygiene extends to every new header.** A single `proxy_set_header` per
name, sourced from the auth subrequest variable, REPLACES any client-smuggled
inbound header of that name — nginx does not pass inbound client headers as proxy
headers unless explicitly set, and the last `proxy_set_header` for a name wins.
So each of `X-Owner-Id`/`X-Owner-Name`/`X-Owner-Picture` gets the same one-set
treatment `X-Owner-Email` already had: a client cannot smuggle any of them.

**Unique variable names per service.** All service fragments are `include`d into
the one apex server block, so every `auth_request_set` variable is
service-prefixed to stay unique: the bearer tier uses `$notify_owner_id`,
`$notify_owner_name`, `$notify_owner_picture` (beside the existing
`$notify_owner`/`$notify_client`); the session tier uses
`$notify_session_owner_id`, `$notify_session_owner_name`,
`$notify_session_owner_picture` (beside the existing `$notify_session_owner`).

**How it coexists with the rest of the fragment.** notify's fragment also has:

1. `location = /srv/notify/.well-known/oauth-protected-resource` — unauthenticated
   PRM bootstrap (exact-match), forwards no identity.
2. `location /srv/notify/static/` — the session-gated static path for the landing
   assets (D8), which forwards no identity today and is untouched here.

(notify is an event-plane **consumer**, not a producer, so — unlike crm — its
fragment carries **no** `/feed` location at all.)

nginx longest-prefix/exact-match rules make an **exact-match `=` location win over
a prefix location** for that one path regardless of source order, so the
`= /srv/notify/` exact match wins for the **bare root only**, while the
`/srv/notify/` prefix still serves every other sub-path under the mount. The PRM
exact match is unaffected (a different exact path).

**Why `/_session-authn`, not `/_authn`.** A browser carries the
`dashboard_session` cookie, not a bearer token, so the bearer gate (`/_authn`)
would refuse it. `/_session-authn` is the dashboard-owned, loopback-only internal
location (defined in `dashboard/etc/nginx.conf` as
`proxy_pass …/internal/session-authn`) that validates the cookie against the
web-session store and returns `200` + the owner headers on success or `401` on
failure. This is the **same** mechanism `sites` already uses for its private
static tier. The gate is deliberately **coarse**: any logged-in dashboard user
passes; there is no per-resource check — fine for a name+version page.

**Why the simple form (no 429 re-emit) on the session location.** The bearer
prefix block carries the `$authn_status`/`error_page 500 = @notify_authn_500`
machinery because token introspection can return `429` (rate-limited) which
`auth_request` collapses to a synthetic `500`. The session check is a cookie
lookup, not a rate-budgeted token introspection, so the landing location maps the
session subrequest's allow/deny directly (`200`→serve, `401`→`@login_bounce`),
matching the `sites` private-tier location, which likewise omits it.

**Downstream consumption is out of scope.** This Decision governs only what the
**fragment forwards**. Whether the notify Go binary reads `X-Owner-Id` (a later
suite-wide conversion) is not decided here; the fragment simply makes the header
available on the upstream request.

**Templating.** The loopback port is a **literal** in the fragment (`PORT=3201`,
this service's fixed registry port), so nothing substitutes it at install —
`opsctl setup notify` on the box / `nginx/run` in dev drop the fragment verbatim,
exactly like the existing locations. The fragment is **not** a vhost — no
`server{}`, no `listen`, no cert; it is dropped at
`/etc/nginx/conf.d/locations/notify.conf` and included by the dashboard's apex
server block, which is also where `/_session-authn` is defined.

**Rejected.**
- **Gating the landing page with `/_authn`** — a browser has no bearer token; it
  would always 401. Wrong audience.
- **Making the landing page fully public (no auth_request)** — the brief forbids
  it; a logged-out browser must get gated. The page is gated, just coarsely.
- **A prefix `location /srv/notify/` for the page** — would collide with the
  existing bearer prefix; the exact-match `=` is what cleanly carves the bare root
  out of the prefix without touching the rest.
- **Adding the `@notify_authn_500` 429 re-emit to the session location** — the
  session check is not rate-budgeted; the extra machinery is dead weight (the
  `sites` private tier omits it for the same reason).
- **Forwarding only `X-Owner-Email` and letting the new owner headers die at
  nginx** — they never reach the upstream, so no downstream conversion could ever
  read them. Every gated location forwards the full owner set the hooks emit.
- **Passing the owner headers straight through from the client request instead of
  from the auth subrequest** — that would let a client smuggle identity. Each
  header is set once from its `auth_request_set` variable.
- **Reusing one variable name across the bearer and session tiers (or across
  services)** — fragments share one apex server block; collisions would cross the
  tiers. Names are tier- and service-prefixed.

**Verification.** (Proven by a test that reads `notify/etc/nginx.conf` from disk
and asserts over its content — see design's Testing strategy.)
- R-NGNX-3N6X — `notify/etc/nginx.conf` contains an **exact-match** location for
  the bare mount root: a `location = /srv/notify/ {` block (distinct from the
  prefix `location /srv/notify/ {`).
- R-NGNX-5P8Y — that exact-match block gates with `auth_request /_session-authn`
  (the cookie/session validator) and **not** `auth_request /_authn` (the bearer
  validator).
- R-NGNX-7Q1Z — that block proxies to the loopback upstream **root**
  (`proxy_pass http://127.0.0.1:3201/;` — trailing slash), so the upstream
  sees `/`.
- R-NGNX-9R3B — the pre-existing locations are retained alongside it: the
  bearer-gated `location /srv/notify/ {` prefix with `auth_request /_authn` and
  the `location = /srv/notify/.well-known/oauth-protected-resource` PRM bootstrap
  both still appear in the file.
- R-M9EO-BQGX — the **bearer-gated** `location /srv/notify/` prefix captures all
  four owner headers from the `/_authn` subrequest and forwards each once: it
  contains `auth_request_set $notify_owner_id $upstream_http_x_owner_id;`,
  `auth_request_set $notify_owner_name $upstream_http_x_owner_name;`, and
  `auth_request_set $notify_owner_picture $upstream_http_x_owner_picture;` (beside
  the existing `$notify_owner` ← `x_owner_email`), and a `proxy_set_header` for
  each of `X-Owner-Id $notify_owner_id`, `X-Owner-Email $notify_owner`,
  `X-Owner-Name $notify_owner_name`, and `X-Owner-Picture $notify_owner_picture`,
  while still forwarding `X-Client-Id $notify_client` (a block missing any of the
  three new owner headers, or forwarding an owner header from a source other than
  its `$notify_owner_*` subrequest variable, fails).
- R-MAMK-PI7M — the **session-gated** exact-match `location = /srv/notify/`
  captures all four owner headers from the `/_session-authn` subrequest and
  forwards each once: it contains `auth_request_set $notify_session_owner_id
  $upstream_http_x_owner_id;`, `auth_request_set $notify_session_owner_name
  $upstream_http_x_owner_name;`, and `auth_request_set $notify_session_owner_picture
  $upstream_http_x_owner_picture;` (beside the existing `$notify_session_owner` ←
  `x_owner_email`), and a `proxy_set_header` for each of `X-Owner-Id
  $notify_session_owner_id`, `X-Owner-Email $notify_session_owner`, `X-Owner-Name
  $notify_session_owner_name`, and `X-Owner-Picture $notify_session_owner_picture`
  (a block missing any of the three new owner headers, or forwarding an owner
  header from a source other than its `$notify_session_owner_*` subrequest
  variable, fails).
