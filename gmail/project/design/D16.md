# Decision 16 — Attachment content endpoint: loopback `GET /attachment` (content plane holder)

**Problem.** gmail can enumerate a message's attachments (filename, size, mime
type — D10's `read`/`thread` render) but must also serve their bytes: an
emailed PDF is otherwise unreachable by any consumer. The suite's content plane
(`docs/content-plane-design.md`) makes gmail a **holder**: a service that owns
bytes exposes a loopback-private `GET` content endpoint, and everything
agent-facing carries only references. This Decision is gmail's holder half; the
references stamped into `read`/`thread` results are D17.

The addressing constraint is external and hard (see
`project/research/research.md`): the Gmail API's `attachmentId` **rotates on
every `messages.get`** — it is a per-response retrieval token, not a durable
handle — while a message's MIME **`partId` is immutable** for the life of the
message. A reference must therefore carry only immutable addressing
(`message_id` + `part_id`), and the retrieval token must be minted and spent
inside the handler's own request.

**Decision.** gmail serves attachment bytes from a loopback-private
`GET /attachment` endpoint addressed by **`message_id` + `part_id`** (the
stable MIME tree position, `MessagePart.PartID`). The service is a connector:
it fetches from Google **per request** and keeps **no local blob cache** —
stateless, exactly like every other gmail read path.

1. **The client method is unchanged** (`internal/gmail/client.go`):

   ```go
   // AttachmentGet fetches one attachment body via
   // GET /messages/{messageID}/attachments/{attachmentID} and returns the
   // DECODED bytes (Gmail returns {attachmentId, size, data} with data in
   // URL-safe base64; decode tolerates absent padding — try padded
   // base64.URLEncoding, fall back to RawURLEncoding). Empty ids →
   // ErrValidation; a Google 404 → ErrNotFound.
   func (c *Client) AttachmentGet(ctx context.Context, messageID, attachmentID string) ([]byte, error)
   ```

   The `attachmentID` it takes is always the **fresh token** the handler just
   read out of its own `MessageGet` response — never a value from the request
   URL.

2. **The handler lives in the domain package** (`internal/gmail/attachment.go`,
   mirroring dropbox's `internal/dropbox/content.go`), behind a narrow seam so
   handler tests inject a fake and never touch the network:

   ```go
   // AttachmentSource is the slice of the client the handler drives.
   type AttachmentSource interface {
       MessageGet(ctx context.Context, id, format string) (Message, error)
       AttachmentGet(ctx context.Context, messageID, attachmentID string) ([]byte, error)
   }

   // AttachmentHandler returns the loopback GET /attachment handler.
   func AttachmentHandler(src AttachmentSource) http.Handler
   ```

   Request flow:
   - **Primary guard first** (copied from dropbox's `/content` and `/feed`):
     if the request carries an nginx-injected identity header —
     `X-Owner-Email` **or** `X-Forwarded-Proto` — respond **404**. The
     endpoint is loopback-private by handler guard, never by nginx.
   - Read `message_id` and `part_id` from the query (net/http
     percent-decodes them). Either missing/blank → **404**. There is **no**
     `attachment_id` parameter and no legacy acceptance of one: a URL built
     to the pre-part_id shape simply lacks `part_id` and 404s. (Nothing
     consumed the old shape and no live data exists; no compatibility path.)
   - `MessageGet(message_id, "full")`, then walk the MIME part tree
     (`MessagePart.Parts`, recursive) for the part whose `PartID` equals
     `part_id`. `ErrNotFound` from the client, no matching part, or a
     matching part whose `Body.AttachmentID` is empty (an inline part — no
     fetchable body at Google) → **404**. The walk yields the part's
     `MimeType` (the Content-Type source) **and its current
     `Body.AttachmentID`** — the retrieval token, minted by this very
     response, that makes the fetch resolvable despite rotation.
   - `AttachmentGet(message_id, <that fresh token>)` → the decoded bytes;
     write them with `Content-Type` = the matched part's mime type
     (`application/octet-stream` when the part carries none) and
     `Content-Length` = the decoded byte count. The body is buffered (Gmail
     returns the whole base64 body in one response; there is no upstream
     stream to relay), which is fine at Gmail's ~25 MB message ceiling.
   - Any **other** upstream error (Google 5xx, network fault, token refresh
     failure) → **502** — a connector's transient upstream fault is a gateway
     condition, distinguishable from not-found so a consumer can retry.
     `ErrInvalidGrant` also surfaces as 502 (the poll daemon already fails
     loudly on it; the endpoint must not mask it as absence).

3. **Mounting** (composition root, `cmd/gmail/main.go` `Handlers`): registered
   verbatim, **not** behind `rt.RequireIdentity` — the handler is the guard,
   exactly like dropbox's `/content` routes:

   ```go
   rt.Handle("GET /attachment", gm.AttachmentHandler(client))
   ```

4. **nginx defence-in-depth** (`gmail/etc/nginx.conf`): an exact-match
   `location = /srv/gmail/attachment { return 404; }` beside the existing
   `= /srv/gmail/feed` 404 — belt to the handler's braces, optional per the
   content-plane doc and never the mechanism.

**Addressing honesty.** Both components of the reference are immutable for the
life of the message: the message id, and the MIME tree position `part_id`. The
only ephemeral value — Google's `attachmentId` token — never leaves the
handler's own request. A reference therefore resolves for as long as the
message exists; the only 404 a once-valid reference can earn is the message
itself being deleted. There is no rev/pin dimension (see D17's deviation
note): Gmail attachments are immutable per message, so "the bytes moved on"
cannot happen.

**Rejected.**
- **Addressing by `attachment_id` (the original shape of this Decision)** —
  falsified live: the id rotates per `messages.get`, so the handler's own
  refetch never sees the URL's id and every reference 404s, including a
  seconds-old one. This is the bug this rewrite removes.
- **Accepting both `part_id` and legacy `attachment_id` URLs** — a
  compatibility path for a shape nothing ever successfully consumed; the
  legacy id also cannot be resolved correctly even when accepted.
- **Filename as the part locator** — not unique within a message;
  `partId` is (research doc).
- **A local blob cache / mirror of fetched attachments** — gmail is a
  connector; Google is the source of truth and every other gmail read is
  per-request. A cache adds state, retention, and invalidation for no
  identified consumer need.
- **Content-Type from a `mime_type` query parameter stamped into the URL** —
  makes the served type caller-asserted; the part walk provides both the
  correct type and the existence check, and is needed anyway to mint the
  fresh retrieval token.
- **Uniform 404 for every failure (dropbox's mapping)** — gmail's upstream
  can fail transiently; collapsing that to 404 would tell consumers a real
  attachment does not exist. 502 keeps absence and fault distinguishable.
- **Gating the route behind `RequireIdentity`** — backwards: the route must be
  reachable *only* by bare loopback callers; identity headers are precisely
  the reject signal.
- **Streaming Google's base64 through a decoder without buffering** — possible
  but pointless: Content-Length would be unknowable and the payload ceiling is
  small; buffering keeps the handler simple and the Content-Length exact.

**Verification** (client claims on the client's existing offline substrate — a
stub `http.Client` RoundTripper / redirected host vars, no network; handler
claims against the **real** `AttachmentHandler` over `httptest` with a fake
`AttachmentSource`). The end-to-end proof against the real Gmail API is D19's
live check; the ids here pin the mechanism offline, including the rotation
behavior a static fake cannot expose.

- R-WVZH-M0IY — `Client.AttachmentGet` issues
  `GET /messages/{id}/attachments/{aid}` (both path segments URL-escaped) with
  the bearer, decodes the `{size, data}` response's URL-safe base64 `data`
  (padded and unpadded both decode) and returns exactly the original bytes; a
  Google 404 maps to `ErrNotFound`; an empty `messageID` or `attachmentID`
  returns `ErrValidation` without any HTTP call.
- R-WX7D-ZS9N — the identity-header guard discriminates: a request **with**
  `X-Owner-Email: a@b.c` gets 404, a request **with** `X-Forwarded-Proto:
  https` gets 404, and the byte-identical request **without** either header
  gets 200 with the attachment bytes — proving the endpoint is loopback-private
  by handler guard, not by accident.
- R-3G57-009Q — resolution survives token rotation: against a fake whose
  `MessageGet` returns a **different** `Body.AttachmentID` for the target part
  on every call (never the value any earlier call returned), a
  `GET /attachment?message_id=…&part_id=…` responds 200 with the body
  byte-equal to the attachment bytes, `Content-Type` equal to the matched
  part's `mimeType`, and `Content-Length` equal to the decoded byte count —
  and the fake records that `AttachmentGet` was called with exactly the token
  its own `MessageGet` response just minted (proving the handler resolves the
  token fresh and an implementation keyed on a request-supplied or stored
  token fails).
- R-3HD3-DS0F — a request missing `message_id`, missing `part_id`, or with
  either blank gets 404 without any call on the `AttachmentSource` (the fake
  records zero calls); a legacy-shaped URL carrying only
  `message_id` + `attachment_id` is among them (no `part_id` → 404, zero
  calls).
- R-3IKZ-RJR4 — unknown-resource mapping: `MessageGet` returning `ErrNotFound`
  → 404; a message fetched successfully but containing **no** part whose
  `PartID` equals the requested `part_id` → 404; a matching part whose
  `Body.AttachmentID` is empty (inline part) → 404 with **no**
  `AttachmentGet` call; a non-not-found upstream error (e.g. a wrapped 503) →
  **502**, never 200 and never 404 — absence and upstream fault stay
  distinguishable.
- R-X22Z-IV8F — `gmail/etc/nginx.conf` contains the exact-match
  `location = /srv/gmail/attachment { return 404; }` while the pre-existing
  locations remain (the PRM well-known, the `= /srv/gmail/feed` 404, the
  session-gated `= /srv/gmail/` root, the session-gated `/srv/gmail/static/`,
  and the bearer-gated `/srv/gmail/` prefix) — proven by the existing
  content-assertion pattern over the shipped fragment (D4's substrate).
