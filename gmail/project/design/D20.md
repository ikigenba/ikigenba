# Decision 20 — Structured MCP adoption: `StructuredResult`, output schemas, typed error codes, and the shared loopback guard

**Problem.** gmail's ten mailbox tools were built when an MCP result was
agent-shaped: every domain success returned `appkitmcp.JSONResult(v)` (a Go
value marshalled to a JSON string inside a single text block) and every failure
returned `appkitmcp.ErrorResult(msg)` — a bare human string with no machine
code. The attachment endpoint guarded itself with a hand-copied
`X-Owner-Email` **or** `X-Forwarded-Proto` predicate
(`internal/gmail/attachment.go`).

The suite has since settled the **structured MCP** contract
(`docs/structured-mcp-design.md`): MCP is the single verb surface for agents
**and** machines, so every domain verb's result must carry `structuredContent`
(a plain JSON object a deterministic caller consumes without parsing prose)
mirrored by a text block (the agent rendering), each tool must declare an
`outputSchema`, tool errors must carry a code from the suite's closed
vocabulary, and the loopback guard must key on `X-Forwarded-Proto` **only**
(caller-asserted identity headers are ignored). appkit shipped that contract
(protocol `2025-06-18`, `Tool.OutputSchema`, `StructuredResult`,
`ErrorResult(code, msg)`, `server.LoopbackOnly` / `Router.HandleLoopback`);
`JSONResult` is deleted and `ErrorResult` is re-signed, so gmail no longer
compiles until it adopts. This Decision is gmail's adoption. It **formalizes the
output gmail already emits** — no result field is added, removed, renamed, or
reshaped — and swaps the hand-copied attachment guard for the shared chassis
guard.

**Decision.** Four coordinated, behavior-preserving changes in `internal/mcp`
(results, schemas, error codes) and one in `internal/gmail` + `cmd/gmail`
(guard).

### 1. `JSONResult` → `StructuredResult` on every domain success

Every `return appkitmcp.JSONResult(v)` in `internal/mcp/tools.go` becomes
`return appkitmcp.StructuredResult(v)`. The value `v` is **unchanged** — the same
`map[string]any` the tool built before. `StructuredResult` returns
`(map[string]any, error)` exactly as `JSONResult` did, so each call site keeps
its two-value `return`; the marshal error it now surfaces propagates up the
handler's existing `error` return (never swallowed, never mapped to a tool
error). After this change **no `JSONResult` token remains** in gmail's
non-test Go source.

### 2. Declare an `outputSchema` per domain tool

Each of the ten `appkitmcp.Tool` values in `Tools()` gains an `OutputSchema`
field: a hand-authored `map[string]any` JSON-schema literal that mirrors the
JSON the tool already emits, verbatim. The literals are built with the existing
`obj(props, required...)` helper (house style, mirroring appkit's
`objectSchema`), plus one small private helper `messageSchema()` for the
message-render object that `read` and `thread` share so the two never drift.
`headers` is an open object (`additionalProperties: true`); an attachment's
`content_url` is present only for attachment-backed parts, so it is a property
but **not** required. The mirrored shapes (types straight off the emitted
structs — `internal/gmail/types.go`):

- **`list`** — `{messages: [{id:string, thread_id:string}], next_page_token:string,
  result_size_estimate:integer}`; required `messages`, `next_page_token`,
  `result_size_estimate`.
- **`read`** — `messageSchema()`: `{id:string, thread_id:string,
  label_ids:[string], snippet:string, internal_date:string,
  headers:object(additionalProperties:true), attachments:[{filename:string,
  size:integer, mime_type:string, content_url:string}]}`; the attachment object
  requires `filename`, `size`, `mime_type` (not `content_url`); the message
  object requires `id`, `thread_id`, `label_ids`, `snippet`, `internal_date`,
  `headers`, `attachments`.
- **`thread`** — `{id:string, snippet:string, messages:[messageSchema()]}`;
  required `id`, `snippet`, `messages`.
- **`labels`** — `{labels:[{id:string, name:string, type:string}]}`; required
  `labels`.
- **`send`** — `{id:string, thread_id:string, label_ids:[string]}`; required all
  three.
- **`draft`** — `{id:string, message:{id:string, thread_id:string}}`; required
  `id`, `message`.
- **`label`**, **`unlabel`**, **`trash`** — `{id:string, label_ids:[string]}`;
  required `id`, `label_ids`.
- **`delete`** — `{id:string, deleted:boolean}`; required `id`, `deleted`.

All ten domain tools declare a schema; gmail has **no prose-exception tools**
(no `guide`/`describe`, and no raw-content **MCP** tool — attachment bytes are
served by the loopback HTTP endpoint, not an MCP tool). The chassis `health`
and `reflection` declare their own schemas centrally (appkit); gmail does
nothing for them.

### 3. Typed error codes on every `ErrorResult` call site

`appkitmcp.ErrorResult(code, msg)` now takes a code from the closed vocabulary.
The mapping, named exhaustively:

- **Argument-validation misses** — the tool's own pre-call guards
  (`id is required`, `label_id is required`, `to is required`,
  `subject is required`, and a malformed-JSON `decodeArgs` failure surfaced as a
  tool error) → `appkitmcp.ErrValidation`.
- **Client/upstream errors** — the `err` returned by any `Client` method is
  mapped by a single private helper `errorResultFor(err)`:
  - `errors.Is(err, gm.ErrValidation)` → `ErrValidation`,
  - `errors.Is(err, gm.ErrNotFound)` → `ErrNotFound`,
  - **default** (covers `gm.ErrInvalidGrant` — a revoked refresh token makes the
    mailbox unreachable — and any other Gmail HTTP/network/transport fault) →
    `ErrSourceUnavailable`.

  ```go
  func errorResultFor(err error) map[string]any {
      switch {
      case errors.Is(err, gm.ErrValidation):
          return appkitmcp.ErrorResult(appkitmcp.ErrValidation, err.Error())
      case errors.Is(err, gm.ErrNotFound):
          return appkitmcp.ErrorResult(appkitmcp.ErrNotFound, err.Error())
      default:
          return appkitmcp.ErrorResult(appkitmcp.ErrSourceUnavailable, err.Error())
      }
  }
  ```

  Every `return appkitmcp.ErrorResult(err.Error()), nil` in a tool handler
  becomes `return errorResultFor(err), nil`. `ErrConflict`, `ErrTooLarge`, and
  `ErrInternal` are unused by gmail's surface (no create-conflict, no size gate,
  and a marshal failure is an `error` return, not a coded tool error). The human
  `msg` text is unchanged; only the code is added.

### 4. Swap the hand-copied attachment guard for the shared chassis guard

`internal/gmail/attachment.go`'s `AttachmentHandler` **drops** its opening
`if r.Header.Get("X-Owner-Email") != "" || r.Header.Get("X-Forwarded-Proto")
!= ""` block entirely — the handler becomes pure attachment logic (param read,
MIME walk, byte fetch, status mapping), never inspecting request headers. The
loopback privacy moves to the mount: `cmd/gmail/main.go` registers the route
with `rt.HandleLoopback("GET /attachment", gm.AttachmentHandler(client))`
instead of `rt.Handle`, so `server.LoopbackOnly` fronts it and keys on
`X-Forwarded-Proto` **only**. This narrows the guard to its real meaning ("did
this cross nginx?"): a bare loopback caller that asserts `X-Owner-Email` for its
own identity is now **served**, where the old predicate 404'd it. nginx stamps
`X-Forwarded-Proto` on every proxied request, so front-door reachability is
still refused. The defence-in-depth `location = /srv/gmail/attachment { return
404; }` nginx fragment (D16) is unaffected.

**Rejected.**
- **A second, mirrored loopback domain API for machine callers** — the suite's
  settled rejection (`docs/structured-mcp-design.md`): a permanent second
  surface to guard and keep in sync, to save a machine caller one JSON-RPC
  envelope. `structuredContent` on the existing MCP surface is the whole point.
- **Reshaping any result to be "cleaner" while adopting** — out of scope and
  forbidden: this Decision formalizes the *existing* output. A shape change is a
  separate Decision with its own ids and its own behavioral tests.
- **Collapsing all client errors to one code** (e.g. everything `internal`) —
  loses the machine caller's ability to branch: a deleted message
  (`not_found`), a bad argument (`validation`), and a revoked token
  (`source_unavailable`) are operationally different. The three-way split
  mirrors D16's 404/502 distinction on the HTTP holder.
- **Mapping `ErrInvalidGrant` to `internal`** — a revoked token is a
  *source-availability* condition (the mailbox is unreachable until re-consent),
  not a gmail bug; `source_unavailable` tells the caller it may be transient,
  consistent with the endpoint's 502.
- **Keeping the guard inside `AttachmentHandler` (swapping only the predicate to
  `X-Forwarded-Proto`)** — leaves a hand-copied guard to drift; the suite
  contract puts the guard in appkit (`server.LoopbackOnly`) and services mount
  through `rt.HandleLoopback`. Keeping it local would re-diverge on the next
  chassis change.

**Verification** (through the composition-root seam — the `appkit/mcp` handler
built by `NewHandler` over a `fakeClient`, driven with `tools/list` and
`tools/call` over `httptest`, the real transport; the guard claim drives the
route as mounted, `server.LoopbackOnly(gm.AttachmentHandler(fake))` over
`httptest`, the real guard). No live Gmail call is needed — the service's own
MCP/HTTP layer is the real substrate for every claim here.

- R-8K29-UF3R — `tools/list` through the assembled handler surfaces a **non-nil
  `outputSchema` for each of gmail's ten domain tools** (`list`, `read`,
  `thread`, `labels`, `send`, `draft`, `label`, `unlabel`, `trash`, `delete`);
  a table-driven check over the listed tools fails if any domain tool omits its
  schema (gmail declares no prose-exception domain tool, so none is exempt).
- R-8LA6-86UG — every domain tool's **success** result carries a
  `structuredContent` value that is deep-equal to the JSON parsed from the same
  result's mirrored text block (both present in one result) — proving the
  machine rendering is emitted alongside the agent rendering; a handler still
  returning a text-only `JSONResult`-era envelope (no `structuredContent`) fails.
- R-8MI2-LYL5 — an **argument-validation** miss returns a tool-error result
  (`isError` true) whose `structuredContent.code` equals `validation`: `read`
  with no `id`, `label` with no `label_id`, and `send` with no `to` each
  produce code `validation`, and none calls the `Client` (the fake records zero
  calls) — proving the pre-call guard, not an upstream round-trip.
- R-8NPY-ZQBU — a tool whose `Client` call returns `gm.ErrNotFound` (e.g. `read`
  of an id the fake 404s) returns a tool-error result with
  `structuredContent.code` equal to `not_found` — not `validation`, not
  `source_unavailable`.
- R-8OXV-DI2J — a tool whose `Client` call returns a **non-not-found** fault
  returns `structuredContent.code` equal to `source_unavailable`: a
  `gm.ErrInvalidGrant` and a generic transport error each map to
  `source_unavailable`, while a `gm.ErrNotFound` on the same tool still maps to
  `not_found` — proving `errorResultFor`'s default branch is
  `source_unavailable` and does not collapse the not-found case.
- R-8Q5R-R9T8 — the attachment route as mounted
  (`server.LoopbackOnly(gm.AttachmentHandler(fake))`) discriminates on
  `X-Forwarded-Proto` **only**: a `GET /attachment?message_id=…&part_id=…`
  bearing `X-Forwarded-Proto: https` gets a bare 404 and `AttachmentHandler`
  never runs (the fake `AttachmentSource` records zero calls), while the
  byte-identical request **without** `X-Forwarded-Proto` but **with**
  `X-Owner-Email: a@b.c` set is **served 200** with the attachment bytes —
  proving the guard keys on the front-door marker alone and no longer rejects a
  caller-asserted identity header.
- R-8RDO-51JX — **structural:** no `JSONResult` token survives in gmail's
  non-test Go source:
  `grep -rn 'JSONResult' internal cmd --include='*.go' | grep -v _test.go`
  returns empty.
