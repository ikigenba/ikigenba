# Decision 19 — Live attachment round-trip check (`-tags live`, real Gmail API)

**Problem.** D16/D17's original shape shipped fully green on offline fakes and
was broken against the real API on first live use: the fakes returned stable
`attachmentId` values, so the one property that mattered — rotation — was
unfalsifiable on that substrate. The claim "a minted `content_url` resolves to
the attachment's bytes" hinges on Google's real contract (token rotation,
prompt-token acceptance, real MIME `partId` values) and therefore needs a
verification id whose test drives the **real dependency**, per the design
doctrine that a mock-only capability is itself the smell.

**Decision.** One live end-to-end test, self-contained and self-cleaning,
excluded from the normal suite by a build tag.

1. **Location and gating.** `internal/gmail/live_test.go`, first line
   `//go:build live`. The normal suite (`go test ./...`) never compiles it;
   the live check runs explicitly:

   ```sh
   cd gmail && go test -tags live -run TestLiveAttachmentRoundTrip ./internal/gmail/
   ```

   Credentials come from the environment (`GMAIL_CLIENT_ID`,
   `GMAIL_CLIENT_SECRET`, `GMAIL_REFRESH_TOKEN` — the same vars the service
   reads, already wired by `.envrc`). If any are absent the test **fails
   loudly** with a message naming the missing var — it does not skip: whoever
   invokes `-tags live` asked for the live check, and a silent skip would
   report a pass that proved nothing.

2. **Flow — send, mint, resolve, verify, clean up.** Each run creates its own
   fixture and removes it; no standing test email exists to go stale:

   - `GetProfile` → the connector's own `emailAddress`. The test sends **to
     self** and hardcodes no mailbox anywhere (which account the connector is
     authorized as is an ops choice, not a code fact).
   - `MessagesSend` a raw MIME message with a unique subject
     (`live-attachment-<random hex>`) and one small generated attachment
     (fixed random bytes, ~1 KiB, `application/octet-stream`).
   - Poll `MessageGet(sentID, "full")` (the send response carries the id)
     until the message's payload includes the attachment part, bounded by a
     deadline (~60s, short sleeps) — a bounded poll keeps the exit condition
     deterministic: resolve in time or fail.
   - Build the reference exactly as D17 does (the part's `partId` +
     message id), stand up the real `AttachmentHandler` over `httptest` with
     the **real** `Client`, and `GET` the reference.
   - Assert 200 and body **byte-equal** to the attachment bytes that were
     sent. This is the claim no fake can prove: Google accepted a `part_id`
     resolved to a fresh token and returned the original bytes.
   - **Cleanup runs unconditionally**: a `defer` issues
     `MessageDelete(sentID)` (permanent — the consent scope is full-mail, so
     the mailbox is left with no trace, not a Trash entry) the moment the
     send succeeds, so even an assertion failure or poll timeout removes the
     fixture.

3. **Role.** This test is both the phase's acceptance proof and the standing
   post-deploy smoke for the attachment surface: re-running it against a
   freshly deployed box's credentials is the "gmail attachment → bytes" check,
   replacing manual MCP smoke sequences.

**Rejected.**
- **A standing fixture email the test searches for** — deletable/stale-able
  out-of-band; the failure mode is indistinguishable from a real regression.
  Send-per-run makes the fixture's existence part of the test.
- **`t.Skip` when credentials are absent** — a skipped live check looks green
  in the one context where someone explicitly asked for live proof.
- **Cleanup via `MessageTrash`** — leaves the mailbox accumulating trashed
  test messages for 30 days; the full-mail scope permits real deletion.
- **A runbook of manual MCP steps instead of a test** — not mechanically
  checkable, drifts, and cannot be a phase's deterministic exit condition.
- **Running the live check inside the normal suite** — the suite must stay
  green offline (build loops, CI, no-credential checkouts); live proof is an
  explicit, operator-invoked act.

**Verification** (substrate: the **real Gmail API** via the real `Client` and
real `AttachmentHandler`; the observable outcome is a completed live
round-trip, not a configured value).

- R-3NGL-AMPW — `go test -tags live -run TestLiveAttachmentRoundTrip
  ./internal/gmail/` with real credentials in the environment exits 0, having
  observably: sent a self-addressed message with a generated attachment,
  polled it visible, resolved a D17-shaped `content_url`
  (`message_id` + `part_id`, no attachment token) through the real handler
  against live Google, received the byte-identical attachment body, and
  deleted the message it created (a subsequent `MessageGet` of the fixture id
  returns `ErrNotFound`); with any credential var absent it fails naming the
  variable; without `-tags live` the file does not compile into the suite.
