# Decision 4 — nginx fragment: the session-gated `= /srv/gmail/` landing location and the fragment's four-header identity-forwarding contract

**Decision.** gmail's nginx fragment (`gmail/etc/nginx.conf`) session-gates the
human landing page with an **exact-match** location for the bare mount root, and
— across **every location that forwards identity today** — captures and forwards
the dashboard's **four owner headers** (`X-Owner-Id`, `X-Owner-Email`,
`X-Owner-Name`, `X-Owner-Picture`), not `X-Owner-Email` alone.

## The exact-match session-gated location

The landing location sits alongside (does not replace) the existing locations. It
gates on the browser session cookie and proxies the page to the service root:

```
# Landing page — human web UI, gated by the dashboard browser SESSION (cookie),
# validated by the dashboard-owned /_session-authn internal location (a sibling of
# /_authn). Exact-match `=` so it never shadows the gated `/srv/gmail/` prefix below
# or the MCP endpoint. A failed session check yields 401 (bounced to sign-in by the
# D15 `error_page 401 = @login_bounce;` opt-in).
location = /srv/gmail/ {
    auth_request /_session-authn;
    error_page 401 = @login_bounce;                       # D15
    auth_request_set $gmail_session_owner         $upstream_http_x_owner_email;
    auth_request_set $gmail_session_owner_id      $upstream_http_x_owner_id;
    auth_request_set $gmail_session_owner_name    $upstream_http_x_owner_name;
    auth_request_set $gmail_session_owner_picture $upstream_http_x_owner_picture;
    proxy_set_header X-Owner-Email   $gmail_session_owner;
    proxy_set_header X-Owner-Id      $gmail_session_owner_id;
    proxy_set_header X-Owner-Name    $gmail_session_owner_name;
    proxy_set_header X-Owner-Picture $gmail_session_owner_picture;
    proxy_pass http://127.0.0.1:3202/;   # trailing slash → upstream sees "/"
    proxy_set_header Host $host;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_http_version 1.1;
}
```

nginx longest-prefix/exact-match rules make an **exact-match `=` location win
over a prefix location** for that one path regardless of source order. So this
`= /srv/gmail/` exact match wins for the **bare root only**, while the
`/srv/gmail/` prefix still serves `/srv/gmail/mcp` and every other sub-path under
the mount. The two PRM/feed exact matches are unaffected (different exact paths).

**Why `/_session-authn`, not `/_authn`.** A browser carries the
`dashboard_session` cookie, not a bearer token, so the bearer gate (`/_authn`)
would refuse it. `/_session-authn` is the dashboard-owned, loopback-only internal
location (defined in `dashboard/etc/nginx.conf` as
`proxy_pass …/internal/session-authn`) that validates the cookie against the
web-session store and returns `200` + the owner headers on success or `401` on
failure. This is the **same** mechanism `sites` already uses for its private
static tier. The gate is deliberately **coarse**: any logged-in dashboard user
passes; there is no per-resource check — fine for a name+version page.

**Why the simple form (no 429 re-emit).** The bearer prefix block carries the
`$authn_status`/`error_page 500 = @gmail_authn_500` machinery because token
introspection can return `429` (rate-limited) which `auth_request` collapses to a
synthetic `500`. The session check is a cookie lookup, not a rate-budgeted token
introspection, so the landing location maps the session subrequest's allow/deny
directly (`200`→serve, `401`→bounce/401), matching the `sites` private tier.

## Four-header identity forwarding (this pass)

The dashboard's introspection hooks (`/_authn` bearer/PAT tier and
`/_session-authn` session tier) now emit **four** owner headers on every allow,
fail-closed: `X-Owner-Id` (durable owner key, never empty), `X-Owner-Email`
(display/audit, never empty), `X-Owner-Name` (percent-encoded, MAY be empty), and
`X-Owner-Picture` (percent-encoded, MAY be empty). Previously the fragment
captured and forwarded only `X-Owner-Email`; the other three died at nginx. This
Decision extends **every location in the fragment that forwards identity** to
capture (`auth_request_set`) and forward (`proxy_set_header`) all four:

- **Session-gated landing** (`location = /srv/gmail/`, above): captures the four
  owner headers into `$gmail_session_owner` (email, existing),
  `$gmail_session_owner_id`, `$gmail_session_owner_name`,
  `$gmail_session_owner_picture`, and forwards each as its `X-Owner-*` header.
- **Bearer-gated prefix** (`location /srv/gmail/`, `auth_request /_authn`):
  captures the four owner headers into `$gmail_owner` (email, existing),
  `$gmail_owner_id`, `$gmail_owner_name`, `$gmail_owner_picture`, and forwards
  each as its `X-Owner-*` header. Its `X-Client-Id` forwarding
  (`auth_request_set $gmail_client …` + `proxy_set_header X-Client-Id
  $gmail_client`) and its `@gmail_authn_500` 429 re-emit machinery are
  **unchanged**:

```
location /srv/gmail/ {
    auth_request /_authn;
    auth_request_set $gmail_owner         $upstream_http_x_owner_email;
    auth_request_set $gmail_owner_id      $upstream_http_x_owner_id;
    auth_request_set $gmail_owner_name    $upstream_http_x_owner_name;
    auth_request_set $gmail_owner_picture $upstream_http_x_owner_picture;
    auth_request_set $gmail_client        $upstream_http_x_client_id;
    … $authn_status / $authn_retry / error_page 500 = @gmail_authn_500 (unchanged) …
    proxy_set_header X-Owner-Email   $gmail_owner;
    proxy_set_header X-Owner-Id      $gmail_owner_id;
    proxy_set_header X-Owner-Name    $gmail_owner_name;
    proxy_set_header X-Owner-Picture $gmail_owner_picture;
    proxy_set_header X-Client-Id     $gmail_client;
    proxy_pass http://127.0.0.1:3202/;   # trailing slash strips the mount
    …
}
```

**Variable naming.** New per-location variables follow the fragment's existing
`$gmail_owner` / `$gmail_session_owner` convention: `$gmail_owner_id`,
`$gmail_owner_name`, `$gmail_owner_picture` (bearer) and
`$gmail_session_owner_id`, `$gmail_session_owner_name`,
`$gmail_session_owner_picture` (session). The names are **service-prefixed**
(`gmail_`) so they stay unique when every service fragment is included into the
one apex server block.

**Identity hygiene extends to the new headers.** As with `X-Owner-Email`, a
single `proxy_set_header X-Owner-Id|X-Owner-Name|X-Owner-Picture` from the auth
subrequest variable **replaces** any client-smuggled inbound header of that name
on the upstream request — nginx does not pass inbound client headers as proxy
headers unless explicitly set, and the last `proxy_set_header` for a name wins.

**Out of scope.** Downstream consumption of `X-Owner-Id` (or the new headers) by
the gmail binary is a later suite-wide conversion — not this pass. This pass is
fragment forwarding plus its content-assertion tests only.

**Locations that forward NO identity are untouched.** The PRM well-known
bootstrap (`location = /srv/gmail/.well-known/oauth-protected-resource`), the
`= /srv/gmail/feed` and `= /srv/gmail/attachment` 404 stubs, the static asset
tier (`location /srv/gmail/static/`, session-gated but forwards no owner header),
and the `@gmail_authn_500` re-emit stay exactly as they are.

**Templating.** The loopback port is a **literal** in the fragment (`PORT=3202`,
this service's fixed registry port); nothing substitutes it at install. The
fragment is **not** a vhost — no `server{}`, no `listen`, no cert; it is dropped
at `/etc/nginx/conf.d/locations/gmail.conf` and included by the dashboard's apex
server block, which is also where `/_session-authn` and `@login_bounce` are
defined.

**Rejected.**
- **Forwarding only `X-Owner-Email` (the status quo)** — the dashboard now emits
  four fail-closed owner headers; dropping three of them at nginx strands the
  durable `X-Owner-Id` key and the display fields the service will need. The
  fragment must forward what the introspection hook resolves.
- **Un-prefixed variable names (`$owner_id`, `$session_owner_id`)** — all service
  fragments are included into one apex server block; an un-prefixed variable would
  collide across fragments. Names stay `$gmail_`-prefixed.
- **Adding the owner headers to the bearer block via a shared `include`** — nginx
  `include` cannot parameterize the per-location variable names, and the fragment
  is shipped verbatim (no templating); the explicit per-location directives are
  the only form that loads directly.
- **Touching `X-Client-Id` or the 429 re-emit** — orthogonal to owner-identity
  forwarding; the bearer client id and rate-limit machinery are retained
  verbatim.
- **Gating the landing page with `/_authn`** — a browser has no bearer token; it
  would always 401. Wrong audience.
- **Making the landing page fully public (no auth_request)** — a logged-out
  browser must get `401` (bounced by D15). The page is gated, just coarsely.

**Verification.** (Proven by the existing nginx content-assertion test
(`cmd/gmail/nginx_test.go`) that reads `gmail/etc/nginx.conf` from disk and
asserts over its content; nginx is not run by the suite.)
- R-NGNX-3B6C — `gmail/etc/nginx.conf` contains an **exact-match** location for the
  bare mount root: a `location = /srv/gmail/ {` block (distinct from the prefix
  `location /srv/gmail/ {`).
- R-NGNX-5D8E — that exact-match block gates with `auth_request /_session-authn`
  (the cookie/session validator) and **not** `auth_request /_authn` (the bearer
  validator).
- R-NGNX-7F1G — that block proxies to the loopback upstream **root**
  (`proxy_pass http://127.0.0.1:3202/;` — trailing slash), so the upstream
  sees `/`.
- R-NGNX-9H3J — the pre-existing locations are retained alongside it: the
  bearer-gated `location /srv/gmail/ {` prefix with `auth_request /_authn`, the
  `location = /srv/gmail/feed { return 404; }`, and the
  `location = /srv/gmail/.well-known/oauth-protected-resource` PRM bootstrap all
  still appear in the file.
- R-MTLH-R7Z9 — the **bearer-gated prefix** `location /srv/gmail/` (the
  `auth_request /_authn` tier) captures **all four** owner headers into
  service-prefixed variables —
  `auth_request_set $gmail_owner $upstream_http_x_owner_email`,
  `auth_request_set $gmail_owner_id $upstream_http_x_owner_id`,
  `auth_request_set $gmail_owner_name $upstream_http_x_owner_name`,
  `auth_request_set $gmail_owner_picture $upstream_http_x_owner_picture` — and
  forwards each with a matching `proxy_set_header X-Owner-Email $gmail_owner`,
  `proxy_set_header X-Owner-Id $gmail_owner_id`,
  `proxy_set_header X-Owner-Name $gmail_owner_name`,
  `proxy_set_header X-Owner-Picture $gmail_owner_picture`; its existing
  `proxy_set_header X-Client-Id $gmail_client` is still present.
- R-MUTE-4ZPY — the **session-gated exact-match** `location = /srv/gmail/`
  captures **all four** owner headers into service-prefixed variables —
  `auth_request_set $gmail_session_owner $upstream_http_x_owner_email`,
  `auth_request_set $gmail_session_owner_id $upstream_http_x_owner_id`,
  `auth_request_set $gmail_session_owner_name $upstream_http_x_owner_name`,
  `auth_request_set $gmail_session_owner_picture $upstream_http_x_owner_picture`
  — and forwards each with a matching
  `proxy_set_header X-Owner-Email $gmail_session_owner`,
  `proxy_set_header X-Owner-Id $gmail_session_owner_id`,
  `proxy_set_header X-Owner-Name $gmail_session_owner_name`,
  `proxy_set_header X-Owner-Picture $gmail_session_owner_picture`.
