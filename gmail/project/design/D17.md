# Decision 17 — Attachment references in `read`/`thread` results (content plane references)

**Problem.** With D16's holder endpoint in place, the agent-facing surface must
carry the reference: the content plane's rule is *agents pass references;
services move bytes*. The reference must be **durable** — and per the external
constraint D16 records (see `project/research/research.md`), the Gmail
`attachmentId` is a per-response token that rotates on every fetch, so it must
not appear in the reference at all. The message id and the MIME `partId` are
the immutable pair.

**Decision.** The `read` and `thread` tools' attachment entries gain a
complete, URL-encoded `content_url` addressed by `message_id` + `part_id`.
Metadata fields (`filename`, `size`, `mime_type`) are unchanged. The entries
carry **no `attachment_id` field**: the token is ephemeral, useless to every
consumer, and publishing it invites exactly the stored-token bug D16's rewrite
removes.

1. **The reference shape.** For a MIME part with a filename, a non-empty
   `PartID`, **and** a non-empty `Body.AttachmentID`, the rendered entry
   becomes:

   ```json
   {
     "filename":    "2026-06-inv.pdf",
     "size":        184230,
     "mime_type":   "application/pdf",
     "content_url": "http://127.0.0.1:3202/attachment?message_id=18f2a1b3c4d5e6f7&part_id=2"
   }
   ```

   `content_url` is complete and URL-encoded (built with `url.Values.Encode`
   over `message_id` + `part_id`); the consumer never constructs or rewrites
   it. A filename-bearing part **without** an attachment id (an inline part
   whose body data rides in the message itself) or **without** a part id
   stays a metadata-only entry — no `content_url`: there is nothing at the
   endpoint for it to serve.

2. **The base URL is resolved by registry name at the composition root**
   (dropbox's exact pattern; gmail's own loopback guard test,
   `cmd/gmail/loopback_guard_test.go`, forbids any `127.0.0.1:3xxx` literal in
   Go source — **including test expectations**, which must derive the port from
   `registry.MustPort("gmail")`):

   ```go
   // cmd/gmail/main.go, in Handlers:
   port, err := config.EnvOrInt(os.Getenv, "GMAIL_PORT", registry.MustPort("gmail"))
   ip   := config.EnvOr(os.Getenv, "GMAIL_IP", "127.0.0.1")
   contentBase := "http://" + ip + ":" + strconv.Itoa(port)
   ```

3. **The seam threads through the tool table** (as already realized —
   D10's signatures):

   ```go
   func Tools(client Client, contentBase string) []appkitmcp.Tool
   func NewHandler(client Client, contentBase string, rt *appkit.Router) (http.Handler, error)
   ```

   `renderMessage`/`collectAttachments` take the base and stamp the reference
   per attachment part, carrying the part's `PartID` into the query. An empty
   `contentBase` is a wiring error and panics at the `NewHandler` seam, like a
   nil client.

4. **Tool descriptions state the current truth.** `read` and `thread` say
   attachment entries carry a loopback `content_url` for byte fetch by
   services (not by the agent), addressed durably (no ephemeral ids). No
   other descriptor or schema changes; the surface stays exactly twelve tools
   (D10's R-9NYN-SVIR).

**Documented deviation-by-omission from the full content-plane reference
tuple.** The plane's reference is `{content_url, content_hash, size, rev?}`.
gmail's reference is **`{content_url, size, mime_type}`** (plus `filename` as
metadata):

- **No `content_hash`** — Google exposes no content hash for an attachment
  without fetching the bytes; stamping one would force a full blob download
  per `read`. Consumers that need integrity verify what they fetched by size
  or hash it themselves post-fetch.
- **No `rev`, no pin dimension** — Gmail attachments are immutable per
  message; the bytes a reference names can never "move on," so the `rev`→409
  pin contract has nothing to pin. A reference resolves for as long as its
  message exists.

This is an honest narrowing, recorded here as gmail's conformance statement to
`docs/content-plane-design.md` (whose conformance map row for gmail this
Decision and D16 realize).

**Rejected.**
- **Carrying `attachment_id` in the entry (the original shape of this
  Decision)** — the id rotates per fetch (falsified live; research doc), so
  it can never be used for anything except immediately misleading a consumer;
  as reference addressing it 404s by construction.
- **Keeping `attachment_id` as informational metadata** — a standing trap
  with zero legitimate use; the content plane's consumers never construct
  URLs, so nothing needs the raw handle.
- **Stamping `content_hash` by prefetching each attachment during `read`** —
  turns every message read into N blob downloads; the tool is a metadata read.
- **Returning attachment bytes inline (`content_base64`) as dropbox's `get`
  does** — exactly the by-value crossing the content plane exists to retire;
  bytes never enter model context.
- **A relative or template `content_url` the consumer completes** — the plane
  requires the URL appear complete and encoded; consumers never construct.
- **Emitting `content_url` with empty values for inline or part-id-less
  parts** — noise that invites consumers to fetch a URL that must 404;
  absence states the truth.

**Verification** (through the assembled `appkit/mcp` handler over the existing
`fakeClient` — the D10 substrate; expected URLs built from
`registry.MustPort("gmail")`, never a port literal).

- R-3JSW-5BHT — a `read` of a message whose MIME tree carries an attachment
  part (filename, size, mime type, non-empty `PartID`, non-empty
  `Body.AttachmentID`) renders that entry with `filename`/`size`/`mime_type`
  unchanged **plus** `content_url` equal to
  `http://127.0.0.1:<registry gmail port>/attachment?<encoded query>` whose
  parsed query round-trips exactly the original `message_id` and the part's
  `partId` — and the entry contains **no** `attachment_id` key and no
  occurrence of the fake's `Body.AttachmentID` value anywhere in the rendered
  result (the ephemeral token does not leak).
- R-3L0S-J38I — `thread` renders the same reference shape: each message's
  attachment entries carry a `content_url` addressed by **that message's** id
  (two messages in one thread yield two distinct `message_id` values in their
  URLs), each with its own part's `partId`, and no `attachment_id` key
  anywhere.
- R-3M8O-WUZ7 — a filename-bearing part whose `Body.AttachmentID` is empty
  (inline data), and a filename-bearing part whose `PartID` is empty, each
  render metadata only: the entry has `filename`/`size`/`mime_type` and
  **no** `content_url` or `attachment_id` key.
