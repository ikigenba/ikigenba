# Decision 15 — Data model: a `public` boolean and owner id/email, retiring the publish lifecycle

**Decision.** The `sites` row drops the publish-lifecycle state and becomes a
plain record of *what exists, at what visibility, owned by whom*:

| column | type | meaning |
|---|---|---|
| `name` | TEXT PK | slug (unchanged grammar `^[a-z0-9][a-z0-9-]{0,62}$`, reserved-name guard) |
| `public` | INTEGER NOT NULL DEFAULT 0 | visibility boolean: `1` = public, `0` = private |
| `owner_id` | TEXT NOT NULL | stable owner key (the nginx-forwarded `X-Owner-Id`, appkit `Identity.OwnerID`); captured at create. Sites does not owner-scope any query (the slug `name` is the global handle), so no logic reads it today — it is stored so the suite keys on the stable id and any future owner grouping has it |
| `owner_email` | TEXT NOT NULL | owner email that created the site — a **write-once snapshot** captured from the request headers at row creation, never updated afterward, never read for logic. Display only. Replaces the retired `created_by` |
| `source_path` | TEXT | dropbox-sync provenance (nullable; **unchanged**) |
| `created_at` | TEXT NOT NULL | UTC creation timestamp |
| `updated_at` | TEXT NOT NULL | UTC last-update timestamp |

The retired columns — `tier`, `published`, `published_at` — remain removed. A
site's visibility is the single `public` boolean; there is no "unpublished" state.

**Owner keying (suite owner-id conversion, `docs/owner-id-design.md`).** Email is
becoming user-mutable (the dashboard's GitHub alternate login), so it can no
longer be an identity key. `owner_id` is the stable key the suite standardizes on;
`owner_email` is a display snapshot only. Sites happens to owner-scope *nothing*
(its `list` and landing page show every site on the box, and `name` is a global
primary key), so no query, filter, or authorization comparison keys on either
column — but the pair is stored on every row per the suite standard so the
identity is captured at the stable id and the display email travels beside it.
Because a service cannot map a stored email back to an owner id (only the
dashboard holds that mapping), the conversion **rebuilds the table empty**.

**Schema change is one new immutable migration that rebuilds the table** (per
Conventions; created with `bin/create-migration sites <name>`, never hand-named).
The frozen `002_sites.sql`, `20260609135943_add_source_path.sql`,
`20260708010236_add_public_created_by.sql`, and
`20260708012637_drop_publish_lifecycle.sql` are **untouched**. The new migration
**drops and recreates** `sites` with the final shape above — rows are dropped (per
the suite decision; safe during the migration window, sites holds no live customer
data), `created_by` is gone, `owner_id` and `owner_email` are `NOT NULL`, and an
index sits on `owner_id`:

```sql
DROP TABLE sites;
CREATE TABLE sites (
    name        TEXT    PRIMARY KEY,
    public      INTEGER NOT NULL DEFAULT 0,
    owner_id    TEXT    NOT NULL,   -- stable owner key (X-Owner-Id); captured at create
    owner_email TEXT    NOT NULL,   -- write-once display snapshot; replaces created_by
    source_path TEXT,
    created_at  TEXT    NOT NULL,
    updated_at  TEXT    NOT NULL
) STRICT;
CREATE INDEX idx_sites_owner ON sites(owner_id);
```

The rebuild is deterministic on any database (dev, fresh, or the live box) and
does not depend on anyone remembering to wipe `state/`. `STRICT` is preserved so
SQLite enforces the column affinity.

**Store contract** (`internal/sites/store.go`). The `Site` struct and CRUD lose
the `CreatedBy` field and gain the owner pair:

```go
type Site struct {
    Name       string
    Public     bool
    OwnerID    string     // stable owner key, captured at create
    OwnerEmail string     // write-once display snapshot
    SourcePath string     // dropbox-sync provenance (unchanged)
    CreatedAt  time.Time
    UpdatedAt  time.Time
}

// Create inserts a fresh row at the caller-chosen visibility: `public` is stored
// verbatim (there is no store-side default). ownerID is captured from the request
// Identity's stable id; ownerEmail is the write-once display snapshot captured
// from the same Identity. Neither is derived from the other. The MCP `create`
// tool is where "private by default" lives — it passes public=false when its
// argument is omitted (D20). Returns ErrExists on a duplicate slug.
func (s *Store) Create(ctx context.Context, name, ownerID, ownerEmail string, public bool) (Site, error)

// SetVisibility flips a site's public flag (and bumps updated_at); it never
// touches owner_id/owner_email. The on-disk directory move is D16's concern,
// invoked in lockstep. Returns ErrNotFound.
func (s *Store) SetVisibility(ctx context.Context, name string, public bool) error
```

`Get`/`List`/`Delete`/`SetSourcePath` keep their shapes (their column lists change
from `created_by` to `owner_id, owner_email`). `List` stays ordered by `name`.

**Rejected.**
- **Keeping `created_by` (email) as the identity.** Email is becoming mutable; the
  suite keys on the stable `owner_id`. Retained as `owner_email`, a display-only
  snapshot.
- **Omitting `owner_id` because sites scopes nothing on it.** The suite standard is
  that every owner-scoped row stores the stable id and the display email; storing
  only the email would leave sites the one service that cannot join to a stable
  identity, and would strand any future owner-grouping fix with no id to group on.
  The column is cheap and captured for free at create.
- **Migrating existing rows (email → id backfill) instead of dropping them.** Only
  the dashboard knows the email→id mapping; sites cannot backfill. Suite-settled:
  rebuild empty, deterministic on any database.
- **An additive `ALTER TABLE … ADD COLUMN owner_id` + `RENAME COLUMN created_by
  TO owner_email`.** `owner_id` is `NOT NULL` with no honest value for the rows we
  are required to drop anyway; a drop-and-recreate is the one migration that lands
  the exact final shape (and the `NOT NULL` owner columns) on every database, old
  or fresh, without a bogus backfill default.
- **Keeping `tier TEXT` / a `published` boolean.** Already rejected with the
  publish lifecycle: a site's visibility is the single `public` boolean; a site
  that exists is served.

**Verification.**
- R-QSLO-SAIQ — `Store.Create(ctx, name, ownerID, ownerEmail, false)` stores
  `Public == false` and `Store.Create(ctx, name, ownerID, ownerEmail, true)`
  stores `Public == true` (the store persists the requested visibility verbatim,
  with no store-side default), proven against a real migrated SQLite DB.
- R-QTTL-629F — `Store.SetVisibility(ctx, name, true)` then `Get` returns
  `Public == true`, and `SetVisibility(ctx, name, false)` returns it to
  `Public == false`; `updated_at` advances; a missing slug yields `ErrNotFound`.
- R-Z3ZN-5BFE — `Store.Create(ctx, name, ownerID, ownerEmail, public)` persists
  `owner_id == ownerID` and `owner_email == ownerEmail` **exactly as passed** (the
  email is a snapshot, not derived, defaulted, or used as the key); a subsequent
  `Get`/`List` returns a `Site` whose `OwnerID` and `OwnerEmail` equal the values
  passed. Two `Create`s with the **same `ownerEmail` but different `ownerID`**
  persist **distinct `owner_id`** values (the id is captured from the identity, not
  derived from the email) — a build that stored the email into `owner_id`, or
  derived one column from the other, fails. (Proven against a real migrated SQLite
  DB with distinct, unrelated id and email values.)
- R-Z57J-J363 — after the full migration set runs, `pragma table_info(sites)`
  reports columns including `owner_id` and `owner_email`, and **excludes**
  `created_by`, `tier`, `published`, and `published_at` (proven against the real
  SQLite the runner uses — the substrate that actually enforces the column set).
