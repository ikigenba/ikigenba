# Decision 15 — Data model: a three-value `visibility` enum and owner id/email

**Decision.** The `sites` row records *what exists, at which visibility, owned by
whom*. Visibility is a **TEXT enum of exactly three values** — `public`,
`private`, `unlisted` — enforced by SQLite:

| column | type | meaning |
|---|---|---|
| `name` | TEXT PK | slug (grammar `^[a-z0-9][a-z0-9-]{0,62}$`, reserved-name guard). For public/private sites the owner chose it; for unlisted sites it is the generated 30-char token (D27) — the token satisfies the same grammar, so one column and one grammar serve all three visibilities |
| `visibility` | TEXT NOT NULL CHECK (`visibility IN ('public','private','unlisted')`) | the site's visibility, exactly one of the three |
| `owner_id` | TEXT NOT NULL | stable owner key (the nginx-forwarded `X-Owner-Id`, appkit `Identity.OwnerID`); captured at create. Sites does not owner-scope any query (the slug `name` is the global handle), so no logic reads it — it is stored so the suite keys on the stable id and any future owner grouping has it |
| `owner_email` | TEXT NOT NULL | owner email that created the site — a **write-once snapshot** captured from the request headers at row creation, never updated afterward, never read for logic. Display only |
| `source_path` | TEXT | dropbox-sync provenance (nullable; **unchanged**) |
| `created_at` | TEXT NOT NULL | UTC creation timestamp |
| `updated_at` | TEXT NOT NULL | UTC last-update timestamp |

The former `public` INTEGER boolean is **retired**: `1` becomes `'public'`, `0`
becomes `'private'`, and the new `'unlisted'` value joins them as a peer. The
long-retired `tier`/`published`/`published_at` stay gone. There is no lifecycle
flag and no fourth state.

**The Go visibility type** (`internal/sites`). A dedicated string type keeps the
enum closed at the Go layer too:

```go
type Visibility string

const (
    Public   Visibility = "public"
    Private  Visibility = "private"
    Unlisted Visibility = "unlisted"
)

// ParseVisibility maps a wire string to the enum; anything outside the three
// values is ErrInvalidVisibility (a new sentinel beside ErrInvalidSlug etc.).
func ParseVisibility(s string) (Visibility, error)
```

**Schema change is one new immutable migration that rebuilds the table** (per
Conventions; created with `bin/create-migration sites <name>`, never hand-named).
All previously committed migrations stay frozen. The new migration recreates
`sites` with the shape above and **carries the existing rows across**, mapping
the boolean to the enum:

```sql
CREATE TABLE sites_new (
    name        TEXT    PRIMARY KEY,
    visibility  TEXT    NOT NULL CHECK (visibility IN ('public','private','unlisted')),
    owner_id    TEXT    NOT NULL,
    owner_email TEXT    NOT NULL,
    source_path TEXT,
    created_at  TEXT    NOT NULL,
    updated_at  TEXT    NOT NULL
) STRICT;
INSERT INTO sites_new (name, visibility, owner_id, owner_email, source_path, created_at, updated_at)
    SELECT name, CASE public WHEN 1 THEN 'public' ELSE 'private' END,
           owner_id, owner_email, source_path, created_at, updated_at
    FROM sites;
DROP TABLE sites;
ALTER TABLE sites_new RENAME TO sites;
CREATE INDEX idx_sites_owner ON sites(owner_id);
```

`STRICT` is preserved; the `CHECK` makes SQLite itself the enum's enforcer, so a
bad value cannot be persisted even by a buggy caller.

**Store contract** (`internal/sites/store.go`). The `Site` struct carries the
enum in place of the retired boolean, and `SetVisibility` grows an optional
rename — the one store-level primitive the unlisted transition matrix (D20)
needs, executed as a **single UPDATE** so name and visibility can never change
out of lockstep:

```go
type Site struct {
    Name       string
    Visibility Visibility
    OwnerID    string     // stable owner key, captured at create
    OwnerEmail string     // write-once display snapshot
    SourcePath string     // dropbox-sync provenance (unchanged)
    CreatedAt  time.Time
    UpdatedAt  time.Time
}

// Create inserts a fresh row at the caller-stated visibility, stored verbatim
// (there is no store-side default — the MCP layer requires the caller to state
// it, D20). ownerID/ownerEmail are captured from the request Identity as
// before. Returns ErrExists on a duplicate slug.
func (s *Store) Create(ctx context.Context, name, ownerID, ownerEmail string, v Visibility) (Site, error)

// SetVisibility updates a site's visibility and, when newName is non-empty,
// renames the row in the same single UPDATE (the unlisted transitions rename;
// public↔private passes ""). Bumps updated_at; never touches
// owner_id/owner_email/created_at. Returns ErrNotFound for a missing slug and
// ErrExists when newName collides with an existing row.
func (s *Store) SetVisibility(ctx context.Context, name string, v Visibility, newName string) error
```

`Get`/`List`/`Delete`/`SetSourcePath` keep their shapes (their column lists carry
`visibility` instead of `public`). `List` stays ordered by `name`. The on-disk
directory move that accompanies a visibility change is D16's `Layout.Move`,
invoked in lockstep by the MCP handler (D20).

**Rejected.**
- **Keeping the `public` boolean and marking unlisted some other way** (a second
  flag, a nullable token column, a naming convention alone). Two columns that
  must agree is drift waiting to happen; the operator's settled contract is one
  three-value visibility, and one enum column states it exactly.
- **An INTEGER enum (0/1/2).** The TEXT values are self-describing in every
  query, dump, and debug session, match the wire vocabulary of the MCP surface
  verbatim, and cost nothing at this scale. The CHECK closes the set either way.
- **Dropping rows in the rebuild (as the owner-id migration did).** That
  migration had no honest value for its new NOT NULL columns; this one has a
  total, deterministic mapping (`1→public`, `0→private`), so the rows carry
  across. Gratuitous data loss is not simplicity.
- **A separate `rename` store method beside `SetVisibility`.** Every rename in
  the contract happens *because of* a visibility transition (D20); two calls
  would open a gap where name and visibility disagree. One UPDATE, one moment.

**Verification.**
- R-H0LI-XQXI — `Store.Create` persists the passed visibility **verbatim for
  each of the three values**: three creates with `Public`, `Private`, and
  `Unlisted` read back (via `Get` and `List`) as exactly those values, proven
  against a real migrated SQLite DB. An implementation that collapses unlisted
  into public, or defaults any value, fails.
- R-H1TF-BIO7 — `Store.SetVisibility(ctx, name, v, "")` updates the visibility
  and keeps the name; `SetVisibility(ctx, name, v, "newname")` updates **both
  name and visibility** so a subsequent `Get` of the old name is `ErrNotFound`
  and of the new name returns the new visibility; `updated_at` advances in both
  cases; a missing slug yields `ErrNotFound`; a `newName` colliding with an
  existing row yields `ErrExists` and leaves both rows unchanged.
- R-Z3ZN-5BFE — `Store.Create(ctx, name, ownerID, ownerEmail, v)` persists
  `owner_id == ownerID` and `owner_email == ownerEmail` **exactly as passed** (the
  email is a snapshot, not derived, defaulted, or used as the key); a subsequent
  `Get`/`List` returns a `Site` whose `OwnerID` and `OwnerEmail` equal the values
  passed. Two `Create`s with the **same `ownerEmail` but different `ownerID`**
  persist **distinct `owner_id`** values (the id is captured from the identity, not
  derived from the email) — a build that stored the email into `owner_id`, or
  derived one column from the other, fails. (Proven against a real migrated SQLite
  DB with distinct, unrelated id and email values.)
- R-H31B-PAEW — after the full migration set runs, `pragma table_info(sites)`
  reports a `visibility` TEXT column and **no** `public` column (nor
  `created_by`/`tier`/`published`/`published_at`); and a direct INSERT with
  `visibility = 'bogus'` is **rejected by SQLite itself** (the CHECK constraint
  errors), proven against the real SQLite the runner uses.
- R-H498-325L — the migration **maps existing rows, not drops them**: executing
  the frozen migration SQL files in order up to (but excluding) the new one,
  inserting one row with `public = 1` and one with `public = 0`, then executing
  the new migration file, leaves both rows present with `visibility = 'public'`
  and `visibility = 'private'` respectively. (Driven by executing the real
  migration `.sql` files against a real SQLite — the manual-application harness
  exists precisely to catch a rebuild that silently drops rows.)
