# Decision 19 — The landing page lists the sites that exist

**Decision.** The landing route grows from a static version card into the
service's owner-facing index: it lists the sites the store holds, alongside the
version. The route stays `GET /{$}` at the bare mount root, session-gated at nginx
(D4), rendered in-process from the DB (always accurate, no generated file on
disk).

The `GET /{$}` handler queries `store.List(ctx)` and passes a view model to the
existing `share/www/landing.html` template through `rt.WWW().Render`:

```go
type landingView struct {
    Service string
    Version string
    Sites   []siteRow
}

type siteRow struct {
    Slug      string
    URL       string // absolute front-door URL; identical to the MCP `url` field
    Public    bool   // rendered as "public" / "private"
    CreatedBy string
    CreatedAt string // pre-formatted UTC for display
}
```

`landing.html` renders one row per `siteRow` — slug, public/private, creator,
created-at — and renders cleanly (no rows / an explicit empty state) when `Sites`
is empty. The layout and styling are D6.

**The slug is a link to the site.** Each row's slug renders as
`<a href="{{.URL}}">{{.Slug}}</a>`. `URL` is the site's **absolute front-door
URL**, built by the handler the same way the MCP layer builds its `url` field
(`internal/mcp/tools.go`'s `siteURL`): `baseURL + <public|private> + "/" + slug +
"/"`, where the segment is `sites.PublicSeg`/`sites.PrivateSeg` per `Public` and
`baseURL` carries its trailing slash. There is **one** notion of a site's URL: the
landing link and the tool-reported `url` are the same string. Absolute (not
relative) so it matches the tool `url` and never encodes a second scheme; the
private link works for the viewer because the landing page is already behind the
same `/_session-authn` gate.

`baseURL` is currently computed inside the MCP wiring block at the composition
root (`cmd/sites/main.go`: `strings.TrimSuffix(rt.ResourceID(), "mcp")`). It is
**lifted** so the landing handler and the MCP handler share the one computed
value rather than deriving it twice — behavior-preserving for the MCP surface.

The session-gate (D4) and the name+version heading (D6) are unchanged.

There is **no** `state/www/index.html` file: the index is the live-rendered
landing page, not a generated artifact, so it can never fall out of sync with the
store.

**Rejected.**
- **A generated `state/www/index.html` file rewritten on every create/delete.** A
  second source of truth that must be regenerated and can drift; live render from
  the DB is always correct with no regeneration step.
- **Serving the index publicly.** The list of a box's sites (including private
  ones and who made them) is owner information; it stays behind the existing
  `/_session-authn` gate, same as the version card.
- **Walking the filesystem to build the list.** The DB is authoritative for what
  exists and each site's visibility (D15); `store.List` is the source, not a
  directory scan.
- **A relative slug link (`public/<slug>/`).** Rejected in favor of the absolute
  URL so the landing link is byte-identical to the tool-reported `url` — one
  canonical URL per site, not a second scheme. The only case where relative would
  differ is a direct loopback hit no browser reaches; and a wrong `ResourceID()`
  is a pre-existing bug that already corrupts the tool `url`, best fixed once, not
  masked here.

**Verification.** (Rendered over the repo-real `share/www` tree via
`appkit/web.Load` with fixed inputs.)
- R-RAW6-IUN5 — rendering `landing.html` with a `Sites` slice of two sites (one
  public, one private, each with a distinct creator and created-at) produces
  output containing, for each site, its slug, its public/private label, its
  creator, and its created-at.
- R-RC42-WMDU — rendering `landing.html` with an empty `Sites` slice returns
  without error and still contains the service version (the page degrades to the
  heading with no site rows).
- R-WMWB-7EWX — driving the `GET /{$}` handler (constructed with a fixed
  `baseURL`) against a store seeded with a **public** site `X` and a **private**
  site `Y` renders `X`'s slug inside an anchor whose `href` is
  `<baseURL>public/X/` and `Y`'s slug inside an anchor whose `href` is
  `<baseURL>private/Y/` — the absolute front-door URL the MCP `url` field would
  report for the same site. A wrong implementation that renders the slug as plain
  text, drops the trailing slash, uses the wrong visibility segment, or points
  both rows at one href fails. (The handler + real store substrate, in-process —
  not a fixed-`URL` template render, which could not falsify a wrong handler-side
  URL computation.)
