# Decision 26 — the bearer MCP endpoint forwards all four owner identity headers

**Decision.** sites's nginx fragment (`sites/etc/nginx.conf`) gains ownership of
the identity headers the **bearer-gated** MCP endpoint `= /srv/sites/mcp` forwards
to the loopback process. That endpoint already captures `X-Owner-Email` and
`X-Client-Id` from the dashboard's bearer introspection hook (`/_authn`) and
forwards them; the dashboard now emits **four** owner headers on every allow,
fail-closed, and the endpoint must capture and forward all four (plus the
unchanged `X-Client-Id`).

The four owner headers the `/_authn` hook emits:

- `X-Owner-Id` — the durable owner key, **never empty**.
- `X-Owner-Email` — display/audit, **never empty**.
- `X-Owner-Name` — percent-encoded (headerEncode), **MAY be empty**.
- `X-Owner-Picture` — percent-encoded, **MAY be empty**.

The endpoint captures each with `auth_request_set` into `sites_`-prefixed bearer
variables and re-emits each with a single `proxy_set_header`, alongside the
existing `X-Client-Id` and the untouched `$authn_status` / `error_page 500 =
@sites_authn_500` rate-limit re-emit machinery:

```
location = /srv/sites/mcp {
    auth_request /_authn;
    auth_request_set $sites_owner         $upstream_http_x_owner_email;
    auth_request_set $sites_owner_id      $upstream_http_x_owner_id;
    auth_request_set $sites_owner_name    $upstream_http_x_owner_name;
    auth_request_set $sites_owner_picture $upstream_http_x_owner_picture;
    auth_request_set $sites_client        $upstream_http_x_client_id;

    # rate-limit re-emit machinery (unchanged) — capture the real subrequest
    # status and Retry-After, intercept the synthetic 500 in @sites_authn_500.
    auth_request_set $authn_status $upstream_status;
    auth_request_set $authn_retry  $upstream_http_retry_after;
    error_page 500 = @sites_authn_500;

    proxy_set_header X-Owner-Email   $sites_owner;
    proxy_set_header X-Owner-Id      $sites_owner_id;
    proxy_set_header X-Owner-Name    $sites_owner_name;
    proxy_set_header X-Owner-Picture $sites_owner_picture;
    proxy_set_header X-Client-Id     $sites_client;

    proxy_pass http://127.0.0.1:3004/mcp;   # exact path; upstream serves /mcp
    proxy_set_header Host $host;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_http_version 1.1;
}
```

**Why this is a new Decision, not an extension.** The two session-gated
identity-forwarding locations are already owned — the landing root `= /srv/sites/`
by D4 and the private tier `/srv/sites/private/` by D18 — and their four-header
forwarding is stated there in place. The **bearer** endpoint's identity forwarding,
by contrast, was baseline appkit infrastructure that **no Decision owned**: it is
referenced only as pre-existing context in D4/D8/D18/D24 (e.g. D4's R-NGNX-9W4Z
asserts the endpoint keeps `auth_request /_authn`, not what identity it forwards).
Making it emit the three new headers is a new, falsifiable behavior with no
existing home, so it gets its own Decision here. Ownership stays partitioned by
location, exactly as this fragment's decisions already are (D4 landing root, D18
tiers, D24 the login-bounce opt-in).

**Identity hygiene, per header.** nginx does not forward an inbound client header
as a proxy header unless explicitly set, and the last `proxy_set_header` for a
name wins, so a single set from the auth-subrequest variable **replaces** any
client-smuggled inbound value for that name. Because every one of the four owner
names (and `X-Client-Id`) is set exactly once from its `auth_request_set`
variable, a client can smuggle none of them.

**Variable naming.** The bearer variables are `sites_`-prefixed
(`$sites_owner`, `$sites_owner_id`, `$sites_owner_name`, `$sites_owner_picture`,
`$sites_client`) so they never collide with another service's fragment in the
shared apex server block. They are distinct from the session locations'
`$sites_session_owner*` names; the bearer and session tiers never share a request,
but the distinct names keep the fragment readable and collision-free.

**Scope.** This Decision owns only the `= /srv/sites/mcp` block's identity-header
forwarding. The rate-limit re-emit machinery, the bearer gate, and the endpoint's
exact-match path are unchanged. The PRM bootstrap, the public/static tiers, and the
`@sites_authn_500` re-emit forward no identity and are untouched.

**Rejected.**
- **Forwarding only `X-Owner-Email` (the status quo).** Drops `X-Owner-Id` (the
  durable key downstream will key on), `X-Owner-Name`, and `X-Owner-Picture` at
  nginx; the new headers would die at the trust boundary. Rejected.
- **Folding this into D4 or D18.** Their titles and scope are the session-gated
  landing root and the static tiers respectively; the bearer endpoint is a
  different location with a different gate (`/_authn`) and its own rate-limit
  machinery. Attaching it to either would overload a decision whose purpose is a
  different location. Rejected in favor of a location-scoped Decision.
- **Reusing the `$sites_session_owner*` names for the bearer capture.** The bearer
  and session tiers are distinct locations with distinct gates; sharing names would
  blur which gate produced the value and invites confusion. Distinct `$sites_owner*`
  names match the endpoint's existing `$sites_owner` / `$sites_client` convention.
  Rejected.

**Verification.** (Proven by a test that reads `sites/etc/nginx.conf` from disk and
asserts over the `location = /srv/sites/mcp` block's content — the same
content-assertion pattern D4/D8/D18/D24 use; nginx is not run by the suite.)
- R-7L9K-C2Y0 — the bearer endpoint block `location = /srv/sites/mcp` **captures
  all four owner headers** with `auth_request_set` — `$sites_owner` ←
  `$upstream_http_x_owner_email`, `$sites_owner_id` ← `$upstream_http_x_owner_id`,
  `$sites_owner_name` ← `$upstream_http_x_owner_name`, `$sites_owner_picture` ←
  `$upstream_http_x_owner_picture` — and **forwards each** with its own
  `proxy_set_header`: `X-Owner-Email $sites_owner`, `X-Owner-Id $sites_owner_id`,
  `X-Owner-Name $sites_owner_name`, `X-Owner-Picture $sites_owner_picture`, while
  still forwarding `X-Client-Id $sites_client` and keeping `auth_request /_authn`.
  A block that forwards only `X-Owner-Email`/`X-Client-Id` (drops any of the three
  new owner headers) or omits any of the four owner `auth_request_set` captures
  fails this.
