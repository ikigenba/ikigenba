# Decision 23 — Browser wiring proof: one minimal headless-Chrome test (chromedp)

**Decision.** The one claim the goja tier structurally cannot see — that
`initController` (D22) actually connects the tested logic to a live DOM — is
proven by a **single real-browser test**: `github.com/chromedp/chromedp` (pure
Go, no node/npm, drives Chrome over the DevTools protocol; see
`project/research/research.md`) launches one headless Chrome, loads a seeded
landing page from an in-test HTTP server, and touches **each interactive
control exactly once**. The governing principle is **minimal chromedp**: logic
is tested in goja, never in the browser; the browser asserts only that each
control's wiring does *something* correct once. "Minimal" is measured in
browser sessions, not assertions — the expensive, flake-bearing step is the one
Chrome launch; the actions inside it are milliseconds.

**The harness — real page, no auth, no suite.** The test builds an
`httptest.Server` whose handler is the **real landing render**: the
`landingHandler` (D19) over a real migrated store seeded with fixed rows, plus
the repo-real `share/www` static assets (`tokens.css`, fonts,
`landing.js`) — the same bytes production serves. **Authentication is
structurally absent**: the session cookie is consumed entirely at nginx, a
layer this test deliberately excludes; the sites process serves `GET /{$}`
ungated and the controller makes zero network requests (D22). Identity, if a
page ever renders it, arrives via server render from trusted headers — one more
seeded input, never a live auth chain. The nginx gate itself is D4/D18's
concern, already proven by their content-assertion ids. For the copy step the
harness grants the browser context **clipboard permission** for the test origin
(a CDP `Browser.grantPermissions` for `clipboardReadWrite`) so the scenario can
read the clipboard back; the `httptest` origin is `127.0.0.1`, a secure context
where `navigator.clipboard` is available.

Illustrative shape (seams only):

```go
srv := httptest.NewServer(landingTestHandler(t, seedRows(...)))  // real render + real static
allocCtx, _ := chromedp.NewExecAllocator(ctx, chromedp.DefaultExecAllocatorOptions[:]...)
tabCtx, _   := chromedp.NewContext(allocCtx)                     // + context.WithTimeout
chromedp.Run(tabCtx,
    chromedp.Navigate(srv.URL),
    chromedp.WaitVisible(`#site-search`),        // controller booted (unhid the controls)
    chromedp.SendKeys(`#site-search`, "dsb"),    // real keystrokes → real input events
    chromedp.Click(`th[data-sort-key="name"]`),  // real clicks
    chromedp.Evaluate(`...read slugs from tbody...`, &slugs),  // DOM read-back
)
```

**One session, one straight-line scenario.** The seeded store holds **twelve
sites** — slug `docs` named `Docs`, slug `dashboard` named `Ops Panel`, slug
`blog` named `Blog`, and nine fillers (e.g. slug `site-01` … `site-09` named
`Filler 01` … `Filler 09`; no other slug or name contains the subsequence
`dsb`) with distinct, non-lexical creation timestamps, and with **name
order ≠ slug order** (e.g. `Ops Panel` sorts after `Docs` while `dashboard`
sorts before `docs`) — so a single dataset exercises every step:

1. **Boot** — after `Navigate`, the search input and the pager become visible
   (the controller parsed the island and removed `hidden`). The shipped-empty
   stub of v0.13.0 fails here by timeout, before any assertion.
2. **Filter** — type `dsb` → exactly one row remains, its **name anchor**
   reading `Ops Panel`. `dsb` is a subsequence (not substring) of the row's
   **slug** `dashboard` and of no other row's name or slug — so this one step
   pins the browser to the goja-proved fuzzy semantics, to slug-field matching,
   and to the name being what the rebuilt row displays.
3. **Sort** — clear the query, click the **Name** header → rows in
   **display-name-ascending** order (an order distinct from slug order by
   construction of the seed — a controller sorting slugs fails) and
   `aria-sort="ascending"` on that header; click it again → exact reverse and
   `aria-sort="descending"`.
4. **Clear** — after the above, click Clear → search box empty, full list back
   in default created-at-descending order, first page.
5. **Page** — with 12 rows the pager is visible reading `Page 1 of 2`; click
   Next → `Page 2 of 2` and the `<tbody>` shows the remaining 2 rows; Prev
   returns to page 1.
6. **Copy** — before clicking, assert the copy button's **icon renders**: the
   glyph inside the JS-rebuilt button is an element in the **SVG namespace**
   (`namespaceURI === "http://www.w3.org/2000/svg"`) with a **non-zero rendered
   box** (its `getBoundingClientRect()` width and height are both `> 0`) — the
   `createElement("svg")`+`innerHTML` defect (D22) leaves an HTML-namespaced,
   unrendered node and fails here. Record the table's rendered **width**, then
   click one row's copy button → reading the clipboard back
   (`navigator.clipboard.readText()`) returns **that row's absolute front-door
   URL**, and the button's label reads `Copied`. The session is granted
   clipboard permission for the test origin; the page is served from
   `127.0.0.1`, a secure context where `navigator.clipboard` is available, so the
   real async-clipboard path (not the fallback) is exercised. This is the one
   step that proves the copy wiring actually reaches the system clipboard — a
   claim no structural or goja test can see. Finally, with the label showing
   `Copied`, assert the table's rendered width is **byte-identical to the
   pre-click measurement** — the `Copy`→`Copied` swap must not reflow the layout
   (D6's width-stable control).

**Chrome is a hard requirement of the green bar.** The suite requires a
`google-chrome` binary on `PATH` of the box running `go test ./...` (present:
`/usr/bin/google-chrome`; the deploy box never runs tests). No Chrome, or
Chrome fails to launch → the test **fails** — never skips. A skipped test reads
as green to the verify gate and silently un-proves the wiring; rejected. One
narrow reliability concession: the harness may retry the **browser launch**
once (launch flake is the known failure mode; a second attempt distinguishes
"Chrome hiccuped" from "Chrome broken/absent") — but a scenario assertion that
fails, fails; **no scenario retries**. The whole session runs under a
`context.WithTimeout` (~30s) so a hung page fails instead of hanging the suite.
The browser is chromedp's own: headless (`--headless=new`), a fresh temporary
user-data-dir, isolated from any desktop Chrome; nothing appears on screen and
no profile is touched.

**The dependency boundary.** `chromedp` joins `sites/go.mod` as a **test-only**
dependency, exactly the goja precedent: imported only from `*_test.go`, so the
shipped binary links zero bytes of it. That boundary is enforced mechanically,
not by intention: the production import graph must be free of both test-only
deps.

**Rejected.**
- **Playwright / any node-based runner.** The node ecosystem in a pure-Go repo:
  `package.json`, `node_modules`, driver churn. chromedp does what this needs
  over the same DevTools protocol with zero node (research.md).
- **A goja DOM shim.** A hand-rolled fake `document` is a mock that passes
  whatever it is taught; it cannot falsify real event wiring. The substrate must
  be able to make the test fail.
- **`t.Skip` when Chrome is absent.** A soft gate: an environment
  misconfiguration silently un-proves the wiring and reads as green to the
  verify step. The requirement is hard and documented.
- **Scenario retries.** Retrying assertions hides real intermittent bugs; only
  the launch (an environment step, not a behavior) gets one retry.
- **One browser session per assertion.** Pays the launch cost (and its flake
  surface) five times for no added independence — the steps share one dataset by
  design.
- **Testing logic boundaries in the browser** (pagination arithmetic, clamps,
  tie-breaks, case folding). goja owns logic (D22); duplicating it here couples
  broad coverage to the flakiest substrate and slows the loop.
- **Driving the live `bin/start` suite / the nginx front door.** Would drag the
  session-cookie auth chain into the test for zero added proof of *this*
  Decision's claim; the gate is D4/D18's, already content-asserted.

**Verification.** (R-87B9-J644, R-06IK-15YH, R-08YC-SPFV, R-8AYY-OHC7,
R-8DER-G0TL, R-NN9H-UKP3, R-VYEF-053C, and R-VZMB-DWU1 are one chromedp
scenario — a single headless-Chrome session over the seeded `httptest` landing
page described above; R-8EMN-TSKA is a mechanical import-graph assertion.)

- R-87B9-J644 — **boot:** navigating to the seeded page, the search input and
  the pager region become **visible** (bounded wait) — the controller parsed the
  island and removed the `hidden` markers. An empty/stubbed `initController`
  (the shipped v0.13.0 defect), a 404'd or throwing `landing.js`, or a
  mis-wired island id all fail here by timeout.
- R-06IK-15YH — **filter wiring:** typing `dsb` into the search input (real key
  events) leaves exactly **one** row in the table body, whose **name anchor**
  text is `Ops Panel` — among the 12 seeded rows only the `dashboard`-slugged
  row carries the subsequence `dsb` (in its slug), and the visible text the row
  shows is its display name, not its slug. A page whose input listener never
  fires, whose dispatch chain is broken, whose `<tbody>` is never re-rendered,
  or whose rebuilt rows display the slug fails.
- R-08YC-SPFV — **sort wiring:** clicking the **Name** header orders the
  visible rows **display-name-ascending** — an order the seed makes distinct
  from slug order, so a controller sorting the slug field fails — and stamps
  `aria-sort="ascending"` on that header; clicking it again yields the exact
  reverse order and `aria-sort="descending"`. A dead header listener, a toggle
  that never flips, or a controller that never stamps `aria-sort` fails.
- R-8AYY-OHC7 — **clear wiring:** after filtering and sorting, clicking Clear
  empties the search input and restores the full first-page listing in the
  default created-at-descending order. A dead Clear listener or a partial reset
  fails.
- R-8DER-G0TL — **pager wiring:** with 12 seeded rows the pager is visible
  reading `Page 1 of 2`; clicking Next changes the readout to `Page 2 of 2` and
  the table body to the remaining 2 rows; clicking Prev returns to the first
  10. A dead pager listener or a readout that never updates fails.
- R-NN9H-UKP3 — **copy wiring:** in the same session (clipboard permission
  granted, page served from `127.0.0.1` — a secure context), clicking a row's
  copy button, then reading the clipboard back via
  `navigator.clipboard.readText()`, returns **that row's absolute front-door
  URL** (byte-identical to the row's name-anchor `href`), and the clicked
  button's label reads `Copied`. A dead copy listener, a button that copies the
  wrong string (or nothing), or one that never shows the `Copied` confirmation
  fails. This drives the **real system clipboard** — the substrate a mock or a
  structural render cannot falsify.
- R-VYEF-053C — **icon renders:** in the same session, after boot, the copy
  glyph inside a row's JS-rebuilt copy button is an element in the **SVG
  namespace** (`namespaceURI === "http://www.w3.org/2000/svg"`) whose
  `getBoundingClientRect()` reports a **non-zero** width and height (both `> 0`).
  A controller that builds the icon with `document.createElement("svg")` +
  `innerHTML` produces an HTML-namespaced node the browser does not paint (zero
  box) and fails — the exact defect that left the shipped v0.14.1 copy button
  iconless. This drives the **real browser layout** — a claim no structural or
  goja test can see.
- R-VZMB-DWU1 — **no reflow on copy:** in the same session, the table's rendered
  width is measured immediately **before** clicking a row's copy button and again
  **after** the label has swapped to `Copied`, and the two are **equal** — the
  `Copy`→`Copied` confirmation must not change the button, cell, or table width.
  A control whose label is not width-reserved (so `Copied` is wider than `Copy`)
  grows the button and reflows the table, and fails. This drives the **real
  browser layout** — a width change a structural render cannot observe.
- R-8EMN-TSKA — **dependency boundary:** the production import graph of the
  binary (`go list -deps ./cmd/sites`) contains **neither**
  `github.com/chromedp/chromedp` **nor** `github.com/dop251/goja` — both are
  test-only and the shipped binary links zero bytes of either. A stray non-test
  import of either fails.
