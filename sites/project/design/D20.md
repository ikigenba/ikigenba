# Decision 20 ŌĆö MCP surface: drop publish/unpublish, thread the owner from Identity, flip visibility, serve the live folder

**Decision.** The `ikigenba_sites_*` tool table drops the publish lifecycle and
reflects the new model: a site is created (private by default), its files are
edited *live* in its served directory, its visibility is flipped with one tool,
and it is deleted.

**Tools removed:** `publish`, `unpublish`.

**Tools added:** `set_visibility` ŌĆö flip a site publicŌåöprivate.

```
set_visibility(name, public: bool)
  ŌåÆ Store.SetVisibility(name, public) + Layout.Move(name, public), in lockstep.
    Returns the site's JSON projection with its new "url".
```

**Tools changed:**

- `create(name, public?)` ŌĆö records the creator from the request `Identity`
  (`docs/owner-id-design.md`): the stable `Identity.OwnerID` (nginx-forwarded
  `X-Owner-Id`) into `owner_id` and the `Identity.OwnerEmail` display snapshot into
  `owner_email`; neither is a tool argument. It creates the site in the
  caller-chosen visibility in **one insert**: the optional `public` boolean
  defaults **false** (private) when omitted, and the site's row and directory are
  born at that visibility (`store.Create(ctx, name, id.OwnerID, id.OwnerEmail,
  public)` + `os.MkdirAll(SiteDir(public, name))`). No intermediate private state
  and no post-create move ŌĆö `set_visibility`/`Layout.Move` remain the
  *flip-an-existing-site* path only, never part of birth.

  ```
  create(name, public: bool = false)
    ŌåÆ store.Create(name, id.OwnerID, id.OwnerEmail, public) + MkdirAll(SiteDir(public, name))
      Returns the site's JSON projection; "url" is public/<name>/ when public
      else private/<name>/.
  ```
- `delete(name)` ŌĆö removes the row **and** the site's on-disk directory
  `SiteDir(site.Public, name)` (no unpublish step; no working tree). Idempotent.
- `list` ŌĆö describes sites in public/private + creator terms; the lifecycle text
  drops "working tree" and "publish/unpublish". (The self-describing tool is
  `guide`, not `describe` ŌĆö see D21.)
- `sync` **requires the site to already exist.** It is a content-reconciliation
  tool, not a birth path: it `Get`s the target slug and, when the slug is absent,
  returns the `not_found` error envelope **without creating a row or a
  directory** ŌĆö the caller must `create` the site first. The former
  create-or-reuse branch (which inserted a row with **empty** owner columns) is
  removed, so every site now has a real creator. For a site that *does* exist,
  `sync` reconciles as before (overwrite every upstream file, delete every site
  file absent upstream) into the site's live served directory and leaves its
  visibility unchanged.
- The **file tools** (`file_write`, `file_read`, `file_edit`, `file_glob`,
  `file_grep`, `file_list`, `mkdir`) and **`sync`** operate on the site's **live
  served directory** ŌĆö `SiteDir(site.Public, name)` resolved from the DB row ŌĆö not
  a working tree. A write is immediately visible under the served tree. The tools
  first `Get` the site to learn its visibility, then confine under that directory
  via `internal/files.ConfinePath` (unchanged confinement, new root).

**`renderSite`** projects the fields ŌĆö `name`, `public`, `owner_id`,
`owner_email`, `url`, `created_at`, `updated_at` ŌĆö where `url` is
`<baseURL>public/<name>/` when public else `<baseURL>private/<name>/`. Per the
suite owner-id conversion (Decision 5, `docs/owner-id-design.md`), the owner
projection **keeps `owner_email`** (the retired `created_by` key becomes
`owner_email`) and **adds `owner_id`**, so `list`/`create`/`set_visibility`
results surface both. `tier`/`published`/`published_at` remain out of the
projection. `errResult` drops the `ErrInvalidTier` case. (The `outputSchema`
carrying these fields is D25's.)

**Rejected.**
- **Keeping `publish`/`unpublish` as no-ops or aliases of `set_visibility`.** The
  operator removed the lifecycle; leaving dead verbs invites the old mental model.
  A site that exists is served; visibility is the only control.
- **A separate working directory the file tools edit.** Removed with D16 ŌĆö edits
  are live; the tools address the served directory directly.
- **Taking the owner (`owner_id`/`owner_email`) as a tool argument.** The creator
  is the authenticated caller; reading both from the request `Identity` prevents
  spoofing and needs no argument. The stable id is the suite key; the email rides
  along as the display snapshot.
- **Making `public` the *default* on `create`, or a separate `create_and_publish`
  tool.** `create` accepts an optional `public` boolean but **defaults private**:
  a fat-fingered or hallucinated `create` must never auto-expose content to the
  world (the safe-default asymmetry ŌĆö an accidentally-private site is a harmless,
  instantly-fixable inconvenience; an accidentally-public one is a confidentiality
  failure). A distinct `create_and_publish` verb is rejected outright: it
  resurrects the "publish" vocabulary the whole rework deleted and adds a tool for
  no capability the optional flag doesn't already give.
- **Letting `sync` create the site (create-or-reuse).** Folding creation into
  `sync` is what forced the empty-`created_by` insert and blurred "content
  reconciliation" with "bring a site into being." `create` is the single birth
  path; `sync` reconciles an existing site. The two-call flow (`create` then
  `sync`) is the same shape as the manual path and keeps every site owned by a
  real creator.
- **A `public` argument on `sync`.** Once `sync` cannot create, a visibility
  argument on it is moot (there is no birth to set visibility for) and would
  tempt a content re-sync into silently re-flipping a live site's visibility.
  Visibility is chosen at `create` and changed only by `set_visibility`.

**Verification.**
- R-RDBZ-AE4J ŌĆö the tool table returned by `Tools(...)` contains **no** tool named
  `publish` and **no** tool named `unpublish`.
- R-Z6FF-WUWS ŌĆö calling `create` with a request `Identity` carrying
  `X-Owner-Id: id-alice` and `X-Owner-Email: alice@example.com` persists
  `owner_id == "id-alice"` and `owner_email == "alice@example.com"`, both surfaced
  (as `owner_id` and `owner_email`) on the returned site and on a subsequent
  `list`. Two callers presenting the **same `X-Owner-Email` but different
  `X-Owner-Id`** create sites carrying **distinct `owner_id`** (create reads the
  stable id from `Identity.OwnerID`, not the email) ŌĆö a build still recording the
  email as the owner key, or dropping `owner_id` from the result, fails. (Real
  handler+DB ŌĆö the identity comes from the gate, the persistence and the projection
  are the real substrate.)
- R-554R-3MBC ŌĆö calling `create(name, public:true)` creates the row **and** the
  directory at `SiteDir(true, name)` in one step (no `SiteDir(false, name)`
  directory is ever created for it) and returns a site whose `public` is true and
  whose `url` ends `public/<name>/`; calling `create(name)` with `public` omitted
  yields `public == false`, a directory at `SiteDir(false, name)`, and a `url`
  ending `private/<name>/`.
- R-56CN-HE21 ŌĆö calling `sync` with a `source_path`/`slug` naming a site that does
  not exist returns the `not_found` error envelope and creates **no** row (a
  subsequent `get`/`list` still shows the slug absent) and **no** directory under
  either `SiteDir(true, name)` or `SiteDir(false, name)`; calling `sync` for an
  existing site still reconciles its files and leaves its `public` flag unchanged.
- R-RGZO-FPCM ŌĆö calling `set_visibility(name, true)` on a private site flips the
  stored `public` flag to true, relocates its directory to `SiteDir(true, name)`,
  and returns a site whose `url` ends `public/<name>/`; `set_visibility(name, false)`
  reverses all three.
- R-RI7K-TH3B ŌĆö a `file_write` to a site resolves under `SiteDir(site.Public, name)`
  (the live served directory): after writing `index.html` to a public site, the
  file exists at `SiteDir(true, name)/index.html` and is served by D17's handler ŌĆö
  there is no separate working-tree copy.
- R-RJFH-78U0 ŌĆö `delete(name)` removes both the row (a subsequent `get` yields
  not-found) and the on-disk directory `SiteDir(site.Public, name)` (it no longer
  exists).
