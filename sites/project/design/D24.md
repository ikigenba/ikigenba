# Decision 24 — the session-gated locations opt into the apex `@login_bounce`: a logged-out human navigation goes to sign-in, not a bare 401

**Decision.** sites's nginx fragment (`sites/etc/nginx.conf`) opts its **three
browser-facing, session-gated locations** into the dashboard-owned apex
`@login_bounce` primitive by adding **one line** — `error_page 401 =
@login_bounce;` — to each. Today a logged-out browser navigating to any of them
gets the bare nginx **401** the `auth_request /_session-authn` deny produces; with
the opt-in, the apex `@login_bounce` (a dashboard-owned external contract, defined
in the apex `server` block exactly where `/_session-authn` is) turns a logged-out
**navigation** into a `302` to `/login?return_to=…` and returns the person to the
page after they sign in, while a scripted `fetch`/XHR still gets the clean `401` it
can handle. `@login_bounce` makes that navigate-vs-fetch distinction itself (off
`Sec-Fetch-Mode`); sites only has to reference it.

The three locations that gain the line — every session-gated (`auth_request
/_session-authn`) location in the fragment:

```
location = /srv/sites/ {                 # landing root (proxied to the process)
    auth_request /_session-authn;
    error_page 401 = @login_bounce;      # ← added
    …
}
location /srv/sites/static/ {            # landing assets
    auth_request /_session-authn;
    error_page 401 = @login_bounce;      # ← added
    …
}
location /srv/sites/private/ {           # private static tier
    auth_request /_session-authn;
    error_page 401 = @login_bounce;      # ← added
    …
}
```

**What deliberately does NOT get the line.** The change is confined to the
browser/session tiers:

- `location /srv/sites/public/` — the **public** tier has no `auth_request` and
  never 401s; a bounce there would be meaningless.
- `location = /srv/sites/mcp` — the **bearer** MCP endpoint (`auth_request
  /_authn`) must keep returning `401` with its `WWW-Authenticate` challenge so MCP
  clients can discover the AS; redirecting an MCP call to an HTML login page would
  break the OAuth bootstrap.
- `location = /srv/sites/.well-known/oauth-protected-resource` and
  `location @sites_authn_500` — unauthenticated PRM bootstrap and the bearer
  rate-limit re-emit; neither is a browser session gate.

The change is **purely additive** — one line per session-gated location, every
existing directive (`auth_request /_session-authn`, the `proxy_pass`, the headers)
retained and nothing rewritten. It adds **no new suite dependency** beyond the one
sites already leans on: it referenced apex-owned `/_session-authn` before, and now
also references apex-owned `@login_bounce` (both provided by the dashboard's apex
block, per D20 in the dashboard's `project/`).

**Rejected.**
- **Adding `error_page 401 = @login_bounce;` to `= /srv/sites/mcp`** — the bearer
  endpoint's 401 is a protocol signal (the `WWW-Authenticate` challenge); bouncing
  it to an HTML login page breaks MCP client discovery. The opt-in is browser-only.
- **Adding it to the public tier** — no `auth_request`, so no 401 to intercept;
  dead directive.
- **Having sites define its own bounce (a local named location / `if`)** — the
  bounce is a suite-wide contract owned once by the apex (D20); duplicating it in
  every service fragment is exactly the copy-paste the shared `@login_bounce`
  exists to avoid.
- **Bouncing all requests regardless of `Sec-Fetch-Mode`** — a logged-out
  browser-side `fetch` to a private path would then receive login HTML with a
  `200` instead of a `401` it can handle; the navigate-vs-fetch split (in
  `@login_bounce`) is deliberate and this fragment inherits it for free.

**Verification.** (Proven by a Go test that reads `sites/etc/nginx.conf` from disk
and asserts over its content — the content-assertion pattern D04/D18 already use;
nginx is not run by the suite.)
- R-XVIT-1NXD — in `sites/etc/nginx.conf`, each of the three session-gated
  locations — the exact-match landing root `= /srv/sites/`, the landing-asset tier
  `/srv/sites/static/`, and the private tier `/srv/sites/private/` — contains
  **both** `auth_request /_session-authn;` and `error_page 401 = @login_bounce;`.
- R-XWQP-FFO2 — the public tier `location /srv/sites/public/` and the bearer
  endpoint `= /srv/sites/mcp` do **not** contain `error_page 401 = @login_bounce;`
  (the opt-in is confined to the session tiers; the bearer 401 is left intact).
- R-XXYL-T7ER — the change is additive: every pre-existing location still appears
  and every session-gated location retains its `auth_request /_session-authn;` and
  its `proxy_pass` — no existing directive is removed or rewritten by the opt-in.
