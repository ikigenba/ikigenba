# Decision 21 — MCP self-discovery: routing `instructions` + a `guide` tool (replacing `describe`)

**Decision.** sites adopts the suite's MCP self-discovery convention
(`docs/mcp-discovery-convention.md`, piloted on crm) so a connecting agent can
find sites and drive it from the connection alone. Two channels change; no domain
verb's *behavior* changes.

**Tier 0 — the `initialize` `instructions`.** The `Instructions` const in
`internal/mcp/mcp.go` is rewritten to name sites's domain in the vocabulary a
user actually types (the routing hooks that let tool search match sites without
an external alias skill), state the verb flow, and point at the `guide` tool —
and it no longer mentions `describe`. Illustrative:

```go
const Instructions = "Host small static websites on this box — web pages, " +
    "landing pages, marketing or docs sites, hand-written HTML/CSS/JS — each " +
    "served at its own URL, either public (anyone) or private (logged-in " +
    "dashboard users). Create a site, write its files, and set it public or " +
    "private; edits are live immediately, no publish step. Call guide for the " +
    "site model and worked examples."
```

**Tier 2 — a `guide` tool replaces `describe`.** The lightweight `describe` tool
and its `toolDescribe()` JSON blob are **deleted**. In their place a single
read-only `guide` tool returns an embedded usage document as text, exactly as the
crm pilot does:

```go
//go:embed guide.md
var guideDoc string

// guide: flat, read-only, input-free; returns the embedded document verbatim.
desc(tool("guide"), "Return the sites usage guide — the site model, slug and \
confinement rules, and worked basic/advanced examples (create a public page in \
one call, import from Dropbox). Read once before your first create.",
    obj(map[string]any{}),  // no properties, no required
    func(ctx, args, _ ) (map[string]any, error) { return appkitmcp.TextResult(guideDoc), nil })
```

`guide` takes no inputs, reads nothing, mutates nothing, and does not error. The
document is embedded (versioned and shipped with the binary), so the guide an
agent reads always matches the running service. The surface stays **15 tools**:
`guide` replaces `describe` 1:1 (see D13), plus `create`/`sync` keep their names
(their schemas gain/lose args under D20). No tool is added or removed net.

**`guide.md` contents** (one screen; the reference bulk that must not live in
Tier 0/1):

1. **Model** — a site = slug + `public`/`private` + creator; files served live at
   the site's URL; **`create` is the only way a site comes into being** — every
   other tool (`file_*`, `mkdir`, `sync`, `set_visibility`, `delete`) requires the
   site to already exist and returns `not_found` otherwise.
2. **Slug rules** — 1–63 chars, lowercase alphanumeric + hyphen, must start
   alphanumeric; reserved names rejected.
3. **Confinement** — file paths are relative to the site root; absolute paths and
   `..` escapes are rejected (`path_escapes_working_dir`).
4. **Basic — a public page in two calls** — `create(name:"launch", public:true)`
   → `file_write(site:"launch", file_path:"index.html", content:"…")` → visit the
   returned `url`.
5. **Basic — a private site** — `create(name:"runbook")` (defaults private) →
   write files → served only to logged-in users; promote later with
   `set_visibility(name:"runbook", public:true)`.
6. **Advanced — import from Dropbox** — `create(name:"marketing")` first (`sync`
   will not create it), then `sync(source_path:"/sites/marketing")`, then
   `set_visibility(name:"marketing", public:true)`.
7. **Error self-correction** — `not_found` → "call `create` first";
   `already_exists`; `invalid_slug`; `reserved_name`.

**Tier 1 — keyword-forward the two descriptions that change.** Under D20 the
`create` and `sync` descriptions are rewritten anyway; they lead with the domain
vocabulary tool search indexes (`create` → "static website/site"; `sync` →
"import … from a Dropbox-mirrored folder"). `sync`'s wording deliberately avoids
"publish"/"deploy" — reintroducing that vocabulary would re-imply the deleted
publish step.

**The guide is referenced in exactly two places** — the Tier 0 `Instructions`
and the `guide` tool's own description — and **nowhere else**. It is deliberately
not repeated in every tool description (that would re-inflate Tier 1 for no gain);
an agent that ignores the pointer and calls a write tool cold self-corrects from
the `not_found` envelope.

**Rejected.**
- **Keeping `describe` alongside a new `guide`.** Two orientation tools is the
  redundancy the convention collapses; the crm pilot has only `guide`. `describe`
  is removed, not kept.
- **A `guide` MCP *resource* instead of a tool.** Most clients do not surface
  resources into the agent's context; the tool is the channel with the hard
  delivery guarantee (`tools/list` in every client).
- **Repeating the guide pointer in every tool description.** Re-inflates Tier 1
  on every connection; the always-loaded Tier 0 pointer already reaches the agent
  before it first needs the guide.
- **A parameterized/topic-scoped guide.** Premature: sites's guide fits one
  screen. Start flat and input-free; scope only if it outgrows a screen.

**Verification** (through the assembled `appkit/mcp` handler over a temp DB, temp
`SITES_ROOT`, and a fake mirror client — the same seam D13/D20 use; plus a
content assertion over the repo-real embedded `guide.md`).
- R-57KJ-V5SQ — `tools/list` through the assembled handler contains a tool named
  `guide` and **no** tool named `describe`; `guide`'s `InputSchema` is an object
  schema that declares **no** `required` fields; and `tools/call guide` with
  omitted arguments returns a **single non-empty text content block** (not an
  error envelope) — proving `guide` is present, flat, read-only, and input-free.
- R-58SG-8XJF — the embedded `guideDoc` text contains all four load-bearing
  anchors: the create-first rule (a site must exist / `create` is the only birth
  path), a public-in-one-call example that invokes `create` with `public`, the
  Dropbox flow that shows `create` **before** `sync`, and the confinement token
  `path_escapes_working_dir` — proving the reference bulk actually lives in Tier 2.
- R-5A0C-MPA4 — the `Instructions` string contains the routing word `website` and
  the pointer word `guide`, and does **not** contain `describe` — proving Tier 0
  carries user vocabulary and points at the guide, with the old tool name gone.
- R-5B89-0H0T — the guide is referenced in exactly two channels: `Instructions`
  contains `guide` and the `guide` tool's own `Description` is non-empty; and for
  **every other tool** in the table, its `Description` does not contain the
  substring `guide` — proving Tier 1 was not re-inflated with the pointer.
- R-5CG5-E8RI — the `sync` tool's `Description` contains neither `publish` nor
  `deploy` (the deleted-vocabulary guard) while staying findable (it contains
  `Dropbox`) — proving the reworded description does not resurrect the publish
  step.
