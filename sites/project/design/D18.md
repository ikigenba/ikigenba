# Decision 18 — nginx fragment proxies the public/private tiers to the process (no `alias`)

**Decision.** `sites/etc/nginx.conf`'s two static-tier locations stop serving
files off disk and instead **proxy to the sites process**, which now serves the
bytes (D17). The visibility gate stays at nginx (per Conventions — sites runs no
session logic):

```
# Public site tier — NO auth. Proxied to the process (was: alias off disk).
location /srv/sites/public/ {
    proxy_pass http://127.0.0.1:3004/public/;   # trailing slash preserves the subpath
    proxy_set_header Host $host;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_http_version 1.1;
}

# Private site tier — dashboard SESSION gated, then proxied to the process.
location /srv/sites/private/ {
    auth_request /_session-authn;
    auth_request_set $sites_session_owner         $upstream_http_x_owner_email;
    auth_request_set $sites_session_owner_id      $upstream_http_x_owner_id;
    auth_request_set $sites_session_owner_name    $upstream_http_x_owner_name;
    auth_request_set $sites_session_owner_picture $upstream_http_x_owner_picture;
    proxy_set_header X-Owner-Email   $sites_session_owner;
    proxy_set_header X-Owner-Id      $sites_session_owner_id;
    proxy_set_header X-Owner-Name    $sites_session_owner_name;
    proxy_set_header X-Owner-Picture $sites_session_owner_picture;
    proxy_pass http://127.0.0.1:3004/private/;
    proxy_set_header Host $host;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_http_version 1.1;
}
```

The private tier forwards **all four** owner identity headers, not just email. The
`/_session-authn` hook emits four owner headers on every allow, fail-closed:
`X-Owner-Id` (durable key, never empty), `X-Owner-Email` (never empty), and the
percent-encoded `X-Owner-Name` / `X-Owner-Picture` (either MAY be empty). The block
captures each with `auth_request_set` into the `sites_`-prefixed session variables
(`$sites_session_owner` for email — name kept stable — plus
`$sites_session_owner_id`, `$sites_session_owner_name`,
`$sites_session_owner_picture`) and re-emits each with one `proxy_set_header` per
name. That single set per name is the per-header identity-hygiene property: it
replaces any client-smuggled inbound value for that header, so none of the four can
be spoofed. These are the same variable names the landing root (D4) uses;
`auth_request_set` binds per-location, so the reuse across the two session-gated
locations is safe. The **public** tier below has no `auth_request` and forwards no
identity — it is unchanged.

The `alias`, `autoindex off`, and `index index.html` directives are **removed**
from both blocks — directory/index semantics now live in the Go handler (D17). The
fragment **no longer references the on-disk state path** (`/opt/sites/…/www`):
nginx needs no filesystem access to sites' state.

Everything else in the fragment is **unchanged**: the PRM well-known bootstrap,
the bearer-gated `= /srv/sites/mcp`, the session-gated landing root `= /srv/sites/`
(D4), the landing `/srv/sites/static/` assets, and the `@sites_authn_500` re-emit
all stay. The public/private locations keep their **paths** and **auth posture**
(public ungated, private `/_session-authn`) — only their bodies change from
`alias` to `proxy_pass` — so D4's `R-NGNX-9W4Z` coexistence assertion remains true.

The loopback port stays a literal (`3004`, sites' fixed registry port) — nginx
reads config directly and cannot consult the Go `registry` (per the product
addendum), so the fragment is shipped verbatim.

**Rejected.**
- **Keeping `alias` and having nginx serve the files.** The redundant-state path
  the whole change removes: nginx off disk requires the symlink/served tree that
  D16 deletes, and it re-encodes DB state nginx can't read.
- **Moving the private session gate into the sites process.** Would make sites do
  cookie/session logic, breaking the "nginx is the sole trust boundary" invariant.
  The gate stays at nginx; the process trusts it.
- **Dropping the trailing slash on `proxy_pass`.** The trailing slash is what maps
  `/srv/sites/public/<slug>/<path>` to the process's `/public/<slug>/<path>`; without
  it the subpath would be mangled.

**Verification.** (Proven by a test reading `sites/etc/nginx.conf` from disk.)
- R-R78H-DJF2 — the `location /srv/sites/public/` block `proxy_pass`es to
  `http://127.0.0.1:3004/public/` and contains **no** `auth_request` directive
  (public tier stays unauthenticated).
- R-R8GD-RB5R — the `location /srv/sites/private/` block contains
  `auth_request /_session-authn` and `proxy_pass http://127.0.0.1:3004/private/`.
- R-7NPD-3MFE — the `location /srv/sites/private/` block **captures all four owner
  headers** with `auth_request_set` — `$sites_session_owner` ←
  `$upstream_http_x_owner_email`, `$sites_session_owner_id` ←
  `$upstream_http_x_owner_id`, `$sites_session_owner_name` ←
  `$upstream_http_x_owner_name`, `$sites_session_owner_picture` ←
  `$upstream_http_x_owner_picture` — and **forwards each** with its own
  `proxy_set_header`: `X-Owner-Email $sites_session_owner`,
  `X-Owner-Id $sites_session_owner_id`, `X-Owner-Name $sites_session_owner_name`,
  and `X-Owner-Picture $sites_session_owner_picture`. A block that forwards only
  `X-Owner-Email` or omits any of the four captures fails this.
- R-R9OA-52WG — neither tier block contains an `alias` directive, and the fragment
  contains no reference to the on-disk state path (no `/opt/sites/` ... `/www`
  path and no `alias` anywhere), while the pre-existing landing root
  `= /srv/sites/`, the `= /srv/sites/mcp`, the PRM well-known, and
  `@sites_authn_500` locations still appear.
