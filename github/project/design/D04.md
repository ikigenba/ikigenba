# Decision 4 ‚Äî The MCP tool surface

## Decision

`internal/mcp` is the JSON-RPC 2.0 tool surface, structurally the same as
`gmail/internal/mcp`: a `Handler` mounted at `POST /mcp` behind
`rt.RequireIdentity`, carrying **no** token logic (nginx has already introspected
the request and injected identity). It speaks JSON-RPC over plain HTTP POST
(`initialize`, `tools/list`, `tools/call`), responding `application/json`.

**Bare verb names.** Tools register **bare** verbs (no service prefix), exactly
like `gmail` (`toolPrefix = ""`): the suite discovery layer in `prompts`
re-qualifies each bare verb to `ikigenba_github_<verb>`. The surface is the 13
domain verbs (D3) plus the two chassis verbs:

```
repos_list  repo_get
pr_list  pr_get  pr_comment  pr_review  pr_merge
issue_list  issue_get  issue_create  issue_comment  issue_update
file_get  file_put
health  reflection
```

Each has a JSON input schema declaring its parameters (e.g. `pr_get` requires
`repo` and `number`; `pr_review` requires `repo`, `number`, `event`, optional
`body`). `tools/call` decodes the arguments against the schema and dispatches to
the matching `internal/gh.Client` method.

```go
type Identity struct { OwnerEmail, ClientID string } // from X-Owner-Email / X-Client-Id

type Handler struct {
    client  GitHubClient // the D3 Client (interface, for offline tests)
    version string
    service string
    health  func(context.Context) (map[string]any, error)
    logger  *slog.Logger
}
```

**`health` is the live auth proof.** The `health` verb returns the chassis health
envelope *and* drives a real authenticated GitHub call (D2: resolve the org
installation / mint a token). Success means the app credentials actually work
against `@ikigenba`; failure is reported loudly in the envelope, never a silent
OK. This is the seam that realizes D2's `R-DMUT-QF4A` end to end.

**`reflection` reports an empty event graph.** github is neither producer nor
consumer, so `reflection` self-describes **zero** published events and **zero**
subscriptions. The `Handler` is constructed with an empty
`eventplane/outbox.Registry` and a nil subscriptions provider (the only use of the
`eventplane` types).

**Owner provenance ‚Üí logs, never GitHub.** On every **write** verb
(`pr_comment`, `pr_review`, `pr_merge`, `issue_create`, `issue_comment`,
`issue_update`, `file_put`), the dispatcher emits one structured `slog` line
carrying the `X-Owner-Email`, the `X-Client-Id`, the verb, and the target
(repo + number/path) ‚Äî the audit record of who triggered the action. That owner
identity is passed to `slog` **only**; it is never forwarded into any GitHub
request body (D3 already omits owner fields). Read verbs need no such line.

**Error surfacing.** A `Client` error (`ErrNotFound`, `ErrInvalid`, `ErrAppAuth`,
or a transport failure) becomes a JSON-RPC tool result with `isError: true` and a
clear message ‚Äî never a panic or a dropped connection.

## Rejected

- **Service-prefixed tool names** (`ikigenba_github_pr_get` at registration).
  Rejected to match the suite convention: peers register bare verbs and the
  `prompts` discovery layer qualifies them; double-qualifying would break
  discovery.
- **A generic passthrough `github_request` tool.** Rejected: an agent handed raw
  HTTP-to-GitHub bypasses the fixed, reviewable verb set and the bot-only
  attribution guarantees; the explicit verb surface is the product surface.
- **Stamping owner identity into the GitHub write** (comment trailer, commit
  author). Rejected by product ‚Äî repos are mostly public and owner PII must not be
  published; the log line is the audit substitute.
- **Reporting a placeholder event in `reflection`.** github publishes nothing;
  reflecting a fake event would misdescribe the service.

## Verification

- `R-EEWI-J569` ‚Äî `initialize` returns a protocol/serverInfo result, and
  `tools/list` advertises exactly the 15 verbs above (13 domain + `health` +
  `reflection`) as **bare** names, each with a non-empty input schema. Asserted
  offline against the `Handler` with a fake client.
- `R-EHCB-AONN` ‚Äî `tools/call` for a verb with a missing or malformed required
  argument (e.g. `pr_get` without `number`) returns a JSON-RPC error / `isError`
  result and does **not** call the client or panic. Asserted offline.
- `R-EIK7-OGEC` ‚Äî The dispatcher reads caller identity from the `X-Owner-Email`
  and `X-Client-Id` request headers (populated by `RequireIdentity`) and carries no
  bearer/token parsing of its own. Asserted offline by setting headers and
  observing them threaded to the log/dispatch.
- `R-EJS4-2851` ‚Äî Each **write** verb emits exactly one structured log line
  carrying the caller's `X-Owner-Email`, the verb, and the target; and the GitHub
  request the write produces carries **no** owner-identifying field (cross-checked
  with D3). A read verb emits no such provenance line. Asserted offline with a
  captured `slog` handler and the fake client's recorded request.
- `R-EL00-FZVQ` ‚Äî The `health` verb returns the health envelope and its success
  reflects a **real** authenticated GitHub call (D2 `R-DMUT-QF4A`): against the live
  suite it succeeds only when the app can authenticate to `@ikigenba`, and reports
  failure loudly (a non-OK envelope) when it cannot. Offline, the auth call is
  faked; the live proof is the `bin/start` + `health` smoke.
- `R-EM7W-TRMF` ‚Äî The `reflection` verb reports an **empty** published-event set
  and **empty** subscriptions (github is neither producer nor consumer). Asserted
  offline against the `Handler`.
- `R-ENFT-7JD4` ‚Äî A `Client` error (`ErrNotFound` / `ErrInvalid` / `ErrAppAuth` /
  transport) from a `tools/call` becomes an `isError` tool result with a
  descriptive message; the transport stays up and returns a well-formed JSON-RPC
  response. Asserted offline with a fake client returning each error.
