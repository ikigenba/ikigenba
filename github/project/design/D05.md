# Decision 5 — The loopback `GET /pr` twin for scripts

## Decision

`scripts` is deterministic and does not speak MCP/JSON-RPC; like its dropbox
consumer (`GET /content`), it reaches a peer through a plain loopback HTTP route.
`internal/gh` exposes one such route — `GET /pr` — the loopback twin of the
`pr_get` MCP verb, returning a pull request as JSON.

It is registered **verbatim, not behind `RequireIdentity`** (mirroring dropbox's
`/content`/`/list`), and **self-guards**: the perimeter is "it is on loopback."
Any request carrying an nginx-injected identity header — `X-Owner-Email` **or**
`X-Forwarded-Proto` (nginx always sets the latter) — was proxied through the
public, authenticated front door, which must never reach this route, so the
handler **404s it**. This is the exact guard copied from
`dropbox/internal/dropbox/content.go`.

```go
// PRHandler returns the loopback GET /pr handler: ?repo=<name>&number=<n> →
// the pull request as JSON. Unauthenticated + loopback-only; self-guards by
// 404ing any request that carries nginx identity headers.
func (c *Client) PRHandler() http.Handler
```

Contract:

- `GET /pr?repo=<name>&number=<n>` on loopback (no identity headers) →
  `200 application/json` with the same `PRDetail` shape `pr_get` returns (built by
  the same `Client.PRGet`).
- Any request with `X-Owner-Email` or `X-Forwarded-Proto` present → `404`, no PR
  data.
- Missing/malformed `repo` or `number` → `400`; a PR that GitHub reports absent
  (`ErrNotFound`) → `404`. No path leaks a `500` for a client/not-found condition
  and no input crashes the handler.

The nginx fragment (D6) additionally makes the public mount return `404` for
`/srv/github/pr` (defence in depth), like the `/feed` block in other services —
but the handler self-guard is the primary defence and stands even if the fragment
is wrong. `github` has no `/feed`, so `/pr` is the only loopback-private route.

## Rejected

- **Behind `RequireIdentity`.** Then `scripts` would have to present a
  dashboard-issued token it does not have; the loopback-only, self-guarded route
  is exactly how dropbox already serves deterministic consumers.
- **Twins for every verb now.** YAGNI: only a proven PR-read path is needed today
  (product). More twins are added per-need, each a thin handler over the same
  `Client`. `GET /pr` establishes the pattern.
- **A separate REST server / port.** The route lives on the same loopback server as
  `/mcp`, distinguished only by being mounted without the identity gate — one
  process, one port (3203).

## Verification

- `R-EPVL-Z2UI` — `GET /pr?repo=<name>&number=<n>` with **no** identity headers
  returns `200` and a JSON body matching the `pr_get` PR shape for that repo/number
  (built via `Client.PRGet`). Asserted offline with a stubbed transport and a
  request carrying no identity headers.
- `R-ER3I-CUL7` — A request to `/pr` carrying `X-Owner-Email` **or**
  `X-Forwarded-Proto` returns `404` and **no** PR data — the public front door can
  never read a PR through this route. Asserted offline for each header.
- `R-ETJB-4E2L` — A missing/malformed `repo` or `number` returns `400`; a PR the
  client reports `ErrNotFound` returns `404`; neither a bad input nor a not-found
  yields a `500` or a panic. Asserted offline across the bad-input and
  not-found cases.
