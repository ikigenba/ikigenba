# Decision 10 — The loopback `GET /token` twin: installation tokens for repos' git plumbing

**Problem.** The repos service performs real git transport (clone, fetch,
push) and needs a short-lived installation token per operation — but this
connector is the **single custodian** of the App private key, and the token
today never leaves `internal/gh` (it exists only as a Bearer header inside
`doOnce`). repos must not hold the key, and no second service may mint
tokens.

**Decision.** Twin the D05 `GET /pr` pattern with a token route.

1. **Exported accessor** (`internal/gh/token_export.go` or alongside
   `client.go`):

```go
func (c *Client) Token(ctx context.Context) (token string, expiresAt time.Time, err error)
```

   Delegates to the existing `tokenSource` cache path (`c.ts.token(ctx,
   false)` semantics) and surfaces the expiry `mintTokenLocked` already
   tracks — same cache, same `tokenSlack` refresh, same single forced
   re-mint on staleness, no second minting path.

2. **Loopback route** (`internal/gh/token_route.go`, registered in the
   composition root beside `GET /pr`):

```go
rt.HandleLoopback("GET /token", client.TokenHandler())
```

   `TokenHandler` does no header inspection (the chassis `LoopbackOnly`
   guard owns that: front-door requests carrying `X-Forwarded-Proto` get a
   bare 404 before the handler runs). Success: `200 application/json`
   `{"token": "...", "expires_at": "<RFC3339>"}`. Failure (`ErrAppAuth`,
   network): `502` with a generic body — never the underlying detail.

3. **Never logged.** The token value follows the private key's existing
   rule: no slog line, no error string, no response other than the JSON body
   may carry it. Consistent with bot-only posture, an installation token
   carries no owner PII by construction.

4. **Defence in depth at the edge:** the D06 nginx fragment gains
   `location = /srv/github/token { return 404; }` beside the existing `/pr`
   block, so the route is closed at nginx even if the chassis guard were
   misconfigured.

Posture note (accepted, not new): `LoopbackOnly` keys on `X-Forwarded-Proto`
only, so any loopback process on the box can obtain a token — the
single-owner-box trust posture every loopback twin already carries.

**Rejected.**
- **Handing repos the App private key** — two key custodians is the exact
  outcome D02 exists to prevent.
- **Scoped/repo-restricted token minting** (`repositories`/`permissions` in
  the mint request) — the runner, not the agent, holds the token; v1's
  branch-namespace enforcement is runner code, and token-side scoping
  returns if agents ever get push access (recorded post-v1 concern).
- **repos calling `POST /mcp` for a token tool** — a token is not an
  owner-facing verb; the loopback twin keeps it off the agent-visible MCP
  surface entirely.
- **Proxying git's HTTP through the connector** — enormous surface (smart
  HTTP protocol) for no custody gain over a short-lived token.

**Verification** (substrate: the offline harness — stubbed `apiBase` mint
flow, the assembled router over `httptest` for guard behavior; log capture
for the never-logged claim).

- R-GSI7-P8NI — `Client.Token` returns the same cached token the header path
  uses until `tokenSlack` before expiry and a fresh one after (clock-driven
  stub), with `expiresAt` equal to the mint response's `expires_at`; a mint
  failure surfaces `ErrAppAuth`.
- R-GTQ4-30E7 — through the assembled router: a loopback `GET /token`
  returns `200` with JSON keys exactly `token` and `expires_at`; the same
  request with `X-Forwarded-Proto: https` gets the guard's bare 404 before
  the handler runs; a mint failure returns `502` with no token material.
- R-GUY0-GS4W — with a capturing slog handler installed, a successful and a
  failing `GET /token` (and a direct `Client.Token` call) emit no log line
  containing the token value; the `502` body does not contain it either.
- R-GW5W-UJVL — the nginx content-assertion test proves
  `location = /srv/github/token { return 404; }` is present in
  `etc/nginx.conf` alongside the existing `/pr` block.
