# Decision 1 — A stateless connector on the appkit chassis

## Decision

`github` is a new deployable service module built on the shared `appkit` chassis,
structurally the same as `gmail` and `dropbox` but **stateless** and **off the
event plane**. `cmd/github/main.go` is a two-line composition root:

```go
func main() { appkit.Main(githubapp.Spec()) }
```

`internal/githubapp.Spec()` returns the service declaration. Unlike `gmail`, it
sets **no** `Feed`, **no** `Producer`, **no** `Workers`, and **no** `Events`:
`github` is neither producer nor consumer, has no background daemon, and publishes
no event. Its shape:

```go
appkit.Spec{
    App:        "github",
    Mount:      "/srv/github/",
    Port:       3203,
    MCP:        true,
    Migrations: db.FS,      // bootstrap-only schema (see below)
    Handlers:   func(rt *appkit.Router) error { /* build client; mount routes */ },
    // no Feed, no Events, no Producer, no Workers
}
```

**Stateless, but the chassis still opens a DB.** `appkit`'s `serve`/`migrate`
verbs unconditionally open and migrate the SQLite database (`appkit/verbs.go`
calls `db.Open` then `db.Migrate`). `github` owns no domain state — no poll
cursor (no daemon), no outbox (no producer), and the installation token lives in
memory. So `internal/db` embeds exactly **one** migration,
`001_schema_migrations.sql` (the bootstrap tracking table every service starts
from), and no domain tables. `internal/db` mirrors `gmail/internal/db` (an
embedded `FS` plus thin `Open`/`Migrate` helpers delegating to `appkit/db`).

**Config boundary.** `Handlers` reads `IKIGENBA_APP_ID`, `IKIGENBA_GITHUB_ORG`,
and `IKIGENBA_APP_PRIVATE_KEY` from the environment once and constructs the
`internal/gh` client (D2/D3), then mounts the MCP surface behind
`rt.RequireIdentity` (D4) and the loopback PR route without a gate (D5). This is
the only site that reads the environment.

Scope of *this* Decision: the module skeleton, the `Spec` with its
non-producer/stateless posture, the composition root, and the bootstrap-only
`internal/db`. The auth source, the REST surface, the tool surface, the loopback
twin, and the landing page are D2–D6. **Wiring the service into the suite (the
root `go.work` `use ./github`, the root `bin/start`, the dashboard's apex nginx
include) is outside this module's tree and therefore outside this `project/`'s
scope** — it is separate suite-level work, not a phase here.

This Decision is **structural** — it fixes where the code lives and the service's
posture (stateless, off the event plane), not a runtime behavior a unit test
asserts. Its proof is the green standalone build plus the two smokes in its plan
phase (the binary exposes the fixed verb set; `manifest`/`schema` reflect a
non-producer with a bootstrap-only schema).

## Rejected

- **Model github on gmail verbatim (producer + poll daemon + outbox).** gmail
  polls Gmail and emits `mail.*`; github's inbound signal already arrives through
  the `webhooks` service, and adding a github producer would duplicate that plane
  and risk echo loops (github acts → emits → a prompt reacts → acts again). Product
  fixes github as neither producer nor consumer, so `Feed`/`Producer`/`Workers`/
  `Events` are all omitted.
- **Skip the database entirely.** Not possible without forking the chassis:
  `appkit` always opens and migrates a DB on `serve`. The cost of the
  bootstrap-only schema is one tracking table and zero domain tables, far cheaper
  than diverging from the chassis.
- **A background token-refresh worker.** Unnecessary: the token source (D2)
  refreshes lazily on demand within an expiry slack, so no `Workers` daemon is
  needed to keep the token warm.

## Verification

*(none — structural.)* This Decision defines placement and posture, not a behavior
a unit test asserts. Its acceptance is mechanical and lives in the plan phase's
done-bar: `GOWORK=off go build ./...` from `github/` exits 0; the built binary's
`manifest` verb reports `MCP=true` with **no** `FEED=` line and no outbox
retention keys (github is not a producer); and `internal/db` contains exactly one
`.sql` migration file. The service's runtime behavior carries its ids under D2–D6.
