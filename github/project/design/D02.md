# Decision 2 — App authentication: the installation-token source

## Decision

`internal/gh` holds a `tokenSource` that turns the GitHub App credentials into a
short-lived **installation access token**, mirroring the structure of `gmail`'s
`tokenSource` but doing the GitHub App flow instead of a refresh-token exchange.
It is safe for concurrent use; a mutex serializes refreshes. Nothing here touches
disk or logs a credential.

**The flow (all stdlib):**

1. **Mint an app JWT.** Build a JWT with header `{"alg":"RS256","typ":"JWT"}` and
   claims `{"iat": now-60s, "exp": now+600s, "iss": <APP_ID>}`, sign it RS256 over
   `base64url(header) + "." + base64url(claims)` with the RSA private key parsed
   from `IKIGENBA_APP_PRIVATE_KEY` (`pem.Decode` → `x509.ParsePKCS1PrivateKey` or
   `ParsePKCS8PrivateKey`; `rsa.SignPKCS1v15` with `crypto.SHA256`). `iat` is
   backdated 60s for clock skew; `exp` is ≤ 600s ahead (GitHub's hard ceiling).
2. **Resolve the installation.** With the app JWT as `Authorization: Bearer`, GET
   `https://api.github.com/orgs/<ORG>/installation` → the org's installation
   object, whose `id` is the installation id. Resolved lazily and cached (the org's
   installation id is stable for the process lifetime).
3. **Mint the installation token.** With the app JWT as bearer, POST
   `https://api.github.com/app/installations/<id>/access_tokens` → `{"token":...,
   "expires_at": "<RFC3339>"}`. That **installation token** — not the app JWT — is
   the bearer for every REST call in D3.
4. **Cache + refresh.** Cache the installation token and its `expires_at`; reuse it
   while more than a slack (60s) from expiry; re-mint when within the slack. A
   `401` from a REST call forces a one-time re-mint and retry.

```go
type tokenSource struct {
    appID      string
    org        string
    signer     *rsa.PrivateKey
    httpClient *http.Client

    mu       sync.Mutex
    instID   string    // resolved once, cached
    token    string    // installation token
    expiry   time.Time // from GitHub's expires_at
}

// token returns a valid installation token, minting/refreshing as needed.
// force=true (used on a 401) discards the cache and re-mints unconditionally.
func (t *tokenSource) token(ctx context.Context, force bool) (string, error)

var ErrAppAuth = errors.New("github: app authentication failed")
```

Illustrative host vars (kept as `var`, not `const`, so tests redirect them at a
stub; production uses the real host):

```go
var apiBase = "https://api.github.com"
```

**Fail loudly.** A `401`/`404` on the installation-resolve or token-mint call
means the app id, private key, or installation is wrong; the source returns
`ErrAppAuth` (wrapping GitHub's message, never the key) and does **not** spin or
retry — the credentials need a human fix. This matches `gmail`'s `invalid_grant`
posture.

## Rejected

- **A JWT library (`golang-jwt`).** The app JWT is a fixed two-segment RS256 sign
  over tiny JSON; stdlib `crypto/rsa` + `crypto/sha256` + `encoding/base64` do it
  in a few lines. The suite carries zero general-purpose SDKs; adding one for this
  is unjustified.
- **A statically configured installation id.** Resolving it at startup from the
  org name via `GET /orgs/<org>/installation` needs no extra config and cannot
  drift out of sync with the org; a pinned id would be one more thing to keep
  correct.
- **`go-github`'s transport.** Same dependency-posture reason; the flow is small
  and we implement only the two auth calls plus D3's endpoints.
- **A background refresher.** Lazy refresh within the slack keeps the token warm
  without a daemon (D1 has no `Workers`).

## Verification

- `R-DLMX-CNDL` — The minted app JWT has header `alg=RS256` and claims `iss` equal
  to the configured app id, `exp` no more than 600s after `iat`, and `iat`
  backdated relative to now; its signature **verifies against the public key
  derived from the configured private key** (a wrong key or tampered claim fails
  verification). Asserted offline by signing with a test RSA key and verifying with
  `rsa.VerifyPKCS1v15`.
- `R-DMUT-QF4A` — **(live substrate: real GitHub via `bin/start`.)** Against the
  real `@ikigenba` GitHub App, resolving the installation and minting a token
  succeeds and yields a non-empty installation token; this is the observable that
  the app id, private key, and org installation are all valid. Driven end-to-end by
  the `health` verb (D4) against the running suite — the proof that cannot rest on
  a stub. Failure (wrong key/app) surfaces as `ErrAppAuth`, not a hang or empty
  success.
- `R-DO2Q-46UZ` — The bearer used on D3 REST calls is the **installation token**
  returned by `/access_tokens`, not the app JWT: given a stub that returns a
  distinguishable installation token, a subsequent REST request carries exactly
  that token in its `Authorization` header. Asserted offline against the stub.
- `R-DPAM-HYLO` — A cached token whose `expires_at` is more than the 60s slack away
  is reused without a re-mint; a token within the slack of `expires_at` triggers a
  re-mint. The expiry honored is GitHub's returned `expires_at`, not a hardcoded
  hour. Asserted offline by advancing a test clock across the slack boundary and
  counting stub mint calls.
- `R-DQII-VQCD` — A `401` from a REST call forces exactly **one** token re-mint and
  a single retry of the call; a second consecutive `401` returns an error rather
  than looping. Asserted offline with a stub that 401s then succeeds (retry wins)
  and a stub that always 401s (bounded, returns error).
- `R-DRQF-9I32` — A `401`/`404` on the installation-resolve or token-mint call
  returns `ErrAppAuth` wrapping GitHub's status, performs no retry loop, and never
  includes the private key material in the error. Asserted offline against a stub
  that rejects the mint.
