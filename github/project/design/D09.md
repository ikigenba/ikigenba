# Decision 9 — Issue-execution support verbs: `pr_create`, `issue_comments`, `label_add`, `label_remove`

**Problem.** The repos service's runner drives its issue protocol through
this connector, and four contracts are missing: creating a pull request,
reading an issue's comment thread, and mutating labels **atomically** — the
existing `issue_update` label path is a whole-set `PATCH` replace, race-prone
for a label state machine running concurrently with human labeling.

**Decision.** Four additions, each the established thin shape (D3 client
method + D4/D8 MCP verb), org-scoped, bot-attributed.

1. **Client methods** (`internal/gh/client.go`):

```go
func (c *Client) PRCreate(ctx context.Context, repo, title, head, base, body string) (PR, error)
    // POST /repos/{org}/{repo}/pulls  {title, head, base, body} → existing PR type
func (c *Client) IssueComments(ctx context.Context, repo string, number int) ([]Comment, error)
    // GET /repos/{org}/{repo}/issues/{number}/comments → existing Comment type (ascending)
func (c *Client) LabelAdd(ctx context.Context, repo string, number int, labels []string) ([]Label, error)
    // POST /repos/{org}/{repo}/issues/{number}/labels  {labels: [...]}
func (c *Client) LabelRemove(ctx context.Context, repo string, number int, label string) error
    // DELETE /repos/{org}/{repo}/issues/{number}/labels/{label}
```

   `LabelRemove` needs the one missing HTTP helper: a `c.delete(ctx, parts,
   out)` wrapper over the same `doJSON` path `get/post/put/patch` use (same
   headers, same 401-retry, same `githubStatusError` mapping — a 404, e.g.
   removing a label the issue doesn't carry, maps to `ErrNotFound`; a 422
   head-branch problem on `PRCreate` maps to `ErrInvalid`).

2. **MCP verbs** (`internal/mcp/tools.go`, bare names, both schemas each,
   `decodeAndValidate` inputs, `clientResult`/`codeFor` outputs, and the
   `GitHubClient` interface extended by the four methods):
   - `pr_create` `{repo, title, head, base, body?}` → the PR view.
   - `issue_comments` `{repo, number}` → `{items: [comment]}`.
   - `label_add` `{repo, number, labels: [string, …]}` → `{labels: [label]}`.
   - `label_remove` `{repo, number, label}` → `{removed: true}`.

3. **Constraints carried forward, unchanged:** the three writes each emit
   exactly one `logWrite` slog line (owner_email/client_id/verb/target);
   **no owner-identifying field ever enters a request body** (bot-only
   attribution, D03) — `pr_create` sends only `{title, head, base, body}`,
   the label verbs only the label names. `issue_comments` is a read: no
   `logWrite`.

**Rejected.**
- **Extending `issue_update` for atomic label ops** — a replace and an
  atomic mutation are different contracts; overloading one verb with a mode
  flag hides the race the atomic endpoints exist to close.
- **A combined `label_swap` verb** — GitHub has no atomic swap endpoint; a
  connector verb pretending otherwise would paper over the two-call reality
  the caller must reason about.
- **Returning a bespoke PR-create shape** — the existing `PR` type is the
  connector's PR view; one shape per noun.

**Verification** (substrate: the established offline harness — injected
`http.RoundTripper`/`apiBase` stub asserting exact method, path, headers,
and body; the MCP layer through the assembled handler with the fake
`GitHubClient`).

- R-GJYX-0UGN — `PRCreate` issues `POST /repos/{org}/{repo}/pulls` with body
  exactly `{title, head, base, body}` (no author/committer/owner field) and
  decodes the created PR; a 422 maps to `ErrInvalid`.
- R-F70H-NRU9 — `IssueComments` issues
  `GET /repos/{org}/{repo}/issues/{n}/comments` and decodes the comment list
  in order; a 404 maps to `ErrNotFound`.
- R-GL6T-EM7C — `LabelAdd` issues `POST …/issues/{n}/labels` with body
  `{labels: [...]}` and decodes the returned label set.
- R-GMEP-SDY1 — `LabelRemove` issues `DELETE …/issues/{n}/labels/{label}`
  through the new `delete` helper with the standard headers and 401-retry;
  a 404 maps to `ErrNotFound`.
- R-GNMM-65OQ — the `pr_create` tool validates its four required strings
  (failure → `validation`, client never called), returns the PR as
  `structuredContent` on success, emits exactly one `logWrite` line carrying
  the caller identity, and maps client errors through `codeFor`.
- R-F88E-1JKY — the `issue_comments` tool validates `{repo, number}`,
  returns `{items}` wrapped, and emits **no** `logWrite` line.
- R-GOUI-JXFF — the `label_add` tool validates a non-empty `labels` array,
  returns the label set, and emits exactly one `logWrite` line.
- R-GQ2E-XP64 — the `label_remove` tool validates its inputs, returns
  `{removed: true}`, emits exactly one `logWrite` line, and surfaces the
  not-on-issue 404 as code `not_found`.
