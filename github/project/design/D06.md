# Decision 6 — The landing page and nginx fragment

## Decision

`github` serves the suite's uniform v1 landing page and ships its path-routed
nginx fragment, mirroring `gmail`'s landing (`internal/web`) but with **no**
consent flow and **no** OAuth routes (the GitHub App is pre-installed; there is no
per-user consent).

**`internal/web`** is a copy of the established pattern: an embedded
`landing.html` template rendered with the service name and running version, plus
an embedded `static/` tree (the Carbon `tokens.css` and the woff2 fonts) served
from `/static/`. Wiring in `Handlers` (D1):

```go
rt.Handle("GET /{$}", web.LandingHandler(rt.Service(), rt.Version()))
rt.Handle("GET /static/", http.StripPrefix("/static/", web.StaticHandler()))
```

The page shows exactly the **service name (`github`) and running version** on the
suite design system — nothing interactive, no GitHub data.

**`etc/nginx.conf`** is the path-routed location fragment (not a vhost), mirroring
`gmail/etc/nginx.conf` with literal `MOUNT=/srv/github/` and port `3203`:

- `= /srv/github/.well-known/oauth-protected-resource` → open PRM bootstrap (no
  auth_request).
- `= /srv/github/` → bare mount root, gated by `auth_request /_session-authn`
  (dashboard browser session), proxied to `/` — the landing page.
- `/srv/github/static/` → session-gated static assets.
- `/srv/github/` (prefix) → every other path gated by `auth_request /_authn` (the
  bearer gate) with the identity-header hygiene and the 429-faithful `error_page`
  block, proxied to loopback `:3203` — this covers `/mcp`.
- `= /srv/github/pr` → `return 404`: the loopback-private twin (D5) must never be
  reachable through the public front door (defence in depth; the handler
  self-guard is primary).

github has **no `/feed`**, so — unlike the producer services — the fragment
carries no `/srv/github/feed` block; nothing under the mount serves a feed.

**`etc/manifest.env`** declares `APP=github`, `MOUNT=/srv/github/`, `PORT=3203`,
`MCP=true`, `DEFAULT=false`, and **no** `FEED=` line and no outbox retention keys
(non-producer, D1). `etc/deploy.env` mirrors the other connectors' workstation
routing.

## Rejected

- **A consent binary (`cmd/consent`) and OAuth callback routes.** gmail needs them
  for per-user Google OAuth; the GitHub App is installed once at the org, so there
  is no consent step and no callback — omitted entirely.
- **A `/feed` block in the fragment.** github publishes no events and mounts no
  feed; a block would guard a route that does not exist.
- **A shared landing handler.** The suite convention is each app owns its landing
  page, template, and embedded assets under its own tree; github embeds its own
  copy of `tokens.css` and the fonts.

## Verification

- `R-EVZ3-VXJZ` — `GET /` returns `200 text/html` containing the service name
  `github` and the running version (from `rt.Service()` / `rt.Version()`).
  Asserted offline against `LandingHandler`.
- `R-EX70-9PAO` — `GET /static/tokens.css` returns the embedded stylesheet with
  `Content-Type: text/css`, a `.woff2` asset returns `font/woff2`, and a
  traversal path (`../`) or unknown asset returns `404`. Asserted offline against
  `StaticHandler`.
- `R-EYEW-NH1D` — The `etc/nginx.conf` fragment session-gates the bare mount root
  (`auth_request /_session-authn` on `= /srv/github/`), bearer-gates the mount
  prefix (`auth_request /_authn` on `/srv/github/`), returns `404` for
  `= /srv/github/pr`, opens the PRM well-known, and contains **no**
  `/srv/github/feed` block. Asserted by a fragment parse/grep test
  (`internal/web/nginx_test.go`, like gmail's) over the shipped file.
