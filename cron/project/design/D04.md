# Decision 4 — nginx fragment: the session-gated `= /srv/cron/` location and the full owner-identity forwarding contract

**Decision.** cron's nginx fragment (`cron/etc/nginx.conf`) gains **one new
location** — an **exact-match** location for the bare mount root that
session-gates the human landing page and proxies it to the service's root — and,
on **every** location that forwards identity, captures and forwards the **four**
owner headers the dashboard's introspection hooks now emit on every allow
(`X-Owner-Id`, `X-Owner-Email`, `X-Owner-Name`, `X-Owner-Picture`), not just
`X-Owner-Email`. The new exact-match location sits alongside (does not replace)
the existing locations:

```
# Landing page — human web UI, gated by the dashboard browser SESSION (cookie),
# validated by the dashboard-owned /_session-authn internal location (a sibling of
# /_authn). Exact-match `=` so it never shadows the gated `/srv/cron/` prefix below
# or the MCP endpoint. A failed session check yields 401.
location = /srv/cron/ {
    auth_request /_session-authn;
    auth_request_set $cron_session_owner         $upstream_http_x_owner_email;
    auth_request_set $cron_session_owner_id      $upstream_http_x_owner_id;
    auth_request_set $cron_session_owner_name    $upstream_http_x_owner_name;
    auth_request_set $cron_session_owner_picture $upstream_http_x_owner_picture;
    proxy_set_header X-Owner-Email   $cron_session_owner;
    proxy_set_header X-Owner-Id      $cron_session_owner_id;
    proxy_set_header X-Owner-Name    $cron_session_owner_name;
    proxy_set_header X-Owner-Picture $cron_session_owner_picture;
    proxy_pass http://127.0.0.1:3005/;   # trailing slash → upstream sees "/"
    proxy_set_header Host $host;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_http_version 1.1;
}
```

**The four-header owner-identity forwarding contract.** The dashboard's
introspection endpoints (`/_authn` bearer/PAT hook and `/_session-authn` session
hook) now emit four owner headers on every allow, fail-closed: `X-Owner-Id` (the
durable owner key, never empty), `X-Owner-Email` (display/audit, never empty),
`X-Owner-Name` (percent-encoded, may be empty), and `X-Owner-Picture`
(percent-encoded, may be empty). Every location in cron's fragment that forwards
identity today must capture (`auth_request_set`) and forward
(`proxy_set_header`) **all four** — the two that forward identity are:

- the **session-gated** exact landing location `location = /srv/cron/` (above),
  using the `$cron_session_owner{,_id,_name,_picture}` variables;
- the **bearer-gated** service prefix `location /srv/cron/` (item 3 below),
  using the `$cron_owner{,_id,_name,_picture}` variables, plus its existing
  `X-Client-Id` (`$cron_client`), unchanged.

Variable names stay service-prefixed and unique (all fragments are included into
one apex `server` block, so `$cron_*` names must not collide with another
service's). The **identity-hygiene** property extends to all four: a single
`proxy_set_header` per header name, sourced from the auth-subrequest variable,
**replaces** any client-smuggled inbound header of that name — nginx does not
pass inbound client headers as proxy headers unless explicitly set, and the last
`proxy_set_header` for a name wins, so a client cannot smuggle any owner header.
The **static** asset tier (`location /srv/cron/static/`) forwards **no** identity
today and is left untouched; the PRM bootstrap, the `/feed` 404, and the 429
re-emit named location likewise forward no identity and are unchanged.

Downstream consumption of `X-Owner-Id` by the cron binary is **out of scope**
here (a later suite-wide conversion); this Decision governs fragment forwarding
and its content-assertion tests only.

**How it coexists with the existing fragment.** cron's fragment already has,
verbatim, the same three machine locations the crm template carries:

1. `location = /srv/cron/.well-known/oauth-protected-resource` — unauthenticated
   PRM bootstrap (exact-match).
2. `location = /srv/cron/feed { return 404; }` — the loopback feed is never
   reachable through nginx (exact-match).
3. `location /srv/cron/ { auth_request /_authn; … }` — the bearer-gated **prefix**
   for the MCP endpoint and everything else under the mount, including its
   four-owner-header capture/forward (`$cron_owner{,_id,_name,_picture}` from the
   `x_owner_email`/`x_owner_id`/`x_owner_name`/`x_owner_picture` subrequest
   headers) plus its `$cron_client`/`X-Client-Id` capture, and the
   `error_page 500 = @cron_authn_500` rate-limit re-emit (with the
   `@cron_authn_500` named location that faithfully re-surfaces a `429`).

nginx longest-prefix/exact-match rules make an **exact-match `=` location win
over a prefix location** for that one path regardless of source order. So the new
`= /srv/cron/` exact match wins for the **bare root only**, while the
`/srv/cron/` prefix (item 3) still serves `/srv/cron/mcp` and every other sub-path
under the mount. The two PRM/feed exact matches are unaffected (different exact
paths). Nothing in the existing fragment is removed or rewritten — the change is
**purely additive**.

**Why `/_session-authn`, not `/_authn`.** A browser carries the
`dashboard_session` cookie, not a bearer token, so the bearer gate (`/_authn`)
would refuse it. `/_session-authn` is the dashboard-owned, loopback-only internal
location (defined in `dashboard/etc/nginx.conf` as
`proxy_pass …/internal/session-authn`) that validates the cookie against the
web-session store and returns `200` + `X-Owner-Email` on success or `401` on
failure. This is the **same** mechanism `sites` already uses for its private
static tier (`sites/etc/nginx.conf` `location /srv/sites/private/`). The gate is
deliberately **coarse**: any logged-in dashboard user passes; there is no
per-resource check — fine for a name+version page.

**Why the simple form (no 429 re-emit).** The bearer prefix block carries the
`$authn_status`/`error_page 500 = @cron_authn_500` machinery because token
introspection can return `429` (rate-limited) which `auth_request` collapses to a
synthetic `500`. The session check is a cookie lookup, not a rate-budgeted token
introspection, so the landing location does not need the 429 re-emit dance — it
maps the session subrequest's allow/deny directly (`200`→serve, `401`→401),
matching the `sites` private-tier location, which likewise omits it.

**Templating.** The loopback port is a **literal** in the fragment (`PORT=3005`,
cron's fixed registry port), so nothing substitutes it at install — `opsctl setup
cron` on the box / `nginx/run` in dev drop the fragment verbatim, exactly like the
existing locations. The fragment is **not** a vhost — no
`server{}`, no `listen`, no cert; it is dropped at
`/etc/nginx/conf.d/locations/cron.conf` and included by the dashboard's apex
server block, which is also where `/_session-authn` is defined.

**Rejected.**
- **Gating the landing page with `/_authn`** — a browser has no bearer token; it
  would always 401. Wrong audience.
- **Making the landing page fully public (no auth_request)** — the brief forbids
  it; a logged-out browser must get `401`. The page is gated, just coarsely.
- **A prefix `location /srv/cron/` for the page** — would collide with the
  existing bearer prefix; the exact-match `=` is what cleanly carves the bare root
  out of the prefix without touching the rest.
- **Adding the `@cron_authn_500` 429 re-emit to the session location** — the
  session check is not rate-budgeted; the extra machinery is dead weight (the
  `sites` private tier omits it for the same reason).
- **Forwarding only `X-Owner-Email` (the pre-four-header shape)** — the dashboard
  now emits `X-Owner-Id`/`X-Owner-Name`/`X-Owner-Picture` too; capturing only the
  email drops the durable owner key and display fields at nginx, so every
  identity-forwarding location must carry all four.
- **Adding the four owner headers to the static asset tier or the PRM/feed/429
  locations** — those forward no identity today; widening them would leak owner
  headers onto surfaces that deliberately carry none. The four-header treatment is
  confined to the two locations that already forward identity.

**Verification.** (Proven by a test that reads `cron/etc/nginx.conf` from disk and
asserts over its content — see design's Testing strategy.)
- R-NGNX-3B6C — `cron/etc/nginx.conf` contains an **exact-match** location for the
  bare mount root: a `location = /srv/cron/ {` block (distinct from the prefix
  `location /srv/cron/ {`).
- R-NGNX-5D8E — that exact-match block gates with `auth_request /_session-authn`
  (the cookie/session validator) and **not** `auth_request /_authn` (the bearer
  validator).
- R-NGNX-7F1G — that block proxies to the loopback upstream **root**
  (`proxy_pass http://127.0.0.1:3005/;` — trailing slash), so the upstream
  sees `/`.
- R-NGNX-9H3J — the pre-existing locations are retained alongside it: the
  bearer-gated `location /srv/cron/ {` prefix with `auth_request /_authn`, the
  `location = /srv/cron/feed { return 404; }`, and the
  `location = /srv/cron/.well-known/oauth-protected-resource` PRM bootstrap all
  still appear in the file.
- R-8ALX-VK6V — the **session-gated** exact landing location `= /srv/cron/`
  captures and forwards all four owner headers: it contains
  `auth_request_set $cron_session_owner $upstream_http_x_owner_email`,
  `auth_request_set $cron_session_owner_id $upstream_http_x_owner_id`,
  `auth_request_set $cron_session_owner_name $upstream_http_x_owner_name`,
  `auth_request_set $cron_session_owner_picture $upstream_http_x_owner_picture`,
  and a matching `proxy_set_header` for each of `X-Owner-Email`
  (`$cron_session_owner`), `X-Owner-Id` (`$cron_session_owner_id`), `X-Owner-Name`
  (`$cron_session_owner_name`), and `X-Owner-Picture`
  (`$cron_session_owner_picture`).
- R-8BTU-9BXK — the **bearer-gated** service prefix `location /srv/cron/`
  captures and forwards all four owner headers with the `$cron_owner`-family
  variables: it contains
  `auth_request_set $cron_owner $upstream_http_x_owner_email`,
  `auth_request_set $cron_owner_id $upstream_http_x_owner_id`,
  `auth_request_set $cron_owner_name $upstream_http_x_owner_name`,
  `auth_request_set $cron_owner_picture $upstream_http_x_owner_picture`, and a
  matching `proxy_set_header` for each of `X-Owner-Email` (`$cron_owner`),
  `X-Owner-Id` (`$cron_owner_id`), `X-Owner-Name` (`$cron_owner_name`), and
  `X-Owner-Picture` (`$cron_owner_picture`), while its existing
  `proxy_set_header X-Client-Id $cron_client` is retained.
