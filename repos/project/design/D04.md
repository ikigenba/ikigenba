# Decision 4 — Repo lifecycle & git custody

**Decision.** `internal/repos/git.go` owns every git invocation in the
service; nothing else shells out to git. It wraps the real `git` binary
(missing tooling fails loudly):

```go
type Git struct {
    stateRoot string            // <state>/repos
    tokens    TokenSource       // github loopback GET /token (below)
    runner    func(ctx context.Context, dir string, env []string, args ...string) ([]byte, error)
}
func (g *Git) Clone(ctx, cloneURL, name string) error            // → state/repos/<name>
func (g *Git) Freshen(ctx, name, defaultBranch string) error     // fetch + reset --hard origin/<default>
func (g *Git) WorktreeAdd(ctx, name, branch, path, defaultBranch string) error
func (g *Git) WorktreeRemove(ctx, name, path string) error
```

`WorktreeAdd` runs `git worktree add` with the git working directory set to the
canonical clone (`state/repos/<name>`), so a **relative** `path` would be
resolved against the clone, not the process cwd — nesting the worktree under
the clone and desynchronizing it from every later git call (`Push`,
`HasCommits`, the check runner) that resolves the same string against the
process cwd. The `path` handed to `WorktreeAdd` is therefore always
**absolute**: it derives from the absolute state root (D9), so git resolves it
to one location regardless of which directory the invocation runs in.

The absolute-root guarantee is a **testable seam**, not inline composition-root
code — a package-level resolver the composition root calls and a test can drive
directly:

```go
// ResolveStateRoot reads REPOS_STATE_DIR (default "state") and returns it as an
// absolute path (via filepath.Abs against the process cwd). The composition
// root calls this once and hands the result to NewGit (joined with "repos")
// and to the runner; nothing below the composition root re-reads the env or
// keeps a relative root.
func ResolveStateRoot(getenv func(string) string) (string, error)
```

```go
func (g *Git) Push(ctx, worktreePath, branch string) error
func (g *Git) BranchExists(ctx, name, branch string) (bool, error) // local + ls-remote
```

**Credential custody.** The github service is the single custodian of the App
key; repos never sees it. `TokenSource` fetches a short-lived installation
token from github's loopback `GET /token` (github D10) at
`registry.BaseURL("github")`, caching it until near expiry. Authenticated
operations (`Clone`, `Freshen`, `Push`) inject it **per invocation** as
`-c http.extraHeader=Authorization: Basic <b64(x-access-token:TOKEN)>` —
the token is never written to `.git/config`, any file under the clone or
worktree, or a persistent environment variable, so the session agent's
workspace holds no credential material. (On this single-owner box the
loopback twin is reachable by any local process — the same accepted posture
as every loopback surface; custody here is defense in depth, not a second
trust boundary.)

**Lifecycle.**
- **Lazy provisioning:** intake (D3) calls `Service.EnsureRepo(ctx,
  RepoFacts{Name, CloneURL, DefaultBranch, OwnerID, OwnerEmail})` — inserts the
  row (recording `owner_id` as the scoping/provenance key and `owner_email` as
  the display snapshot, both from the webhooks payload) and clones on first
  contact; idempotent when the row exists.
- **Explicit onboarding:** the MCP `clone` verb (D7) takes a repo `name`,
  derives `https://github.com/<REPOS_GITHUB_ORG>/<name>.git`, and calls the
  same `EnsureRepo` with the caller's `owner_id`/`owner_email` from
  `server.Identity`; a repo that already exists is `conflict`.
- **v1 is clone-only:** every repo has a GitHub remote; there is no `init`.
- **Before each session:** `Freshen` pins the canonical clone to
  `origin/<default_branch>`; sessions branch from that tip (D5). The
  canonical tree is never an agent's cwd and never has a session branch
  checked out.
- **Removal:** the MCP `delete` verb removes the repo row and
  `state/repos/<name>` entirely; sessions and transcripts survive (D9).

**Rejected.**
- **Credential helper baked into the clone's git config** (the pre-v1
  sketch) — a worktree shares the canonical repo's config, so the agent
  could read or invoke the helper from inside its session; per-invocation
  header injection keeps credential material out of everything the agent
  can reach.
- **Tokens in remote URLs** (`https://x-access-token:<tok>@github…`) — the
  token lands in `.git/config` and in error output; rejected for the same
  reason.
- **go-git instead of the git binary** — worktrees, credential-header
  injection, and exact CLI parity are what the design leans on; the binary
  is guaranteed on the box and is what sessions themselves use.
- **Deleting a repo tombstones the row but keeps the clone** — the clone is
  recoverable from GitHub at any time (clone-only v1); keeping dead trees
  contradicts "delete means the disk is reclaimed".

**Verification** (real substrate: the real `git` binary against local bare
fixture remotes in `t.TempDir()`; the token source as a recording `httptest`
stub).

- R-EXFQ-NUVB — `EnsureRepo` for an unseen repo creates the row and a
  canonical clone at `state/repos/<name>` whose `origin` is the given URL; a
  second call with the same facts changes nothing and errors nothing.
- R-EYNN-1MM0 — the MCP-facing path: `Service.CloneRepo` for `"acme"` under
  `REPOS_GITHUB_ORG=ikigenba` records `clone_url`
  `https://github.com/ikigenba/acme.git` and stamps the caller's `owner_id`
  and `owner_email` on the row; cloning an already-tracked name returns the
  domain conflict sentinel.
- R-EZVJ-FECP — after the fixture remote's default branch advances,
  `Freshen` moves the canonical clone's `origin/<default>` and local default
  ref to the remote tip (asserted by SHA equality), even when the canonical
  clone had drifted.
- R-F13F-T63E — an authenticated `Push` against a stub that requires the
  `Authorization` header succeeds and, afterward, no file under the canonical
  clone or the worktree (including `.git/config` and
  `.git/worktrees/**/config`) contains the token string; the `TokenSource`
  was consulted (recorded call) rather than any ambient credential.
- R-F3J8-KPKS — end-to-end over the real plumbing: clone a fixture remote,
  freshen, add a worktree on a new branch, commit in the worktree, push, and
  observe the commit on the bare remote's branch; remove the worktree and
  the canonical clone remains intact.
- R-C9CO-ODYU — `ResolveStateRoot` turns a **relative** configured root into
  an absolute one, and that absolute root keeps worktree paths cwd-independent
  end to end: (a) `ResolveStateRoot(getenv)` with `getenv` returning `""`
  (default `state`) returns an absolute path (`filepath.IsAbs` true) equal to
  `<cwd>/state`; (b) driving the real git plumbing from the resolved root —
  clone a fixture remote, `WorktreeAdd`, then `HasCommits` (which runs git with
  its working directory set to the returned worktree path) — succeeds even when
  the process working directory is changed to a temp dir **different from the
  canonical clone** between creation and inspection. Feeding the *unresolved
  relative* root instead reproduces the regression (worktree nested under
  `state/repos/<name>/…`, inspection failing under the process cwd), so the
  resolver is what the assertion turns on.
