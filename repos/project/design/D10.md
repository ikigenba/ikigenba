# Decision 10 — nginx fragment & the canonical landing page

**Decision.** repos ships the suite's standard four-tier location fragment at
`etc/nginx.conf` (a location fragment, no `server{}`/cert/listen; `MOUNT`
`/srv/repos/` and `PORT` `3007` literal in the file, dev port-rewritten by
`nginx/run`), modeled verbatim on gmail's:

1. `location = /srv/repos/.well-known/oauth-protected-resource` — open (PRM
   bootstrap, no `auth_request`).
2. `location = /srv/repos/feed { return 404; }` — the event plane is
   loopback-only and never reachable through nginx. (v1 has no other
   loopback-only route.)
3. Session-gated human tier: `location = /srv/repos/` and
   `location /srv/repos/static/` — `auth_request /_session-authn`,
   `error_page 401 = @login_bounce`. The exact landing location
   (`= /srv/repos/`) forwards the owner identity from the session subrequest;
   the static-assets location (`/srv/repos/static/`) forwards **no** identity
   and is unchanged by the header contract below.
4. Bearer tier: `location /srv/repos/` (prefix) — `auth_request /_authn`,
   identity headers **replaced** from subrequest vars, proxy to
   `http://127.0.0.1:3007/` (trailing slash strips the mount), with the
   standard 429-recovery block.

**Owner-identity header contract.** The dashboard introspection hooks
(`/_authn`, `/_session-authn`) now emit four owner headers on every allow,
fail-closed: `X-Owner-Id` (durable key, never empty), `X-Owner-Email`
(display/audit, never empty), `X-Owner-Name` (percent-encoded, may be empty),
and `X-Owner-Picture` (percent-encoded, may be empty). Every location that
forwards identity today captures all four with `auth_request_set` and replaces
them on the upstream request with a single `proxy_set_header` each (identity
hygiene: one set per name overrides any client-smuggled inbound header). The
bearer tier additionally forwards `X-Client-Id` from `$repos_client`,
unchanged. Variables are service-prefixed so the fragment stays unique inside
the shared apex server block:

- Bearer prefix (`location /srv/repos/`): `$repos_owner` (email, existing),
  `$repos_owner_id`, `$repos_owner_name`, `$repos_owner_picture` — set as
  `X-Owner-Email` / `X-Owner-Id` / `X-Owner-Name` / `X-Owner-Picture`; plus
  `$repos_client` → `X-Client-Id` (unchanged).
- Session landing (`location = /srv/repos/`): `$repos_session_owner` (email,
  existing), `$repos_session_owner_id`, `$repos_session_owner_name`,
  `$repos_session_owner_picture` — set as the same four `X-Owner-*` headers.

Downstream consumption of `X-Owner-Id` by the repos binary is out of scope for
this pass; this contract governs the fragment's capture-and-forward only.

Landing page: `Spec.WWW: true`; `share/www/landing.html` is the canonical
cron/crm Carbon layout (eyebrow, display service name, one-line lead, the
Service/Version/API panel) rendered by `rt.WWW()` at `GET /{$}` with
`{{.Service}}`/`{{.Version}}`; `share/www/static/` carries `tokens.css` and
the vendored fonts. Only this service's three text fields differ from the
canonical template.

**Rejected.**
- **Skipping the landing page** — every suite app serves the uniform page;
  its absence reads as a broken deploy.
- **Exposing session output over a public route** — all reads go through the
  bearer-gated `/mcp`; no bespoke HTTP read surface.

**Verification.**
- R-G1OF-AAC8 — a content-assertion `nginx_test.go` over the shipped
  `etc/nginx.conf` proves: the PRM exact location with no `auth_request`;
  `= /srv/repos/feed` returning 404; both session-gated locations with
  `error_page 401 = @login_bounce`; the bearer prefix tier replacing the
  identity headers; and the `127.0.0.1:3007` upstream.
- R-UZVS-S08C — `nginx_test.go` proves the session landing block
  (`location = /srv/repos/`) captures all four owner headers from the
  `/_session-authn` subrequest and forwards them, each via its own
  service-prefixed variable: `auth_request_set $repos_session_owner
  $upstream_http_x_owner_email` with `proxy_set_header X-Owner-Email
  $repos_session_owner`, and likewise `$repos_session_owner_id` →
  `X-Owner-Id`, `$repos_session_owner_name` → `X-Owner-Name`, and
  `$repos_session_owner_picture` → `X-Owner-Picture`. A block forwarding only
  `X-Owner-Email` (today's behavior) fails.
- R-V13P-5RZ1 — `nginx_test.go` proves the bearer prefix block
  (`location /srv/repos/`) captures all four owner headers from the `/_authn`
  subrequest and forwards them, each via its own service-prefixed variable:
  `$repos_owner` → `X-Owner-Email`, `$repos_owner_id` → `X-Owner-Id`,
  `$repos_owner_name` → `X-Owner-Name`, `$repos_owner_picture` →
  `X-Owner-Picture`, while `X-Client-Id` from `$repos_client` stays forwarded
  unchanged. A block forwarding only `X-Owner-Email` + `X-Client-Id` (today's
  behavior) fails.
- R-G2WB-O22X — `GET /{$}` through the assembled handler renders the landing
  with service name `repos` and the running version, byte-parity with the
  canonical layout apart from the three service-specific text fields
  (mirroring the crm-clone web test set).
- R-G448-1TTM — `GET /static/tokens.css` (and one vendored font) serve with
  correct content types from `share/www/static/` through the chassis.
