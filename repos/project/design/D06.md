# Decision 6 — The issue protocol: labels, the check gate, and runner-side GitHub I/O

**Decision.** Everything GitHub-facing is **runner-side, deterministic Go**
(`internal/repos/protocol.go`); the agent only does the work. The protocol
drives the github service — never GitHub directly — through a thin loopback
client:

```go
type GitHubPeer struct {                    // internal/repos/ghpeer.go
    baseURL string                          // registry.BaseURL("github")
    http    *http.Client
}
// one method per consumed verb, each a JSON-RPC tools/call POST to /mcp with
// asserted identity headers X-Owner-Email: <session owner>, X-Client-Id: "repos:<session id>"
func (p *GitHubPeer) IssueGet(ctx, id Identity, repo string, n int) (Issue, error)
func (p *GitHubPeer) IssueComments(ctx, id Identity, repo string, n int) ([]Comment, error)
func (p *GitHubPeer) IssueComment(ctx, id Identity, repo string, n int, body string) error
func (p *GitHubPeer) LabelAdd(ctx, id Identity, repo string, n int, labels []string) error
func (p *GitHubPeer) LabelRemove(ctx, id Identity, repo string, n int, label string) error
func (p *GitHubPeer) PRCreate(ctx, id Identity, repo, title, head, base, body string) (PR, error)
```

(Consumed contract: github D9 verbs + D2 loopback identity assertion. On this
box a loopback caller asserts identity headers; nginx owns them at the front
door.)

**Read-path wire shape.** The github verbs answer as MCP structured content, and
MCP requires the structured result to be a JSON **object** at its root — a
top-level array is not valid. So github wraps every **list** result in a named
field: `issue_comments` returns `{"items": [...]}`. The peer's list reads must
decode that envelope and return the inner slice — `IssueComments` unmarshals
into an `{ Items []Comment }` wrapper and returns `Items`, never straight into a
bare `[]Comment`. Single-object verbs (`issue_get` → `Issue`, `pr_create` →
`PR`) are unwrapped bare objects and decode directly. Decoding a list verb as a
bare array is the defect this pins: it fails against github's real output with
`cannot unmarshal object into Go value of type []…`.

**Field-name fidelity (the created-PR address).** Decoding a bare object is
necessary but not sufficient — the struct's JSON tags must match github's
*actual* field names. github answers `pr_create` with the created PR's web
address in **`html_url`** (github D9 mirrors GitHub's REST field names), never a
bare `url`; so `PR` must decode `URL string ``json:"html_url"``` or the address
arrives empty while decode still "succeeds". An empty `pr.URL` after a
*successful* `pr_create` is a corrupt/unknown state, so `Success` **fails loud**:
it returns an error and the session ends `failed` with a clear reason, rather
than posting an empty (no-op) PR-link comment and recording a blank `pr_url`.
The old `json:"url"` tag silently degrading to an empty comment plus a blank
`sessions.pr_url` — on a run that still reported `succeeded` — is the defect this
pins; the whole downstream chain (comment body, `FinishSession` `pr_url`, the
outcome event's `PRURL`) already threads `pr.URL` and lights up once it is
non-empty.

**Label lifecycle** (issue-driven sessions; label names are the product
constants):

1. **Ack, at admission:** swap `execute` → `executing` (`LabelRemove` +
   `LabelAdd`) and comment the session id — the ack doubles as the visible
   double-trigger guard. A session that only queued (D5 gates) comments
   `queued` once at enqueue instead, and acks when admitted.
2. **Success:** push the branch (D4), `PRCreate` with head
   `ikibot/issue-<N>[.k]`, base = default branch, body carrying `Fixes #<N>`,
   the session id, and the check summary; comment the PR link (the created PR's `html_url`, asserted
   non-empty — see field-name fidelity above); remove `executing` (and `failed`
   if present).
3. **Failure** (engine error, TTL, check failure): push the branch anyway
   when it has commits (the crime scene stays inspectable), swap `executing`
   → `failed`, comment the reason (for a check failure: the check's tail
   output).
4. **Retry** is a fresh `execute` label: intake (D3) starts a new session
   (attempt+1); its ack removes `failed`.
5. Manual (MCP-started) sessions skip all label traffic; they still push and
   open a PR (no `Fixes` line — no issue), with the same check gate.

**Errored runs surface their own reason.** The check gate below applies **only
when the run completed cleanly** (the agent's conversation drained without an
engine error and inside the TTL). When the run itself failed — an engine error,
a setup failure such as the issue fetch (D6 read path) erroring, or the TTL
expiring — that failure's reason **is** the session's reason and is surfaced
verbatim; the runner does not run the check gate, does not open a PR, and does
**not** overwrite the reason with a commit verdict. It still pushes the branch
when the worktree has commits (the crime scene stays inspectable) and skips the
push when it has none. The `"no commits produced"` verdict is reserved strictly
for a **clean** completion that produced nothing — it must never mask the real
error of a run that failed before it could commit.

**The check gate — the in-repo definition of done.** After the agent's
conversation ends cleanly, the runner (never the transcript) decides:

- `worktree/.ikibot/check` exists and is executable → the runner runs it with
  cwd = the worktree, remaining session TTL as its bound, capturing combined
  output to `state/sessions/<id>/check.log`. Exit 0 → success path (PR). Non-
  zero → failure path: branch pushed, **no PR**, `failed` + check output
  comment.
- No check declared → success path, and the PR body states **"no check
  declared"** — the agent's self-assessment (its final message) is the body's
  summary either way.
- A session whose agent **completes cleanly** but produced **no commits** is a
  failure ("no commits produced"), nothing pushed, no PR.

**Rejected.**
- **Agent-performed push/PR with scoped credentials** — the settled
  credential-less division of labor: issue text is untrusted input to the
  agent; the runner holding all GitHub I/O makes exfiltration a non-path.
- **Trusting the transcript's claim that checks passed** — the runner
  executes `.ikibot/check` itself; a transcript is an account, not evidence.
- **Draft PRs on check failure** — a failed check means no PR at all; a
  half-artifact invites merging unverified work. The pushed branch is the
  inspection surface.
- **`issue_update` whole-set label replace** — race-prone for a label state
  machine (concurrent human labeling would be clobbered); atomic
  `label_add`/`label_remove` verbs were added to github (its D9) precisely
  for this.
- **Talking REST to GitHub directly from repos** — the github service is the
  suite's single GitHub API actor and token custodian; two actors means two
  credential custodies and two rate-limit owners.

**Verification** (substrate: a recording `httptest` stub implementing the
github `/mcp` wire shape — asserting exact verb names and arguments — plus
real git fixture remotes and a scripted fake engine; the real-GitHub half of
the contract is proven on github's side, and live end-to-end runs against
`bin/start`). **The stub must mirror github's *actual* output shape, never the
client's assumption** — list verbs answer `{"items": [...]}`, and a stub that
returns a bare array (or an always-empty thread) is what let the `issue_comments`
decode bug ship green; read-path assertions therefore drive the real envelope
with a non-empty comment thread.

- R-FDAF-MVIC — at admission of an issue-session the peer receives, in
  order: `label_remove` (`execute`), `label_add` (`executing`), and an
  `issue_comment` containing the session id — each call carrying
  `X-Owner-Email` = the session's owner and `X-Client-Id` =
  `repos:<session id>`.
- R-FEIC-0N91 — a session whose scripted agent commits and whose fixture repo
  declares a passing `.ikibot/check`: the branch is on the remote, exactly
  one `pr_create` was received with head `ikibot/issue-<N>`, base the default
  branch, and a body containing `Fixes #<N>` and the session id; then a
  PR-link comment and `label_remove` (`executing`) follow; the session row
  ends `succeeded` with `pr_url` set.
- R-FFQ8-EEZQ — same setup with a failing check: the branch **is** pushed, no
  `pr_create` occurs, the labels end swapped to `failed`, the failure comment
  contains the check output's tail, `check.log` holds the full combined
  output, and the session row ends `failed`.
- R-FGY4-S6QF — a fixture repo with no `.ikibot/check`: the PR is created and
  its body contains "no check declared"; the check gate never executes
  anything.
- R-FI61-5YH4 — a scripted agent that produces no commits: nothing is pushed,
  no PR, `failed` with reason "no commits produced".
- R-FKLT-XHYI — a retry after a recorded failure: the new session's ack
  removes the `failed` label, and its work lands on `ikibot/issue-<N>.2`.
- R-FLTQ-B9P7 — a manual session (no issue): zero label or comment calls
  reach the peer; the PR is created without a `Fixes` line; the check gate
  behaves per the fixture exactly as in the issue-driven cases.
- R-2V8C-1FO6 — a run that errors **before producing commits** (e.g. a scripted
  `FetchIssue`/engine error) ends `failed` with the failure comment carrying
  **that error verbatim**, never masked as "no commits produced"; the check
  gate never executes and no `pr_create` reaches the peer. Distinct from
  R-FI61-5YH4, which is the clean-completion-with-no-commits case. Substrate:
  the recording `httptest` github stub (assert zero `pr_create` and the exact
  reason on the failure comment) plus a scripted engine that returns an error.
- R-894D-CUA2 — `FetchIssue` against a github stub that answers `issue_comments`
  with github's **real** list envelope `{"items": [...]}` carrying **≥2**
  comments returns the issue title, body, and **every** comment in order (a
  bare-array or empty-thread stub does not model the real contract and is
  rejected); the pinned `instructions.md` contains all of them. The old decode
  into a bare `[]Comment` fails this with `cannot unmarshal object into Go value
  of type []repos.Comment`. Substrate: the corrected recording `httptest` stub
  emitting github's real MCP structured-content shape. Real-substrate proof (a
  live-acceptance step, not this id): with the fix deployed, re-triggering the
  `execute` label on `ikigenba/agentrepl` issue #1 drives a session that fetches
  the real multi-comment thread from the live github service, passes
  `FetchIssue`, writes `instructions.md`, and runs the agent — the wall this
  session previously hit.
- R-APSC-24AL — `Success` against a github stub whose `pr_create` answers
  github's **real** created-PR shape `{"number":N,"html_url":"https://…"}` sets
  the session's `pr_url` to that `html_url`, sends exactly one `issue_comment`
  whose body **is** that non-empty URL, and ends the session `succeeded`. A stub
  answering `{"number":N,"url":"…"}` (the pre-fix `json:"url"` assumption) leaves
  `pr.URL` empty, so the assertion turns on the `html_url` tag and that variant
  must fail. And a stub whose `pr_create` returns no address (empty `html_url`)
  drives `Success` to **fail loud**: it returns an error, no `issue_comment`
  carrying an empty body is sent, and the session ends `failed` — never
  `succeeded` with a blank `pr_url`. Substrate: the corrected recording
  `httptest` github stub emitting github's real created-PR shape. Real-substrate
  proof (a live-acceptance step, not this id): with the fix deployed,
  re-triggering the `execute` label on `ikigenba/agentrepl` drives a session to a
  real PR whose `html_url` lands both as a comment on the issue and in the
  session's `pr_url`.
