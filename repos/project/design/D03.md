# Decision 3 — GitHub-fact intake: the webhooks consumer & dispatch table

**Decision.** GitHub facts enter as suite events. The GitHub App's single
org-wide webhook delivers to the webhooks service (a `github-hmac` hook,
webhooks D17); repos consumes that hook's events and dispatches them through
one **event-name → handler table** so later signals are added rows, not
redesigns.

Wiring: one `appkit.Consumer{Source: "webhooks", Subscriptions:
[]consumer.Subscription{{Source: "webhooks", Filter: "webhooks:received/" +
hookName, Description: …}}, Handler: intake.Handler(...)}` where `hookName`
comes from `REPOS_GITHUB_HOOK` (default `github`). The chassis owns the feed
connection and cursor; the handler is at-least-once.

`internal/repos/intake.go`:

```go
type Intake struct {
    store    *Store
    svc      *Service          // lazy repo provisioning (D4)
    enqueuer Enqueuer          // session construction routed to the runner (D5)
    botLogin string            // REPOS_BOT_LOGIN, default "ikigenba[bot]"
    log      *slog.Logger
}
func (in *Intake) Handle(ctx context.Context, ev consumer.Event) error

// Enqueuer is the D5 session-construction seam. It lives here (not in
// internal/runner) because runner imports repos, not the reverse; the
// composition root wires the runner as its implementation.
type Enqueuer interface {
    Enqueue(context.Context, SessionRequest) (Session, error)
}

// the dispatch table — keyed by the GitHub event name
var ghHandlers = map[string]func(*Intake, context.Context, ghDelivery) error{
    "issues": (*Intake).handleIssues,
}
```

`Handle` decodes the webhooks payload (`{name, owner_id, owner_email,
received_at, content_type, body, headers}`), base64-decodes `body` into the
GitHub delivery, and reads the event name from `headers["x-github-event"]`.
The payload's owner field has followed the suite's owner-id conversion
(`docs/owner-id-design.md`, webhooks D5): the single `owner` (email) is
**replaced** by the pair `owner_id` (the for-whom key intake attributes and
keys sessions on) + `owner_email` (the display snapshot intake stores
unchanged). Then, in order:

1. **Malformed** (undecodable payload/body, missing event header) → wrap
   `consumer.ErrSkip` (poison: log loudly, advance).
2. **Not attributable** (decoded payload carries an empty/absent `owner_id`) →
   wrap `consumer.ErrSkip` (poison: log loudly, advance) — without the
   for-whom key no session can be attributed, the same skip semantics as a
   malformed payload. (Post-conversion webhooks always emits a non-empty
   stored `owner_id`, so this is a legacy/corrupt-payload guard.)
3. **Unknown event name** (no table row) → return nil (advance silently — the
   org hook delivers many event types; only table rows are signals).
4. **Loop suppression, day one:** `sender.login == in.botLogin` → return nil.
   The bot's own label writes come back as deliveries; they must never
   dispatch.
5. `handleIssues`: only `action == "labeled"` with `label.name == "execute"`
   proceeds. Then:
   - `issue.state != "open"` → ignore (log at info). Executing against a
     closed issue is never useful; re-open to re-dispatch.
   - An active (queued or running) session already exists for
     `(repo, issue.number)` → ignore (double-trigger guard; the `executing`
     label plus this check makes re-delivery harmless).
   - Otherwise: lazily provision the repo (D4) from the delivery's
     `repository` object, then construct the session **only** through the
     `Enqueuer` seam (D5) — never a hand-rolled `store.InsertSession`, so the
     `LogPath`, the dispatch doorbell, the branch/attempt naming, and the
     queued comment are all set in exactly one place. The request carries the
     issue number, `owner_id` = the webhooks payload's `owner_id` (the for-whom
     key the session is attributed and keyed on, and the identity the runner
     will assert to the github service, D6), and `owner_email` = the payload's
     `owner_email` (stored as the display snapshot, never read for logic); the
     attempt, branch, and id are the `Enqueuer`'s to compute, not intake's.

Handler errors that indicate a broken box (DB failure, clone failure of a
kind retry could fix) return the error → the consumer stalls and replays
(at-least-once); everything classified above advances. The queue itself (cap,
FIFO comment) is D5's concern.

**Rejected.**
- **Shape-sniffing the delivery body instead of `x-github-event`** — GitHub
  puts the event name in a header; guessing from body fields is fragile and
  fails silently on new event types. (This is why webhooks D17 forwards the
  header.)
- **A per-repo webhook** — the App's single org-wide hook covers every repo
  by construction; per-repo hooks would need per-repo onboarding, defeating
  lazy provisioning.
- **Label-text suppression instead of sender identity** — comparing
  `sender.login` to the bot login suppresses every future bot-authored event
  uniformly, not just the ones we remembered to special-case.
- **Dedup by `x-github-delivery`** — a delivery-id table is machinery v1
  doesn't need; the active-session-per-issue guard already makes re-delivery
  a no-op.

**Verification** (real substrate: handlers driven with recorded GitHub
delivery fixtures over a real migrated SQLite; enqueue observed in the DB; a
recording stub stands in for the github peer).

- R-EQ4C-D8F5 — the consumer declaration exposes exactly one subscription,
  source `webhooks`, filter `webhooks:received/github` under default env (and
  `webhooks:received/<name>` when `REPOS_GITHUB_HOOK=<name>`), visible
  through the chassis reflection surface.
- R-ERC8-R05U — an `issues`/`labeled`/`execute` delivery for an open issue in
  an unseen repo ends with a repo row created and one `queued` session for
  `(repo, issue)` attributed to the payload owner (R-IG68-6F1D pins the
  id-keyed attribution and the snapshot); the handler returns nil.
- R-ESK5-4RWJ — the same delivery with `sender.login` equal to the configured
  bot login creates no repo and no session and returns nil; a non-bot sender
  under a non-default `REPOS_BOT_LOGIN` is likewise judged against the
  configured value.
- R-ETS1-IJN8 — a delivery whose `issue.state` is `closed`, and a delivery
  whose `label.name` is not `execute` (e.g. `executing`), each create no
  session and return nil.
- R-EUZX-WBDX — with an active session already recorded for `(repo, issue)`
  (status `queued` in one case, `running` in another), a fresh `execute`
  delivery creates no second session; with only a terminal session recorded,
  it creates a new one with `attempt` incremented.
- R-EW7U-A34M — a payload that is not valid JSON, a body that is not valid
  base64/JSON, and a payload missing `x-github-event` each return an error
  wrapping `consumer.ErrSkip`; an unknown event name (`push`) returns nil;
  a DB failure (closed handle) returns a non-ErrSkip error (stall).
- R-IG68-6F1D — an `issues`/`labeled`/`execute` open-issue delivery whose
  webhooks payload carries `owner_id` = X and `owner_email` = E enqueues one
  session whose stored `owner_id` is X (the scoping/for-whom key) and whose
  `owner_email` is E (the display snapshot). A delivery whose payload carries
  the **same `owner_email` E but a different `owner_id` Y** attributes the
  session to Y — an implementation reading `owner_email` for attribution
  fails. And a decoded payload with an **empty/absent `owner_id`** creates no
  repo and no session and returns an error wrapping `consumer.ErrSkip`
  (not attributable). (Substrate: recorded GitHub delivery fixtures wrapped in
  the converted webhooks payload over a real migrated SQLite; the enqueued
  session's `owner_id`/`owner_email` read back from the DB.)
