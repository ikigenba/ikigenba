# Decision 4 — nginx fragment: the exact-match session-gated `= /srv/ledger/` location

**Decision.** ledger's nginx fragment (`ledger/etc/nginx.conf`) gains **one new
location** — an **exact-match** location for the bare mount root that
session-gates the human landing page and proxies it to the service's root. It
sits alongside (does not replace) the existing locations:

```
# Landing page — human web UI, gated by the dashboard browser SESSION (cookie),
# validated by the dashboard-owned /_session-authn internal location (a sibling of
# /_authn). Exact-match `=` so it never shadows the gated /srv/ledger/ prefix below
# or the MCP endpoint. A failed session check yields 401.
location = /srv/ledger/ {
    auth_request /_session-authn;
    auth_request_set $ledger_session_owner         $upstream_http_x_owner_email;
    auth_request_set $ledger_session_owner_id      $upstream_http_x_owner_id;
    auth_request_set $ledger_session_owner_name    $upstream_http_x_owner_name;
    auth_request_set $ledger_session_owner_picture $upstream_http_x_owner_picture;
    proxy_set_header  X-Owner-Email   $ledger_session_owner;
    proxy_set_header  X-Owner-Id      $ledger_session_owner_id;
    proxy_set_header  X-Owner-Name    $ledger_session_owner_name;
    proxy_set_header  X-Owner-Picture $ledger_session_owner_picture;
    proxy_pass http://127.0.0.1:3101/;   # trailing slash → upstream sees "/"
    proxy_set_header Host $host;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_http_version 1.1;
}
```

**How it coexists with the existing fragment.** ledger's fragment already has:

1. `location = /srv/ledger/.well-known/oauth-protected-resource` — unauthenticated
   PRM bootstrap (exact-match).
2. `location /srv/ledger/ { auth_request /_authn; … }` — the bearer-gated
   **prefix** for the MCP endpoint and everything else under the mount, including
   its `error_page 500 = @ledger_authn_500` rate-limit re-emit.
3. `location @ledger_authn_500 { … }` — the named location the prefix's
   `error_page` falls back to, which re-emits a faithful `429` (with `Retry-After`)
   when token introspection was rate-limited.

nginx longest-prefix/exact-match rules make an **exact-match `=` location win
over a prefix location** for that one path regardless of source order. So the new
`= /srv/ledger/` exact match wins for the **bare root only**, while the
`/srv/ledger/` prefix (item 2) still serves `/srv/ledger/mcp`, `/srv/ledger/feed`,
and every other sub-path under the mount. The PRM exact match (item 1) and the
`@ledger_authn_500` named location (item 3) are unaffected. Nothing in the existing
fragment is removed or rewritten — the change is **purely additive**.

**Why `/_session-authn`, not `/_authn`.** A browser carries the
`dashboard_session` cookie, not a bearer token, so the bearer gate (`/_authn`)
would refuse it. `/_session-authn` is the dashboard-owned, loopback-only internal
location (defined in `dashboard/etc/nginx.conf` as
`proxy_pass …/internal/session-authn`) that validates the cookie against the
web-session store and returns `200` + `X-Owner-Email` on success or `401` on
failure. This is the **same** mechanism `sites` already uses for its private
static tier (`sites/etc/nginx.conf` `location /srv/sites/private/`). The gate is
deliberately **coarse**: any logged-in dashboard user passes; there is no
per-resource check — fine for a name+version page.

**Why the simple form (no 429 re-emit).** The bearer prefix block carries the
`$authn_status`/`error_page 500 = @ledger_authn_500` machinery because token
introspection can return `429` (rate-limited) which `auth_request` collapses to a
synthetic `500`. The session check is a cookie lookup, not a rate-budgeted token
introspection, so the landing location does not need the 429 re-emit dance — it
maps the session subrequest's allow/deny directly (`200`→serve, `401`→401),
matching the `sites` private-tier location, which likewise omits it.

**Templating.** The loopback port is a **literal** in the fragment (`PORT=3101`, this service's fixed registry port), so nothing substitutes it at install — `opsctl setup ledger` on the box / `nginx/run` in dev drop the fragment verbatim, exactly like the existing locations. The fragment is **not** a vhost — no
`server{}`, no `listen`, no cert; it is dropped at
`/etc/nginx/conf.d/locations/ledger.conf` and included by the dashboard's apex
server block, which is also where `/_session-authn` is defined.

**Full owner-identity forwarding (all four `X-Owner-*` headers).** The dashboard's
introspection hooks now emit **four** owner headers on every allow, fail-closed:
`X-Owner-Id` (durable owner key, never empty), `X-Owner-Email` (display/audit,
never empty), `X-Owner-Name` (percent-encoded, may be empty), and `X-Owner-Picture`
(percent-encoded, may be empty) — plus `X-Client-Id` on the bearer tier as before.
Every location in this fragment that forwards identity today captures **all four**
owner headers with `auth_request_set` and forwards **all four** with
`proxy_set_header`, so none of the new headers die at nginx:

- **The bearer prefix `location /srv/ledger/`** (`auth_request /_authn`) captures
  `$ledger_owner` (email), `$ledger_owner_id`, `$ledger_owner_name`, and
  `$ledger_owner_picture` from `$upstream_http_x_owner_email` /
  `$upstream_http_x_owner_id` / `$upstream_http_x_owner_name` /
  `$upstream_http_x_owner_picture`, and forwards `X-Owner-Email`, `X-Owner-Id`,
  `X-Owner-Name`, `X-Owner-Picture` (plus `X-Client-Id` from `$ledger_client`,
  unchanged). Its `$authn_status`/`$authn_retry` rate-limit machinery and
  `@ledger_authn_500` re-emit are untouched.
- **The session exact-match `= /srv/ledger/`** (`auth_request /_session-authn`)
  captures `$ledger_session_owner` (email), `$ledger_session_owner_id`,
  `$ledger_session_owner_name`, and `$ledger_session_owner_picture` from the same
  four upstream headers, and forwards all four `X-Owner-*` headers. The session
  tier carries no `X-Client-Id` (there is no bearer client), unchanged.

The variable names are **service-prefixed** (`$ledger_owner_id`,
`$ledger_session_owner_id`, …) so they stay unique when every service fragment is
included into the one apex `server` block. The identity-hygiene property is
preserved and extended to the new headers: a single `proxy_set_header` per name
from the auth-subrequest variable **replaces** any client-smuggled inbound
`X-Owner-*` header entirely (nginx does not pass inbound client headers as proxy
headers unless explicitly set, and the last `proxy_set_header` for a name wins), so
a client cannot smuggle owner identity through any of the four.

The locations that forward **no** identity today are unchanged: the PRM bootstrap
`location = /srv/ledger/.well-known/oauth-protected-resource`, the session-gated
static tier `location /srv/ledger/static/` (a coarse cookie gate that proxies
assets without owner headers), and the `@ledger_authn_500` re-emit. Downstream
consumption of `X-Owner-Id` by the ledger binary is out of scope for this
Decision — this is fragment forwarding only.

**Rejected.**
- **Gating the landing page with `/_authn`** — a browser has no bearer token; it
  would always 401. Wrong audience.
- **Making the landing page fully public (no auth_request)** — the brief forbids
  it; a logged-out browser must get `401`. The page is gated, just coarsely.
- **A prefix `location /srv/ledger/` for the page** — would collide with the
  existing bearer prefix; the exact-match `=` is what cleanly carves the bare root
  out of the prefix without touching the rest.
- **Adding the `@ledger_authn_500` 429 re-emit to the session location** — the
  session check is not rate-budgeted; the extra machinery is dead weight (the
  `sites` private tier omits it for the same reason).
- **Forwarding only `X-Owner-Email` and dropping the other three owner headers at
  nginx** — the introspection hooks now emit `X-Owner-Id`/`X-Owner-Name`/
  `X-Owner-Picture` too; leaving them uncaptured strands the durable owner key and
  profile fields at the trust boundary. Every identity-forwarding location must
  capture and forward all four.
- **Reusing one un-prefixed variable name (e.g. `$owner_id`) across services** —
  all fragments are included into a single apex `server` block, so variable names
  must be service-prefixed to stay unique; a shared name would collide.

**Verification.** (Proven by a test that reads `ledger/etc/nginx.conf` from disk
and asserts over its content — see design's Testing strategy.)
- R-NGNX-2B4C — `ledger/etc/nginx.conf` contains an **exact-match** location for
  the bare mount root: a `location = /srv/ledger/ {` block (distinct from the prefix
  `location /srv/ledger/ {`).
- R-NGNX-4D6E — that exact-match block gates with `auth_request /_session-authn`
  (the cookie/session validator) and **not** `auth_request /_authn` (the bearer
  validator).
- R-NGNX-6F8G — that block proxies to the loopback upstream **root**
  (`proxy_pass http://127.0.0.1:3101/;` — trailing slash), so the upstream
  sees `/`.
- R-NGNX-8H1J — the pre-existing locations are retained alongside it: the
  bearer-gated `location /srv/ledger/ {` prefix with `auth_request /_authn`, its
  `@ledger_authn_500` named re-emit location, and the
  `location = /srv/ledger/.well-known/oauth-protected-resource` PRM bootstrap all
  still appear in the file.
- R-FLV3-9RX8 — the bearer prefix `location /srv/ledger/` (`auth_request /_authn`)
  captures **all four** owner headers with service-prefixed variables —
  `auth_request_set $ledger_owner $upstream_http_x_owner_email;`,
  `$ledger_owner_id $upstream_http_x_owner_id;`,
  `$ledger_owner_name $upstream_http_x_owner_name;`,
  `$ledger_owner_picture $upstream_http_x_owner_picture;` — and forwards all four
  with `proxy_set_header X-Owner-Email $ledger_owner;`,
  `X-Owner-Id $ledger_owner_id;`, `X-Owner-Name $ledger_owner_name;`,
  `X-Owner-Picture $ledger_owner_picture;`, while still forwarding
  `X-Client-Id $ledger_client;`. A fragment that forwards only `X-Owner-Email`
  (dropping any of `X-Owner-Id`/`X-Owner-Name`/`X-Owner-Picture`) fails this.
- R-FN2Z-NJNX — the session exact-match `location = /srv/ledger/`
  (`auth_request /_session-authn`) captures **all four** owner headers with
  service-prefixed variables —
  `auth_request_set $ledger_session_owner $upstream_http_x_owner_email;`,
  `$ledger_session_owner_id $upstream_http_x_owner_id;`,
  `$ledger_session_owner_name $upstream_http_x_owner_name;`,
  `$ledger_session_owner_picture $upstream_http_x_owner_picture;` — and forwards
  all four with `proxy_set_header X-Owner-Email $ledger_session_owner;`,
  `X-Owner-Id $ledger_session_owner_id;`, `X-Owner-Name $ledger_session_owner_name;`,
  `X-Owner-Picture $ledger_session_owner_picture;`. A fragment that forwards only
  `X-Owner-Email` on the landing root fails this.
