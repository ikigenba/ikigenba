# Decision 4 — Version production: `bump` / `ship` emit `v`-prefixed SemVer + the release bundle

**Decision.** The version *producers* are brought in line with the `v`-everywhere,
full-SemVer-2.0 contract (D03), and `ship` now produces the **versioned bundle**
(D02), not a bare binary.

- **`bin/bump`** reads `<svc>/VERSION`, increments the requested field
  (`major|minor|patch`) **carrying the `v`**, and writes the `v`-prefixed result
  back (still a path-limited commit of only that file, still on `main`). It
  **rejects** a `VERSION` file that is not valid `v`-prefixed SemVer rather than
  silently coping (the old "bare X.Y.Z" contract is gone). The pre-release
  identifier, when one is set, is preserved/advanced per SemVer.
- **`bin/ship`** builds `main` static for `linux/amd64` (`GOWORK=off`), stamps the
  binary with the **full** version (the `VERSION` core plus a `+<SHA>`
  build-metadata suffix from the built commit, via `-ldflags`), then **assembles a
  bundle** and `scp`s the bundle (not a bare binary) to the box `/tmp`.

**The release bundle.** The unit of delivery is one `tar.gz`, **named with the
full version**, carrying every shipped tier in the **exact versioned-tier layout
opsctl places on the box** (D02) — the archive *is* the on-box tree for that one
version, so `stage` only has to unpack and move it, never remap a flat layout:

```
<svc>-v<full>.tar.gz
  libexec/<svc>-v<full>          the static binary (a bare FILE, full v-SemVer name)
  etc/v<full>/nginx.conf         the service's nginx location fragment
  etc/v<full>/manifest.env       the service's authored, portable manifest (D11; verbatim)
  share/v<full>/...              shipped read-only resources
```

where `v<full>` is the same full v-prefixed SemVer 2.0 identity (D03) the binary
stamps, the bundle is named with, and `opsctl stage <svc> v<full>` is invoked
with — every tier keys off that one string, so the unpacked tree matches what
`stage` reads (`libexec/<svc>-v<full>`, `etc/v<full>/{nginx.conf,manifest.env}`,
`share/v<full>/`) by construction. **`share/v<full>/` is always present** — an
**empty** directory when the service ships no resources — because opsctl `stage`
requires the `share/<version>` tier to exist; a service with no `<svc>/share/`
still gets an empty `share/v<full>/` in the bundle, never a missing tier.

**Bundle naming schema.** The filename is exactly:

```
<svc>-<version>.tar.gz
```

where `<svc>` is the canonical service name and `<version>` is the **full
v-prefixed SemVer 2.0 identity** (D03) — `vMAJOR.MINOR.PATCH[-prerelease]+<sha>`,
the same string the binary stamps and `libexec/<svc>-<version>` carries. Examples:

```
crm-v0.7.1+a1b2c3d.tar.gz
wiki-v0.7.1-rc.2+a1b2c3d.tar.gz
```

The `+<sha>` build metadata is always present in a shipped bundle (`ship` always
stamps the built commit). The same filename is the box `/tmp` staging name and the
`opsctl stage --artifact <path>` argument (D10). `<svc>` and `<version>` are passed
to `stage` explicitly, so the name is **constructed and validated**, never parsed
back apart.

`stage` (D10) unpacks it onto the box as `libexec/<svc>-v<full>`,
`etc/v<full>/nginx.conf`, `etc/v<full>/manifest.env`, and `share/v<full>/…` (D02) —
the same paths the bundle already carries, so `stage` places the tiers directly
with no flat-to-tiered remap. The service **authors its shipped tiers in source
mirroring the install tree** — `<svc>/etc/nginx.conf`, `<svc>/etc/manifest.env`
(D11), and an optional `<svc>/share/…` — and `ship` packages exactly those (the
built binary under `libexec/`, the two `etc/` files under `etc/v<full>/`, and the
`share/` contents under `share/v<full>/`, empty when the service ships none). The
`manifest.env` is **authored and portable** (no box paths; D11) and is packaged
**verbatim** — `ship` does not generate, stamp, or rewrite it.

The version embedded in the binary, the bundle filename, the bundle's binary, and
opsctl's deploy preflight (D03 gate) all carry the same full `v`-prefixed string,
so `<svc> version`, the staged `libexec/` filename, and the stage/deploy preflight
agree by construction.

**Rejected.**
- *Ship a bare binary and apply config/resources out of band.* The gap this work
  closes: a fragment change then never rides the version and cannot roll back with
  the binary. The bundle carries all shipped tiers as one versioned unit. Rejected.
- *Fix only the consumer side (opsctl) and leave `bump`/`ship` emitting bare
  numbers.* opsctl's `v`-everywhere gate would then reject the suite's own
  artifacts — "any legal SemVer" is a constraint on **both** ends. Rejected.
- *Carry `-rel`/`+SHA` as out-of-band metadata rather than in the version string.*
  Defeats a single canonical identity the filesystem and `version` verb share.
  Rejected.
- *Author the fragment/resources somewhere other than the service source tree
  (e.g. a central opsctl dir).* Splits a service across two repos' worth of intent
  and breaks "the service owns its shipped tiers." Rejected for `<svc>/etc/` +
  `<svc>/share/`.
- *Flat bundle (`<svc>`, `nginx.conf`, `manifest.env`, `share/…` at the root) with
  `stage` remapping it into the versioned tiers.* The original form of this
  Decision. It split the layout knowledge across two producers — `ship` wrote flat,
  opsctl `stage` had to know the flat→tiered mapping — and the two drifted: `stage`
  as built requires the tiered layout (`libexec/<svc>-v<full>`, `etc/<v>/…`,
  `share/<v>/`) and rejects the flat bundle `ship` produces, so no real ship→stage
  round-trip works. Rejected: the bundle now carries the exact tiered tree, so
  there is one layout definition and `stage` only unpacks-and-places.
- *Fix the drift on the opsctl side instead (teach `stage` to accept the flat
  bundle).* Would keep two layout shapes in play (a flat wire format plus the tiered
  on-box tree) and put the mapping in the privileged on-box tool. Making the wire
  format equal the on-box tree removes the mapping entirely and keeps the change
  off the box. Rejected for the ship-side fix.

**Verification.**

This Decision mints **no requirement ids**: `bump` and `ship` are `bin/` shell
tooling, which per Conventions is deliberately untested (the former
`bin/*.test.sh` tier was removed) and verified manually, outside the build
loop, when created or changed. The behaviors that verification exercises —
`bump patch` on `v0.7.1` writing `v0.7.2` and rejecting a bare `0.7.1`;
`ship` stamping `v<core>+<sha>` into the binary's `version` output and the
bundle name; the bundle carrying the versioned-tier layout
(`libexec/<svc>-v<full>`, `etc/v<full>/{nginx.conf,manifest.env}` verbatim,
`share/v<full>/` present even for a share-less service) — remain normative
above. The bundle layout's in-gate proof lives on the consumer side: opsctl
`stage`'s `requireBundlePaths` behavior (D10's ids) rejects a malformed
bundle, so a bundle that deploys has round-tripped the contract.
