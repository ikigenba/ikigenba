# Decision 12 — Per-app secrets parameters, pushed from `.envrc`

**Decision.**

Every deployable service's deployed secrets live in **one SSM parameter per
app** — `/ikigenba/<ACCOUNT>/app-config/<app>` (SecureString, Standard tier,
region `us-east-2`) — whose value is a **flat JSON object** `{ "KEY":
"value", … }`. A secret-less app holds an explicit `{}`: "no secrets" is a
deliberately seeded state, never an absence. Parameter *existence* is
script-managed (`put-parameter --overwrite` creates on first push; see
research); nothing per-app is Terraform-managed, so adding a service to the
suite requires no infrastructure change.

The **box-side consumer is an external contract**, already live (see
research): the platform launcher (`metaspot/templates/ikigenba-launch`,
installed out-of-band) fetches the app's own parameter, exports each key over
`manifest.env` (SSM wins on collision), and hard-fails on any fetch error
**including `ParameterNotFound`** — a never-seeded app must not start. The
suite repo does not own the launcher; this Decision only fixes the parameter
path and value shape both sides meet at.

**The `.envrc` parse rule (strict, no exceptions).** An app's `.envrc` is the
single source of truth for the secrets it needs. A line matching exactly

    export NAME="$(cat ~/.secrets/NAME)"

— same `NAME` on both sides — marks a pushed secret. Nothing else is pushed:
plain exports are local-only by construction, and the repo-root `.envrc` is
never parsed (it is not an app), so workstation-only secrets (`IKIGENBA_TOKEN`,
the local-MCP-to-box token) are exported there and inherited locally via
`source_up` without ever riding to the box. There is no opt-out annotation: a
secret that must stay local belongs in the root `.envrc`, not the app's.
`wiki/.envrc`'s former `for`-loop form is unrolled to plain lines to conform.

**The push tool: `bin/push-secrets`.** One bash script at repo root (beside
`bump`/`ship`; same style), the only seeding procedure:

- `bin/push-secrets <app>` — parse `<app>/.envrc` by the strict rule; resolve
  each name (same-named env var overrides, else `~/.secrets/<NAME>` contents,
  else abort; whitespace-stripped; empty value aborts); assemble the flat JSON
  with `jq`; write it via a `chmod 600` temp file (trap-cleaned) with
  `aws ssm put-parameter --name /ikigenba/<ACCOUNT>/app-config/<app>
  --type SecureString --overwrite --value file://<tmp>`. No `.envrc` or no
  matching lines → push `{}`.
- `bin/push-secrets --all` — sweep every deployable app (the top-level
  directories carrying a `VERSION`); report per-app; exit non-zero on any
  failure.
- `bin/push-secrets --dry-run [<app>|--all]` — print each app's parameter
  name and pushed key **names** without calling AWS.
- `ACCOUNT` comes from `<app>/etc/deploy.env` (present in all fourteen apps);
  `AWS_PROFILE` defaults to `$ACCOUNT`, `AWS_REGION` to `us-east-2`, both
  overridable. Requires a live SSO session; a failed AWS call aborts loudly.
- Output shows key names and **masked** values only (first 4 + last 4 chars);
  a full secret value never appears on a command line, in output, or in a log.

Because the parameter is per-app, the write path is a plain overwrite of the
app's own parameter: there is **no fetch, no merge, no sibling preservation**
— the read-modify-write ritual (and its clobber risk) that the shared-blob
era's seven per-service `bin/secrets` scripts existed to manage is deleted
with them. `sops/seed-secrets.md` is rewritten around `bin/push-secrets` as
the only procedure.

**Rejected.**

- *Advanced-tier (8 KB) shared blob.* Defers the same wall for a monthly fee;
  keeps every write a clobber risk for all fourteen apps. Rejected.
- *Terraform-managed per-app parameters.* Makes every new service a
  `terraform apply`; today a new service touches no infrastructure. Rejected —
  the push tool creates parameters, Terraform's only involvement is the
  one-time IAM path grant (already applied).
- *Launcher tolerates `ParameterNotFound` for secret-less apps.* Softens the
  hard-fail contract: a typo'd path or never-seeded secret-carrying app would
  silently start without secrets — the exact failure mode the launcher exists
  to prevent. Rejected for the explicit `{}` seed.
- *A per-service manifest file (or opt-out annotations) instead of parsing
  `.envrc`.* A second list to keep in sync with the first; the `.envrc` *is*
  the list, and the root-`.envrc` home for workstation-only secrets keeps the
  rule exception-free. Rejected.
- *Implementing the push tool in Go with unit tests.* Operator decision: the
  tool is bash like its `bin/` siblings, deliberately untested, verified once
  outside the build loop. Rejected for this iteration.

**Verification.**

This Decision mints **no requirement ids** — deliberately. Its box-side half
is an external contract already verified live (see research: all fourteen
services healthy on per-app parameters, `github_auth: ok` proving end-to-end
secret flow). Its repo-side half is `bin/` shell tooling, which per
Conventions is untested by decision: `bin/push-secrets` is verified **once,
manually, outside the build loop** by the operator/agent — `--dry-run` output
compared against every app's `.envrc`, then one live push compared (by key
names only) against the seeded parameters. The build phases realizing this
Decision therefore carry structural deterministic checks, not id-tagged
tests.
