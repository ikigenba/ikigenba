# Decision 8 — Per-service adoption & live-box migration

**Decision.** This redesign touches **every** service; D08 owns the uniform
adoption contract and the one-time migration of the live `int` box. Per service:

1. **Authored shipped tiers.** Each service grows, in its source tree mirroring the
   install layout (D04): `<svc>/etc/nginx.conf` (its location fragment) and
   `<svc>/etc/manifest.env` (its **authored, portable** manifest — no box paths,
   D11), plus an optional `<svc>/share/…`. `ship` packages exactly these into the
   bundle; nothing is generated on the box.
2. **Layout move.** DB → `state/<svc>.db`; generation sidecar →
   `cache/<svc>.db.generation`; durable file space (if any) under `state/`;
   everything else stays non-state (D05). The paths are composed in-binary from the
   box-global `IKIGENBA_ROOT=/opt` (D11) — there is **no** per-unit `<APP>_DB_PATH`
   stamping anymore; the unit inherits `IKIGENBA_ROOT` from `/etc/ikigenba/env`.
3. **Boot reconstruction.** The service recreates its non-state on startup (D06)
   so a restore can wipe it.
4. **Binary placement.** Built and deployed as `libexec/<svc>-v<full>` with
   `bin/run` selecting it (D02); `VERSION` is `v`-prefixed (D03/D04).
5. **Backup/restore.** Inherits `opsctl backup/restore <svc>` (D07) — no
   per-service script.

Services with extra durable data make a **conscious classification** (the point of
the model): `prompts` (`sandboxes/` durable → `state/`; `runs/` logs → decide),
`scripts` (`runs/` trees → decide), `sites` (its `/opt/sites/www/{served/public,
served/private,working}` → `/opt/sites/state/www/{public,private}` (+ any working
content under `state/`); the `web` group + nginx fragment repointed per D01),
`dashboard` (apex TLS cert via the dedicated cert stream, D07).

**Live-box migration** (sequenced, since `int.ikigenba.com` is the live account
and is touched only on an explicit deploy request): opsctl gains a one-time
conversion that, per service:

- moves `data/<svc>.db*` → `state/<svc>.db*` and the sidecar → `cache/`;
- converts `releases/`+`current` → `libexec/`+`bin/run` (D02);
- converts the old single `etc/manifest.env` (the box-generated one) and any
  loose `etc/nginx.conf` into the versioned, symlink-selected form
  `etc/<v>/{nginx.conf,manifest.env}` + `etc/current → <v>`, and `share/<v>` +
  `share/current` (D02);
- **retires the dead tiers** — the local `backups/` dir (no on-box backup of
  record, D07) and the flat `data/` dir (split into `state/`+`cache/`, D05) are
  removed once their contents have been moved.

The converter is **idempotent** (safe to re-run; a half-converted tree completes)
and **non-destructive** (no data dropped — moves precede any removal, and a removal
only fires once the destination exists). The existing OLD→new conversion precedent
in opsctl (the legacy `bin/run` wrapper-script handling) is the model.

This is largely an **orchestration/coordination** decision; its per-mechanism
proofs live in D01–D07, D10, D11. The behavior it owns is that the adoption is
complete and the migration is safe.

**Rejected.**
- *Adopt a few services and leave the rest on the old layout.* The whole value is
  uniformity (one backup path, one layout); a partial rollout keeps the
  3-incompatible-ways problem alive. Rejected — every service adopts.
- *Big-bang in-place rename on the live box with no idempotent converter.* A
  failed mid-rename leaves a service unbootable with no safe re-run. Rejected for
  an idempotent, non-destructive converter.
- *Keep box-generating `etc/manifest.env` during the converter.* That preserves
  the very on-box generation step this redesign deletes (D11); the converter instead
  drops the generated file in favor of the shipped, authored one. Rejected.

**Verification.**
- R-4LKF-FB23 — for **each** service, a freshly set-up `/opt/<svc>/` (DB under
  `state/`, sidecar under `cache/`, binary under `libexec/`, `bin/run`/`etc/current`/
  `share/current` symlinks, the **authored** `etc/<v>/manifest.env` shipped not
  generated) boots and passes its `health` check. Substrate: the running suite
  (per-service boot smoke; `bin/start` + health, or the service binary against a
  temp root).
- R-4MSB-T2SS — the one-time converter, run against a fixture of the **old**
  layout (`data/<svc>.db`, `releases/<v>/`, `current`, a box-generated
  `etc/manifest.env`, and a stray `backups/` dir), produces the new layout — DB
  intact under `state/`, the live binary reachable via `bin/run`,
  `etc/current`/`share/current` resolving, the authored `etc/<v>/manifest.env` in
  place — and **removes** the `data/` and `backups/` tiers; re-running it is a
  no-op (idempotent). Substrate: opsctl unit test against a temp `OPSCTL_ROOT`
  seeded with the old layout.
