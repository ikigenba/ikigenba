# Decision 7 — opsctl-owned `backup` / `restore`, S3-only (stop·snapshot·start)

**Decision.** Backup/restore is a **first-class opsctl verb**, uniform across
**all** services — `opsctl backup <svc>` / `opsctl restore <svc>` — and **S3 is
the only durable store**. opsctl is the only on-box actor that can `systemctl`
(the service binary runs as the app user and cannot), so the stop/start
orchestration must live here.

**There is exactly one backup/restore implementation, and it is off-box.** This
retires, removed outright (no stub, no tombstone):

- the three per-service `bin/backup`/`bin/restore` shell scripts (crm, prompts,
  scripts) and the dashboard's in-binary `Spec.Backup`/`Spec.Restore` S3 branch;
- the appkit **local** `backup --out` / `restore --from` verb and the
  `Spec.Backup` / `Spec.Restore` hooks entirely. There is **no on-box backup path
  of any kind** — the old local pre-migration `.db` snapshot is gone (its job moves
  to "every deploy takes an S3 backup first", D10). The service binary's verb set
  shrinks to **`serve / version / migrate / schema`** (the `manifest` verb is also
  dropped — D11; backup/restore are dropped here) — none of which can be done from
  opsctl, which is why they stay in the binary; backup/restore need none of the
  binary's compiled-in knowledge, so they do not.

**Backup** = stop unit · archive `state/` (D05) → upload · start unit. S3 layout
keeps the good crm/dashboard conventions (not prompts/scripts' historyless sync):

```
s3://$IKIGENBA_BACKUP_BUCKET/<svc>/
  <svc>-<version>.<UTC>.tar.gz   immutable, one per run; the key embeds the
                                 producing version (D03) so rollback can read it
  latest                         pointer → newest snapshot key
```

The snapshot key **embeds the producing version** — the live deployed version at
backup time, which opsctl reads from `bin/run`'s target (D02). This is what lets
`opsctl rollback -N` (D10) resolve a snapshot to the exact binary version to swap
back to, from the key alone.

Count-based retention (keep newest N, operator-configurable; **N defaults to 30** —
the product contractual constant, used here by value) on the snapshot prefix;
`latest` is never pruned. **There is no `pre-restore/` namespace** — a backup
carries no metadata about *why* it was taken; a nightly backup (D09) and a
pre-deploy backup (D10) are byte-identical in kind, distinguished only by
timestamp.

**Restore** = resolve key (default `latest`, or an explicit key) · **always
interactive, no `--yes` bypass** · stop unit · **replace `state/`** from the
snapshot · **reset the non-state region** (D05/D06: `cache/` wiped, sidecar gone) ·
start unit. Restore takes **no** safety snapshot of its own — the single generic
backup is the only backup, and rollback's safety comes from the deploy-time backup
(D10), not a restore-time one.

**Apex TLS cert (dashboard / the DEFAULT app).** The one durable artifact outside
`/opt/<svc>/` — the apex Let's Encrypt cert at `/etc/letsencrypt/` — is backed up
and restored as a **dedicated snapshot stream** alongside dashboard's state
stream, because **reissue does not scale for mass recovery**: every
`<account>.ikigenba.com` is a subdomain of the single registered domain
`ikigenba.com`, and Let's Encrypt's 50-certs/week limit is **per registered
domain**, so a fleet-wide rebuild that reissued certs would exhaust the shared
quota. Restoring a cert from S3 consumes **zero** issuance quota.

```
s3://…/dashboard/cert/cert.<UTC>.tar.gz   tars /etc/letsencrypt/{archive,renewal,live}/<domain> (paths relative to /)
s3://…/dashboard/cert/latest              pointer, resolved INDEPENDENTLY of the state pointer
```

- Backup produces the cert object **in addition to** the state object; the cert
  stream has its own `latest` and retention.
- A dashboard restore lays the cert back via `tar -C /` and **never invokes
  certbot** (no reissue) — and resolves `cert/latest` independently of the chosen
  state snapshot, so a DB rollback never drags the cert backward.
- **Secrets:** the cert archive contains the TLS **private key**
  (`archive/<domain>/privkey*.pem`). opsctl handles these as **opaque bytes —
  never read, printed, or logged**. The backup bucket's privacy is load-bearing.

**Testability seam.** S3 sits behind an `ObjectStore` interface so opsctl logic and
the backup/restore round-trip are proven **in-gate, unprivileged, with no external
services**: the retention / `latest` logic and the full `state/` archive → `Put` →
`Get` → restore round-trip run against the in-memory `fakeStore` with a **real
`tar` and real filesystem**, proving the archive format and the `ObjectStore`
contract. That `aws` actually transfers bytes to a real S3 bucket is a separate
**on-box/manual** check outside the green gate (never a `t.Skip`-gated test, which
would count as uncovered):

```go
type ObjectStore interface {
    Put(ctx, key string, r io.Reader) error
    Get(ctx, key string, w io.Writer) error
    List(ctx, prefix string) ([]ObjInfo, error) // ObjInfo{Key, Size}
    Delete(ctx, key string) error
}
// awsCLIStore: shells `aws s3 / s3api` using the box instance role; region from
// IKIGENBA_AWS_REGION (default us-east-2), bucket from IKIGENBA_BACKUP_BUCKET.
// fakeStore: in-memory, for opsctl unit tests of retention/latest/round-trip logic.
```

The stop/start path uses the existing `System` seam. The unattended cadence — the
`opsctl backup --all` box sweep and its 3 AM timer — is **D09**; the
backup-before-deploy call and rollback's restore are **D10**; restore is never
scheduled.

**Rejected.**
- *Keep S3 in per-service `bin/*` scripts.* Three incompatible implementations and
  they can't `systemctl` cleanly; consolidating in opsctl gives one implementation
  and real stop/start. Rejected.
- *Keep the appkit local `backup --out`/`restore --from` verb for deploy/rollback.*
  A backup on the same disk as the data is not a backup; the verb was the last
  on-box backup path and the whole violation of "a backup is off the box."
  Deploy's pre-step is now an ordinary S3 backup (D10). Rejected — deleted outright.
- *Keep a `pre-restore/` safety snapshot.* A second backup *kind* and namespace for
  no gain: rollback relies on the deploy-time backup, and a standalone restore is
  gated on an interactive human. One generic backup, one generic restore. Rejected.
- *Fold S3 into each binary as a `backup` verb.* The binary cannot stop its own
  unit, and the D05 layout convention removes the only reason in-binary made sense.
  Rejected — opsctl owns it.

**Verification.**
- R-4GOT-W83B — `opsctl backup` invokes `System.Stop(<svc>)` **before** taking the
  snapshot and `System.Start(<svc>)` **after** (ordering asserted via the seam); on
  a snapshot failure it still restarts the unit. Substrate: opsctl unit test with
  stubbed `System` + fake `ObjectStore`.
- R-4HWQ-9ZU0 — `opsctl backup` puts an immutable timestamped object **and** updates
  `latest` to that key; with N+1 snapshots present, the retention pass deletes the
  oldest down to N and **never** deletes `latest`. Substrate: opsctl unit test
  against the fake store with a **real `tar`/filesystem** producing the archived
  object — unprivileged, in-gate. (That `aws` actually accepts the object and
  `latest` reads back is the on-box check, outside the green gate.)
- R-4J4M-NRKP — `opsctl restore` refuses to proceed without interactive
  confirmation — **no `--yes`/non-interactive bypass path exists** — and takes **no**
  pre-restore safety snapshot (the restore path makes no `Put` to S3). Substrate:
  opsctl unit test (confirmation gate) + fake store (asserting no extra object
  written).
- R-4KCJ-1JBE — full round-trip: `backup` then `restore` reproduces `state/`
  byte-for-byte (DB + durable files) through the `ObjectStore` seam with a **real
  `tar` and real filesystem**, proving the archive format and the `ObjectStore`
  contract round-trip. Substrate: opsctl test, real `tar`/filesystem + `fakeStore`;
  unprivileged, in-gate. (Real-`aws` wire interop is the on-box check.)
- R-QQNU-T5M7 — a service binary's verb dispatcher exposes **no** `backup` and
  **no** `restore` case (both verbs removed — an invocation is an unknown-verb
  error), and the appkit `Spec` has **no** `Backup`/`Restore` hook field. (The full
  kept verb set is asserted by D11.) Substrate: appkit verb-dispatch unit test
  (`backup`/`restore` → unknown-verb error) + a compile-level assertion that `Spec`
  carries no backup/restore hook.
- R-82FY-GAL6 — the snapshot object key embeds the **producing version**:
  `opsctl backup <svc>` writes the object under
  `<svc>/<svc>-<version>.<UTC>.tar.gz` where `<version>` is the live deployed
  version read from `bin/run`'s target (D02/D03), so the version is recoverable from
  the key alone — a key with no embedded version fails this. This is the fact D10
  rollback reads to choose the binary to swap to. Substrate: opsctl unit test
  (assert the `Put` key against a seeded `bin/run` target) + fake store.
- R-TAOX-5LKS — `opsctl backup dashboard` produces a cert object under
  `dashboard/cert/` **and** its own `cert/latest`, distinct from the state
  object/pointer (the two streams are independently keyed). Substrate: opsctl unit
  test (fake store) + real `tar`/filesystem for the cert archive; unprivileged,
  in-gate. (Real-`aws` interop is the on-box check.)
- R-TBWT-JDBH — a dashboard restore lays `/etc/letsencrypt/{archive,renewal,live}/
  <domain>` back from `cert/latest` matching the backed-up bytes, **without invoking
  certbot** (no reissue — `System`'s certbot/issuance seam is never called during
  restore), and resolves the cert pointer independently of the chosen state
  snapshot. Substrate: real filesystem round-trip (cert tree) + stubbed `System`
  asserting no issuance call; cert bytes via the `ObjectStore` seam (`fakeStore`).
  Unprivileged, in-gate.
