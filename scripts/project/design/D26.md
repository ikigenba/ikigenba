# Decision 26 — `describe` teaches the runtime contract

**Problem.** `docs/structured-mcp-design.md` closes the scripts row with:
"`describe` teaches the runtime contract, so authoring agents write against
`suite` instead of hand-rolling HTTP." An agent authoring a script body sees
only what `describe` tells it; if the `suite` module isn't taught there, agents
will keep writing raw `urllib` against loopback ports — exactly the drift the
module exists to end. `describeText`'s RUNTIME CONTRACT section still describes
the pre-`suite` world (payload on stdin/`$EVENT_JSON`, stdlib only, and nothing
else).

**Decision.** `internal/mcp/describe.go`'s `describeText` is revised in place
(a documentation tool stays prose — D19's classification unchanged). The
revised text keeps the existing sections and grows the runtime contract:

- **The `suite` module**, importable in every run (`import suite`), with its
  five surfaces, each in a sentence with a one-line example:
  - `suite.event()` — the trigger payload as a dict, verbatim; `{}` for a
    manual run;
  - `suite.mcp(service, tool, args)` — call any suite service's MCP tool;
    returns the structured result (dict), or text for prose tools;
  - `suite.fetch(content_url, dest)` — land the bytes behind a content URL
    (from an event payload or a tool result) in a local file;
  - `suite.files.*` — the account's **file share** (list, stat, get, put,
    delete, move, mkdir): the durable, shared file store — files there
    persist, sync, and are what the owner and other workflows see; the run
    dir is private working space. Writing a product to the share is how a
    script publishes it durably (and how watching workflows trigger).
  - the error model: suite-service failures raise `suite.ToolError` with
    `.code` from the closed vocabulary (`validation`, `not_found`,
    `conflict`, `too_large`, `source_unavailable`, `internal`) — catch it and
    branch on `.code`, or let it crash the run (the failure lands in
    `stderr.log` and the run is marked failed).
- **Products travel by reference:** files a run writes into its run dir are
  addressable afterward — `run_fs_list` entries carry a `content_url` other
  services fetch (D25) — so a script hands results onward by writing a file,
  not by printing bytes.
- The framing stays **service-agnostic** for the share ("the file share",
  never the backing service's name — the D24 posture) and the existing
  sections (WHAT IT IS, LIFECYCLE, TRIGGERS) stay truthful and are not
  restated here.

**Rejected.**
- **Teaching the contract in the `initialize` instructions field** — every
  client pays it on every connection; `describe` is the on-demand home
  (`describe.go`'s existing posture).
- **A separate `runtime` tool** — a second documentation surface to keep in
  sync; `describe` already owns "call this first".
- **Enumerating peer services and their tools in the text** — the machine
  surface is self-describing (an agent with a service's tools loaded knows
  what a script may call — `docs/structured-mcp-design.md`); a hand-kept list
  would drift.

**Verification** (through the assembled MCP handler — the `describe` tool's
returned text, asserted by substring):

- R-IOUA-M8K1 — the `describe` result names the `suite` module and each of its
  surfaces (`suite.event`, `suite.mcp`, `suite.fetch`, `suite.files`), names
  `ToolError` and at least the `not_found` and `source_unavailable` codes,
  states that non-directory `run_fs_list` entries carry a `content_url`, and
  contains no backing-service name in the file-share guidance (the share
  paragraph names "file share", not the service behind it).
