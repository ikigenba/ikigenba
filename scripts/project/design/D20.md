# Decision 20 — Error-taxonomy enrichment: `too_large` and `source_unavailable` become real domain sentinels

**Problem.** D19's structured-error mapping deliberately covered only the
classifications the domain actually raises — `script.ErrNotFound`,
`script.ErrValidation`, and a default `internal` arm — and named two known
coarse spots as out of scope: the 1 MiB import cap is wrapped in
`ErrValidation` (`internal/script/service.go`, `maxImportBytes`), and a
dropbox-mirror fetch failure inside `import` surfaces as an unclassified error
and therefore maps to `internal`. Both conditions have exact codes in the
suite's closed vocabulary (`too_large`, `source_unavailable`); a machine caller
retrying a transient mirror outage or shrinking an oversized file cannot
distinguish either from a bad argument or a scripts bug. prompts made the same
move in its D27 (added `prompt.ErrTooLarge` / `prompt.ErrSourceUnavailable`);
this Decision is scripts' mirror of it.

**Decision.** Add two sentinel errors to the domain and classify `Import`'s two
coarse failure arms with them; extend the MCP mapping helper by two
`errors.Is` arms. No new tools, no schema change, no wire-surface change beyond
the two error codes.

1. **Domain sentinels** — alongside the existing pair in
   `internal/script/model.go`:

   ```go
   var (
       ErrNotFound          = errors.New("script: not found")
       ErrValidation        = errors.New("script: validation")
       ErrTooLarge          = errors.New("script: too large")
       ErrSourceUnavailable = errors.New("script: source unavailable")
   )
   ```

2. **`Service.Import` reclassifies its two arms** (message text otherwise
   unchanged):
   - the `maxImportBytes` cap wraps `ErrTooLarge` instead of `ErrValidation`
     (`"%w: %q is %d bytes, over the 1 MiB import limit"`);
   - a `Fetcher.Fetch` failure wraps `ErrSourceUnavailable`
     (`fmt.Errorf("%w: fetch %q: %v", ErrSourceUnavailable, sourcePath, err)`)
     instead of returning the fetch error unclassified.
   The UTF-8 rejection and the `source_path is required` guard remain
   `ErrValidation` — they are bad input, not size or availability.

3. **`structuredError` gains two arms** (D19 owns the helper; the full ladder
   is now):
   - `errors.Is(err, script.ErrNotFound)` → `mcp.ErrNotFound`
   - `errors.Is(err, script.ErrValidation)` → `mcp.ErrValidation`
   - `errors.Is(err, script.ErrTooLarge)` → `mcp.ErrTooLarge`
   - `errors.Is(err, script.ErrSourceUnavailable)` → `mcp.ErrSourceUnavailable`
   - otherwise → `mcp.ErrInternal`

`conflict` stays unused, and deliberately so: `import` upserts on
`(owner_id, source_path)` — re-importing a path is an update, never a collision —
and no other verb has a conflict state.

**Rejected.**
- *String-sniffing the wrapped messages at the MCP layer* — brittle, and D19
  already rejected it; the domain is the only honest classifier.
- *Reclassifying inside `internal/mcp` by call-site knowledge* (e.g. "any
  `import` failure that isn't validation is source_unavailable") — wrong the
  moment `Import` grows another failure mode; the sentinel keeps the
  classification where the knowledge lives.
- *Folding the UTF-8 rejection into `too_large`* — it is a content-type
  rejection, not a size bound; it stays `validation`.

**Verification.**
- R-CBF4-AYEU — `import` of a mirror file larger than 1 MiB returns
  `isError: true` with `structuredContent.code == "too_large"` (not
  `"validation"`), and the message still names the byte count and the 1 MiB
  limit.
- R-CCN0-OQ5J — `import` whose mirror fetch fails (injected `Fetcher` returning
  an error) returns `structuredContent.code == "source_unavailable"` (not
  `"internal"`).
- R-CDUX-2HW8 — `import` of invalid-UTF-8 mirror bytes still returns
  `structuredContent.code == "validation"` — pinning that only the size and
  fetch arms were reclassified.
