# Decision 28 — Owner-id keying: rebuild the `scripts` table, rekey scoping and snapshot on `owner_id`

> **Suite conversion (operator-sequenced).** This Decision is scripts' slice of
> the suite-wide owner-id conversion (`docs/owner-id-design.md`). It assumes
> appkit's shipped identity contract (its D13): `server.Identity` carries
> `OwnerID`/`OwnerEmail`/`OwnerName`/`OwnerPicture`/`ClientID`, and the identity
> gate refuses `POST /mcp` (401) exactly when `X-Owner-Id` is empty — email is
> no longer gated on. Until appkit's flip lands, scripts' `/mcp` tests are red;
> this Decision's phase leaves the module green again keyed on the id.

**Problem.** scripts keys ownership on the caller's **email**. The `scripts`
domain table stores `owner_email TEXT NOT NULL` (`002_scripts.sql`, "from
`X-Owner-Email` at create"), every owner-scoped verb (`list`, `get`, `update`,
`delete`, `run`, and the trigger verbs' `ownsScript` gate) filters on that
column, the `import` upsert's partial-unique index is
`idx_scripts_source ON scripts(owner_email, source_path)`
(`20260609135007_add_source_path.sql`), and the `Script`/`ScriptDetail` MCP
results expose the stored email as their `OwnerEmail` field. Email is becoming
user-mutable (the dashboard's GitHub alternate login), so it can no longer be
the identity key: a changed email would orphan a person's scripts. The dashboard
already mints a stable opaque owner id and nginx forwards it fail-closed as
`X-Owner-Id` (appkit `Identity.OwnerID`) on every authenticated request; scripts
must key on it.

`scripts` is the **only** owner-scoped table. `runs` and `script_triggers`
carry no owner column — both are scoped transitively through
`script_id REFERENCES scripts(id)` — so rekeying the `scripts` table rekeys the
whole domain; the other tables need no owner change.

**Decision.** `owner_id` becomes the sole scoping and lookup key; `owner_email`
becomes a write-once display snapshot; the domain table is rebuilt by one new
migration.

1. **One new timestamped migration** minted with
   `bin/create-migration scripts owner_id_keying` (never a hand-picked number;
   the frozen `002_scripts.sql` and `20260609135007_add_source_path.sql` are
   **not touched**). Inside the migration window it **drops and recreates**
   `scripts` with the *effective current* column set plus `owner_id`:

   ```sql
   -- rebuilt scripts table (rows dropped: a service cannot map stored emails
   -- to owner ids — only the dashboard knows that mapping)
   CREATE TABLE scripts (
       id          TEXT PRIMARY KEY,   -- ULID
       owner_id    TEXT NOT NULL,      -- X-Owner-Id at create: the sole scoping/lookup key
       owner_email TEXT NOT NULL,      -- X-Owner-Email at create: write-once display snapshot, never read for logic
       name        TEXT,
       body        TEXT NOT NULL,
       config_json TEXT NOT NULL,
       source_path TEXT,               -- carried forward from 20260609135007; nullable (hand-authored = NULL)
       created_at  TEXT NOT NULL,
       updated_at  TEXT NOT NULL
   );
   CREATE INDEX idx_scripts_owner ON scripts(owner_id);
   CREATE UNIQUE INDEX idx_scripts_source          -- the import-upsert invariant, rekeyed
       ON scripts(owner_id, source_path) WHERE source_path IS NOT NULL;
   ```

   Rows are dropped, per the suite decision (only ledger holds live customer
   data; scripts' `state/` is disposable during the migration window). Because
   `runs` and `script_triggers` FK-reference `scripts(id)` and hold only
   wipeable migration-window state, the migration clears their rows too so no
   dangling FK survives the rebuild (builder's choice of `DELETE`/rebuild — run
   and trigger rows are migration-window state per D17). The rebuild is
   deterministic on any database (dev, fresh, or the live box) and does not
   depend on anyone remembering to wipe `state/`.

2. **`owner_id` is the only identity read for logic.** Every query, filter, and
   authorization comparison in `internal/script` (`store.go`/`service.go`) keys
   on `owner_id`: `ListByOwner`, the `ownsScript` gate that guards `get`,
   `update`, `delete`, `run`, `set_trigger`, and `clear_trigger`, and the
   `import` upsert scope. Two callers presenting the same `X-Owner-Email` but
   different `X-Owner-Id` are **different** owners; two presenting the same
   `X-Owner-Id` are the same owner regardless of email. No store method accepts
   or filters on `owner_email`.

3. **`owner_email` is snapshotted at create, never updated, never read for
   logic.** `create` and `import` capture `Identity.OwnerID` (the key) and
   `Identity.OwnerEmail` (the snapshot) onto the new row from the authenticated
   identity nginx established — never from tool arguments. `update` changes body
   and config only; neither owner column is ever rewritten.

4. **The domain model and MCP results carry both.** The untagged `Script` /
   `ScriptDetail` structs (`internal/script/model.go`) gain an `OwnerID` field
   beside the existing `OwnerEmail`. Because these structs carry **no `json`
   tags** they marshal with Go field names verbatim (D19's casing rule), so the
   `get`/`list`/`update` `structuredContent` exposes the stored owner as
   **`OwnerID` and `OwnerEmail`** (PascalCase) — not snake-cased. D19 owns the
   output-schema fidelity and enumerates `OwnerID` among the `Script` keys; this
   Decision owns that both stored values are present on the surface.

The DB columns are named `owner_id`/`owner_email` (the suite-standard snake
names, Decision 3 of the suite design); the wire keys follow scripts' untagged
struct convention (`OwnerID`/`OwnerEmail`, PascalCase) unchanged from how
`OwnerEmail` already ships today. There is **no** refresh mechanism, resolve
endpoint, or identity cache: rows created before and after an email change
display different emails and the key never moves — the conscious suite trade.

**Rejected.**
- **Migrating existing rows (email → id backfill) instead of dropping them.**
  Only the dashboard knows the email→id mapping; the service cannot backfill.
  Suite-settled: rebuild empty inside the migration window, deterministic on any
  database.
- **Keying on `(owner_id, owner_email)` or falling back to email.** Re-anchors
  logic to the mutable display field; `owner_id` alone is the contract, and the
  snapshot is display-only.
- **Snake-casing the new `OwnerID` wire key for tidiness.** Would diverge from
  the untagged `Script` struct's actual bytes (`OwnerEmail` already ships
  PascalCase); D19's casing doctrine mirrors the wire, not a preference.
- **Adding owner columns to `runs`/`script_triggers`.** They are scoped through
  `script_id`; a redundant owner column would be a second, drift-prone copy of a
  fact the FK already carries.
- **Dropping `owner_email` entirely.** The suite keeps the email as a display
  snapshot so any future grouping/re-display fix has the data on the row; and
  the MCP results already expose it.

**Verification** (real substrates throughout: migration claims apply scripts'
full embedded migration set (`db.FS`) through the appkit runner to a fresh real
SQLite database and read back `PRAGMA table_info`/`index_list` + row counts;
scoping and snapshot claims drive the assembled `appkit/mcp` handler — the
D13/D19 harness — over a real `script.Service` on a temp-file SQLite DB via
`httptest`, with the authenticated identity supplied as `X-Owner-Id` plus
`X-Owner-Email`).

- R-Q2LM-XR9W — applying the full embedded migration set to a fresh real SQLite
  database (seeded with a pre-conversion `scripts` row before the new migration
  runs) yields a `scripts` table that has both `owner_id` and `owner_email` as
  `NOT NULL` columns, still has `source_path`, has a **UNIQUE** index over
  `(owner_id, source_path)` (not over `owner_email`) and a plain index on
  `owner_id`, and holds **zero rows** — the rebuild drops pre-conversion data
  rather than carrying rows with no owner id; and `002_scripts.sql` and
  `20260609135007_add_source_path.sql` remain byte-identical to their committed
  (frozen) bodies.
- R-Q3TJ-BJ0L — a `create` (and an `import`) persists a row whose `owner_id`
  equals the caller's `X-Owner-Id` and whose `owner_email` equals the caller's
  `X-Owner-Email`, exactly as supplied with **distinct, unrelated** id and email
  values — the email is stored as a snapshot, not derived from the id nor used
  as the key. (Real handler+DB — create with unrelated id/email, read the row
  back by SQL.)
- R-Q51F-PARA — `list` returns exactly the scripts whose `owner_id` is the
  caller's `X-Owner-Id` and never one owned by a different id, **even when both
  owners present the same `X-Owner-Email`** — an implementation still filtering
  on email fails. (Real handler+DB — two owners with identical emails but
  distinct ids.)
- R-Q69C-32HZ — `get`, `update`, `delete`, and `run` targeting a script owned by
  a **different** `owner_id` each act on nothing and return `not_found` (the
  `ownsScript` gate keyed on `owner_id`), **even when both callers present the
  same `X-Owner-Email`**; the owning caller then succeeds. (Real handler+DB —
  same-email / different-id two-owner setup; proves mutation scoping keys on the
  id, not the email.)
- R-Q7H8-GU8O — the `get`, `list`, and `update` `structuredContent` each expose
  the stored owner as **both** `OwnerID` and `OwnerEmail` (PascalCase, mirroring
  the untagged `Script` struct), equal to the row's stored values — a result
  exposing only the email fails. (Real handler+DB.)
- R-Q8P4-ULZD — the `import` upsert keys on `(owner_id, source_path)`:
  re-importing the same `source_path` under the same `owner_id` updates the one
  row (no duplicate), while the same `source_path` imported under a **different**
  `owner_id` that presents the **same** `X-Owner-Email` creates a distinct row —
  the partial-unique index scopes on the id, not the email. (Real handler+DB —
  the real SQLite UNIQUE constraint is the substrate.)
