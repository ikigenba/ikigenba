# Decision 4 — nginx fragment: the exact-match session-gated `= /srv/scripts/` location

**Decision.** scripts's nginx fragment (`scripts/etc/nginx.conf`) gains **one new
location** — an **exact-match** location for the bare mount root that
session-gates the human landing page and proxies it to the service's root. It
sits alongside (does not replace) the existing locations:

```
# Landing page — human web UI, gated by the dashboard browser SESSION (cookie),
# validated by the dashboard-owned /_session-authn internal location (a sibling of
# /_authn). Exact-match `=` so it never shadows the gated /srv/scripts/ prefix
# below or the MCP endpoint. A failed session check yields 401.
location = /srv/scripts/ {
    auth_request /_session-authn;
    auth_request_set $scripts_session_owner         $upstream_http_x_owner_email;
    auth_request_set $scripts_session_owner_id      $upstream_http_x_owner_id;
    auth_request_set $scripts_session_owner_name    $upstream_http_x_owner_name;
    auth_request_set $scripts_session_owner_picture $upstream_http_x_owner_picture;
    proxy_set_header X-Owner-Id      $scripts_session_owner_id;
    proxy_set_header X-Owner-Email   $scripts_session_owner;
    proxy_set_header X-Owner-Name    $scripts_session_owner_name;
    proxy_set_header X-Owner-Picture $scripts_session_owner_picture;
    proxy_pass http://127.0.0.1:3003/;   # trailing slash → upstream sees "/"
    proxy_set_header Host $host;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_http_version 1.1;
}
```

**Identity forwarding — all four owner headers.** The dashboard's
`/_session-authn` hook now emits **four** owner identity headers on every allow,
fail-closed — `X-Owner-Id` (durable owner key, never empty), `X-Owner-Email`
(display/audit, never empty), `X-Owner-Name` (percent-encoded, may be empty), and
`X-Owner-Picture` (percent-encoded, may be empty). This session location captures
each into its own `$scripts_session_owner*` variable and forwards it with a single
`proxy_set_header` per name, so all four reach the loopback upstream. The
service-prefixed variable names keep the fragment's variables unique across every
fragment included into the one apex server block, and one `proxy_set_header` per
name replaces any client-smuggled inbound header of that name (identity hygiene).
The session tier forwards **no** `X-Client-Id` (that is bearer-only). Downstream
consumption of `X-Owner-Id` by the scripts binary is **out of scope** here (a
later suite-wide conversion); this is fragment forwarding + its content
assertions only.

**How it coexists with the existing fragment.** scripts's fragment already has:

1. `location = /srv/scripts/.well-known/oauth-protected-resource` — unauthenticated
   PRM bootstrap (exact-match).
2. `location = /srv/scripts/feed { return 404; }` — the loopback event-plane feed
   is never reachable through nginx (exact-match).
3. `location /srv/scripts/ { auth_request /_authn; … }` — the bearer-gated
   **prefix** for the MCP endpoint and everything else under the mount, including
   its `@prompts_authn_500` rate-limit re-emit.

nginx longest-prefix/exact-match rules make an **exact-match `=` location win over
a prefix location** for that one path regardless of source order. So the new
`= /srv/scripts/` exact match wins for the **bare root only**, while the
`/srv/scripts/` prefix (item 3) still serves `/srv/scripts/mcp` and every other
sub-path under the mount. The two PRM/feed exact matches are unaffected (different
exact paths). Nothing in the existing fragment is removed or rewritten — the
change is **purely additive**.

**Why `/_session-authn`, not `/_authn`.** A browser carries the
`dashboard_session` cookie, not a bearer token, so the bearer gate (`/_authn`)
would refuse it. `/_session-authn` is the dashboard-owned, loopback-only internal
location (defined in `dashboard/etc/nginx.conf` as
`proxy_pass …/internal/session-authn`) that validates the cookie against the
web-session store and returns `200` + `X-Owner-Email` on success or `401` on
failure. This is the **same** mechanism `sites` already uses for its private
static tier (`sites/etc/nginx.conf` `location /srv/sites/private/`). The gate is
deliberately **coarse**: any logged-in dashboard user passes; there is no
per-resource check — fine for a name+version page.

**Why the simple form (no 429 re-emit).** The bearer prefix block carries the
`$authn_status`/`error_page 500 = @prompts_authn_500` machinery because token
introspection can return `429` (rate-limited) which `auth_request` collapses to a
synthetic `500`. The session check is a cookie lookup, not a rate-budgeted token
introspection, so the landing location does not need the 429 re-emit dance — it
maps the session subrequest's allow/deny directly (`200`→serve, `401`→401),
matching the `sites` private-tier location, which likewise omits it.

**Templating.** The loopback port is a **literal** in the fragment (`PORT=3003`,
scripts's fixed registry port), so nothing substitutes it at install — `opsctl
setup scripts` on the box / `nginx/run` in dev drop the fragment verbatim, exactly
like the existing locations. The fragment is **not** a vhost — no
`server{}`, no `listen`, no cert; it is dropped at
`/etc/nginx/conf.d/locations/scripts.conf` and included by the dashboard's apex
server block, which is also where `/_session-authn` is defined.

**Rejected.**
- **Gating the landing page with `/_authn`** — a browser has no bearer token; it
  would always 401. Wrong audience.
- **Making the landing page fully public (no auth_request)** — the brief forbids
  it; a logged-out browser must get `401`. The page is gated, just coarsely.
- **A prefix `location /srv/scripts/` for the page** — would collide with the
  existing bearer prefix; the exact-match `=` is what cleanly carves the bare root
  out of the prefix without touching the rest.
- **Adding the `@prompts_authn_500` 429 re-emit to the session location** — the
  session check is not rate-budgeted; the extra machinery is dead weight (the
  `sites` private tier omits it for the same reason).
- **Editing the existing `/srv/scripts/` prefix block** — unnecessary and risky;
  the change is purely an added exact-match location.

**Verification.** (Proven by a test that reads `scripts/etc/nginx.conf` from disk
and asserts over its content — see design's Testing strategy.)
- R-NGNX-2A5Q — `scripts/etc/nginx.conf` contains an **exact-match** location for
  the bare mount root: a `location = /srv/scripts/ {` block (distinct from the
  prefix `location /srv/scripts/ {`).
- R-NGNX-4B7R — that exact-match block gates with `auth_request /_session-authn`
  (the cookie/session validator) and **not** `auth_request /_authn` (the bearer
  validator).
- R-NGNX-6C9S — that block proxies to the loopback upstream **root**
  (`proxy_pass http://127.0.0.1:3003/;` — trailing slash), so the upstream
  sees `/`.
- R-NGNX-8D1T — the pre-existing locations are retained alongside it: the
  bearer-gated `location /srv/scripts/ {` prefix with `auth_request /_authn`, the
  `location = /srv/scripts/feed { return 404; }`, and the
  `location = /srv/scripts/.well-known/oauth-protected-resource` PRM bootstrap all
  still appear in the file.
- R-LYZ1-YEW5 — the exact-match session location `= /srv/scripts/` captures **all
  four** owner headers into service-prefixed variables — `auth_request_set
  $scripts_session_owner $upstream_http_x_owner_email`, `auth_request_set
  $scripts_session_owner_id $upstream_http_x_owner_id`, `auth_request_set
  $scripts_session_owner_name $upstream_http_x_owner_name`, `auth_request_set
  $scripts_session_owner_picture $upstream_http_x_owner_picture` — and forwards
  each with a single `proxy_set_header` per name: `X-Owner-Id
  $scripts_session_owner_id`, `X-Owner-Email $scripts_session_owner`, `X-Owner-Name
  $scripts_session_owner_name`, `X-Owner-Picture $scripts_session_owner_picture`
  (and forwards **no** `X-Client-Id`, which is bearer-only).
