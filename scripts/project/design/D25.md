# Decision 25 — Content-plane holder: run files at `GET /run-content`, references on `run_fs_list`

**Problem.** The content plane makes scripts a **holder**: a run's product (an
extracted JSON, a generated report) must be addressable by a loopback content
URL so it travels onward **by reference** — another run's `suite.fetch`, a
prompts run's `Fetch`, a share `put(source_url=…)` — instead of round-tripping
bytes through `run_fs_read` and a caller's context. Today a run's files are
reachable only through the line-oriented MCP readers; nothing can name their
bytes. This is prompts' D22 lifted onto scripts' run dirs.

**Decision.** Two halves, exactly the D22 shape:

1. **The endpoint: `GET /run-content?run_id=<id>&path=<run-dir-relative path>`**
   — a new `Service.RunContentHandler() http.Handler` in
   `internal/script/content.go`:
   - **Loopback-only guard, chassis-owned:** mounted in `registerRoutes` via
     `rt.HandleLoopback("GET /run-content", svc.RunContentHandler())` — the
     shared `server.LoopbackOnly` wrap, keyed on `X-Forwarded-Proto` only
     (nginx stamps it on every proxied request → bare 404; a loopback caller is
     served). The handler itself reads no request header; the route is
     otherwise unauthenticated — one box, one owner, the perimeter is loopback.
     **No owner scoping** (an event-driven caller has no identity; the same
     trust model as the share's holder route). The guard's behavior is the
     chassis's, verified in appkit — not re-proven here.
   - **Resolution through the existing confinement seam:** `run_id` must be a
     single clean path element (no separator, no `.`/`..`), then the file
     resolves via the same `resolveWithin(runDir(runID), path)` machinery
     `RunFsList`/`RunFsRead` already use. The open file is served with
     `http.ServeContent` (extension-derived `Content-Type`, `Content-Length`,
     Range/conditional for free), passing the file's modtime.
   - **Error mapping never leaks:** an invalid or unknown `run_id`, an absent
     path, a directory, or a confinement/escape error → bare **404** (never
     500, never a path echo). **No `rev` parameter**: run files are unversioned
     working files, so the plane's optional rev→409 pin has nothing to pin — a
     documented deviation-by-omission (prompts D22 and gmail made the same
     one). A consumer needing integrity hashes what it fetched.
   - **Everything under the run dir is served, no exclusions** — runner
     artifacts (`main.py`, `config.json`, the logs) included: all of it is
     already owner-visible via `run_fs_list`/`run_fs_read`, a carve-out would
     be policy protecting nothing, and anything listed with a `content_url`
     must actually be fetchable.

2. **`run_fs_list` entries carry the reference.** `FileEntry` gains
   `ContentURL string \`json:"content_url,omitempty"\``; each **non-directory**
   entry carries the complete, URL-encoded loopback URL
   (`url.Values.Encode` over `run_id` + the entry's run-dir-relative path) —
   the consumer never constructs or rewrites it. Directory entries carry none
   (nothing at the endpoint to name). No `content_hash` (stamping one would
   read every file on every list — the same rejection gmail and prompts
   recorded) and no `rev` (above). The domain stays URL-ignorant: the MCP
   layer populates the field after `RunFsList` returns. The base is resolved
   by registry name at the composition root and threaded through the tool
   table — gmail D17 / prompts D22's exact seam:

   ```go
   // cmd/scripts/main.go, registerRoutes:
   contentBase := registry.BaseURL("scripts") // registry stays composition-root-only (D10)

   // internal/mcp (signatures revised in place):
   func Tools(svc *script.Service, contentBase string) []appkitmcp.Tool
   func NewHandler(svc *script.Service, contentBase string, rt *appkit.Router) (http.Handler, error)
   ```

   `run_fs_list`'s tool description states that non-directory entries carry a
   loopback `content_url` for byte fetch **by services** (a run's
   `suite.fetch`, a share `put(source_url)`), not by the agent. The
   `run_fs_list` `outputSchema` (D19) is extended in place: the `FileEntry`
   item schema declares `content_url` as an optional string property. No tool
   is added or removed — the wire surface stays **18 tools**.

**Rejected.**
- **Serving bytes through MCP** (`run_fs_read` returning whole files/base64) —
  the by-value crossing the plane retires; `run_fs_read` stays the
  line-oriented convenience it is.
- **An env override for the content base** — the base names *this service's
  own* port, fixed by the registry; a knob nobody can meaningfully set is
  drift surface. Tests build expected URLs from `registry.MustPort("scripts")`.
- **Opaque minted tokens instead of `(run_id, path)`** — holders address by
  their own key; run_id + run-dir path *is* scripts' key, already visible via
  `run_fs_list`, and a token store adds state for no confinement gain.
- **A `rev` pin** — run files have no revision identity; 409 would be fiction.
- **Excluding runner artifacts from the endpoint** — a policy surface guarding
  files the MCP readers already expose, at the cost of listed-but-unfetchable
  entries.

**Verification** (real substrates: the handler claims drive the real
`RunContentHandler` over a real temp run-dir tree via `net/http/httptest`; the
`run_fs_list` claim goes through the assembled `appkit/mcp` handler over a real
`script.Service` + temp DB, expected URLs built from
`registry.MustPort("scripts")`, never a port literal).

- R-IIQS-PDUK — `GET /run-content?run_id=…&path=…` (path URL-encoded, e.g.
  `reports%2Fout.pdf`) for an existing run file returns 200 with a body
  byte-identical to the file, `Content-Length` equal to its size, and a
  `Content-Type` derived from the extension.
- R-IJYP-35L9 — an unknown `run_id`, a `run_id` containing a path separator,
  an absent `path`, a directory `path`, and an escaping `path`
  (`../../secrets`) each return **404** — never 500 and never the resolved
  path in the response body.
- R-IMEH-UP2N — `run_fs_list` through the assembled MCP handler returns, for
  each non-directory entry, a `content_url` equal to
  `http://127.0.0.1:<registry scripts port>/run-content?<encoded query>` whose
  parsed query round-trips exactly the run's id and the entry's run-dir
  relative path (proving the encoding, including a subdirectory entry), while
  directory entries carry **no** `content_url` key — and fetching a returned
  `content_url` against the mounted handler yields that file's bytes (the
  reference is live, not decorative).
- R-INME-8GTC — in `tools/list`, `run_fs_list`'s `outputSchema` declares
  `content_url` as a property of its entry item schema — the declared machine
  contract includes the reference — while `run_fs_list` results still
  deep-equal their mirrored text block (D19's R-C0G0-V0QL invariant holds with
  the new field present).
