# Decision 24 — `suite.files`: the file share's filesystem API, service-agnostic

**Problem.** The account's durable, shared file state lives in the file share
(the suite's filesystem authority). A run that produces something the owner or
another workflow should see — an extraction landed beside its source PDF, a
report in a watched folder — must write it **to the share**, not just its own
run dir; and a run reacting to share events needs path-addressed reads,
listings, and metadata. The sanctioned channel is the share's **loopback
filesystem API** (`dropbox/docs/filesystem-api.md`), moving bytes disk-to-disk
— not the share's MCP surface (base64 crossings, 25 MiB caps). This is
prompts' D26 translated to the scripts runtime, extended to the API's full
verb set.

**Decision.** A namespace object in `suite.py` wrapping all seven routes of the
filesystem API, addressed at `SUITE_FILES_BASE_URL` (D21):

```python
suite.files.list(path=None, cursor=None, limit=None)  # GET  /list
suite.files.stat(path)                                # GET  /stat
suite.files.get(share_path, dest)                     # GET  /content?path=
suite.files.put(source, share_path)                   # PUT  /content?path=  (body streamed)
suite.files.delete(path)                              # DELETE /content?path=
suite.files.move(src, dest)                           # POST /move?from=&to=
suite.files.mkdir(path)                               # POST /mkdir?path=
```

(`move` takes `src`/`dest` — `from` is a Python keyword.)

- **Service-agnostic.** The module speaks of "the file share"; neither the code
  nor its docstrings contain the backing service's name (D21 pins the base URL
  indirection). The surface survives a backing change.
- **Caller identity.** Every request carries `X-Client-Id:
  scripts:$SUITE_SCRIPT_ID` and **no** `X-Owner-Email` and no
  `X-Forwarded-Proto` (the header the share's loopback guard 404s on) — the
  share stamps that client id as the emitted event's `origin`, so a run-written
  file triggers watching workflows exactly like an owner-written file, with an
  honest origin.
- **Share paths** are `/`-rooted mirror display paths, URL-escaped into query
  params, validated by the share itself — full share access, no scripts-side
  scoping (the box is single-tenant; a fence here restricts nothing real —
  prompts D26's exact reasoning).
- **Local paths** (`get`'s `dest`, `put`'s `source`) follow D21's posture:
  resolved as `open()` would, no confinement, native `OSError` for local
  failures (a missing `put` source is the natural `FileNotFoundError`, raised
  before any HTTP).
- **Streaming, both directions, no size caps.** `get` reuses D23's streaming
  core: temp file in `dest`'s directory, SHA-256 on the same pass,
  `os.replace` — a torn stream leaves no partial file. `put` streams the local
  file as the request body with `Content-Length` from its size (never buffered
  whole in memory).
- **Return shapes — verbatim over invented:**
  - `get` → `{"path": <dest as given>, "size": <bytes>, "content_hash":
    <64-hex sha256>}` — scripts' own hash of what it wrote (D23's rule; the
    share's `content_hash` is its own algorithm on its own side);
  - `put` → the share's response JSON **verbatim**
    (`{path, size, content_hash, rev}`);
  - `list` / `stat` → the API's response JSON **verbatim**
    (`{files, next_cursor?}` / the metadata object);
  - `delete` / `move` / `mkdir` → **`None`** (the API returns bare 204;
    echoing arguments back at code that already holds them is invention).
    Success is "it returned"; failure raises. Consequence, inherited from the
    share's contract: `delete` of an absent path is **idempotent success**.
- **Failure mapping** (mechanically from HTTP status, so share-side refinements
  flow through unchanged; each raises `ToolError`): **400** →
  `validation` with the response body's detail text in the message; **404** →
  `not_found` (a stale path — re-derive it, e.g. with `list`); **409** →
  `conflict`; connection refused, timeout, any other status, or a torn body →
  `source_unavailable`. After a failed `get`, no file exists at `dest`.

**Rejected.**
- **The share's MCP surface as the channel** — base64 through the caller with
  a 25 MiB cap; exactly what the content plane retires. (A script may still
  call it via `suite.mcp` — nothing blocks it — but `suite.files` is the
  sanctioned byte channel and the one `describe` teaches.)
- **Dropping `stat` (prompts D26's choice)** — that was model-facing
  minimalism: fewer tools, less prompt surface. For a code API, `stat` is the
  natural cheap existence/metadata check; forcing a paged `list` walk to
  answer "does this file exist" is a gratuitous mismatch with an API that has
  the route.
- **Naming the backing service** (`suite.dropbox.*`, "dropbox" in docstrings)
  — the runner doesn't need the name and the surface would break if the
  backing changed; "file share" is the durable concept.
- **Scripts-side share scoping** — single-tenant box; the run already holds
  the owner's full authority (it could reach the share's MCP `put`
  regardless); a fence costs a policy surface and restricts nothing.
- **Synthesized success echoes for the 204 verbs** — `{"deleted": path}`
  served prompts' *model*, which needs text to read; code holds its arguments.

**Verification** (real substrate: the D21 probe harness against a real
`httptest` loopback server standing in for the share's filesystem API —
recording method, path, query, headers, and body; injected via
`SUITE_FILES_BASE_URL`; local rejections assert **zero** recorded requests).

- R-IBFE-EREE — across all seven verbs' happy paths, every recorded request
  carries `X-Client-Id` exactly `scripts:$SUITE_SCRIPT_ID` and carries **no**
  `X-Owner-Email` and no `X-Forwarded-Proto` header.
- R-ICNA-SJ53 — `get` of a share path the server serves with known bytes
  issues exactly one `GET /content` with the URL-escaped `path` query, streams
  the bytes to `dest` (file bytes equal served bytes), and returns
  `{path, size, content_hash}` with the lowercase-hex SHA-256 of those bytes.
- R-IDV7-6AVS — `put` of an existing local file issues exactly one
  `PUT /content` whose body bytes equal the file's bytes and whose `path`
  query is the URL-escaped `share_path`, and returns the server's
  `{path, size, content_hash, rev}` JSON verbatim — while a missing local
  `source` raises `FileNotFoundError` with **zero** requests.
- R-IF33-K2MH — `list(path=…, cursor=…, limit=…)` issues `GET /list` passing
  all three through as query params and returns the server's
  `{files, next_cursor}` JSON verbatim; `stat(path)` issues `GET /stat?path=`
  and returns the server's metadata object verbatim.
- R-IGAZ-XUD6 — `delete` issues exactly one `DELETE /content?path=`, `mkdir`
  exactly one `POST /mkdir?path=`, and `move` exactly one
  `POST /move?from=&to=` (params URL-escaped), each returning `None` on the
  API's 204.
- R-IHIW-BM3V — the failure mapping against the real server: a **400** with a
  detail body raises `ToolError` `.code == "validation"` with that detail in
  `.message`; a **404** → `"not_found"`; a **409** → `"conflict"`; a
  connection-refused base URL → `"source_unavailable"` — and after a failed
  `get`, no file exists at `dest`.
