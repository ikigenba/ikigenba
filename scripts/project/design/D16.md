# Decision 16 — the bearer tier's identity plumbing: scripts-named variables forwarding all four owner headers

**Decision.** This Decision owns the **bearer prefix** `location /srv/scripts/`'s
identity plumbing in `scripts/etc/nginx.conf`: the `auth_request_set` captures
from the introspection subresponse and the `proxy_set_header` forwards to the
loopback upstream, plus the 429 re-emit named location. It fixes two things —
the prompts-inherited **names** and the **set of identity headers forwarded**.

**Scripts-owned names, not prompts.** The bearer tier carried copy-paste heritage
from the prompts fragment: its identity variables and its 429 re-emit named
location were named after **prompts** — `$prompts_owner` / `$prompts_client`,
`proxy_set_header … $prompts_owner` / `$prompts_client`,
`error_page 500 = @prompts_authn_500;`, and `location @prompts_authn_500`. Every
one is the scripts-owned equivalent — `$scripts_owner`, `$scripts_client`, and
`@scripts_authn_500`. This is **hygiene**: nginx tolerates a duplicate named
location (verified empirically — a config with two `location @prompts_authn_500`
blocks passes `nginx -t`), and `auth_request_set` variables are per-request, set
only in the matched location, so nothing malfunctioned today even with both
fragments installed in the one apex server block. But a fragment that claims
another service's names inside that shared block — two services defining
`@prompts_authn_500`, `$prompts_*` variables misattributing scripts's identity
plumbing — is exactly the silent coupling that turns a future prompts change into
a scripts incident. The scripts names remove the ambiguity at zero behavioral
cost, and the string `prompts` must not occur anywhere in the fragment.

**All four owner headers, not just email.** The dashboard's introspection
endpoints now emit **four** owner identity headers on every allow, fail-closed —
`X-Owner-Id` (durable owner key, never empty), `X-Owner-Email` (display/audit,
never empty), `X-Owner-Name` (percent-encoded, may be empty), and
`X-Owner-Picture` (percent-encoded, may be empty) — plus `X-Client-Id` on the
bearer tier as today. The bearer prefix historically captured and forwarded only
`X-Owner-Email` (via `$scripts_owner`) and `X-Client-Id` (via `$scripts_client`);
the three new owner headers died at nginx. This Decision extends the plumbing to
**capture and forward all four** owner headers, each through its own
service-prefixed `auth_request_set` variable so the names stay unique across all
fragments included into the single apex server block:

```
location /srv/scripts/ {
    auth_request /_authn;
    auth_request_set $scripts_owner         $upstream_http_x_owner_email;
    auth_request_set $scripts_owner_id      $upstream_http_x_owner_id;
    auth_request_set $scripts_owner_name    $upstream_http_x_owner_name;
    auth_request_set $scripts_owner_picture $upstream_http_x_owner_picture;
    auth_request_set $scripts_client        $upstream_http_x_client_id;

    auth_request_set $authn_status $upstream_status;
    auth_request_set $authn_retry  $upstream_http_retry_after;
    error_page 500 = @scripts_authn_500;

    # Identity hygiene: one proxy_set_header per name from the auth subrequest
    # variable REPLACES any client-smuggled inbound header of that name — a
    # client cannot smuggle identity through. Extended to all four owner headers.
    proxy_set_header X-Owner-Id      $scripts_owner_id;
    proxy_set_header X-Owner-Email   $scripts_owner;
    proxy_set_header X-Owner-Name    $scripts_owner_name;
    proxy_set_header X-Owner-Picture $scripts_owner_picture;
    proxy_set_header X-Client-Id     $scripts_client;

    proxy_pass http://127.0.0.1:3003/;
    proxy_set_header Host $host;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_http_version 1.1;
}
```

The identity-hygiene property (a single `proxy_set_header` per name replaces any
client-smuggled inbound header) extends to the three new headers unchanged. The
429 re-emit machinery (`$authn_status` / `error_page 500 = @scripts_authn_500` /
`location @scripts_authn_500`) is untouched. Downstream consumption of
`X-Owner-Id` by the scripts binary is **out of scope** for this Decision (a later
suite-wide conversion); this Decision is fragment forwarding + its content
assertions only.

**Rejected.**
- **Leaving the prompts names as-is** — it works, but a shared server block where
  two fragments define the same named location and pass identity through another
  service's variables is exactly the silent coupling to avoid.
- **Forwarding only `X-Owner-Email` (the old two-header set)** — the durable
  `X-Owner-Id` and the display `X-Owner-Name` / `X-Owner-Picture` would die at
  nginx, so the scripts binary could never receive them; the whole point of the
  dashboard emitting four headers is that services can consume them.
- **Reusing one variable for several headers, or unprefixed names** — every
  fragment is included into one apex server block; a bare `$owner_id` would
  collide across services. Service-prefixed names (`$scripts_owner_id`, …) keep
  each fragment's variables unique.
- **Folding this into the D15 bounce opt-in or D4** — D15 is the session-tier
  bounce opt-in and D4 is the session-tier location; the bearer tier's identity
  plumbing is a distinct owner. One owner per concern.

**Verification.** (Proven by extending the existing nginx content-assertion test
(`cmd/scripts/main_test.go`) that reads `scripts/etc/nginx.conf` from disk;
nginx is not run by the suite.)
- R-4EOV-BQWQ — in `scripts/etc/nginx.conf`, the bearer prefix `location
  /srv/scripts/` sets and forwards identity through scripts-named variables
  (`auth_request_set $scripts_owner` / `$scripts_client`, `proxy_set_header
  X-Owner-Email $scripts_owner` / `X-Client-Id $scripts_client`), its 429
  re-emit is `error_page 500 = @scripts_authn_500;`, and `location
  @scripts_authn_500` is defined.
- R-4FWR-PINF — the string `prompts` does not occur anywhere in
  `scripts/etc/nginx.conf` (no prompts-named variable, named location, or
  comment reference survives the rename).
- R-LXR5-KN5G — the bearer prefix `location /srv/scripts/` captures **all four**
  owner headers into service-prefixed variables — `auth_request_set
  $scripts_owner_id $upstream_http_x_owner_id`, `auth_request_set
  $scripts_owner_name $upstream_http_x_owner_name`, `auth_request_set
  $scripts_owner_picture $upstream_http_x_owner_picture` (alongside the existing
  `$scripts_owner` ← `$upstream_http_x_owner_email`) — and forwards each with a
  single `proxy_set_header` per name: `X-Owner-Id $scripts_owner_id`,
  `X-Owner-Email $scripts_owner`, `X-Owner-Name $scripts_owner_name`,
  `X-Owner-Picture $scripts_owner_picture` (and `X-Client-Id $scripts_client`,
  unchanged).
