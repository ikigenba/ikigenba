# Decision 21 — The runner-injected `suite` Python module: embed, materialize, environment

**Problem.** The structured MCP revision (`docs/structured-mcp-design.md`) names
scripts as the suite's **first machine consumer**: a run's `main.py` should call
suite services through their MCP verbs, fetch content-plane bytes, and read/write
the account's file share — without hand-rolling HTTP, header conventions, and
error parsing in every script. Today a run receives only the raw event payload
(stdin / `$EVENT_JSON`) and knows nothing else: no service addresses, no
identity, no client. The payoff surface is a **`suite` Python module** a script
simply imports; this Decision owns the module's chassis — how it ships, how it
reaches the run dir, what runtime facts the runner injects, the module-wide
error model, and `suite.event()`. The verb clients ride on top: `suite.mcp`
(D22), `suite.fetch` (D23), `suite.files` (D24).

**Trust posture (design fact).** A run is **not sandboxed**: `python3 main.py`
executes as the service user with unrestricted filesystem and network access —
the trust decision happened when the owner authored the script. `suite` is
therefore a **contract and convenience surface, not a confinement boundary**:
it never fences local paths (unlike prompts' sandbox tools), and nothing it
validates is a security mechanism. The suite-side guards (nginx, the loopback
guard, the share's own path confinement) are the real boundaries and live
elsewhere.

**Decision.**

1. **One embedded stdlib-only source file.** The module is a single Python file,
   `scripts/internal/runner/suite.py`, embedded in the scripts binary:

   ```go
   // internal/runner/runner.go
   //go:embed suite.py
   var suitePy string
   ```

   It uses only the Python ≥ 3.11 standard library (`json`, `os`,
   `urllib.request`, `urllib.parse`, `hashlib`, `tempfile`, `shutil`) — the
   runtime contract's "stdlib only" holds for the module itself. `suite.files`
   is a namespace object defined inside this one file (no package, no second
   file).

2. **Materialized beside `main.py` on every run.** `Runner.execute` writes
   `suite.py` (0600) into the run dir in the same materialization step that
   writes `main.py` and `config.json`, before the interpreter starts. The run
   dir is the process cwd, so `import suite` resolves with no `PYTHONPATH`
   games. Every run gets the binary's own copy — module and service can never
   skew.

3. **Runtime facts travel as environment variables**, following the existing
   `EVENT_JSON` convention. `Runner.execute` appends to `cmd.Env`:

   - `SUITE_SERVICES` — a JSON object mapping every registered service name to
     its loopback origin (`{"crm": "http://127.0.0.1:3100", …}`).
   - `SUITE_FILES_BASE_URL` — the file share's loopback origin (the same
     resolved value the composition root already wires into `svc.Fetcher`:
     `DROPBOX_BASE_URL` env override, else the registry default). A separate
     variable so the module stays service-agnostic — `suite.py` never contains
     the string `dropbox`.
   - `SUITE_SCRIPT_ID` — `run.ScriptID`; `SUITE_RUN_ID` — `run.ID`;
     `SUITE_OWNER_ID` — the script row's `OwnerID` (the owner key `suite.mcp`
     asserts as `X-Owner-Id`, the field the callee's gate now requires — D22,
     appkit D13); `SUITE_OWNER_EMAIL` — the script row's `OwnerEmail` (the
     display snapshot). The runner already fetches the row via `ScriptForRun`
     and it now carries both owner columns (D28).

   Per D10's convention the `registry` module stays a composition-root import:
   `runner.New` grows a parameter carrying the injected environment —
   `New(store, dataDir, ttl, suiteEnv []string)` where `cmd/scripts/main.go`
   builds `suiteEnv` from `registry.Services` + the resolved files base. The
   per-run variables
   (`SUITE_SCRIPT_ID`/`SUITE_RUN_ID`/`SUITE_OWNER_ID`/`SUITE_OWNER_EMAIL`) are
   appended per spawn.

4. **Loud outside the runner.** Every `suite` function that needs a runtime
   fact reads its variable lazily and raises
   `RuntimeError("suite: not running under the scripts runner (missing <VAR>)")`
   when it is absent. No fallbacks, no guessed defaults.

5. **`suite.event()`** returns `json.loads(os.environ["EVENT_JSON"])`, parsed
   once and cached — the trigger payload **verbatim**, no reshaping, so the
   payload each producer documents in its `reflection` is exactly what the
   script sees. A manual run (the runner's literal `"{}"` input) yields `{}`,
   which is idiomatically falsy: `if not suite.event():` distinguishes manual
   from triggered.

6. **The module-wide error model: exceptions.** `suite.py` defines

   ```python
   class ToolError(Exception):
       def __init__(self, code, message):
           super().__init__(f"{code}: {message}")
           self.code = code        # closed vocabulary: validation | not_found |
           self.message = message  # conflict | too_large | source_unavailable | internal
   ```

   Every suite-service failure in D22–D24 raises `ToolError`; transport
   failures (connection refused, timeout, torn body) map to
   `source_unavailable`. A script that ignores a failure crashes with a
   traceback in `stderr.log` and the run fails — fail loudly. **Local** errors
   keep their native Python types: a programming error (unknown service name)
   is `ValueError`/`TypeError`; a local filesystem error is its native
   `OSError`. `ToolError` is reserved for the suite-service interaction itself.

7. **Local paths are not confined.** Relative paths in D23/D24 resolve against
   the process cwd (the run dir) exactly as `open()` does; absolute paths are
   honored. Rationale as the trust posture above — a fence here restricts
   nothing real (the same script bypasses it with `open()` one line later) and
   would cost validation code and error cases. Keeping run products *in* the
   run dir (so `/run-content` can serve them, D25) is convention taught by
   `describe` (D26), not mechanism.

8. **HTTP posture (module-wide, used by D22–D24):** every request sets a
   30-second socket-operation timeout (bounds connect and each read — a stalled
   peer errors; a flowing large body is never cut) and **no overall body
   deadline** — the run TTL already bounds the whole process. No retries
   anywhere: a caller that wants retry writes the loop.

**Rejected.**
- **Baking the registry table into `suite.py` at build time** — a second copy of
  the registry that drifts silently; env injection keeps the Go registry the
  single runtime source of truth and matches the `EVENT_JSON` convention.
- **A materialized `suite_env.json` file** — a third run-dir artifact carrying
  what env vars already carry; no gain.
- **A Python package / multiple files / a pip dependency** — one stdlib file is
  the whole contract; packaging machinery would buy nothing and break the
  "stdlib only" runtime promise.
- **Result-style error returns (`ok, err` tuples)** — makes ignoring errors the
  default, exactly backwards for deterministic automation; exceptions crash the
  run loudly unless deliberately caught.
- **Sandbox-style path confinement in the module** — see the trust posture: the
  script already runs unconfined; a suite-side fence is policy surface
  protecting nothing (the same reasoning prompts' D26 used to reject per-run
  share scoping).
- **A `suite.config()` accessor** — `config.json` holds runner config
  (`interpreter`, `timeout_secs`), not script parameters; nothing a script
  legitimately reads. It stays reachable via `json.load` if ever needed.

**Verification** (real substrates: Go tests in `internal/runner` exec the real
`python3` — the precedent `runner_test.go` set — on probe scripts that
`import suite`, in a temp run dir materialized exactly as the runner
materializes it, with test-controlled env; see the spine's testing-strategy
bullet for the probe harness).

- R-HVKP-FQRD — a spawned run's dir contains `suite.py` **byte-identical** to
  the embedded source, written before the interpreter starts: a run whose body
  is `import suite` (and nothing else) succeeds, and the materialized
  `suite.py` bytes equal the embedded `suitePy` string. A runner that skips
  materialization, or materializes after exec, fails it.
- R-HWSL-TII2 — the child process environment carries the injected facts: a run
  whose body prints `os.environ["SUITE_SERVICES"]`, `SUITE_FILES_BASE_URL`,
  `SUITE_SCRIPT_ID`, `SUITE_RUN_ID`, `SUITE_OWNER_ID`, and `SUITE_OWNER_EMAIL`
  emits (to `stdout.log`) the composition-root-injected services JSON, the files
  base, and exactly the spawning run's script id, run id, and the script row's
  owner id and owner email.
- R-HY0I-7A8R — `suite.event()` returns the trigger payload **verbatim**: a
  probe run spawned with input `{"a": {"b": [1, 2]}, "c": "x"}` observes
  `suite.event()` deep-equal to that object, and a probe spawned with the
  manual-run input `"{}"` observes exactly `{}`. A module that reshapes,
  wraps, or renames payload keys fails it.
- R-HZ8E-L1ZG — outside the runner the module is loud, not lenient: with
  `EVENT_JSON` unset, `suite.event()` raises `RuntimeError` whose message
  names the missing variable; with `SUITE_SERVICES` unset, `suite.mcp(...)`
  raises the same shape (probe runs `python3` directly with a scrubbed env,
  asserting the raised type and message substring, no fallback value).
