# Decision 23 — `suite.fetch(content_url, dest)`: the content-plane acceptor

**Problem.** Bytes never ride MCP: event payloads and tool results carry
`content_url` **references** (a dropbox event's file, a gmail attachment,
another run's product), and a consumer follows the reference to get bytes.
A run needs the acceptor half — land the bytes behind a content URL on its own
disk — without hand-rolling the plane's confinement and integrity rules. This
is prompts' D21 (`Fetch`) translated to the scripts runtime; the confinement
rules are lifted verbatim because they define what a content URL *is*, while
the destination side drops prompts' sandbox fence per D21's trust posture.

**Decision.** One function in `suite.py`:

```python
def fetch(content_url, dest):
    """Fetch a suite content URL to a local file; returns {path, size, content_hash}."""
```

- **URL confinement (the content-plane acceptor rule — validated before any
  network I/O).** The URL must parse, use scheme `http`, have host literally
  `127.0.0.1` or `::1` (no hostnames — `localhost` is rejected; the plane is
  loopback plumbing, not name resolution), and carry an explicit port that
  appears among the ports of the `SUITE_SERVICES` origins (plus
  `SUITE_FILES_BASE_URL`'s port). Any violation raises
  `ToolError("validation", …)` naming the violated rule, with **zero** HTTP
  requests. This is a *contract* check (what a content URL is), not a sandbox —
  D21's trust posture stands.
- **Destination.** `dest` resolves as `open()` would (relative → the run-dir
  cwd; absolute honored); parent directories must exist. Local I/O failures
  raise their native `OSError` (D21's error model) — `ToolError` is reserved
  for the transfer itself.
- **The fetch.** One `GET` with the D21 HTTP posture (30 s socket-op timeout,
  no body deadline, no retries) and **redirects not followed** — a 3xx could
  re-target the fetch outside the confined origin, so any 3xx is a failure
  (`source_unavailable`), via an opener whose redirect handler refuses. The
  body streams to a temp file in `dest`'s directory with the SHA-256 computed
  on the same pass (`hashlib.sha256` fed per chunk), then `os.replace` onto
  `dest` — a torn stream leaves **no partial file at `dest`**. Deliberately
  **no size cap**: bytes land on disk, never in an agent context.
- **Result:** `{"path": <dest as given>, "size": <bytes written>,
  "content_hash": <64-char lowercase-hex SHA-256>}` — plain SHA-256 of the
  bytes written (the plane pins only "64-hex content hash"; prompts D21's exact
  rule, deliberately not dropbox's block-SHA-256).
- **Failure mapping** (each raises `ToolError`; after every failure no file
  exists at `dest`): source responds **404** → `not_found` (the reference is
  stale — re-derive it from the holder); **409** → `conflict` (a holder's
  rev-pin moved on); any 3xx, connection refused, timeout, any other non-2xx,
  or a torn body → `source_unavailable` (transient, retryable —
  distinguishable from absence).

**Rejected.**
- **Fetching arbitrary URLs / hostnames** — internet ingestion is a
  connector's domain job; the plane's confinement is absolute, and following
  redirects would be the same hole via one indirection.
- **A size cap** — caps exist where bodies buffer through model context; this
  path streams disk-to-disk end to end.
- **Confining `dest` to the run dir** — the D21 trust posture: the script
  already writes anywhere with `open()`; a fence here restricts nothing real.
- **Returning bytes to the caller** — the caller has the file; a bytes return
  would invite exactly the in-memory buffering the plane retires.

**Verification** (real substrate: the D21 probe harness against real `httptest`
loopback servers; the allowed-port set is whatever `SUITE_SERVICES` /
`SUITE_FILES_BASE_URL` carry, so tests inject the `httptest` port; confinement
rejections assert the server recorded **zero** requests).

- R-I7RP-9G6B — `fetch` of an allowed-port `httptest` URL serving known bytes
  issues exactly one recorded `GET`, streams them to `dest` (file bytes equal
  served bytes), and returns `{path, size, content_hash}` where `size` is the
  byte count and `content_hash` is the lowercase-hex SHA-256 of those bytes.
- R-I8ZL-N7X0 — URL confinement discriminates with **zero** requests on every
  rejection, each raising `ToolError` `.code == "validation"`: a non-loopback
  host (`http://10.0.0.5:3202/…`); a hostname
  (`http://localhost:<allowed port>/…`); `127.0.0.1` on a port outside the
  injected origins; a non-`http` scheme — while the byte-equivalent URL on
  `127.0.0.1` with an allowed port proceeds to fetch.
- R-IA7I-0ZNP — the failure mapping against the real server: a 404 response →
  `ToolError` `.code == "not_found"`; a 409 → `"conflict"`; a
  connection-refused allowed port → `"source_unavailable"`; a 302 redirect is
  **not followed** (the redirect target records no request) and raises
  `"source_unavailable"` — and after each failure, including a torn body
  (server closes mid-stream), **no file exists at `dest`**.
