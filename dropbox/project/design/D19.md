# Decision 19 — MCP write tools (capped small-file convenience)

**Problem.** The MCP surface is read-only (`list`/`get` + chassis
`health`/`reflection`). The loopback streaming routes (D16) serve on-box services,
but an **off-box agent** (no local mount) can browse and fetch yet cannot create,
overwrite, delete, or move — an asymmetry with no reason to persist once the
underlying write seam exists.

**Decision.** Declare four new domain write tools over the same `Service` write
methods (D16), added to `internal/mcp.Tools`. Bytes go through the **capped
base64** path (base64-in-JSON cannot stream), so large transfers stay on the
loopback routes.

```go
// internal/mcp — declared alongside the existing list/get.
//   put(path, content_base64)  → Service.Write; content_base64 decoded, capped
//                                 at 25 MiB (too_large above it, like get)
//   mkdir(path)                → Service.Mkdir
//   delete(path)               → Service.Delete   (file or dir, recursive)
//   move(from, to)             → Service.Move
func Tools(svc *dropbox.Service) []mcp.Tool // now: list, get, put, mkdir, delete, move
```

- **Same seam, same semantics.** Each tool calls the identical `Service` method
  the loopback route calls, so behavior (confinement, created-vs-modified,
  recursive delete idempotency, one-op move, upload enqueue, origin) is shared —
  the MCP tools add only argument decoding and the sentinel→error-code envelope
  (`not_found`/`conflict`/`validation`/`too_large`/`internal`), reusing the
  existing `toolErr` mapping.
- **Origin.** An MCP write's `origin` is the authenticated caller's
  `client_id` (the same identity `health` already surfaces), threaded into the
  event exactly as the loopback header is (D18).
- **Cap.** `put` rejects a decoded body over 25 MiB with `too_large`, mirroring
  `get` — the base64 path is deliberately the small-file convenience; anything
  large uses `PUT /content`.
- **Surface count.** The declared domain table grows from **2** (`list`,`get`) to
  **6** (`+put,mkdir,delete,move`); with the chassis `health`+`reflection` the
  advertised surface is **8** tools (was 4).

**Rejected.**
- **Keep MCP read-only.** Leaves the off-box asymmetry for no benefit once the
  Service write methods exist; the tools are a thin declaration over them.
- **Stream large writes through MCP.** base64-in-JSON can't stream and would
  reintroduce whole-file residency — the exact thing D14 removes. The cap keeps
  the boundary crisp.
- **A single polymorphic `write` tool.** Overloading create/mkdir/delete/move
  into one tool muddies schemas and error mapping; four verbs match the four
  Service methods one-to-one.

**Verification** (through the composition-root seam — the `appkit/mcp` handler
built by `NewHandler` over a domain `Service`, driven through the real `ServeHTTP`
seam, exactly as the existing `list`/`get` tests do; a fake upload sink stands in
for Dropbox).
- R-KRXK-IQE2 — `tools/list` returns **8** tools: `list`, `get`, `put`, `mkdir`,
  `delete`, `move` (dropbox-declared) plus chassis `health`, `reflection` — the
  expanded partition (a table declaring `health`/`reflection` would fail the
  reserved-name check; a missing write tool fails the count).
- R-KT5G-WI4R — MCP `put` of a small base64 body writes the file (a following
  `get` returns the bytes) and enqueues an upload; a decoded body **over 25 MiB**
  is rejected with `too_large`.
- R-KUDD-A9VG — MCP `mkdir`/`delete`/`move` invoke the same `Service` methods as
  the loopback routes (behavior parity: empty dir listable, recursive delete,
  one-op move), and a failure surfaces through the sentinel→code error envelope.
