# Decision 4 — nginx fragment: the exact-match session-gated `= /srv/dropbox/` location + all four owner identity headers forwarded

**Decision.** dropbox's nginx fragment (`dropbox/etc/nginx.conf`) carries an
**exact-match** location for the bare mount root that session-gates the human
landing page and proxies it to the service's root, alongside the bearer-gated
`/srv/dropbox/` prefix that fronts the MCP endpoint. This Decision owns the
fragment's **identity-header forwarding contract**: every location that forwards
owner identity today captures (`auth_request_set`) and forwards
(`proxy_set_header`) **all four** owner headers the dashboard's introspection
hooks emit — `X-Owner-Id`, `X-Owner-Email`, `X-Owner-Name`, `X-Owner-Picture` —
not just `X-Owner-Email`.

The dashboard introspection endpoints (`/_authn` bearer/PAT hook, `/_session-authn`
session hook) emit four owner headers on every allow, fail-closed: `X-Owner-Id`
(durable owner key, never empty), `X-Owner-Email` (display/audit, never empty),
`X-Owner-Name` (percent-encoded, MAY be empty), `X-Owner-Picture` (percent-encoded,
MAY be empty) — plus `X-Client-Id` on the bearer tier. The fragment must relay all
four owner headers downstream; capturing only `X-Owner-Email` drops the rest at
nginx.

The exact-match session-gated landing location:

```
# Landing page — human web UI, gated by the dashboard browser SESSION (cookie),
# validated by the dashboard-owned /_session-authn internal location (a sibling of
# /_authn). Exact-match `=` so it never shadows the gated /srv/dropbox/ prefix below
# or the MCP endpoint. A failed session check yields 401.
location = /srv/dropbox/ {
    auth_request /_session-authn;
    auth_request_set $dropbox_session_owner         $upstream_http_x_owner_email;
    auth_request_set $dropbox_session_owner_id       $upstream_http_x_owner_id;
    auth_request_set $dropbox_session_owner_name     $upstream_http_x_owner_name;
    auth_request_set $dropbox_session_owner_picture  $upstream_http_x_owner_picture;
    proxy_set_header X-Owner-Email   $dropbox_session_owner;
    proxy_set_header X-Owner-Id      $dropbox_session_owner_id;
    proxy_set_header X-Owner-Name    $dropbox_session_owner_name;
    proxy_set_header X-Owner-Picture $dropbox_session_owner_picture;
    proxy_pass http://127.0.0.1:3200/;   # trailing slash → upstream sees "/"
    proxy_set_header Host $host;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_http_version 1.1;
}
```

**The bearer-gated prefix forwards all four owner headers too.** The
`location /srv/dropbox/` (`auth_request /_authn`) tier already captures
`$dropbox_owner` (from `X-Owner-Email`) and `$dropbox_client` (from `X-Client-Id`);
it gains the same three additional owner captures/forwards, keeping `X-Client-Id`
unchanged:

```
location /srv/dropbox/ {
    auth_request /_authn;
    auth_request_set $dropbox_owner          $upstream_http_x_owner_email;
    auth_request_set $dropbox_owner_id       $upstream_http_x_owner_id;
    auth_request_set $dropbox_owner_name     $upstream_http_x_owner_name;
    auth_request_set $dropbox_owner_picture  $upstream_http_x_owner_picture;
    auth_request_set $dropbox_client $upstream_http_x_client_id;
    …                                         # $authn_status/@dropbox_authn_500 machinery unchanged
    proxy_set_header X-Owner-Email   $dropbox_owner;
    proxy_set_header X-Owner-Id      $dropbox_owner_id;
    proxy_set_header X-Owner-Name    $dropbox_owner_name;
    proxy_set_header X-Owner-Picture $dropbox_owner_picture;
    proxy_set_header X-Client-Id     $dropbox_client;
    …
}
```

**Variable naming and identity hygiene.** New variables follow the fragment's
existing convention: the session tier's `$dropbox_session_owner_id`,
`$dropbox_session_owner_name`, `$dropbox_session_owner_picture` beside the existing
`$dropbox_session_owner`; the bearer tier's `$dropbox_owner_id`,
`$dropbox_owner_name`, `$dropbox_owner_picture` beside the existing `$dropbox_owner`.
Names stay service-prefixed and unique because every service fragment is included
into one apex `server` block. The identity-hygiene property holds for each new
header exactly as for `X-Owner-Email`: a single `proxy_set_header <name>
$<var>;` from the auth subrequest variable REPLACES any client-smuggled inbound
header of that name (nginx does not pass inbound client headers as proxy headers
unless explicitly set, and the last `proxy_set_header` for a name wins), so a
client cannot smuggle identity through any of the four.

**Locations that forward no identity are untouched.** The
`= /srv/dropbox/.well-known/oauth-protected-resource` PRM bootstrap, the
`= /srv/dropbox/content { return 404; }` defence-in-depth stub, the
`/srv/dropbox/static/` asset tier (session-gated but forwards no owner header
today), and the `@dropbox_authn_500` re-emit named location forward no owner
identity and stay exactly as they are.

**How it coexists with the existing fragment.** dropbox's fragment already has,
in source order:

1. `location = /srv/dropbox/.well-known/oauth-protected-resource` — unauthenticated
   PRM bootstrap (exact-match).
2. `location = /srv/dropbox/content { return 404; }` — defence-in-depth: the
   loopback byte endpoint `GET /content` is never reachable through the public
   edge (exact-match; belt to the handler's own identity-header guard).
3. `location /srv/dropbox/ { auth_request /_authn; … }` — the bearer-gated
   **prefix** for the MCP endpoint and everything else under the mount, carrying
   the `$authn_status`/`error_page 500 = @dropbox_authn_500` rate-limit re-emit
   machinery.
4. `location @dropbox_authn_500` — the named location the prefix block's
   `error_page` jumps to, which re-emits a faithful `429` (with `Retry-After`)
   when introspection was actually rate-limited, else a real `500`.

nginx longest-prefix/exact-match rules make an **exact-match `=` location win over
a prefix location** for that one path regardless of source order. So the new
`= /srv/dropbox/` exact match wins for the **bare root only**, while the
`/srv/dropbox/` prefix (item 3) still serves `/srv/dropbox/mcp` and every other
sub-path under the mount. The two PRM/content exact matches (items 1–2) are
unaffected (different exact paths), and the `@dropbox_authn_500` named location
(item 4) is reached only from the prefix block and is untouched. Nothing in the
existing fragment is removed or rewritten — the change is **purely additive**.

> **Delta from the crm template.** dropbox's fragment has **no**
> `= /srv/dropbox/feed { return 404; }` location — its `/feed` is safe purely on
> the handler's identity-header guard (per `dropbox/CLAUDE.md`), so unlike crm
> there is no feed 404 block to assert. The exact-match dropbox carries instead is
> the `= /srv/dropbox/content` defence-in-depth block, plus the
> `@dropbox_authn_500` named location the bearer prefix needs. The retained-locations
> assertion (R-NGNX-8V1W) enumerates *these* — not a feed block.

**Why `/_session-authn`, not `/_authn`.** A browser carries the
`dashboard_session` cookie, not a bearer token, so the bearer gate (`/_authn`)
would refuse it. `/_session-authn` is the dashboard-owned, loopback-only internal
location (defined in `dashboard/etc/nginx.conf` as
`proxy_pass …/internal/session-authn`) that validates the cookie against the
web-session store and returns `200` + `X-Owner-Email` on success or `401` on
failure. This is the **same** mechanism `sites` already uses for its private
static tier (`sites/etc/nginx.conf` `location /srv/sites/private/`). The gate is
deliberately **coarse**: any logged-in dashboard user passes; there is no
per-resource check — fine for a name+version page.

**Why the simple form (no 429 re-emit).** The bearer prefix block carries the
`$authn_status`/`error_page 500 = @dropbox_authn_500` machinery because token
introspection can return `429` (rate-limited) which `auth_request` collapses to a
synthetic `500`. The session check is a cookie lookup, not a rate-budgeted token
introspection, so the landing location does not need the 429 re-emit dance — it
maps the session subrequest's allow/deny directly (`200`→serve, `401`→401),
matching the `sites` private-tier location, which likewise omits it. (It also
takes only the owner from the session subrequest — no `X-Client-Id`, since a
browser session is not a registered MCP client.)

**Templating.** The loopback port is a **literal** in the fragment (`PORT=3200`,
dropbox's fixed registry port), so nothing substitutes it at install — `opsctl
setup dropbox` on the box / `nginx/run` in dev drop the fragment verbatim (dev
copies `etc/nginx.conf` to `nginx/locations/dropbox.conf`, and **must include
`dropbox`** in its service loop), exactly like the existing
locations. The fragment is **not** a vhost — no `server{}`, no `listen`, no cert;
it is dropped at `/etc/nginx/conf.d/locations/dropbox.conf` and included by the
dashboard's apex server block, which is also where `/_session-authn` is defined.

**Rejected.**
- **Gating the landing page with `/_authn`** — a browser has no bearer token; it
  would always 401. Wrong audience.
- **Making the landing page fully public (no auth_request)** — the brief forbids
  it; a logged-out browser must get `401`. The page is gated, just coarsely.
- **A prefix `location /srv/dropbox/` for the page** — would collide with the
  existing bearer prefix; the exact-match `=` is what cleanly carves the bare root
  out of the prefix without touching the rest.
- **Adding the `@dropbox_authn_500` 429 re-emit to the session location** — the
  session check is not rate-budgeted; the extra machinery is dead weight (the
  `sites` private tier omits it for the same reason).
- **Forwarding only `X-Owner-Email` (the status quo)** — the introspection hooks
  now emit `X-Owner-Id`/`X-Owner-Name`/`X-Owner-Picture` too; capturing only the
  email silently drops the durable owner key and display fields at nginx, so the
  service can never see them. The fragment must relay all four.
- **Editing the `= /srv/dropbox/content` defence-in-depth block or the PRM/static/
  re-emit locations** — those forward no owner identity; adding owner headers there
  would be meaningless and is out of scope. Only the two identity-forwarding
  locations (session landing, bearer prefix) gain the new captures/forwards.

**Verification.** (Proven by a test that reads `dropbox/etc/nginx.conf` from disk
and asserts over its content — see design's Testing strategy.)
- R-NGNX-2P4Q — `dropbox/etc/nginx.conf` contains an **exact-match** location for
  the bare mount root: a `location = /srv/dropbox/ {` block (distinct from the
  prefix `location /srv/dropbox/ {`).
- R-NGNX-4R6S — that exact-match block gates with `auth_request /_session-authn`
  (the cookie/session validator) and **not** `auth_request /_authn` (the bearer
  validator).
- R-NGNX-6T8U — that block proxies to the loopback upstream **root**
  (`proxy_pass http://127.0.0.1:3200/;` — trailing slash), so the upstream
  sees `/`.
- R-NGNX-8V1W — the pre-existing locations are retained alongside it: the
  bearer-gated `location /srv/dropbox/ {` prefix with `auth_request /_authn`, the
  `location = /srv/dropbox/content { return 404; }` defence-in-depth block, the
  `location = /srv/dropbox/.well-known/oauth-protected-resource` PRM bootstrap, and
  the `location @dropbox_authn_500` named location all still appear in the file.
- R-KJGZ-FKVP — the exact-match session-gated landing block `location = /srv/dropbox/ {`
  captures and forwards **all four** owner headers: `auth_request_set
  $dropbox_session_owner $upstream_http_x_owner_email;` +
  `auth_request_set $dropbox_session_owner_id $upstream_http_x_owner_id;` +
  `auth_request_set $dropbox_session_owner_name $upstream_http_x_owner_name;` +
  `auth_request_set $dropbox_session_owner_picture $upstream_http_x_owner_picture;`,
  each proxied on with its own `proxy_set_header` (`X-Owner-Email
  $dropbox_session_owner`, `X-Owner-Id $dropbox_session_owner_id`, `X-Owner-Name
  $dropbox_session_owner_name`, `X-Owner-Picture $dropbox_session_owner_picture`).
- R-KKOV-TCME — the bearer-gated prefix block `location /srv/dropbox/ {` captures
  and forwards **all four** owner headers with the bearer-tier variables:
  `auth_request_set $dropbox_owner $upstream_http_x_owner_email;` +
  `$dropbox_owner_id $upstream_http_x_owner_id;` +
  `$dropbox_owner_name $upstream_http_x_owner_name;` +
  `$dropbox_owner_picture $upstream_http_x_owner_picture;`, each proxied on with its
  own `proxy_set_header` (`X-Owner-Email $dropbox_owner`, `X-Owner-Id
  $dropbox_owner_id`, `X-Owner-Name $dropbox_owner_name`, `X-Owner-Picture
  $dropbox_owner_picture`), while its existing `X-Client-Id $dropbox_client`
  forward is retained unchanged.
