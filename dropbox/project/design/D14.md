# Decision 14 — Streaming byte I/O in the mirror + streaming read route

**Problem.** The mirror's byte primitives buffer whole files in RAM: `Mirror.Write`
takes a `[]byte` and `tmp.Write(data)`s it (`mirror.go`), `Mirror.Read` is
`os.ReadFile` (whole file into a `[]byte`), and `Service.Content` / `GET /content`
hand that `[]byte` back. That is acceptable for the download-only mirror it was
built for, but the box is memory-constrained and the new service-facing filesystem
must move large artifacts (images, exports). Loading a 64 MiB file into a `[]byte`
per concurrent read *and* per write is the wrong foundation, and it undercuts the
near-native performance target. The byte paths must move data through a **fixed
copy buffer** whose residency is independent of file size.

**Decision.** Reshape the `Mirror` byte primitives to stream, and convert the read
route to `http.ServeContent`.

```go
// internal/dropbox/mirror.go — streaming primitives.

// WriteFrom streams src into the confined path atomically (temp file in the same
// dir → fsync → rename), computing the content hash on the fly via an
// io.TeeReader into a sha256 hasher fed by io.Copy (a fixed ~32 KiB buffer; the
// whole file is never resident). Returns the Dropbox-form content hash and the
// byte size. Replaces Write([]byte).
func (m *Mirror) WriteFrom(rel string, src io.Reader) (contentHash string, size int64, err error)

// Open returns a streaming handle to the confined file for the read route. The
// *os.File is an io.ReadSeekCloser so http.ServeContent can honor Range requests
// without buffering. Replaces Read([]byte) for the serve path.
func (m *Mirror) Open(rel string) (f *os.File, info os.FileInfo, err error)
```

- **Content hash on the write stream.** The Dropbox block hash (`ContentHash`,
  already implemented in `client.go`) is computed as bytes flow to the temp file,
  not in a second pass — the source is read exactly once. The atomic temp→fsync→
  rename discipline (and its `ErrPathEscape` confinement) is preserved verbatim;
  only the byte source changes from a `[]byte` to an `io.Reader`.
- **Read route streams.** `GET /content` resolves the display path through the
  index (unchanged), then `m.Open`s the mirror file and serves it with
  `http.ServeContent`, which sets content-type/length, supports **Range**
  requests (206 partial), and copies through a fixed buffer. The MCP `get` tool's
  buffered, 25 MiB-capped, base64-in-JSON path is **unchanged** (base64 cannot
  stream) — it stays the small-file convenience for off-box agents; large
  transfers use `/content`.
- **Fixed buffer everywhere.** The Dropbox uploader (D17) likewise re-opens the
  mirror file and streams it; no byte path holds a whole file. Memory per op is a
  fixed copy buffer, independent of file size.

**Rejected.**
- **Keep `[]byte` primitives** — the status quo; whole-file residency on a
  memory-constrained box moving large images is exactly what this work exists to
  fix.
- **Chunk into partial HTTP responses ourselves** — `http.ServeContent` already
  implements correct Range/If-Range/content-type handling; reimplementing it is
  pure risk. Coalescing and chunking of *uploads* over 150 MiB is Dropbox's
  `upload_session` (D17), not the read route's concern.
- **Memory-map the file** — `mmap` couples resident pages to file size and adds
  platform-specific fault handling for no benefit over a streamed copy.

**Verification** (mirror tests over a temp dir; the read route through
`net/http/httptest` against the shipped handler wiring; the local filesystem is
the real substrate — no mock can falsify the atomicity/Range claims).
- R-JV0A-6XDB — `WriteFrom` on a **> 8 MiB multi-block** reader produces a
  byte-identical mirror file and returns the correct streaming `ContentHash` and
  size; no partial file is ever observable (a concurrent `Open` sees either the
  old bytes or the complete new bytes, never a truncated temp).
- R-JW86-KP40 — `GET /content` with a `Range: bytes=` header returns **206** and
  exactly the requested byte slice (proving it serves from a seekable `*os.File`
  via `ServeContent`, not a buffered `[]byte`); a full GET returns 200 and the
  whole file.
- R-JXG2-YGUP — a **64 MiB** file round-trips (`WriteFrom` then full `GET
  /content`) with matching content hashes, exercising the fixed-buffer path
  end-to-end; the `WriteFrom` signature takes an `io.Reader` (not `[]byte`) so no
  caller can reintroduce whole-file residency on the write side.
