# Decision 20 — The `dropbox/docs/` filesystem-API reference + route-coverage guard

**Problem.** Wiring the suite's other services to use dropbox's filesystem is
**out of scope** here (each service adopts it in its own `project/` later), but
those future integrator agents need a precise reference for the API they will call
— every filesystem-interaction endpoint, its method, path, parameters, and
behavior. An undocumented or drifting reference is worse than none: it silently
lies the first time a route is added.

**Decision.** Ship a documentation deliverable under **`dropbox/docs/`** covering
the **filesystem-interaction API only**, and enforce its completeness with a
route-coverage test.

- **Location & authorship.** The reference lives at `dropbox/docs/` — a **shipped
  artifact** in the service tree, a sibling of `internal/`, produced by the build
  loop. (Spec authoring specifies it; the build writes it. It is **not** under
  `project/`.)
- **Scope of the doc = the filesystem-interaction routes.** The endpoints a
  consumer calls to read/write/discover files and directories, regardless of age:
  `GET /content`, `PUT /content`, `DELETE /content`, `POST /mkdir`, `POST /move`,
  `GET /list`, `GET /stat`. It **excludes** the machine-plumbing surface that is
  not how a service does filesystem work: `/feed`, `/health`, `/mcp`, the PRM
  well-known, and the landing page. For each route the doc gives method + path,
  query/body parameters, the success response shape, the error taxonomy
  (`not_found`/`conflict`/`validation`/`too_large`), the streaming/near-native and
  local-commit-then-async-push contract, path confinement, and the `X-Client-Id`
  origin convention.
- **Drift guard.** A test enumerates the filesystem-API routes the router
  actually registers (the canonical route list the composition root mounts) and
  **fails if any is absent** from the docs — so the reference cannot rot as
  file-API routes are added later ("enforce with systems, not willpower"). The
  excluded plumbing routes are explicitly not required.

**Rejected.**
- **Document the whole HTTP surface.** The operator scoped this to the parts other
  services use for filesystem work; `/feed`/`/health`/`/mcp`/PRM/landing are a
  different audience and out of this reference.
- **A prose doc with no coverage check.** An API reference is exactly the doc that
  drifts stale the first time someone adds a route; the mechanical check is the
  point.
- **Put it under `project/`.** `project/` is the design/plan workspace; a shipped
  API reference for integrators is service-tree documentation (`dropbox/docs/`),
  built like any other artifact.

**Verification** (a test in `cmd/dropbox` over the shipped `dropbox/docs/` tree
and the composition root's route list — the real files that ship).
- R-KVL9-O1M5 — `dropbox/docs/` contains the filesystem-API reference documenting
  each in-scope route (`GET`/`PUT`/`DELETE /content`, `POST /mkdir`, `POST /move`,
  `GET /list`, `GET /stat`) with its method, path, parameters, and behavior.
- R-KWT6-1TCU — the coverage test asserts **every** filesystem-API route the
  router registers appears in the doc and **fails** when a registered in-scope
  route is undocumented; the excluded plumbing routes (`/feed`,`/health`,`/mcp`,
  PRM, landing) are not required.
