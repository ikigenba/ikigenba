# Decision 11 — Web surface from `share/www` through the chassis (de-embed)

**Problem.** dropbox's landing page and static assets are embedded in the binary
(`//go:embed` in `dropbox/internal/web/`), like every service's original shape.
The suite is reverting that: web assets are release artifacts that belong on disk
— visible and diffable in the versioned release tree, editable in dev without a
rebuild, served by one chassis mechanism instead of twelve drifting copies. crm
proved the conversion; notify followed; dropbox is another producer converted the
same way. This Decision is the service-side half of appkit's chassis design (its
D5 www-root resolution, D6 `appkit/web`, D7 `Spec.WWW` — consumed here as a fixed
external contract via the committed `replace appkit => ../appkit`).

**Decision.** dropbox serves its web surface from the on-disk www root.

1. **The files move.** `dropbox/internal/web/landing.html` →
   `dropbox/share/www/landing.html`; `dropbox/internal/web/static/` →
   `dropbox/share/www/static/` (same bytes: `tokens.css`, the four woff2 fonts —
   `space-grotesk.woff2`, `ibm-plex-sans.woff2`, `ibm-plex-mono-400.woff2`,
   `ibm-plex-mono-500.woff2`). `bin/ship` already bundles `dropbox/share/` into
   the release `share/<version>` tier; on the box the chassis resolves the root
   to `share/current/www`, in dev to `./share/www` or the `DROPBOX_WWW_PATH`
   override.
2. **The Spec opts in.** `cmd/dropbox/main.go` sets `WWW: true`. The chassis
   loads the site at serve (failing startup loudly if the root is missing),
   auto-mounts `GET /static/`, and exposes the loaded site as `rt.WWW()`.
3. **The landing route renders through the chassis.** The `GET /{$}` mount in
   `Handlers` uses the few-line handler over `rt.WWW().Render(w, "landing.html", …)`
   (D1/D2 carry the exact shape; the crm/notify composition roots are the
   template): it 404s a non-root path and renders `landing.html` with the
   `{Service, Version}` struct from `rt.Service()`/`rt.Version()`. The
   service-side `GET /static/` mount (`web.StaticHandler()`) is **deleted** — the
   chassis owns it.
4. **`dropbox/internal/web/` is deleted** — `web.go` (`LandingHandler`,
   `StaticHandler`), `embed.go`, `landing.html`, `static/`, and its tests
   (`web_test.go`, `nginx_test.go`). The landing/asset tests move to the
   composition-root package (`cmd/dropbox`), loading the repo-real `share/www`
   tree via `appkit/web` (a relative path from the package dir), so they exercise
   the exact files that ship. The nginx fragment tests (`nginx_test.go`) relocate
   to `cmd/dropbox` **unchanged** in their assertions — they already derive the
   proxy target from `registry.BaseURL("dropbox")` after D10, and D10's drift
   guard keeps its registry-derived expectations.
5. **Dev wiring (boundary-crossing).** `bin/start`'s `launch_dropbox` exports
   `DROPBOX_WWW_PATH="$repo/dropbox/share/www"` beside its existing
   `DROPBOX_DB_PATH` / `DROPBOX_MIRROR_PATH` exports, so a repo-root-launched dev
   binary finds the source tree. This one line crosses the `dropbox/` boundary
   and is **verified by the live `bin/start` smoke, not the Go suite** (the
   workspace's standing convention for repo-root collaborators).
6. **Nothing else moves.** The nginx fragment (D4/D8), the landing markup and
   tokens (D6/D7/D8), the MCP surface, the migrations, the loopback `GET /content`
   and `GET /list` byte routes, and the background sync engine are untouched. The
   embedded migrations (`internal/db`) stay embedded — they are not web assets
   (D13 handles that package). The `/content` and `/list` handlers keep their
   in-handler identity-header guards exactly as they are; this Decision changes
   only the human landing surface.

**Rejected.**
- **Keeping the embed** — the suite-wide reversion this work exists to complete;
  embedded assets are invisible on the box and cost rebuilds in dev.
- **`state/www` as the root** — that tree is the sites service's nginx-served
  user content, backed up with `state/`; dropbox's UI assets are release
  artifacts and live in the shipped `share` tier (appkit D5 records the full
  argument).
- **Keeping a thin `internal/web` package** — with the mechanism in `appkit/web`
  there is nothing left but a handler closure; a package holding one closure is
  ceremony. The composition root is the right home for composition.
- **Serving assets from both embed and disk during a transition** — two sources
  of truth; the conversion is atomic within one service.

**Verification** (over the repo-real `dropbox/share/www` tree; D1's R-LAND-\*,
D2's R-ROUT-\*, D3's R-ASST-\*, D7's R-HOME-6P8T, and D8's R-LQXL/LS5H/LTDD/LULA/
LVT6 ids continue to pin the page/asset behaviors through the new substrate, and
D4's R-NGNX-\* ids pin the relocated nginx-fragment assertions).
- R-QO40-U0VH — an `appkit/web` Site loaded from `dropbox/share/www` (the actual
  shipped tree, resolved relative to the test package) renders `landing.html`
  with an injected service and version visible in the output — proving the
  shipped template, not an embedded copy, is the page source.
- R-QPBX-7SM6 — the chassis static mount over the same tree serves
  `GET /static/tokens.css` (`200`, `text/css`) and
  `GET /static/fonts/space-grotesk.woff2` (`200`, `font/woff2`) with **no**
  dropbox-side static handler registered — proving the assets ship on disk and
  the chassis serves them.
