# Decision 17 — Push-up: durable upload queue, Dropbox write client, uploader worker

**Problem.** The mirror was **download-only** ("the box never writes back — no
conflict resolution, no echo loop"). Making the folder authoritative means every
local mutation (D16) must propagate **up** to Dropbox, without blocking the
writing service, without losing a push if Dropbox is down, and without the pushed
change round-tripping back as a phantom download (the echo). And because the box
is low-bandwidth, a superseded write must not waste an upload.

**Decision.** A **durable, SQLite-backed upload queue** feeds a background
**uploader worker** (a `Spec.Workers` goroutine) that calls new **Dropbox write
methods** on the existing `Client`. The suite index is the master; Dropbox is the
replica.

```sql
-- internal/db/migrations/005_upload_queue.sql  (additive; timestamped)
CREATE TABLE upload_queue (
    path            TEXT PRIMARY KEY,   -- one pending op per path ⇒ COALESCING
    op              TEXT NOT NULL,      -- put | mkdir | delete | move
    dest            TEXT,               -- move target (else NULL)
    origin          TEXT,               -- writing client id (for logging)
    enqueued_at     TEXT NOT NULL,
    attempts        INTEGER NOT NULL DEFAULT 0,
    next_attempt_at TEXT NOT NULL,      -- backoff gate
    state           TEXT NOT NULL DEFAULT 'pending',  -- pending | failed
    last_error      TEXT
);
CREATE INDEX idx_upload_queue_due ON upload_queue (state, next_attempt_at);
```

```go
// internal/dropbox/client.go — the write half of the Dropbox API (rpc + content
// hosts, bearer). Overwrite is the conflict policy; upload_session handles >150 MiB.
func (c *Client) Upload(ctx context.Context, path string, src io.Reader, size int64) (rev string, err error) // mode:overwrite
func (c *Client) CreateFolder(ctx context.Context, path string) error
func (c *Client) DeletePath(ctx context.Context, path string) error
func (c *Client) Move(ctx context.Context, from, to string) error

// internal/dropbox/uploader.go — the worker.
func (s *Service) RunUploader(ctx context.Context) error // drains due rows until ctx done
```

- **Coalescing.** The queue is keyed by `path` (PRIMARY KEY): a second write to a
  path **upserts** the row (latest op wins), so the uploader, which **re-reads the
  mirror at push time**, sends only the current bytes. A file overwritten before
  its first upload drains uploads once, to the latest version — intermediate
  versions are skipped on the replica (each local write is still durable and still
  emitted its own event). Uploads are always **whole files** (the temp→rename
  mirror write means the uploader never reads a partial); coalescing skips a
  complete version, it never sends a fragment.
- **Conflict = overwrite (suite wins).** `Upload` uses `mode:overwrite`; Dropbox
  takes the bytes unconditionally and returns a new `rev`. No conflict copies, no
  bespoke resolution — the authoritative folder always reflects suite state.
  (Same-path concurrent human edits are designed away by the read-only/inbox
  convention; if one occurs, the suite write wins by last-writer-order.)
- **Echo suppression via the returned rev.** On success the uploader persists the
  Dropbox-returned `rev` into the `files` row **atomically** with clearing the
  queue entry. The file appears on Dropbox only *after* the upload, so the next
  `list_folder` delta carries **that same rev** — the existing rev/content_hash
  dedup on the download apply path makes it a **no-op** (no re-download, no
  duplicate event). The pushed change never round-trips.
- **>150 MiB uploads.** `Upload` sends a single `files/upload` under the limit and
  switches to `upload_session/start|append_v2|finish` above it, streaming fixed
  chunks from the re-opened mirror file (never resident, D14). Dropbox reassembles
  server-side, so no consumer ever observes a partial.
- **Async, durable, loud on failure.** The worker picks `state='pending' AND
  next_attempt_at<=now` rows; on failure it increments `attempts`, advances
  `next_attempt_at` by **exponential backoff**, and after a poison threshold sets
  `state='failed'` and records `last_error`. A failed row is **never dropped** —
  the local copy is authoritative, so an unpushable change stays pending/failed
  and **loud**. The `Health` reporter gains `pending_uploads`, `failed_uploads`,
  and `oldest_pending_age_seconds` under `details`, feeding both the `health` tool
  and `/health`.

**Rejected.**
- **In-memory queue.** A restart would silently drop un-pushed writes — the
  authoritative local file would never reach the replica. Durability is
  non-negotiable; it lives in the same SQLite the index does.
- **`mode:add` / `update{rev}` (conflict copies).** Litters the authoritative
  folder with `(conflicted copy)` files that flow back down as new files the
  suite must notice and clean up — contradicts "our tools are the authority."
- **Re-upload every intermediate version (no coalescing).** Burns uploads on a
  low-bandwidth link pushing bytes already superseded; the replica only needs to
  converge to current state.
- **Synchronous upload inside the write route.** See D16 — couples suite writes to
  Dropbox latency/availability.

**Verification.** Split across build phases (each names its id slice). Queue and
worker logic are hermetic (temp SQLite, an httptest fake capturing request shape);
the claims that hinge on the **real Dropbox contract** (overwrite accepted,
returned rev, session reassembly, folder/delete/move applied, the rev that a
later `list_folder` actually reports) are proven by a **live smoke** — a
`-tags live` test that runs against the real app folder with the suite refresh
token, asserting the observable outcome via a follow-up `list_folder`. The live
smoke is **not** part of the hermetic green suite; it is a distinct, reachable
check the client/uploader phases require be run.

- R-KC2V-JPR1 — enqueuing two writes to the same path leaves **exactly one**
  `upload_queue` row for that path (PK upsert), carrying the latest op — coalescing.
- R-KDAR-XHHQ — migration `005_upload_queue` is recorded in `schema_migrations`
  and the table enforces one row per path.
- R-KEIO-B98F *(LIVE, real Dropbox)* — `Upload(mode:overwrite)` of a file to the
  real app folder returns a `rev`, and a follow-up `list_folder` shows the file
  present with **that rev** (proves overwrite is accepted and the rev is real).
- R-KFQK-P0Z4 *(LIVE, real Dropbox)* — a file **> 150 MiB** uploads via
  `upload_session start/append_v2/finish` and is retrievable **whole** (byte
  count / content hash match), proving the session path reassembles.
- R-KGYH-2SPT *(LIVE, real Dropbox)* — `CreateFolder`, `DeletePath`, and `Move`
  produce, respectively, the folder / the removal / the relocation, each observable
  via `list_folder`.
- R-KJE9-UC77 — against an httptest fake, `Upload` sends `mode:overwrite` for a
  small file and chunks a `> 150 MiB` source into `upload_session` calls
  (request-shape assertion; no network).
- R-KKM6-83XW — on a successful upload the uploader persists the returned `rev`
  into the `files` row and clears the queue entry atomically; a subsequent
  `list_folder` delta at **that rev** is a **no-op** (no re-download, no duplicate
  event) — echo suppression.
- R-KLU2-LVOL — a failing upload advances `next_attempt_at` by exponential backoff
  and increments `attempts`; after the poison threshold the row is `state='failed'`,
  **retained** (never deleted), with `last_error` set.
- R-KN1Y-ZNFA — `Health.details` reports `pending_uploads`, `failed_uploads`, and
  `oldest_pending_age_seconds` from the queue, and the same reporter feeds
  `/health` and the `health` tool (they cannot diverge).
