# Decision 9 — restore re-asserts the served-tree `web` invariant after replacing state

## Decision

`opsctl restore <app>` wipes `state/`, untars the archived `state/` back into
place, and (per D01) recreates `cache/` owned by the service user before the
deferred unit restart (`backup.go`). The restored `state/` returns with whatever
owner/group **and mode** the snapshot tar captured — restore *replays archive
metadata*, it does not re-establish invariants. For the served `state/www` tree
that is a fidelity trap: a snapshot taken while `www` was mis-grouped (e.g. one
captured after a pre-fix deploy left it `<app>:<app>`), or an older snapshot
predating setgid, restores a tree nginx cannot serve — even though the box is
otherwise healthy. Restore is precisely the recovery path where you cannot assume
the captured state was correct.

Restore already refuses to trust the archive for `cache/` ownership — it
re-establishes `cache/` as `<app>:<app>` rather than replaying it (D01). This
Decision extends that same "reconstitute to the invariant, don't replay" posture
to the served tree: **after `state/` is replaced (and alongside the existing
`cache/` recreate), call `ensureWWWPerms` (D7)** to re-label `state/www`
`<app>:web` and re-apply setgid, so the restored box serves regardless of what
the snapshot's metadata happened to be:

```go
// (*Opsctl).Restore, after replaceStateFromArchive succeeds and near the cache
// recreate/chown (D01), before the deferred unit restart:
if err := o.ensureWWWPerms(ctx, app, l); err != nil {
    return fmt.Errorf("restore: restore www perms: %w", err)
}
```

`ensureWWWPerms` self-guards on `WWWDir` existence, so restore of a
no-served-tree app is a no-op, and it re-applies both the group **and** setgid —
the setgid re-assert is load-bearing here (unlike deploy, restore can untar a tree
that never had setgid), because without it the first post-restore publish would
drift back to `<app>:<app>`. The recreate happens before the deferred restart, so
the service and nginx both see the correct tree the moment the unit comes back.
It depends on the `web` group existing (D6); a restore runs on a box init-box
already provisioned.

**Boundary.** D9 adds exactly one `ensureWWWPerms` call to the restore path,
after `replaceStateFromArchive`. It does not change the state wipe/untar, the
`cache/` recreate (D01), the generation-sidecar removal, the cert restore
(dashboard), or the deferred restart. It reuses D7's helper and D6's group; it
adds no new seam.

## Rejected

- **Trust the archive's ownership/mode for `state/www` (do nothing on restore).**
  Only correct if every snapshot was captured from a healthy tree — the one thing
  a recovery operation cannot assume. A snapshot taken mid-incident or predating
  setgid would restore an unservable tree. Rejected.
- **Re-chown the group but not setgid on restore.** Fixes the restored entries
  once, but an older snapshot's tier dirs lack setgid, so the next publish on the
  recovered box drifts back to `<app>:<app>`. Restore must re-assert setgid, which
  is exactly why it reuses the full `ensureWWWPerms` rather than a bare chown.
  Rejected.
- **Rely on the next `deploy` (D8) to fix perms post-restore.** Leaves the
  freshly-restored box unservable until the operator happens to deploy — a
  recovery that isn't actually recovered. Restore must stand on its own. Rejected.

## Verification

- R-AZ63-Y06Z — **Restore re-asserts the served-tree invariant after replacing
  state.** After `Restore` of an app with a `state/www` tree present, the fake
  `System` records a `ChownTree(app, "web", WWWDir())` and the tier-dir
  `Chmod(02750)` calls, ordered **after** the state replacement and **before** the
  deferred unit restart, in addition to the existing `cache/` chown (D01).
  Falsifiable: today's restore chowns only `cache/` and never touches the served
  tree. *(unit: temp `OPSCTL_ROOT` + fake `System`)*
- R-B0E0-BRXO — **Real-substrate (live box `int.ikigenba.com`).** After a real
  `opsctl restore sites <key>` on the box, an anonymous
  `GET https://int.ikigenba.com/srv/sites/public/<published-site>/` returns
  **200** — the restored tree is servable by nginx regardless of the snapshot's
  captured metadata. Cannot be falsified by the fake `System`; requires a real
  restore + live HTTP fetch. Operator-verified out-of-loop.
