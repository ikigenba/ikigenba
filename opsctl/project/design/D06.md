# Decision 6 — init-box creates the `web` group and makes nginx a member

## Decision

Services that publish files for the front door to serve **directly off disk**
(today only `sites`, via its `state/www/{public,private}` tiers) require the
nginx worker to *traverse and read* that tree. The stock per-app `state/` is
`0750 <app>:<app>`, deliberately unreadable by anyone but the app user, so the
nginx worker — which runs as its own user (`nginx` on the box, from the
`nginx` package `init-box` installs) and is a member of no app group — cannot
reach it. The chosen mechanism is a shared **`web`** system group: the served
tree is group-`web` and group-readable (D7), and the nginx user is a **member of
`web`**.

Establishing that membership is box-global front-door state, not per-app state:
there is one nginx user for the whole box, its group memberships are a property
of the front door, and `init-box` already owns the front door — it installs
nginx + certbot, writes the apex block, and brings nginx up (`initbox.go`). This
Decision therefore makes **`init-box`** responsible for both halves: **ensure the
`web` group exists, and add the `nginx` user to it.** Per-app `setup`/`deploy`
only ever *label* a tree `:web` (D7/D8); they never touch the nginx user.

Ordering matters and `init-box` already has the right shape. It installs the
`nginx` package (creating the `nginx` user) early, then brings nginx up last. The
group + membership are established **after the package install** (the user must
exist) and **before nginx is enabled/started**, so the running worker inherits
`web` on first start — no separate restart on a greenfield box. (A worker only
re-reads its supplementary groups when its master process restarts; on an
*already-running* box a rerun of init-box adds the membership but the live worker
does not pick it up until nginx is restarted — a real-substrate concern captured
in Verification, not something unit tests can observe.)

Adding a user to a group is a new privileged operation, so the `System` seam
gains one method:

```go
// System (seam.go) gains:
//   // AddUserToGroup adds an existing user to a supplementary group (the box
//   // runs `usermod -aG <group> <user>` / `gpasswd -a`). Idempotent: re-adding
//   // an already-member user is a no-op success.
//   AddUserToGroup(ctx context.Context, user, group string) error

// initbox.go, after InstallPackages("nginx", "certbot"), before nginx bring-up:
o.logf("ensure web group + nginx membership")
if err := o.System.EnsureSystemGroup(ctx, "web"); err != nil {
    return fmt.Errorf("init-box: ensure web group: %w", err)
}
if err := o.System.AddUserToGroup(ctx, "nginx", "web"); err != nil {
    return fmt.Errorf("init-box: add nginx to web group: %w", err)
}
```

`EnsureSystemGroup` already exists on the seam (`seam.go`, `groupadd --system`).
The `--skip-cert` / deferred-nginx path is unaffected: the group + membership are
established regardless of whether the cert/bring-up is deferred, since they are a
precondition of nginx serving, not of the cert.

**Boundary.** D6 adds group creation + nginx membership to `init-box` only. It
does not change what nginx serves, the apex block, the cert flow, or any per-app
`setup`/`deploy`/`restore` behavior. The reciprocal act — labelling a served tree
`:web` — is D7 (setup), D8 (deploy), and D9 (restore); D6 only guarantees the
group exists and nginx is in it so those labels grant real access.

## Rejected

- **Put the `web` group + nginx membership in per-app `setup` (sites).** Makes a
  per-app provisioning step mutate the box-global nginx user, and — because a
  live nginx worker only re-reads groups on restart — would force `setup` to
  restart nginx (it currently only *reloads*). The front-door user's groups are
  init-box's concern; setup should only label the tree. Rejected.
- **Keep `EnsureSystemGroup("web")` in setup's worker branch (status quo) and add
  membership there.** That call is already effectively dead (no worker service has
  a served tree) and creates an *empty* group nginx never joins — exactly the box
  state that produced the 403s. Consolidating group ownership in init-box removes
  the dead call; setup relies on the group already existing (D7). Rejected.
- **Make the served tree world-readable (`0755`) so no group membership is
  needed.** Drops the private tier's on-disk protection: any local box user could
  read `state/www/private/**` directly, bypassing the front door's session gate.
  The `web` group keeps the private tier readable only by the app and nginx.
  Rejected (see D7).
- **Hardcode the worker user as `www-data`.** The box runs the `nginx` package,
  whose worker user is `nginx`; `www-data` is a Debian-ism that does not exist
  here. The membership targets the actual `nginx` user init-box installs.
  Rejected.

## Verification

- R-AQMT-9M04 — **init-box establishes the `web` group and nginx membership, in
  order.** After `InitBox`, the fake `System` has recorded an
  `EnsureSystemGroup("web")` **and** an `AddUserToGroup("nginx", "web")`, both
  ordered **after** the `InstallPackages("nginx", …)` call and **before** the
  nginx enable/bring-up (`EnableUnit("nginx", …)` / `NginxReload`). Falsifiable:
  today's `initbox.go` records neither call. Holds on the `--skip-cert` path too
  (group + membership are independent of the cert). *(unit: temp `OPSCTL_ROOT` +
  fake `System`)*
- R-ARUP-NDQT — **Real-substrate (live box `int.ikigenba.com`).** On the box,
  `getent group web` succeeds and `id nginx` lists `web` among nginx's groups.
  This — that the *real* nginx user is actually a `web` member — cannot be
  falsified by the fake `System` (which records any call); it requires the real
  `usermod` on the box. Operator-verified out-of-loop; the observable proof is
  nginx serving a `sites` public page 200 (D8/R-AXY7-K8GA), which requires the
  membership to be real and live.
