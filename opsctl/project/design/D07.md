# Decision 7 — setup provisions the served `www` tree as `<app>:web`, setgid, via one `ensureWWWPerms` helper

## Decision

A service that serves files off disk through the front door (today only `sites`,
selected by `WWWDirsFor` returning a non-empty dir list) gets a `state/www` tree
with exactly two tier subdirs — `public/` and `private/`. A site's files live
**directly** under its tier (`public/<name>/…`, `private/<name>/…`): there is
**no** `working/` staging tree and no publish-symlink indirection. The sites
service collapsed to this single-tree model — the bytes exist at the path they
are served from — so opsctl's on-box view of the tree must match: the served-tree
segments are `public` and `private` only.

The tree must satisfy one **invariant**:

- **owner** `<app>` (the service writes it),
- **group** `web`,
- **tier dirs setgid** (`2750`) so every entry the service later creates under
  them inherits group `web` instead of the service's primary group.

The `web` group and setgid are retained as the established on-box ownership
normalization for the served tree: they keep `state/www` service-owned and
group-consistent across setup, deploy (D8), and restore (D9). (The sites service
now serves each tier through its own loopback process, with nginx proxying to it
rather than `alias`ing the directory, so the `web` group is no longer nginx's
read path; opsctl still normalizes ownership so the tree stays consistent
regardless.) Nothing needs to be world-readable — the private tier's bytes stay
off `other`.

The invariant is defined **once** in a single helper reused by setup, deploy
(D8), and restore (D9). It is idempotent, self-guards on the served tree's
existence, and touches only tier dirs that exist — so a box that still carries a
legacy `working/` directory from an earlier provisioning is simply left untouched
(no migration, no error):

```go
// internal/opsctl/www.go — the single definition of the served-tree invariant.
// Idempotent; safe to re-run. Only touches tier dirs that exist.
func (o *Opsctl) ensureWWWPerms(ctx context.Context, app string, l Layout) error {
    if _, err := os.Stat(l.WWWDir()); err != nil {
        return nil // no served tree for this app — nothing to do
    }
    // group-own the whole served subtree by the web group
    if err := o.System.ChownTree(ctx, app, "web", l.WWWDir()); err != nil {
        return fmt.Errorf("ensure www perms: chown %s -> %s:web: %w", l.WWWDir(), app, err)
    }
    // setgid each tier dir so future entries inherit group web
    for _, d := range []string{l.WWWDir(), l.WWWPublicDir(), l.WWWPrivateDir()} {
        if _, err := os.Stat(d); err != nil {
            continue
        }
        if err := o.System.Chmod(ctx, d, 0o2750); err != nil {
            return fmt.Errorf("ensure www perms: setgid %s: %w", d, err)
        }
    }
    return nil
}
```

The set of served-tree directories is defined in exactly two places that must
agree: `layout.go` exposes the tier paths (`WWWDir`, `WWWPublicDir`,
`WWWPrivateDir` — and **no** `WWWWorkingDir`), and `WWWDirsFor(root, app)` returns
the dirs setup creates for `sites` (`{WWWDir, WWWPublicDir, WWWPrivateDir}`, parent
first). Both enumerate the same two-tier shape; neither mentions `working`.

In `Setup`, the served-tree branch (`len(opts.WWWDirs) > 0`) creates those dirs
`0750` and then calls `ensureWWWPerms`. `setup` assumes the `web` group already
exists — D6 guarantees init-box created it before any `setup` runs; if it is
somehow absent the `ChownTree(... "web" ...)` fails loudly, which is the correct
"failures are loud" behavior, not a silent skip.

**Boundary.** D7 defines the served-tree shape (`public`/`private`, no `working`)
and the shared `ensureWWWPerms` helper + the `Chmod` seam. It does not create the
`web` group or touch the nginx user (D6), does not change the DEFAULT-app branch
(D5 — dashboard gets no www tree), does not change the unit/user/nginx-fragment
steps, and does not change `deploy`/`restore` (D8/D9 reuse the helper). Apps with
no served tree (`WWWDirsFor` empty) are unaffected: `ensureWWWPerms` is only called
on the served-tree branch and self-guards on `WWWDir` existence anyway.

## Rejected

- **Keep a `working/` staging tier (the original three-tier `working`/`public`/
  `private` model).** That tier existed only to stage edits before a publish step
  that rewrote a `<tier>/<name>` symlink into the working tree. The sites service
  removed both the staging tree and the publish step — files are authored and
  served in place — so opsctl carrying a `working/` segment provisions a directory
  the service never uses and, worse, made `ensureWWWPerms` chmod a path that no
  longer exists, aborting deploy. The on-box tree must mirror the two-tier model.
  Rejected.
- **Keep `working/` in the enumerated tier set but self-guard each entry on
  existence (chmod it only if present).** Tolerates the missing directory without
  aborting, but leaves a phantom segment in `layout.go`/`WWWDirsFor` that the
  service never creates and that misleads every future reader about the shape.
  Removing it outright is the single source of truth. Rejected.
- **World-readable `0755 app:app`.** Exposes the private tier on disk to every
  local box user; the `web` group keeps private bytes off `other`. Rejected.
- **Chown the tree but skip setgid.** Fixes existing entries once; the next entry
  the service creates drops back to `<app>:<app>`. Setgid is what makes the
  invariant survive future writes. Rejected.
- **Set setgid via `MkdirAll(0o2750)` at creation instead of a `Chmod` seam call.**
  Go's `MkdirAll` mode is masked by umask and the S_ISGID bit is not reliably
  honored across `mkdir`; an explicit post-create `chmod` through the seam is both
  reliable and unit-observable (the fake records it). Rejected.

- R-QFXB-VARQ — **The served-tree model carries no `working/` segment.**
  `WWWDirsFor("/opt", "sites")` returns exactly the three parent dirs
  `{WWWDir(), WWWPublicDir(), WWWPrivateDir()}` — three entries, none whose path
  ends in `/working` — and a `project/`-excluded grep over `internal/` for
  `WWWWorkingDir` and `www/working` returns no matches (the layout helper and the
  literal segment are gone). Falsifiable: today `WWWDirsFor(sites)` returns a
  `.../www/working` entry and `layout.go` defines `WWWWorkingDir()`. *(unit: direct
  `WWWDirsFor` call + a scoped grep)*
- R-QEPF-HJ11 — **`ensureWWWPerms` labels `state/www` `<app>:web` and setgids
  exactly the `public` and `private` tiers.** After `Setup` of a served-tree app
  (`WWWDirsFor` non-empty, i.e. `sites`), the fake `System` has recorded a
  `ChownTree(app, "web", WWWDir())` **and** a `Chmod(d, 02750)` for each of
  `WWWDir`, `WWWPublicDir`, `WWWPrivateDir` — and has recorded **no** `Chmod` of
  any path ending in `/working`, no `ChownTree(app, app, …)` over the www tree, and
  no `0755` www mode. Falsifiable: today `ensureWWWPerms` records a
  `Chmod(WWWWorkingDir(), 02750)`. *(unit: temp `OPSCTL_ROOT` + fake `System`)*
- R-AUAI-EX87 — **Setup of a no-served-tree app touches no www/`web` state.**
  After `Setup` of an app whose `WWWDirsFor` is empty (a fragment or worker app
  other than sites), the fake `System` records **no** `ChownTree(_, "web", _)` and
  **no** `Chmod` of a www path, and creates no `state/www` tree — the worker
  symlink / fragment-file behavior (D5/D01) is unchanged. Pins that the web-group
  labelling is scoped to served-tree apps only. *(unit: temp `OPSCTL_ROOT` + fake
  `System`)*
