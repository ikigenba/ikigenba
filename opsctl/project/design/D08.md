# Decision 8 — deploy re-asserts the served-tree `web` invariant after the state-ownership chown

## Decision

`deploy` runs as root, and the `migrate` verb it invokes just before the swap
also runs as root, so a freshly-created DB and its `-wal`/`-shm`/`.generation`
sidecars come back owned `root:root`. The unprivileged service user then cannot
take a write lock, so `deploy` hands the tree back with a **recursive**
`ChownTree(app, app, l.StateDir())` (`deploy.go`) before the restart. That sweep
is intentionally broad (it must catch whatever root-owned sidecars `migrate`
created), and `state/www` lives under `state/` — so the sweep **also rewrites the
served tree from `<app>:web` (what D7 set) back to `<app>:<app>`**, silently
locking nginx out again on **every deploy**. This was the mechanism that
re-broke `sites` serving after the initial fix.

This Decision adds one step: **immediately after the state-ownership chown,
re-assert the served-tree invariant by calling `ensureWWWPerms` (D7)** — which
re-labels `state/www` `<app>:web` and re-applies setgid — **whenever the app has a
served tree on disk**:

```go
// deploy.go, immediately after ChownTree(ctx, app, app, l.StateDir()):
if err := o.ensureWWWPerms(ctx, app, l); err != nil {
    return fmt.Errorf("deploy: restore www perms: %w", err)
}
```

`ensureWWWPerms` self-guards on `WWWDir` existence (D7), so this is a **no-op for
apps with no served tree** (including the DEFAULT app, which has none per D5) and
touches `state/www` **whenever the directory exists** — the guard is existence,
not the app name. That is deliberate: the broad chown's *damage* is
existence-based (it clobbers whatever `state/www` sits under `state/`), so the
repair is symmetric — it re-asserts the invariant on exactly the tree the sweep
just disturbed, for whatever app is being deployed. Whether some other app *should*
have a `state/www` at all is a separate question and not gated here.

The re-chown depends on the `web` group existing on the box; D6 guarantees
init-box created it before any deploy. `ChownTree` changes only owner:group and
leaves the directory setgid bit intact, but `ensureWWWPerms` re-applies setgid
regardless, so the invariant holds even if a prior state had lost it.

**Boundary.** D8 adds exactly one `ensureWWWPerms` call after the existing state
chown in `deploy`. It does not change the state chown itself (the DB-ownership fix
stays — its purpose is unrelated), the migrate/stage/swap/restart steps, the apex
block render (D4), or any non-deploy verb. It reuses D7's helper and D6's group;
it introduces no new seam.

## Rejected

- **Narrow the state chown to exclude `state/www` (chown only the DB + its
  sidecars) instead of re-asserting after.** Conceptually clean ("don't touch what
  you didn't break"), but it replaces one robust recursive call with an explicit
  enumeration of `sites.db` + `-wal` + `-shm` + `.generation` — the exact
  fragility the broad sweep was written to avoid, and it silently misses any new
  sidecar a future SQLite/appkit adds. Re-asserting after keeps the sweep robust
  and repairs its one unwanted side effect in a single idempotent call. Rejected.
- **Guard the re-chown on `WWWDirsFor(app) != nil` (i.e. `app == "sites"`) rather
  than on directory existence.** Makes the repair *narrower* than the damage: the
  broad sweep clobbers any `state/www` under `state/`, so keying the repair on the
  app identity would leave a genuinely-served tree mis-owned if the served-app set
  ever grows without someone updating deploy too. Existence keys the repair to the
  same signal as the damage and needs no second source of truth. Rejected.
- **Run `migrate` as the `<app>` user so the DB is born `<app>:<app>` and no
  state chown is needed at all.** The principled root-cause fix (it dissolves the
  reason the chown exists), but opsctl has **no** privilege-drop mechanism today —
  every verb execs as root through the `AppRunner` seam — so it means a new
  run-as-user capability on a load-bearing seam and a larger change than this
  bug warrants. Worth revisiting separately; out of scope here. Rejected.
- **Restart nginx after the deploy re-chown.** Unnecessary: nginx already has
  `web` (D6), and re-labelling files it can already reach by group requires no
  worker restart. Rejected.

## Verification

- R-AVIE-SOYW — **Deploy re-asserts the served-tree invariant after the state
  chown, when the tree exists.** After `Deploy` of an app with a `state/www` tree
  present, the fake `System` records a `ChownTree(app, "web", WWWDir())` (and the
  tier-dir `Chmod(02750)` calls) ordered **strictly after** the
  `ChownTree(app, app, StateDir())` sweep. Falsifiable: today's `deploy.go`
  records the state sweep and **no** subsequent `web` re-chown, so the served tree
  is left `<app>:<app>`. *(unit: temp `OPSCTL_ROOT` + fake `System`)*
- R-AWQB-6GPL — **Deploy of a no-served-tree app re-asserts nothing.** After
  `Deploy` of an app with no `state/www` on disk (e.g. the DEFAULT app), the fake
  `System` records **no** `ChownTree(_, "web", _)` and no www `Chmod` — the
  existence self-guard holds and non-served apps are untouched. *(unit: temp
  `OPSCTL_ROOT` + fake `System`)*
- R-AXY7-K8GA — **Real-substrate (live box `int.ikigenba.com`).** After a real
  `opsctl deploy sites <version>` on the box, an anonymous
  `GET https://int.ikigenba.com/srv/sites/public/<published-site>/` returns **200**
  — i.e. the served tree survived the deploy's state chown readable by nginx. This
  — that the group actually survived and nginx can still read — cannot be
  falsified by the fake `System`; it requires a real deploy + live HTTP fetch.
  Operator-verified out-of-loop.
