# Decision 17 — Per-hook verification schemes: `bearer` and `github-hmac`

**Problem.** Every hook today authenticates one way: a suite-minted bearer
secret compared at ingress (D3/D4). GitHub cannot send one. A GitHub App's
webhook signs each delivery instead — an HMAC-SHA256 of the **raw request
body** under a shared secret, presented as the `X-Hub-Signature-256` header
(`sha256=<hex>`) — and the repos service needs those deliveries on the event
plane. Provider-signature verification was product's named future feature;
this Decision adds it as a **per-hook verification scheme** without touching
bearer hooks.

**Decision.** A hook now carries a `verification` scheme, `bearer` (default,
today's behavior) or `github-hmac`.

1. **Schema** — one NEW timestamped migration (via `bin/create-migration
   webhooks <name>`; frozen migrations untouched) adds:
   `verification TEXT NOT NULL DEFAULT 'bearer'` and `secret TEXT` (nullable).
   For `bearer` hooks nothing changes: `secret_hash` holds the SHA-256
   verifier, `secret` stays NULL. For `github-hmac` hooks the **plaintext
   secret is retained** in `secret` (HMAC verification requires the key
   itself; a one-way hash cannot verify a signature) and `secret_hash` is
   kept as the same SHA-256 fingerprint for symmetry. Single-owner box,
   0600-perm SQLite: key-at-rest is the accepted posture, same as every
   provider secret in `~/.secrets`. Store threading: `Insert`/`GetByName`/
   `scanWebhook`/`UpdateSecret` carry the two new columns; the `db.Webhook`
   value object gains `Verification` but — as before — never carries secret
   material (show-once display semantics unchanged; `list` structurally
   cannot leak).
2. **Create/rotate** (`internal/webhooks`, D3 service + D6 tools) — `create`
   gains optional input `verification` ∈ {`bearer`, `github-hmac`} (default
   `bearer`; anything else → `validation`). The secret is still suite-minted
   (`ms_wh_` + `ids.New()`) and still shown exactly once by `create`/`rotate`
   — for a `github-hmac` hook the owner pastes it into the GitHub App's
   webhook-secret field. `create` and `list` outputs gain `verification`;
   `rotate` re-mints for either scheme (output shape unchanged).
3. **Ingress dispatch** (`internal/webhooks/ingress.go`) — after the name
   lookup, branch on the stored scheme:
   - `bearer`: byte-for-byte today's path (header check *before* body read).
   - `github-hmac`: read the capped body **first**
     (`http.MaxBytesReader`, same `maxBodyBytes`, over-cap → 413), then
     compute `hmac.New(sha256.New, secret)` over the raw bytes and compare
     against the `X-Hub-Signature-256` header (`sha256=` prefix, hex,
     `subtle.ConstantTimeCompare` on the decoded MACs). Missing header,
     malformed header, or MAC mismatch → the same byte-identical `notFound`
     404 every auth failure produces (failure uniformity is preserved across
     schemes). The identity-header rejection and method gate stay ahead of
     both schemes.
4. **Event payload** — for `github-hmac` hooks only, the D5/D15 payload gains
   a `headers` object carrying a two-key allowlist from the request:
   `x-github-event`, `x-github-delivery` (lowercased; absent ones omitted) —
   GitHub puts the event name in a header, and consumers (repos) need it for
   deterministic dispatch. Bearer hooks' payload is **unchanged** (no
   `headers` key). Kind/subject/registry are untouched — still one `received`
   family, subject `/<hook name>`.

**Rejected.**
- **Storing only a hash for github-hmac secrets** — cryptographically
  impossible to verify an HMAC against; the key must be recoverable.
- **A distinct event family for provider hooks** — the routing key already
  discriminates by hook name; consumers subscribe to the hook they care
  about, and one family keeps the registry closed.
- **Carrying all request headers in the payload** — headers are unbounded,
  provider-private, and mostly noise; an explicit allowlist is the whole
  need.
- **Verifying before reading the body for hmac** — the signature *covers*
  the body; the cap (`MaxBytesReader`) bounds the unauthenticated read, the
  same bound the bearer path applies post-auth.
- **A per-scheme ingress URL** (`/in-github/<name>`) — the scheme is a hook
  attribute, not an address; one URL shape keeps `trigger_url` and the nginx
  fragment untouched.

**Verification** (substrates: real migrated temp-file SQLite; the real
ingress handler over `httptest` with fixture GitHub deliveries signed by the
test).

- R-G7RX-751P — applying the full migration set to a fresh real SQLite
  yields `verification` (default `'bearer'`) and nullable `secret` columns;
  the Store round-trips a `github-hmac` hook (scheme + retained secret) and
  a `bearer` hook (NULL secret), and `db.Webhook` still carries no secret
  material.
- R-G8ZT-KWSE — `create` with `verification: "github-hmac"` returns the
  show-once secret and `verification` in its output and `list` shows the
  scheme (secret-free); `create` with an unknown scheme returns code
  `validation` and writes no row; omitted `verification` records `bearer`.
- R-GA7P-YOJ3 — after `rotate` on a `github-hmac` hook, a delivery signed
  with the new secret is accepted and one signed with the old secret gets
  the uniform 404 and no event.
- R-GBFM-CG9S — a POST to a `github-hmac` hook whose `X-Hub-Signature-256`
  is the correct HMAC of the raw body returns 202 only after commit, and the
  recorded event's payload carries `name`, `owner`, `received_at`,
  `content_type`, `body` (base64 of the raw bytes) **and** `headers` with
  exactly the allowlisted `x-github-event`/`x-github-delivery` values from
  the request.
- R-GCNI-Q80H — against the same hook: a missing header, a `sha256=`-less
  header, a wrong-MAC header, and an unknown hook name each return the
  byte-identical `notFound` 404 body and record no event.
- R-GDVF-3ZR6 — a body over `maxBodyBytes` to a `github-hmac` hook returns
  413 and records no event (the cap binds despite the body-first read
  order).
- R-GF3B-HRHV — bearer hooks are regression-pinned: a valid bearer trigger
  still authenticates before the body is read (asserted via a
  Content-Length-lying request or equivalent), returns 202, and its event
  payload has **no** `headers` key; a wrong bearer secret still gets the
  uniform 404.
