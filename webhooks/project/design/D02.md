# Decision 2 — Data model & migrations

## Decision.

The service owns its schema as ordered, forward-only SQL under
`internal/db/migrations/`, applied by the appkit runner. It ships the suite
**bootstrap trio** with integer prefixes (matching every existing service,
gmail being the newest scaffold); all *future* schema changes are added with
`bin/create-migration` (timestamped, immutable):

- `001_schema_migrations.sql` — copied verbatim (appkit's migration-tracking
  table: `version INTEGER PRIMARY KEY`, `applied_at TEXT NOT NULL`).
- `002_webhooks.sql` — the domain schema (below).
- `003_outbox.sql` — the bootstrap copy of the **pre-revision** outbox DDL;
  frozen. The outbox DDL is **owned by the eventplane library**
  (`outbox.SchemaSQL`); the service applies it through its own runner so there
  is a single migration authority per DB file. Since the event-routing revision
  the **current** outbox schema is applied by the newest timestamped
  `*_outbox_routing.sql` migration (D15), which drops and recreates the table
  per the revised `outbox.SchemaSQL`; the DDL drift guard
  (`internal/db/migrations_outbox_test.go`) asserts the **newest** outbox
  migration matches the library constant verbatim (D15's ids pin this).

**Domain schema** — the *effective* current table shape. The frozen
`002_webhooks.sql` created the original email-keyed table; the owner-id
conversion (`docs/owner-id-design.md`) ships one NEW timestamped migration (via
`bin/create-migration webhooks <name>`; frozen migrations untouched) that
**drops and recreates** `webhooks` with the shape below — rows are dropped, per
the suite decision that a service cannot map stored emails to owner ids:

```sql
CREATE TABLE webhooks (
    name              TEXT PRIMARY KEY,   -- routing key in /in/<name>; the natural unique handle
    owner_id          TEXT NOT NULL,      -- stable owner key; scopes list/delete/rotate — the ONLY identity read for logic
    owner_email       TEXT NOT NULL,      -- write-once snapshot at create; display only, never updated, never read for logic
    secret_hash       TEXT NOT NULL,      -- SHA-256 hex verifier (never plaintext) — see D3
    created_at        TEXT NOT NULL,      -- RFC3339Nano UTC
    last_triggered_at TEXT,               -- nullable; set on each successful trigger — see D4/D5
    verification      TEXT NOT NULL DEFAULT 'bearer',  -- per-hook scheme — see D17
    secret            TEXT                -- retained plaintext for github-hmac only — see D17
);
CREATE INDEX idx_webhooks_owner ON webhooks(owner_id);
```

`name` is the **primary key**: the public URL is `/srv/webhooks/in/<name>` with
no owner segment, so `name` is globally unique across all owners. There is no
surrogate id and no rename in v1. `last_triggered_at` is updated on the hot
ingress path inside the same transaction as the event append (one indexed write).

**Owner keying.** `owner_id` (the nginx-forwarded `X-Owner-Id`, appkit
`Identity.OwnerID`) is the sole scoping and lookup key: every query, filter,
and authorization comparison keys on it and only it. `owner_email` is captured
from the request headers at row creation and never touched again — display
data only. Two callers presenting the same email but different owner ids are
**different owners**; two callers presenting the same owner id are the same
owner regardless of email.

**SQL layer** — a concrete `Store` over `*sql.DB` (illustrative signatures, not
full code):

```go
type Webhook struct {                     // carries NO secret material — never leaves the store
    Name            string
    OwnerID         string                // scoping key
    OwnerEmail      string                // display snapshot
    CreatedAt       time.Time
    LastTriggeredAt *time.Time
    Verification    string                // D17
}

type Store struct{ db *sql.DB }

func (s *Store) Insert(ctx context.Context, w Webhook, secretHash string) error          // errors on duplicate name
func (s *Store) GetByName(ctx context.Context, name string) (w Webhook, secretHash string, ok bool, err error)
func (s *Store) ListByOwner(ctx context.Context, ownerID string) ([]Webhook, error)
func (s *Store) Delete(ctx context.Context, ownerID, name string) (deleted bool, err error)          // owner_id-scoped
func (s *Store) UpdateSecret(ctx context.Context, ownerID, name, secretHash string) (updated bool, err error) // owner_id-scoped (D3)
func (s *Store) TouchLastTriggered(tx *sql.Tx, name string, at time.Time) error                    // in-tx (D4/D5)
```

Every `owner`-shaped parameter above is an **owner id**, never an email; no
store method accepts or filters on `owner_email`.

The `Store` is the SQL CRUD seam; the domain `Service` (D1) composes it with the
`Clock` and the injected `Outbox`. `GetByName` returns `secretHash` separately so
the secret verifier never travels inside the `Webhook` value used by `list`.

## Rejected.

- **Surrogate `id` PK + unique `name` (crm-style).** Buys nothing here: there is
  no rename in v1, and `name` is already the handle in the URL, the MCP tool args,
  and the event payload. A surrogate would be a column with no reader.
- **Per-owner name uniqueness.** Would require an owner segment in the public
  trigger URL (`/in/<owner>/<name>`), which we rejected because it leaks the
  owner's email on a public, unauthenticated endpoint. Global uniqueness keeps the
  URL owner-free; the cost is an authenticated-only "name taken" existence signal,
  which is acceptable (default names are random 128-bit, so collisions are ~nil).
- **`Webhook` struct carrying `secret_hash`.** The verifier must never leave the
  store layer; keeping it out of the struct that `list` returns supports the
  show-once / never-retrievable promise (D3) structurally rather than by
  discipline.
- **Dropping `last_triggered_at`.** A single indexed UPDATE on the ingress path is
  cheap and gives `list` a genuinely useful "last fired" signal; the leaner-write
  alternative loses observability for no measurable gain on a downtime-tolerant
  single box.
- **Migrating existing rows (email → id backfill) instead of dropping them.**
  Only the dashboard knows the email→id mapping; the service cannot backfill.
  Suite-settled: rebuild empty (webhooks holds no live customer data during the
  migration window), deterministic on any database.
- **Keying on `(owner_id, owner_email)` or falling back to email.** Re-anchors
  logic to the mutable display field; `owner_id` alone is the contract.

## Verification.

- R-SZ8I-R4EY — Inserting a webhook whose `name` already exists fails on the
  primary-key/UNIQUE constraint and does **not** overwrite or duplicate the
  existing row. (Exercise `Store.Insert` twice with the same name against a real
  temp-file SQLite; the second call must return an error and the original row's
  owner_id/secret_hash must be unchanged — the real DB constraint is the
  substrate.)
- R-L3TX-A4Q3 — Applying the full migration set to a database that already holds
  a pre-conversion `webhooks` row leaves a `webhooks` table that has both
  `owner_id` and `owner_email` as NOT NULL columns (plus `verification`/`secret`,
  D17) and **zero rows** — the rebuild drops pre-conversion data rather than
  carrying rows with no owner id. (Real temp-file SQLite: create the
  pre-conversion shape, seed a row, run the appkit runner over the full
  migration dir, inspect `PRAGMA table_info` and the row count.)
- R-L51T-NWGS — `ListByOwner(idA)` returns exactly the webhooks whose `owner_id`
  is `idA` and never one owned by `idB`, **even when both owners' rows carry the
  same `owner_email`** — an implementation still filtering on email fails this.
  (Seed two owners with identical emails but distinct ids in real SQLite.)
- R-L69Q-1O7H — `Delete(idA, name)` and `UpdateSecret(idA, name, …)` where
  `name` belongs to `idB` act on nothing (`deleted`/`updated` == false, row
  unchanged via `GetByName`), even when both owners share one `owner_email`;
  `Delete(idB, name)` then removes it. (Real SQLite, two owners, same email —
  proves mutation scoping keys on `owner_id`, not name or email.)

(The outbox-DDL drift invariant is owned by D15 since the event-routing
revision: its guard pins the **newest** outbox migration against the revised
`outbox.SchemaSQL` and pins `003_outbox.sql` frozen.)
