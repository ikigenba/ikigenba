# Decision 3 — Webhook identity & secret lifecycle

## Decision.

Name generation/validation, secret minting, the verifier-at-rest, constant-time
verification, and rotation live in `internal/webhooks/secret.go` as small,
copied-pattern helpers (the suite deliberately has no shared crypto package).

```go
const secretPrefix = "ms_wh_"

func newName() string                                // ids.New() — 26-char base32 [A-Z2-7], 128-bit
func newSecret() string                              // secretPrefix + ids.New() — 128-bit
func hashSecret(plaintext string) string             // sha256 hex — the verifier at rest
func verifySecret(presented, storedHash string) bool // subtle.ConstantTimeCompare on the hashes
func validateName(name string) error                 // user-supplied names only; ^[A-Za-z0-9_-]{1,64}$
```

- **Secret.** `ms_wh_` + a single `ids.New()` (128-bit cryptographically-random,
  matching the suite's `ms_aco_`/`ms_oat_` token-plaintext convention). The
  plaintext is returned **exactly once** by `Create`/`Rotate`; only
  `hashSecret(plaintext)` (SHA-256, hex) is persisted in `secret_hash`. No salt /
  bcrypt / argon2: these are full-entropy random tokens, so a fast hash is the
  correct and conventional choice (dashboard PAT/OAuth tokens hash the same way).
- **Verify.** On a trigger, `Store.GetByName` yields the stored `secret_hash`;
  `verifySecret` hashes the presented secret and compares the two hex hashes with
  `crypto/subtle.ConstantTimeCompare`. (Consumed by the ingress handler in D4.)
- **Name.** Empty input → `newName()` (uppercase base32). Supplied input →
  `validateName`: a single clean URL path segment `^[A-Za-z0-9_-]{1,64}$` —
  slashes, dots, spaces, unicode, empty, and >64 chars are rejected. Names are
  case-sensitive (URL paths are).

Service methods this Decision owns (illustrative — not full code):

```go
var ErrNameTaken   = errors.New("webhook name already in use")
var ErrInvalidName = errors.New("invalid webhook name")
var ErrNotFound    = errors.New("webhook not found")

// Create mints a webhook for the owner identified by ownerID (the scoping
// key); ownerEmail is snapshotted onto the row at create for display and never
// read for logic afterward. name == "" → generate; else validateName.
// Returns the stored Webhook plus the plaintext secret (show-once; the caller
// must surface it immediately — it is never retrievable again).
func (s *Service) Create(ctx context.Context, ownerID, ownerEmail, name string) (w Webhook, secret string, err error)

// Rotate issues a fresh secret for the ownerID's webhook and invalidates the
// old one; name/owner_id/owner_email/created_at are unchanged. Returns the new
// plaintext secret (show-once).
func (s *Service) Rotate(ctx context.Context, ownerID, name string) (secret string, err error)
```

`Create` maps a duplicate-name insert (D2's PK constraint) to `ErrNameTaken`;
`Rotate` on a name that is missing or owned by someone else returns `ErrNotFound`
(via `Store.UpdateSecret` reporting `updated == false`). URL minting and MCP
result/error shaping are D6.

## Rejected.

- **bcrypt / argon2 / salted verifier.** Designed to slow brute force against
  low-entropy passwords; pointless for a 128-bit random token and out of step with
  the suite. SHA-256 hex of the high-entropy secret is correct.
- **Double `ids.New()` (~256-bit) secret.** 128 bits is already cryptographically
  ample; the suite's token-plaintext convention is a single `ids.New()`.
  Consistency beats gratuitous extra entropy.
- **Persisting the plaintext (even encrypted) for re-display.** Breaks the
  show-once promise; the recovery path for a lost secret is rotation, not
  retrieval.
- **Normalizing/lowercasing names.** Generated names are uppercase base32;
  normalization would desync user-supplied and generated forms. Validate-only,
  case-sensitive.

## Verification.

- R-37GT-C05G — `Create` returns a plaintext secret beginning `ms_wh_`, and the
  row persisted for that webhook has `secret_hash == sha256hex(secret)` and
  `secret_hash != secret`. (Create against real temp-file SQLite, read the row
  back, recompute the hash — proves the verifier is the hash of the returned
  plaintext, not the plaintext itself.)
- R-38OP-PRW5 — `verifySecret(p, h)` returns true when `p` is the exact secret
  whose hash is `h`, and false for any other `p`, including a value differing by a
  single character. (Unit — pins acceptance to the precise secret, not a prefix or
  partial match.)
- R-39WM-3JMU — After `Rotate(ownerID, name)`, the newly returned secret
  verifies against the stored hash and the previous secret no longer verifies,
  while `name`, `owner_id`, `owner_email`, and `created_at` for that row are
  unchanged. (Real SQLite — create, capture secret, rotate, re-read row and
  re-verify both secrets.)
- R-L7HM-FFY6 — The row persisted by `Create(ownerID, ownerEmail, name)`
  carries `owner_id == ownerID` and `owner_email == ownerEmail` exactly as
  passed — the email is stored as a snapshot, not derived, defaulted, or used
  as the key. (Real SQLite — create with distinct, unrelated id and email
  values and read the row back.)
- R-3CCE-V348 — A `ListByOwner` result item and the `Webhook` value returned by
  `GetByName` expose neither the secret nor `secret_hash` (e.g. JSON-serializing a
  list item yields no field carrying either); plaintext is produced only by
  `Create`/`Rotate`. (Real SQLite — proves show-once / never-retrievable
  structurally.)
- R-3DKB-8UUX — `Create` with an invalid user-supplied name (containing `/`, a
  `.`, a space, empty, or 65 characters) returns `ErrInvalidName` and writes no
  row; a name matching `^[A-Za-z0-9_-]{1,64}$` is accepted and persisted. (Real
  SQLite — assert both the rejection-with-no-row and the accepted persistence.)
- R-3ES7-MMLM — `Create` with an empty name generates a name matching the `ids`
  form (26 characters, alphabet `[A-Z2-7]`), and two successive empty-name creates
  yield different names. (Unit/real SQLite — pins the generated shape and
  uniqueness, not merely non-empty.)
