# Decision 4 — Public ingress endpoint (`/in/<name>`)

## Decision.

The public trigger endpoint. nginx (D7) proxies
`/srv/webhooks/in/<name>` → loopback `/in/<name>` (the `/srv/webhooks/in/`
prefix stripped). The handler is mounted **bare** — NOT wrapped in
`rt.RequireIdentity` — and self-guards. It is the only unauthenticated-by-OAuth
side-effecting surface in the service.

```go
const maxBodyBytes = 1 << 20   // 1 MiB

func NewIngressHandler(svc *Service, log *slog.Logger) http.Handler
```

Handler pipeline, in strict order:

1. **Method gate.** Method != `POST` → `405 Method Not Allowed`. The check is
   name-agnostic, so it leaks nothing about which names exist.
2. **Defense-in-depth.** If `X-Owner-Id`, `X-Owner-Email`, OR `X-Client-Id` is
   present → `404`. Their presence means the request transited the gated front
   door (`auth_request`), which must never reach `/in/`. `X-Owner-Id` is the
   identity header the suite now keys on (`docs/owner-id-design.md`), so it
   signals a leak exactly as the other two do. We do **NOT** reject on
   `X-Forwarded-Proto`: the ingress nginx tier legitimately sets it (D7), so only
   the *identity* headers signal a leak. (This corrects the research note, which
   lumped `X-Forwarded-Proto` in with the identity headers.)
3. **Resolve name.** `name := strings.TrimPrefix(r.URL.Path, "/in/")`; empty →
   `404`.
4. **Authenticate.** Extract the bearer from `Authorization: Bearer <secret>`,
   `Store.GetByName(name)`, `verifySecret(presented, secret_hash)` (D3). ANY
   failure — missing/empty/malformed header, unknown name, wrong secret — returns
   the **identical** `404` with the **identical** body. Authentication completes
   before any request body is read.
5. **Read body under cap.** Wrap the body with
   `http.MaxBytesReader(w, r.Body, maxBodyBytes)` and read it; exceeding the cap →
   `413 Request Entity Too Large`. Because step 4 already authenticated, only
   secret-holders ever reach this, so the `413` cannot be used to probe name
   existence.
6. **Record.** `svc.Record(ctx, wh, body)` performs the durable transactional
   append + `last_triggered_at` touch (D5). On success → `202 Accepted`, body
   `{"status":"accepted"}`.

The owner stamped on the downstream event is the **stored** `wh.OwnerID` /
`wh.OwnerEmail` resolved in step 4 — never any caller-supplied or header value.

**Settled choices:** POST-only (`405` otherwise); credential as
`Authorization: Bearer <secret>`; uniform `404` for every authentication failure;
`202 Accepted` on success; payload cap a 1 MiB constant.

## Rejected.

- **Rejecting on `X-Forwarded-Proto`.** The ingress tier sets it on every real
  call; rejecting on it would `404` all legitimate traffic. Only identity headers
  signal a front-door leak.
- **Reading the body before authenticating.** Lets unauthenticated callers spend
  server IO and probe the payload cap; authenticate first, read second.
- **Distinct statuses for unknown-name vs bad-secret (e.g. 404 vs 401).** Defeats
  enumeration resistance — an outsider could map which names exist. One uniform
  `404` for all auth failures.
- **`401` with a `WWW-Authenticate` challenge.** Invites retries and signals the
  resource is real; wrong posture for a capability-style ingress.
- **Custom header or query-param credential.** A query param leaks the secret into
  proxy/access logs and browser history; `Authorization` is the right channel.
- **Env/manifest-configurable cap in v1.** A constant is simpler and sufficient;
  tunability can be added later without changing the contract.
- **Trusting a caller-supplied owner/identity.** The owner is authoritative from
  the stored record only.

## Verification.

(All exercised via `httptest` driving the real `NewIngressHandler` over a
`Service` backed by real temp-file SQLite + a real outbox table — the auth, the
PK lookup, the body reader, and the append are all the real substrate.)

- R-7ISQ-ZZCF — A `POST /in/<name>` carrying `Authorization: Bearer <correct
  secret>` and a body within the cap returns `202` with body `{"status":
  "accepted"}`. (Proves the success path runs end to end against a real
  DB-backed service.)
- R-7K0N-DR34 — A wrong secret, an unknown name, and a missing `Authorization`
  header each produce **byte-identical** responses (same `404` status and same
  body), and none of the three appends a row to the outbox. (Real DB — asserts
  both indistinguishability and the no-side-effect-on-auth-failure invariant.)
- R-L8PI-T7OV — A `POST /in/<valid name>` (with a correct secret) that also
  carries `X-Owner-Id` (or `X-Owner-Email`, or `X-Client-Id`) is rejected `404`
  and appends no outbox row; the same request carrying only `X-Forwarded-Proto`
  and no identity header is accepted (`202`). Table driven over the three
  identity headers — a build still rejecting only the pre-conversion pair
  admits the `X-Owner-Id` case and fails. (Pins defense-in-depth to all three
  identity headers, not the legitimate forwarded-proto header.)
- R-7MGG-5AKI — With a correct secret, a body of `maxBodyBytes + 1` bytes returns
  `413` and appends no outbox row, while a body of exactly `maxBodyBytes` returns
  `202`. (Pins the cap at the 1 MiB threshold, not a vague "large".)
- R-7NOC-J2B7 — A non-`POST` request (e.g. `GET`) to `/in/<name>` returns `405`
  whether or not `<name>` exists in the store. (Proves the method gate is
  name-agnostic and leaks no existence signal.)
