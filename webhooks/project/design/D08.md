# Decision 8 — Test strategy, harness & dev-onboarding

## Decision.

Testing is layered so each claim is exercised against a substrate that can
falsify it. The spine `design.md` Conventions pin the commands
(`go build ./...`, `go vet ./...`, `go test ./...`); this Decision defines the
layers, the substrate rules, and the harness that makes the real-dependency
claims runnable.

**Four layers:**

| layer | substrate | covers |
|---|---|---|
| unit | pure Go, no DB | secret gen/verify/`validateName` (D3), base64 round-trip (D5) |
| store/service integration | **real temp-file SQLite** (`db.Open`) + injected deterministic `Clock` + real `eventplane/outbox` | D2, D3, D5 — constraints, owner scoping, durable append, registry validation |
| handler integration | `httptest` over a real-SQLite `Service` | D4 (ingress pipeline), D6 (MCP tools) |
| end-to-end | **real nginx** dev front door on `:8080`, suite up via `bin/start` | D7 — the tier routing/auth |

**Substrate rules:**

- **Temp-file SQLite, never `:memory:`.** The outbox startup probe opens a second
  connection to assert `BEGIN IMMEDIATE` serialization, and D5's durable-before-ack
  test reopens the file; `:memory:` supports neither. Tests use
  `t.TempDir()` files, matching the crm convention.
- **Deterministic `Clock`** is injected wherever a timestamp is asserted, so
  `created_at` / `received_at` / `last_triggered_at` are stable.
- **No mocks for the integration layer.** A mocked store or outbox cannot falsify
  a UNIQUE/PK constraint, durability across reopen, owner scoping, or Append-time
  registry validation — it accepts whatever it is handed. Those claims run against
  the real DB and real outbox.

**Verification-gate honesty.** The D7 end-to-end ids depend on real nginx routing
and cannot be proven against a stub or `nginx -t` alone. The verify phase **must
bring the suite up (`bin/start`) and run them for real**; a `t.Skip` taken because
`:8080` is unreachable is a convenience for local runs only — the gate treats an
all-skipped end-to-end layer as a **gap, not a pass**. Each D7 id names the
observable outcome that proves the route actually ran (an HTTP status from a real
request through `:8080`), not merely that a fragment file exists.

**Required dev-harness edits** (the only legitimate repo-root touches; needed both
to run the service and to drive the end-to-end layer):

- `go.work` — add `./webhooks` to the `use (…)` block.
- `bin/start` — add `webhooks` to the build list; add a `launch_webhooks`
  (exports `WEBHOOKS_DB_PATH`, runs on `:3011`); add to the PORTS map and the
  wait-for list.
- `bin/stop` — add `webhooks` to the shutdown list.
- `nginx/run` — add `webhooks` to the fragment `__PORT__`-substitution loop.

**Chassis defaults.** Backup/restore use appkit's default SQLite snapshot verbs
(no `Spec.Backup`/`Spec.Restore` override). webhooks is **producer-only**: no
`Spec.Consumes`, no `Spec.Workers`.

## Rejected.

- **Mocked `Store`/outbox for integration.** Cannot falsify the load-bearing DB
  and outbox claims; a category error per the design-mode substrate rule.
- **`:memory:` SQLite.** Defeats the reopen-based durability test and the
  single-writer startup probe.
- **Proving D7 with `nginx -t` only.** Validates syntax, not routing or auth
  behavior; the real front door on `:8080` is required.
- **Skipping the end-to-end layer in the gate when the suite is down.** Would let
  a broken fragment ship "green"; the gate must run it for real.

## Verification.

This Decision is primarily the **strategy** that proves D1–D7; its own ids cover
only the two onboarding behaviors not owned by another Decision, both against the
real harness.

- R-UELV-YLA4 — With the dev-harness edits applied, `bin/start` builds and
  launches `webhooks` on `:3011` and the running process answers `/health` with
  `200`; `bin/stop` then stops it. (Real harness — proves the service is wired into
  `bin/start`/`bin/stop`/`go.work`, beyond D1's in-isolation boot. A missing build
  or launch entry fails it.)
- R-UFTS-CD0T — With the suite up, the dashboard's service inventory /
  authorization-server resource list includes the webhooks MCP resource
  (`…/srv/webhooks/mcp`), discovered from the committed `etc/manifest.env`
  (`MCP=true`). (Real dashboard reading the real manifest — proves onboarding
  registration; absent or malformed manifest discovery fails it.)
