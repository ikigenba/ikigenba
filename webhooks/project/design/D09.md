# Decision 9 — Human landing page (web package, template & Carbon assets)

## Decision.

webhooks gains a minimal human **landing page** at the mount root, matching the
suite-wide v1 landing every other app now serves. It is a new self-contained Go
package `internal/web/` plus two routes wired at the composition root — no domain
code, no MCP, no token logic. The browser-session gate lives in the nginx
fragment (D7); in-process the routes are **ungated**.

**Package `internal/web/`** (mirrors the cron/crm canonical shape):

```go
// LandingHandler renders the webhooks human landing page: 200 with the
// service name + version at the exact root "/", 404 for any other path.
func LandingHandler(service, version string) http.Handler

// StaticHandler serves webhooks's embedded landing assets under /static/.
func StaticHandler() http.Handler
```

- `embed.go` — `//go:embed landing.html` (the template) and `//go:embed static`
  (its own copy of `tokens.css` + the woff2 fonts); a `mustSub` on the `static`
  subtree; `mime.AddExtensionType(".woff2", "font/woff2")` in `init`.
- `landing.html` — **byte-conformed to the cron canonical template**
  (`cron/internal/web/landing.html`): the same `<head>` (relative
  `href="static/tokens.css"`, two `<link rel="preload" as="font" type="font/woff2"
  crossorigin>` for the display and body fonts), the same CSS, the same
  `<main>` with the top-left `Home` link (`href="/"`) and the
  `eyebrow / h1 / p / dl(Service, Version, API)` structure. Only the
  **per-service data** differs from the canonical: the eyebrow, the one-line
  description, and the `<title>` — everything else is identical markup. Content:
  - eyebrow: `Inbound Webhooks`
  - description: *"Webhooks receives inbound HTTP from third parties on
    per-webhook secret-guarded endpoints and publishes each delivery as a typed
    event to the event plane."*
  - the API detail cell shows `POST /mcp` (matching crm; the per-webhook public
    ingress URL is returned by the MCP tools, not advertised on the page).
- `static/tokens.css` + `static/fonts/*.woff2` — **verbatim copies of the cron
  canonical assets** (byte-identical `tokens.css`, md5
  `63cb5c83f0db30a9646ea40ba5bb469e`; the shared woff2 set). `tokens.css` already
  carries `font-display: optional` and self-served `local`/relative `src`, so the
  FOUT is eliminated by copying the finished asset, not by re-deriving it.

**Route wiring** (`cmd/webhooks/main.go`, inside the existing `Spec.Handlers`
hook, **outside** `RequireIdentity` — the session gate is nginx's job):

```go
rt.Handle("GET /{$}", web.LandingHandler(rt.Service(), rt.Version()))
rt.Handle("GET /static/", web.StaticHandler())
```

The values come from the chassis (`rt.Service()` / `rt.Version()`); the page adds
no new data source. This sits beside the existing `POST /mcp` and `/in/` mounts
and shadows neither (`GET /{$}` is the exact root; `/mcp`, `/in/`, `/feed`, and
the PRM well-known keep their own patterns).

## Rejected.

- **A shared/suite-level landing handler.** The suite decision is that each app
  owns its own page so it can later diverge; webhooks embeds its own template and
  assets and depends on no other service's assets at runtime.
- **Gating the routes in-process (behind `RequireIdentity`).** The gate is coarse
  and browser-session-based; it belongs in nginx (D7). Bearer-gating in-process
  would make the page unreachable to a browser.
- **Surfacing the public `/in/<name>` ingress URL on the page.** The page sits
  behind the owner's dashboard session and the ingress URL is per-webhook (handed
  back by the MCP `create`/`list` tools); advertising a path here adds nothing and
  breaks uniformity with the canonical. The API cell matches crm (`POST /mcp`).
- **Re-authoring the markup/tokens instead of copying the canonical.** Would risk
  visual drift and re-introduce the FOUT the canonical already fixed; copying the
  byte-canonical template + assets is the whole point of the suite-wide template.

## Verification.

The page content is proven against the **real in-process handler** (`httptest`)
and the embedded bytes — no stub. The markup/asset claims are content assertions
over the embedded template and `tokens.css`.

- R-TMJH-V1NP — `LandingHandler("webhooks", v)` serving `GET /` returns `200`
  with `Content-Type: text/html; charset=utf-8` and a body containing the service
  name `webhooks` and the exact version string `v`. (A handler that dropped the
  name or version, or the wrong content-type, fails it.)
- R-TNRE-8TEE — the same handler returns `404` for any non-root path (e.g.
  `GET /nope`). (Proves the landing is bound to the exact root only and does not
  swallow other paths.)
- R-TOZA-ML53 — `StaticHandler()` serves an embedded asset: `GET
  /static/tokens.css` returns `200` with a non-empty body, and a `.woff2` font
  asset is served with `Content-Type: font/woff2`. (Proves the assets are
  embedded and the woff2 mime type is registered — a missing embed or unregistered
  mime fails it.)
- R-TQ77-0CVS — the embedded `landing.html` carries the cron-canonical structure:
  a relative `href="static/tokens.css"` stylesheet link, the top-left
  `Home` link to `/`, and the `eyebrow / <h1>{{.Service}}</h1> / description /
  dl(Service, Version, API `POST /mcp`)` markup with the webhooks-specific eyebrow
  and description. (A page that hard-coded an absolute asset path, dropped the
  Home link, or diverged from the canonical structure fails it.)
- R-TRF3-E4MH — the FOUT is eliminated in the embedded assets: `tokens.css`
  declares `font-display: optional` and self-served (`local(...)`/relative
  `url(...)`) `src` for its faces, and `landing.html`'s `<head>` preloads the
  display and body fonts with `<link rel="preload" as="font" type="font/woff2"
  crossorigin>`. (An asset set that shipped `font-display: swap`, remote font
  `src`, or dropped the preloads fails it.)
