# Decision 15 — Adopt `registry`: resolve crm's own loopback port by name at the composition root

**Problem.** `cmd/crm/main.go` hardcodes its loopback port as the integer literal
`Port: 3100`. "Which port a service answers on" is duplicated across the suite —
in each service's Spec, its `etc/manifest.env`, and its `etc/nginx.conf` — and
drifts silently: a renumber only surfaces at deploy, uncatchable by the compiler.
crm is the reference shape other services copy, so it must not model a bare port
literal. The shared `registry` library (`module registry`, the repo-root leaf
module) is the single source of truth for the port table; crm should ask it for
its address by name and never repeat the number.

**crm consumes no feeds.** crm is an event-plane **producer** (`Feed: "/feed"`),
not a consumer: it has no upstream feed URLs to resolve. This is the one place
crm's adoption is *smaller* than notify's — where notify D9 also re-pointed
`CRM_FEED_URL` / `PROMPTS_FEED_URL` peer-feed defaults at
`registry.BaseURL(src)+"/feed"`, crm has no such defaults at all. crm's Spec
carries no `Consumers`, no peer-address resolution, no `<APP>_<SRC>_FEED_URL`
env override. The **only** loopback-port literal in crm's non-test Go source is
its own `Port: 3100`, and this Decision replaces exactly that one site.

**Decision.** crm writes no loopback port literal in its non-test Go source: it
references itself **by name** through `registry`. Resolution happens **once, at
the composition root** (`cmd/crm/main.go`) — never per request — because the
table is compile-time data: if a port ever changes we rebuild the binaries
anyway. This is **behavior-preserving**: `registry`'s table carries crm's current
value (`crm=3100`), so the resolved value is byte-identical to the literal it
replaces.

**The call site (in `cmd/crm/main.go`).** The appkit `Spec.Port` value becomes a
name lookup — the strict composition-root form that panics on an unknown name (a
programming error):

```go
appkit.Main(appkit.Spec{
    App:   "crm",
    Mount: "/srv/crm/",
    Port:  registry.MustPort("crm"),
    ...
})
```

Every other Spec field (`MCP`, `WWW`, `Feed`, `Migrations`, `Events`,
`ManifestExtras`, the `Handlers`/`Producer` hooks) is unchanged — only the `3100`
literal moves to a `registry` call.

**Module wiring (inside `crm/` only).** `crm/go.mod` gains `require registry
v0.0.0` and the committed `replace registry => ../registry`, mirroring the
existing in-repo `appkit` and `eventplane` replace-siblings. This is the only
build-graph change.

> **External precondition (OUT OF SCOPE — not built by any crm phase).** The
> repo-root `go.work` must carry `use ./registry`, and the `registry` module
> itself must exist and be green. Both are owned at the repo root / by the
> `registry` project and are assumed satisfied. No crm Decision or phase edits
> `../go.work`, `../registry/`, `../bin/`, nginx, or any sibling module — the
> executor runs from `crm/` and cannot reach outside it.

**Resolve-at-startup, not per-request.** `Spec.Port` is read once by appkit at
boot. No handler or request path calls `registry`; the design's Conventions
already state the table is compile-time data, so this adds no runtime cost.

**Rejected.**
- **Per-request resolution / a `registry` lookup in a handler.** The table is
  compile-time constant; looking it up per request buys nothing. Resolve at the
  composition root, let appkit store the value, reuse it.
- **`registry.Port("crm")` (the checked two-value form) at the composition root.**
  An unknown own-name is a build-time programming error, not a recoverable runtime
  condition — `MustPort`'s panic is the correct fail-loud form here.
- **Leaving `Port: 3100` and only fixing the deploy artifacts.** The Spec literal
  is the drift source of record; leaving it keeps the number duplicated in Go and
  defeats the point of a single source of truth.
- **Inventing peer-feed resolution for crm.** crm is a producer and consumes
  nothing; adding `Consumers` or feed-URL defaults would be dead scope with no
  behavior behind it.

**Verification.**
- R-X04D-MBGE — **crm's own listen port is `registry.MustPort("crm")`**, not a
  `3100` integer literal, at the composition root. Falsifiable: the served/emitted
  port must equal `registry.MustPort("crm")`; a hardcoded `Port: 3100` in the Spec
  fails the intent. If the Spec's port is not directly inspectable from a test
  (appkit.Main blocks on serve), this id is satisfied by the manifest drift guard
  in D16 / phase 12 — `manifest.Emit` with `registry.MustPort("crm")` byte-matches
  the committed `etc/manifest.env` (`PORT=3100`) — and is tagged there, with the
  delegation noted in phase 11.
