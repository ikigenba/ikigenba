# Decision 12 — Web surface from `share/www` through the chassis (de-embed)

**Problem.** crm's landing page and static assets were embedded in the binary
(`//go:embed` in `crm/internal/web/`), like every service's. The suite is
reverting that: web assets are release artifacts that belong on disk — visible
and diffable in the versioned release tree, editable in dev without a rebuild,
served by one chassis mechanism instead of twelve drifting copies. crm is the
first service converted; this Decision is the service-side half of appkit's
chassis design (its D5 www-root resolution, D6 `appkit/web`, D7 `Spec.WWW` —
consumed here as a fixed external contract).

**Decision.** crm serves its web surface from the on-disk www root.

1. **The files move.** `crm/internal/web/landing.html` →
   `crm/share/www/landing.html`; `crm/internal/web/static/` →
   `crm/share/www/static/` (same bytes: `tokens.css`, the four woff2 fonts).
   `bin/ship` already bundles `crm/share/` into the release `share/<version>`
   tier; on the box the chassis resolves the root to `share/current/www`, in
   dev to `./share/www` or the `CRM_WWW_PATH` override.
2. **The Spec opts in.** `cmd/crm/main.go` sets `WWW: true`. The chassis loads
   the site at serve (failing startup loudly if the root is missing),
   auto-mounts `GET /static/`, and exposes the loaded site as `rt.WWW()`.
3. **The landing route renders through the chassis.** The `GET /{$}` mount in
   `Handlers` uses the few-line handler over `rt.WWW().Render("landing.html", …)`
   (D1/D2 carry the exact shape). The service-side `GET /static/` mount is
   **deleted** — the chassis owns it.
4. **`crm/internal/web/` is deleted** — handler, embed shim, template, assets,
   and its tests. The landing/asset tests move to the composition-root package
   (`cmd/crm`), loading the repo-real `share/www` tree via `appkit/web` (a
   relative path from the package dir), so they exercise the exact files that
   ship.
5. **Dev wiring (boundary-crossing).** `bin/start`'s `launch_crm` exports
   `CRM_WWW_PATH="$repo/crm/share/www"` beside the existing `CRM_DB_PATH`
   export, so a repo-root-launched dev binary finds the source tree. Like the
   appkit workspace's earlier `bin/start` phases, this one line crosses the
   `crm/` boundary and is verified by the live `bin/start` smoke, not the Go
   suite.
6. **Nothing else moves.** The nginx fragment (D4/D8), the landing markup and
   tokens (D6/D7/D8), the MCP surface, and the migrations are untouched. The
   embedded migrations (`internal/db`) and guide (`internal/mcp/guide.md`) stay
   embedded — they are not web assets.

**Rejected.**
- **Keeping the embed** — the suite-wide reversion this work exists to prove
  out; embedded assets are invisible on the box and cost rebuilds in dev.
- **`state/www` as the root** — that tree is the sites service's nginx-served
  user content, backed up with `state/`; crm's UI assets are release artifacts
  and live in the shipped `share` tier (appkit D5 records the full argument).
- **Keeping a thin `internal/web` package** — with the mechanism in `appkit/web`
  there is nothing left but a handler closure; a package holding one closure is
  ceremony. The composition root is the right home for composition.
- **Serving assets from both embed and disk during a transition** — two sources
  of truth; the conversion is atomic within one service.

**Verification** (over the repo-real `crm/share/www` tree; D1's R-LAND-\* and
D3's R-ASST-\* ids continue to pin the page/asset behaviors through the new
substrate).
- R-MTM5-0PXH — an `appkit/web` Site loaded from `crm/share/www` (the actual
  shipped tree, resolved relative to the test package) renders `landing.html`
  with an injected service and version visible in the output — proving the
  shipped template, not an embedded copy, is the page source.
- R-MUU1-EHO6 — the chassis static mount over the same tree serves
  `GET /static/tokens.css` (`200`, `text/css`) and
  `GET /static/fonts/space-grotesk.woff2` (`200`, `font/woff2`) with **no**
  crm-side static handler registered — proving the assets ship on disk and the
  chassis serves them.
