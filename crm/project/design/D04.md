# Decision 4 — nginx fragment: the session-gated `= /srv/crm/` landing location and the full owner-identity forwarding contract

**Decision.** crm's nginx fragment (`crm/etc/nginx.conf`) carries an **exact-match**
location for the bare mount root that session-gates the human landing page and
proxies it to the service's root, sitting alongside (not replacing) the existing
locations. Every location in the fragment that forwards owner identity today —
the session-gated landing root and the bearer-gated service prefix — captures and
forwards the **four** owner headers the dashboard's introspection hooks emit on
every allow, not the single `X-Owner-Email` they forwarded originally.

**The four owner headers.** The dashboard's introspection endpoints (the
`/_authn` bearer/PAT hook and the `/_session-authn` cookie hook) emit, fail-closed
on every allow, four owner headers:

- `X-Owner-Id` — durable owner key (never empty).
- `X-Owner-Email` — display/audit address (never empty).
- `X-Owner-Name` — percent-encoded display name (MAY be empty).
- `X-Owner-Picture` — percent-encoded avatar URL (MAY be empty).

The bearer hook additionally emits `X-Client-Id` (the OAuth client), as before.
A fragment location that forwarded only `X-Owner-Email` dropped the other three on
the floor; the contract below forwards all four wherever identity is forwarded.

**The session-gated landing location.** Exact-match `=` so it never shadows the
gated `/srv/crm/` prefix below or the MCP endpoint; a failed session check yields
`401` (bounced to sign-in per D17). It captures each owner header from the
`/_session-authn` subrequest into a **session-scoped** variable and forwards it:

```
# Landing page — human web UI, gated by the dashboard browser SESSION (cookie),
# validated by the dashboard-owned /_session-authn internal location (a sibling of
# /_authn). Exact-match `=` so it never shadows the gated `/srv/crm/` prefix below
# or the MCP endpoint. A failed session check yields 401.
location = /srv/crm/ {
    auth_request /_session-authn;
    error_page 401 = @login_bounce;
    auth_request_set $crm_session_owner         $upstream_http_x_owner_email;
    auth_request_set $crm_session_owner_id       $upstream_http_x_owner_id;
    auth_request_set $crm_session_owner_name     $upstream_http_x_owner_name;
    auth_request_set $crm_session_owner_picture  $upstream_http_x_owner_picture;
    proxy_set_header X-Owner-Email   $crm_session_owner;
    proxy_set_header X-Owner-Id      $crm_session_owner_id;
    proxy_set_header X-Owner-Name    $crm_session_owner_name;
    proxy_set_header X-Owner-Picture $crm_session_owner_picture;
    proxy_pass http://127.0.0.1:3100/;   # trailing slash → upstream sees "/"
    proxy_set_header Host $host;
    proxy_set_header X-Forwarded-Proto $scheme;
    proxy_http_version 1.1;
}
```

**The bearer-gated service prefix location.** The pre-existing
`location /srv/crm/ { auth_request /_authn; … }` prefix — fronting `/srv/crm/mcp`
and every other sub-path under the mount, with its `@crm_authn_500` rate-limit
re-emit — captures each owner header from the `/_authn` subrequest into a
**bearer-scoped** variable and forwards it, alongside its unchanged `X-Client-Id`:

```
location /srv/crm/ {
    auth_request /_authn;
    auth_request_set $crm_owner          $upstream_http_x_owner_email;
    auth_request_set $crm_owner_id        $upstream_http_x_owner_id;
    auth_request_set $crm_owner_name      $upstream_http_x_owner_name;
    auth_request_set $crm_owner_picture   $upstream_http_x_owner_picture;
    auth_request_set $crm_client          $upstream_http_x_client_id;
    …                                     # $authn_status/$authn_retry + error_page 500 unchanged
    proxy_set_header X-Owner-Email   $crm_owner;
    proxy_set_header X-Owner-Id      $crm_owner_id;
    proxy_set_header X-Owner-Name    $crm_owner_name;
    proxy_set_header X-Owner-Picture $crm_owner_picture;
    proxy_set_header X-Client-Id     $crm_client;
    proxy_pass http://127.0.0.1:3100/;   # trailing slash strips the /srv/crm/ mount
    …
}
```

**Variable naming — unique per service.** All service fragments are included into
the one apex `server` block, so a variable name that collided across two fragments
would be a redefinition error. The new variables follow the fragment's existing
convention (`$crm_owner` for the bearer email, `$crm_session_owner` for the session
email) with the header appended: bearer `$crm_owner_id` / `$crm_owner_name` /
`$crm_owner_picture`, session `$crm_session_owner_id` / `$crm_session_owner_name` /
`$crm_session_owner_picture`. Each is crm-prefixed and therefore unique across the
apex block.

**Identity hygiene extends to the new headers.** As with `X-Owner-Email`, a single
`proxy_set_header <Name> $<var>;` per owner header REPLACES any client-smuggled
inbound header of that name on the upstream request: nginx does not pass inbound
client headers as proxy headers unless explicitly set, and the last
`proxy_set_header` for a name wins. So one set per name, sourced from the auth
subrequest variable, is sufficient and safe for all four owner headers — a client
cannot smuggle identity through any of them.

**Downstream consumption is out of scope here.** This Decision governs only what
the fragment captures and forwards. Whether crm's binary reads `X-Owner-Id` (a
later suite-wide conversion) is a separate concern; this Decision does not require
the service to consume the new headers.

**How it coexists with the existing fragment.** crm's fragment also has:

1. `location = /srv/crm/.well-known/oauth-protected-resource` — unauthenticated
   PRM bootstrap (exact-match). Forwards **no** identity — unchanged.
2. `location = /srv/crm/feed { return 404; }` — the loopback feed is never
   reachable through nginx (exact-match). No identity — unchanged.
3. `location /srv/crm/static/` — session-gated landing **assets**, proxied to the
   chassis static handler. It forwards **no** owner identity today and stays that
   way; the four-header contract applies only to locations that forward identity.
4. `location @crm_authn_500` — the bearer rate-limit re-emit. No identity —
   unchanged.

nginx longest-prefix/exact-match rules make an **exact-match `=` location win over
a prefix location** for that one path regardless of source order, so `= /srv/crm/`
wins for the bare root only while the `/srv/crm/` prefix still serves `/srv/crm/mcp`
and every other sub-path.

**Why `/_session-authn`, not `/_authn`.** A browser carries the
`dashboard_session` cookie, not a bearer token, so the bearer gate (`/_authn`)
would refuse it. `/_session-authn` is the dashboard-owned, loopback-only internal
location that validates the cookie against the web-session store and returns `200`
plus the owner headers on success or `401` on failure. This is the **same**
mechanism `sites` uses for its private static tier. The gate is deliberately
**coarse**: any logged-in dashboard user passes; there is no per-resource check —
fine for a name+version page.

**Why the simple form on the session location (no 429 re-emit).** The bearer
prefix block carries the `$authn_status`/`error_page 500 = @crm_authn_500`
machinery because token introspection can return `429` (rate-limited) which
`auth_request` collapses to a synthetic `500`. The session check is a cookie
lookup, not a rate-budgeted token introspection, so the landing location maps the
session subrequest's allow/deny directly (`200`→serve, `401`→bounce), matching the
`sites` private-tier location.

**Templating.** The loopback port is a **literal** in the fragment (`PORT=3100`,
this service's fixed registry port), so nothing substitutes it at install —
`opsctl setup crm` on the box / `nginx/run` in dev drop the fragment verbatim. The
fragment is **not** a vhost — no `server{}`, no `listen`, no cert; it is dropped at
`/etc/nginx/conf.d/locations/crm.conf` and included by the dashboard's apex server
block, which is also where `/_session-authn` is defined.

**Rejected.**
- **Gating the landing page with `/_authn`** — a browser has no bearer token; it
  would always 401. Wrong audience.
- **Making the landing page fully public (no auth_request)** — the brief forbids
  it; a logged-out browser must get gated. The page is gated, just coarsely.
- **A prefix `location /srv/crm/` for the page** — would collide with the existing
  bearer prefix; the exact-match `=` cleanly carves the bare root out of the prefix.
- **Adding the `@crm_authn_500` 429 re-emit to the session location** — the session
  check is not rate-budgeted; the extra machinery is dead weight.
- **Forwarding the new owner headers on the static-asset or PRM/feed locations** —
  those forward no identity today and gain nothing from owner headers; the contract
  is confined to the two locations that already forward identity. Widening it would
  leak identity onto surfaces that never had it.
- **A single shared variable name (e.g. `$owner_id`) across fragments** — collides
  in the shared apex block; each owner variable stays crm-prefixed and unique.
- **Concatenating the owner facts into one header** — loses the per-field contract
  the dashboard emits and the identity-hygiene "one set per name" property; each
  header is captured and forwarded on its own.

**Verification.** (Proven by a test that reads `crm/etc/nginx.conf` from disk and
asserts over its content — see design's Testing strategy. nginx is not run by the
suite.)
- R-NGNX-2H5J — `crm/etc/nginx.conf` contains an **exact-match** location for the
  bare mount root: a `location = /srv/crm/ {` block (distinct from the prefix
  `location /srv/crm/ {`).
- R-NGNX-4K7L — that exact-match block gates with `auth_request /_session-authn`
  (the cookie/session validator) and **not** `auth_request /_authn` (the bearer
  validator).
- R-NGNX-6M9N — that block proxies to the loopback upstream **root**
  (`proxy_pass http://127.0.0.1:3100/;` — trailing slash), so the upstream sees `/`.
- R-NGNX-8P1Q — the pre-existing locations are retained alongside it: the
  bearer-gated `location /srv/crm/ {` prefix with `auth_request /_authn`, the
  `location = /srv/crm/feed { return 404; }`, and the
  `location = /srv/crm/.well-known/oauth-protected-resource` PRM bootstrap all
  still appear in the file.
- R-FTCJ-N9K8 — the session-gated landing block (`location = /srv/crm/ {`)
  **captures and forwards all four** owner headers: it contains
  `auth_request_set $crm_session_owner_id $upstream_http_x_owner_id;`,
  `auth_request_set $crm_session_owner_name $upstream_http_x_owner_name;`, and
  `auth_request_set $crm_session_owner_picture $upstream_http_x_owner_picture;`
  (alongside the existing `$crm_session_owner` from `$upstream_http_x_owner_email`),
  and a matching `proxy_set_header` for each of `X-Owner-Id` (`$crm_session_owner_id`),
  `X-Owner-Name` (`$crm_session_owner_name`), `X-Owner-Picture`
  (`$crm_session_owner_picture`), and `X-Owner-Email` (`$crm_session_owner`).
- R-FS4N-9HTJ — the bearer-gated service prefix block (`location /srv/crm/ {`)
  **captures and forwards all four** owner headers: it contains
  `auth_request_set $crm_owner_id $upstream_http_x_owner_id;`,
  `auth_request_set $crm_owner_name $upstream_http_x_owner_name;`, and
  `auth_request_set $crm_owner_picture $upstream_http_x_owner_picture;` (alongside
  the existing `$crm_owner` from `$upstream_http_x_owner_email` and `$crm_client`
  from `$upstream_http_x_client_id`), and a matching `proxy_set_header` for each of
  `X-Owner-Id` (`$crm_owner_id`), `X-Owner-Name` (`$crm_owner_name`),
  `X-Owner-Picture` (`$crm_owner_picture`), and `X-Owner-Email` (`$crm_owner`),
  with `X-Client-Id` (`$crm_client`) still forwarded.
