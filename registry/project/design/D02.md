# Decision 2 — The service table: a slice of structs with typed blocks and frozen seeds

## Decision

The single source of truth is a **slice of structs**, one row per service, each
row asserting a service's name, port, and type-block together:

```go
// Block groups services by type; the port number carries a hint of the kind.
type Block int

const (
    Core       Block = iota // platform services
    Apps                    // domain applications
    Connectors              // bridges to external systems
    Custom                  // unofficial / experimental
)

// Service is one row of the registry: identity (Name), address (Port), and type.
type Service struct {
    Name  string
    Port  int
    Block Block
}

// Services is the authoritative, frozen registry. Assignments are fixed by name;
// adding a service appends a row, it never shifts an existing one.
var Services = []Service{
    // core — 3000+, counts up
    {"dashboard", 3000, Core},
    {"wiki", 3001, Core},
    {"prompts", 3002, Core},
    {"scripts", 3003, Core},
    {"sites", 3004, Core},
    {"cron", 3005, Core},
    {"webhooks", 3006, Core},
    // applications — 3100+, counts up
    {"crm", 3100, Apps},
    {"ledger", 3101, Apps},
    // connectors — 3200+, counts up
    {"dropbox", 3200, Connectors},
    {"notify", 3201, Connectors},
    {"gmail", 3202, Connectors},
    {"github", 3203, Connectors}, // reserved: name owns the number before code exists
}
```

Blocks are declared by half-open-free explicit `[lo, hi]` ranges, so the
membership guard is a pure range check:

```go
// Range returns the inclusive port range a Block owns.
func (b Block) Range() (lo, hi int) // Core:3000-3099 Apps:3100-3199 Connectors:3200-3299 Custom:3000-3099
```

`Custom` shares the numeric `3000` block with `Core` but grows **down** from 3099;
with core occupying only low single digits and custom descending from the top, the
two would have to grow ~45 each to collide, so no separate custom block is
warranted. `Custom` is defined but unpopulated in the seed table.

Assignments are **fixed by default**: a name owns its number and an unrelated edit
never moves it. There is no positional/index-derived port (that would let an
insertion silently shift every later service — the collision-at-deploy failure the
timestamped-migrations rule exists to prevent). Adding a service can only append a
row. Reuse of a retired number is permitted but only via a deliberate, reviewed
edit; in normal operation a retired number stays dead.

Slice-of-structs over a bare `map[string]int` because it keeps identity, address,
and type in one row, gives the guard tests something to iterate, and preserves a
stable declared order; the O(1) lookup map (D3) is built once from this slice.

## Rejected

- **`map[string]int` (name→port only).** Loses the block/type per row, so block
  membership would need a second parallel structure to stay in sync — exactly the
  duplication this project removes. Also has no stable order.
- **Positional/index-derived ports** (e.g. `3000 + index`). An insertion or an
  alphabetization silently renumbers every following service. This is the precise
  failure mode the fixed table is designed to make impossible.
- **A parsed data file (JSON/TOML) as the source of truth.** Would add runtime
  I/O and a parse step (and arguably a dependency) to what is compile-time
  constant data; the Go slice *is* the data.

## Verification

- R-B00K-9JYR — **No two rows in `Services` share a port.** Falsifiable: adding a
  second row with an already-used port must fail this. Pins uniqueness of the
  address, not merely "the table is non-empty".
- R-B18G-NBPG — **No two rows in `Services` share a name.** Falsifiable: a
  duplicate name must fail this.
- R-B2GD-13G5 — **Every row's `Port` lies within its `Block`'s declared
  `Range()`** (Core 3000–3099, Apps 3100–3199, Connectors 3200–3299, Custom
  3000–3099). Falsifiable: a row whose port falls outside its block (e.g. an
  `Apps` service numbered 3005) must fail this; a check that only asserts "port >
  0" does not satisfy it.
- R-B3O9-EV6U — **`dashboard` is present and pinned to port `3000`** (the apex
  `DEFAULT` app / `opsctl --port` default). Falsifiable: any other port for
  `dashboard`, or its absence, must fail this. Names the exact value, not "set".
