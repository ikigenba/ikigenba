# Decision 1 — A standalone, zero-dependency `registry` module at the repo root

## Decision

`registry` is a **new standalone Go module at the repo root** (`registry/go.mod`,
`module registry`, `go 1.26`), not a package inside `appkit`. It is a leaf: every
other module may depend on it and it depends on nothing but the Go standard
library.

The placement is the whole point. `appkit` is the service **chassis** — the
runtime framework services are built on, and it links `modernc.org/sqlite`,
`eventplane`, and more. `opsctl` is the operator CLI that stands *outside*
services; today its `go.mod` requires exactly one thing (`golang.org/x/mod`) and
it deliberately reads the language-neutral `manifest.env` rather than linking the
chassis. Putting the table in `appkit` and having `opsctl` call it would force
`opsctl` to `require appkit` and drag the chassis's whole module graph into its
`go.sum` to answer "what port is `crm`?". A name→port table is shared
*vocabulary*, not chassis behavior, and has no reason to know about SQLite. So it
lives on its own, wired exactly like the existing shared library `eventplane`:
consumers get it via a committed `replace registry => ../registry` plus a
`go.work use ./registry` entry.

**Naming.** The package is `registry`, not `discovery`. The design's own non-goal
is that nothing is discovered at runtime — the table is compile-time data and the
loopback host is a constant. `registry` names a static authoritative table;
`discovery` would imply the runtime lookup this explicitly is not.

Scope of *this* module: create `registry/go.mod` and the flat `registry` package.
Wiring existing consumers (their `go.mod` `replace`, the root `go.work`,
`manifest.env` generation) is downstream adoption work and is **not** built here.

Illustrative `go.mod`:

```
module registry

go 1.26
```

This Decision is **structural** — it fixes where the code lives and its dependency
posture, not a runtime behavior. Its proof is the green standalone build plus the
zero-dependency smoke below; the package's behavior is proven by D2 and D3.

## Rejected

- **Put the table in `appkit`.** Inverts the layering (the operator tool would
  link the service chassis) and pollutes `opsctl`'s dependency graph with the
  chassis's transitive closure (sqlite, eventplane, …) to compile a name→port
  map. The table is vocabulary, not chassis behavior.
- **Name it `discovery`.** Contradicts the explicit non-goal (no runtime
  discovery). A reader importing `discovery` would assume a live lookup that does
  not exist. `registry` correctly connotes a static, authoritative table.
- **Name it `ports`.** Accurate but collides with the repo checkout directory name
  (`ports`), which is confusing in paths and conversation.
- **A nested `internal/registry` package under some module.** `internal/` would
  make it unimportable across module boundaries — the exact opposite of a shared
  leaf library.

## Verification

*(none — structural.)* This Decision defines placement and dependency posture, not
a behavior a unit test asserts. Its acceptance is mechanical and lives in the
plan's phase done-bar: the module builds green in isolation
(`GOWORK=off go build ./...` from `registry/` exits 0) **and** it has zero
third-party dependencies, checked by
`go list -deps -f '{{if not .Standard}}{{.ImportPath}}{{end}}' ./...` producing no
output. The package's runtime behavior carries its ids under D2 and D3.
