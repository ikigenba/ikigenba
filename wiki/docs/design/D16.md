# Decision 16 — MCP surface expansion: control & footprint verbs + paginated lists

**Decision.** The MCP product surface gains four verbs — **`jobs`** (filtered, paginated job list), **`abort`**, **`rerun`**, **`llm_calls`** (filtered, paginated footprint) — and the existing **`subjects`** and **`claims`** verbs are reshaped to cursor-paginated `{items, next_cursor}` envelopes accepting `limit`/`cursor`. The surface grows from eight to **twelve**. Everything is wired with the existing `WithXxxService` Option pattern, dispatched by the `handleToolCall` switch, and shaped by the existing reflection-based result helpers; nginx remains the trust boundary (`RequireIdentity`), wiki does no token logic. This supersedes D10's eight-verb framing (D10 is edited in place: its R-MUQ4 now asserts the twelve-verb set).

## The verbs

| tool | input | → result |
|---|---|---|
| `jobs` *(new)* | `{status?, since?, until?, limit?, cursor?}` | `{items:[{job_id, status, title, received_at, started_at?, finished_at?, error}], next_cursor}` |
| `abort` *(new)* | `{job_id (req)}` | `{job_id, aborted: bool, status}` |
| `rerun` *(new)* | `{job_id (req)}` | `{job_id, status}` or a clean tool error (in-progress / not-found) |
| `llm_calls` *(new)* | `{job_id?, stage?, since?, until?, limit?, cursor?}` | `{items:[{id, job_id, stage, attempt, provider, model, params, request, response, usage, error, started_at, ended_at}], next_cursor}` |
| `subjects` *(reshaped)* | `{type?, q?, limit?, cursor?}` | `{items:[{path, type, name, has_page}], next_cursor}` |
| `claims` *(reshaped)* | `{subject (req), limit?, cursor?}` | `{items:[{id, text, job}], next_cursor}` |

These back onto the D14 service methods (`Abort`, `Rerun`) and the D15 paginated list seams (`ListJobs`, `SubjectStore.List`, `ClaimStore.ListBySubject`, `LLMCallStore.List`). New Options: `WithJobsService`, `WithAbortService`, `WithRerunService`, `WithLLMCallsService`. The `tools()` builder appends each when its service is wired (matching the existing conditional registration).

## Design rules

- **No internal id crosses the boundary.** `job_id`, the `llm_calls` row `id`, and the `claims` `id`/`job` are public handles the wiki already mints and accepts; every subject reference stays a `type/slug` path. `llm_calls` carries **no subject id** — it is keyed by job, stage, and time only.
- **Time filters are RFC3339 strings** (`since`/`until`) parsed to `time.Time`; an unparseable timestamp returns a clean tool error, never a crash.
- **The `status` filter is validated** against the closed five-state set (`pending`/`working`/`done`/`failed`/`aborted`); an unknown value is a clean tool error.
- **`limit`/`cursor`** map onto `page.Params` (D15); the handler renders an undecodable cursor (D15 typed error) as a clean tool error.
- **`llm_calls` returns request/response inline** — the whole point of the footprint; pagination (default 50, D15) bounds the payload, and `job_id`/`stage`/time narrow it.
- **Not-found / not-actionable are results, not crashes** (existing convention): `abort`/`rerun`/`status` on an unknown `job_id` return a clean not-found; `abort` on a terminal job returns `{aborted:false, status:<current>}`; `rerun` on a pending/working job returns the D14 "can't re-run a job in progress" error.
- **Schema validity.** The new no-required verbs (`jobs`, `llm_calls`) and the still-no-required `subjects` emit `inputSchema` via the shared `objectSchema`, which omits `required` when empty (never `"required": null`) — the strict-client guard from D10 R-N4KO, now covering the added verbs.
- **Ungated this release.** All twelve verbs ship on the surface. The product's forward intent — moving part of the surface behind a `--debug` switch — is **not** built now; the four control/footprint verbs (`jobs`, `abort`, `rerun`, `llm_calls`) are its eventual candidates. Recorded as intent, not mechanism.

## How it's tested

Driving the JSON-RPC handler over an in-process server with a stubbed identity (D10 strategy), against the D14/D15 services on a real temp SQLite + scripted mock provider:
- `tools/list` returns the twelve verbs, each `inputSchema` well-formed — the no-required verbs (`jobs`, `subjects`, `llm_calls`) omit `required` entirely (never `null`).
- `jobs` with a `status` filter returns only matching jobs as a `{items, next_cursor}` page; the returned cursor walks the remainder.
- `abort` returns `{aborted:true}` for a pending/working job, `{aborted:false, status:<terminal>}` for a terminal one, and not-found for an unknown id.
- `rerun` returns `{status:"pending"}` for a terminal job, a clean error for a pending/working job, and not-found for an unknown id.
- `llm_calls` filtered by `job_id` returns that job's records with `request`/`response`/`params`/`model` populated, as a page; a `stage` and a time-range filter each narrow it.
- `subjects` and `claims` return `{items, next_cursor}`, accept `limit`/`cursor`, and their items keep `type/slug` paths with no internal id.
- A malformed `since`/`until`, an unknown `status`, or an undecodable `cursor` each yield a clean tool error, not a crash.

**Rejected.**
- **Gating the new verbs behind `--debug` now** — the product defers the debug switch to a later release; building it now is speculative mechanism. Ship ungated, note the candidates. Rejected (deferred).
- **A combined `job <action>` mega-verb** (list/abort/rerun under one tool with a mode arg) — muddies the schema and the result shape, and hides three distinct actions behind one name; separate verbs match the one-verb-one-job surface style (D10). Rejected.
- **Returning only a reference/summary for `llm_calls` (fetch body separately)** — adds a round-trip and a second verb for the exact payload the user wants; inline request/response under pagination is simpler and is the feature. Rejected.
- **A non-paginated `llm_calls`** — the highest-volume table; an unbounded list is the one place pagination is non-negotiable. Rejected.

**Verification.**
- R-37NS-BRXR — `jobs` returns a `{items, next_cursor}` page whose items carry `job_id`/`status`/timestamps; a `status` filter restricts the page to jobs in that state, and the returned `next_cursor` fetches the following page.
- R-38VO-PJOG — `abort` returns `{aborted:true, status:"aborted"}` for a pending or working job, `{aborted:false, status:<terminal>}` for an already-terminal job, and a clean not-found for an unknown `job_id` (never a crash).
- R-3A3L-3BF5 — `rerun` returns `{status:"pending"}` for a terminal job, a clean tool error for a pending/working job, and a clean not-found for an unknown `job_id`.
- R-3BBH-H35U — `llm_calls` returns a `{items, next_cursor}` page of records carrying `request`/`response`/`params`/`provider`/`model`/`attempt`; a `job_id` filter restricts to that job's calls and a `stage` filter restricts to that stage.
- R-3CJD-UUWJ — `subjects` and `claims` return `{items, next_cursor}` envelopes, accept `limit`/`cursor`, and their items carry `type/slug` paths (subjects) / public handles (claims) with no internal subject id.
- R-3EZ6-MEDX — A malformed `since`/`until` timestamp, an unknown `status` value, or an undecodable `cursor` each return a clean tool error rather than a crash or a silent reset.
- R-3G73-064M — `tools/list` returns the new verbs `jobs`/`llm_calls` (and the reshaped `subjects`) with `inputSchema` that omits `required` when empty — never `"required": null` — so a strict client accepts the whole list.
