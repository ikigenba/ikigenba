# Decision 11 — Subject addressing: the `type/slug` public path

**Decision.** Every subject has one public identifier — a `type/slug` **path** (e.g. `entity/acme-corp`) — and the opaque ULID `subjects.id` never leaves the service. The path is **derived, not stored**: it is a pure function of the subject's already-canonical `type` + `norm_name`, so it adds no column, no migration, and no write-path change. Resolution back to a subject is a forward-compare lookup.

This is a thin seam in `internal/wiki` consumed by D12 (the link footer) and D10 (the MCP surface).

```go
// Path is the public identifier for a subject: "<type>/<slug>".
// slug(norm_name) lower-cases nothing further (norm_name is already
// NFKC + casefold + trim + collapsed whitespace + diacritic-stripped, per D3);
// it only makes the name path-safe: each run of spaces or '/' becomes a
// single '-', leading/trailing '-' trimmed. All other runes (incl. non-Latin
// letters) are preserved, so a CJK subject keeps a meaningful slug.
func Path(s Subject) string            // s.Type + "/" + slug(s.NormName)
func slug(normName string) string      // pure, deterministic

// GetByPath resolves a public path back to its subject by forward-compare:
// split into (type, slug), then find the subject whose Type == type AND
// slug(NormName) == slug. Exact only — no fuzzy/alias matching (D3 identity).
//   - no match           → (Subject{}, ErrSubjectNotFound)
//   - exactly one match   → (subject, nil)
//   - two or more matches → (Subject{}, ErrAmbiguousPath)   // fail loudly
func (s *SubjectStore) GetByPath(ctx context.Context, path string) (Subject, error)

var (
    ErrSubjectNotFound = errors.New("wiki: subject not found")
    ErrAmbiguousPath   = errors.New("wiki: ambiguous subject path")
)
```

**Why derived, not stored.** `norm_name` is the subject's identity (D3, `UNIQUE(norm_name)`) and never changes once minted, so a slug computed from it is **stable** without being persisted. Storing a `slug` column would add a migration, a write-path step, and a second source of truth that could drift from `norm_name` — all for no gain, since the value is a pure function of data we already hold. The lookup computes `slug(NormName)` over candidate rows; at phase-1 corpus scale an `O(N)` scan (optionally narrowed by `type`) is acceptable. A stored, indexed `slug` column is a noted later optimization, not a phase-1 need.

**Collisions are rare but real and handled loudly.** Because `slug` collapses spaces and `/`, two distinct `norm_name`s can map to one slug under the same type (e.g. `x men` and `x-men` → `entity/x-men`). `UNIQUE(norm_name)` does **not** prevent this. Rather than silently pick one (silent corruption — a link or lookup would point at the wrong subject), `GetByPath` returns `ErrAmbiguousPath`. The MCP boundary (D10) surfaces that as a clean tool error, never a wrong answer.

**Scope of the path surface.** The path is the subject identifier in every outward-facing place — `subjects` listing, `page`/`claims` lookup inputs, the `status` produced-subjects list (D10), `ask` citations (D9), and the D12 link footer. Job ids and claim ids are separate handles and are out of scope: a job id is the public ingest handle (`ingest` returns it, `status` consumes it), and is not a subject identifier.

**Rejected.**
- **A stored `slug` column with `UNIQUE(type, slug)` minted at subject creation** — adds a migration, a write-path step, and collision-suffixing machinery (`acme-corp-2`); produces a *second* identity that can drift from `norm_name`. The derived path needs none of it and is stable because `norm_name` is. Rejected.
- **Reverse-engineering the slug back to a name (hyphens → spaces, then normalize-match)** — lossy and ambiguous when a real name already contains hyphens. Forward-compare (compute each candidate's slug, match the input) is exact and needs no inversion. Rejected.
- **Silently returning the first match on an ambiguous path** — silent corruption; violates "fail loudly." Rejected for `ErrAmbiguousPath`.
- **Exposing the ULID as a fallback identifier** — the product forbids showing internal ids; a fallback would re-leak them. Rejected.

**Verification.**
- R-ZO9U-QOT8 — `Path(s)` equals `s.Type + "/" + slug(s.NormName)`, where `slug` replaces each run of spaces or `/` with a single `-` and trims leading/trailing `-`, preserving other runes; it is deterministic and distinct non-colliding subjects yield distinct paths.
- R-ZQPN-I8AM — round-trip: `GetByPath(ctx, Path(s))` returns exactly subject `s`, resolving by exact `type` + `slug` match with no fuzzy/alias behavior (a variant or partial path does not resolve to `s`).
- R-ZRXJ-W01B — a path that matches no subject returns `ErrSubjectNotFound` (a clean miss), not a panic or a wrong subject.
- R-ZT5G-9RS0 — when two distinct subjects of the same type slug to the same path, `GetByPath` returns `ErrAmbiguousPath` and never silently selects one.
