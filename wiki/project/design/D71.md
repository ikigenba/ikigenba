# Decision 71 — The committed tune-folder workspace: `autotune/<step>/`, one per pipeline prompt

**Decision.** Every prompt the service submits for inference gets one committed tune folder under `wiki/autotune/`, conforming to the external `autotune` tool's folder contract (research §13): **`extract`**, **`analysis`** (ask question analysis), **`compile`**, and **`synthesis`** (ask answer synthesis). The folders are **data plus one scorer executable** — no Go, no build step, nothing the service links.

```
autotune/
  folders_test.go        test-only Go package (below) — the suite's integrity check
  <step>/                for step in {extract, analysis, compile, synthesis}
    prompt.txt           byte-for-byte copy of the step's production prompt at authoring time
    improve.md           improver system prompt, tailored to the step's failure modes
    score                the step's scorer executable (python3, +x) — D72
    config.json          {"runner": {...}, "improver": {...}} — pins below
    judge-prompt.txt     compile/synthesis only: the fixed judge rubric (read by score)
    fixtures/            scorer self-test fixtures (canned outputs + expected scores) — D72
    README.md            step-specific operator notes: what the cases are, scorer weights, how to run
    .gitignore           runs/ and __pycache__/
    cases/dev/<case>/input.txt + gold.json
    cases/holdout/<case>/input.txt + gold.json
```

- **Prompts are copies, never links.** Each `prompt.txt` starts as a plain copy of the step's in-code prompt (D6/D7/D9/D36). **No mechanical tie exists in either direction** — no test compares them, no embed references them, no tooling syncs them. After a tuning run the operator manually copies a winner into the owning package's prompt file; divergence in the meantime is expected and harmless.
- **Config pins.** Every folder's `config.json` is exactly:

```json
{
  "runner":   {"provider": "openai", "model": "gpt-5.6-luna", "auth": "sub", "effort": "low"},
  "improver": {"provider": "openai", "model": "gpt-5.6-sol",  "auth": "sub", "effort": "high"}
}
```

  The runner mirrors the production sites (D19: luna, effort low) so scores predict serving behavior; the improver is the product's contractual `gpt-5.6-sol`. No `temperature`/`max_tokens` pins — the subscription backend rejects both (research §13); autotune's epsilon noise floor absorbs the variance, and `MaxTokens` remains a production-transport concern the folder cannot mirror (accepted, documented divergence). Auth is the tool's own (`~/.autotune/auth.json`); the repo supplies only `OPENAI_API_KEY` via `.envrc` for the scorers (D1/D72).
- **Case sourcing — one shared universe, splits aligned.** All four case sets draw on the same fictional gold universe. `extract` copies the already-converted case set from the autotune repo's example (15 dev / 8 holdout; research §13) — cases and scorer verbatim, prompt **not** (the example's is deliberately naive). `analysis` converts the existing analysis gold (14 dev / 7 holdout question cases grounded in the extract corpus's splits). `compile` and `synthesis` get **seed corpora derived from the same universe** at similar scale (~14 dev / ~7 holdout): a compile case's `input.txt` is the production-rendered user message (subject identity + citation-tagged claims, built from extract-gold subjects); a synthesis case's is the production-rendered question + page bodies. **The dev/holdout boundary is aligned across all four folders**: material from a holdout universe document never appears in any step's dev case, and vice versa — no cross-step leakage. Seed cases are drafted by the build loop and **count as gold only after operator review**; growth happens through the same review channel, outside build phases. `gold.json` is scorer-only (autotune reads just `input.txt`); its shape is each scorer's contract (D72).
- **The integrity check.** `autotune/folders_test.go` (package `autotune`, a test-only Go package the suite compiles — no non-test source) walks the four folders and asserts the contract, so a broken folder fails `go test ./...` rather than the first paid run. It never reads the in-code prompts (no-tie rule) and never executes a model call; scorer *behavior* checks live in D72's ids.

**Rejected.**
- **Embedding the production prompt from the tune folder** (the D64 pattern) — makes tuning-side edits production edits; the operator decided the two must have zero mechanical connection, with manual copy as the whole adoption step.
- **Keeping folders gitignored workspaces** (the old `/autotune/` ignore) — the folders are now the durable, reviewed tuning assets; only `runs/` and `__pycache__/` are disposable.
- **A shared `cases/` tree referenced by all four folders** — the tool's contract is one self-contained folder; cross-folder references would break `autotune <folder>` portability for a little deduplication of committed text.
- **Mirroring `max_tokens` into the runner config** — rejected by the subscription backend; a hard pin that 400s every case is worse than a documented divergence.

- R-A88T-6WKN — Each folder's `config.json` parses and pins exactly runner `{provider: openai, model: gpt-5.6-luna, auth: sub, effort: low}` and improver `{provider: openai, model: gpt-5.6-sol, auth: sub, effort: high}`, with no `temperature` and no `max_tokens` key in either section.
