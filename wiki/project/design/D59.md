# Decision 59 — Every user-facing subject link is a fully-qualified front-door URL (retire the web-relative rule)

**Problem.** The MCP surface already returns **absolute** front-door URLs for `ask` citations (D56/D84): `https://<account>.ikigenba.com/srv/wiki/subject/<type>/<slug>` in production, `http://localhost:8080/srv/wiki/subject/…` locally. The **web** surface, by contrast, renders subject links as **origin-relative** hrefs (`subject/<type>/<slug>`) — D44/D45 map `wiki.Ref{Path}` → web `Ref{Href:"subject/"+Path}`, and D56 deliberately kept absolute URLs *out* of the web layer. That relative form resolves fine for in-browser navigation, but it is **not copy-paste-reusable**: a human who wants to share a link to a subject page cannot lift the href out of the page and hand it to someone (or an agent) — it has no origin. We want **every user-facing link that targets a subject** — inline links (D58) and both footers, on both web pages — to be a **fully-qualified front-door URL**, the same absolute form MCP already emits, so it is shareable and resolves from anywhere. In-app **navigation affordances** (the "ask another question" home control, the D41 dashboard Home link) are **not** subject links and stay relative.

**Decision.** Derive the front-door **subject base** once for the web handler exactly as the MCP handler does (D56/D84), and compose every web subject `Href` as `base + Path`. This **retires D56's "the web needs relative hrefs" containment rationale**: the web now *wants* the absolute form, so the reason D56 gave for not composing absolute URLs at the web seam no longer holds. The change is confined to the web composition root + `internal/web` `Ref` mapping; `internal/ask` and `internal/wiki` still traffic in bare `type/slug` paths (`ask.Citation.Path`, `wiki.Ref.Path`) — absolute composition stays a rendering concern, as it is for MCP.

- **The web subject base.** In `cmd/wiki/main.go`, where the web handler is assembled from `rt`:

  ```go
  webBase := strings.TrimRight(rt.AuthServer(), "/") + wiki.Mount + "subject/"
  // e.g. "https://acct.ikigenba.com/srv/wiki/subject/"  (prod)
  //      "http://localhost:8080/srv/wiki/subject/"       (local)
  ```

  Identical in shape to the MCP `pageBase` (D84) — `AuthServer` carries no trailing slash, `wiki.Mount` (`"/srv/wiki/"`) has leading+trailing slashes, so the join yields exactly one separator; `TrimRight` is belt-and-suspenders. This one `webBase` feeds both the footer `Ref` mapping and the D58 inline linkifier for the web surfaces.

- **Footer `Href` becomes absolute.** The adapter's `wiki.Ref{Path,Name}` → web `Ref{Href,Name}` mapping changes from `Href:"subject/"+Path` to `Href: webBase + Path` — for the outbound footer, the inbound footer, and the `MentionsIn` (D44) result. Every subject-targeting `<a>` the web renders now carries the fully-qualified URL.

- **Inline links use the same base.** The D58 web linkify call passes `webBase`, so an inline first-occurrence link and its matching footer entry point at the identical absolute URL.

- **Navigation stays relative.** The "ask another question" control (`.` → the mount root / reset home, D43) and the top-left Home link to the dashboard (D41) are **unchanged** — they are app navigation, not subject links, and `.`-relative home is what preserves reset-home behavior under the mount. No navigation affordance is absolutized.

**Downstream doc truth-up.** D44 and D45 are rewritten in place so their footer/`Ref` hrefs read as the absolute `webBase + Path` (the collateral footer-link assertions R-AU31-XI76 and R-PIAB-HZC0 are corrected to expect the absolute URL, the same way Phase 84 corrected the D10 URL assertions). D56 is rewritten to drop the now-retired "web needs relative hrefs" justification while keeping its citation-composition decision intact.

**Rejected.**
- **Root-relative hrefs (`/srv/wiki/subject/…`).** Shareable within one origin but not across (no scheme+host), and still not a URL you can paste into another tool. The fully-qualified form is the one the product asks for. Rejected.
- **Absolutizing the navigation affordances too.** A hard-coded absolute "home" would break the mount-relative reset-home indirection and coupling that `.` gives for free, for a control nobody copies to share. Kept relative. Rejected.
- **Composing absolute URLs at the source (`ask.Citation.Path` / `wiki.Ref.Path`).** Same reason D56 gave and still valid: the bare path is the internal currency, and both MCP and web already localize URL composition at their own rendering seam. Keeping the base at the seam means one place per surface, not a leak through the domain types. Rejected.
- **Reconstructing the origin per-request from `Host`/`X-Forwarded-*`.** D56 rejected this for MCP (trust surface, headers wiki doesn't trust); the web handler is behind the same nginx and uses the same canonical `rt.AuthServer()`. Rejected.

**Verification.**
- R-8I6N-DL0I — **Absolute web subject base, prod-vs-local, with the `subject/` segment (falsifiable across both origins).** With the web handler assembled over `rt.AuthServer() == "https://acct.ikigenba.com"`, a subject `Ref` for path `entity/tsr` renders `href="https://acct.ikigenba.com/srv/wiki/subject/entity/tsr"`; over `rt.AuthServer() == "http://localhost:8080"` the same `Ref` renders `href="http://localhost:8080/srv/wiki/subject/entity/tsr"` — the URL includes the full `/srv/wiki/subject/` prefix, exactly one `/` between parts (no `//`, no missing mount or `subject/` segment). A relative `subject/entity/tsr` href fails this. *(httptest, both origins)*
- R-8JEJ-RCR7 — **Navigation affordances stay relative.** On both the answer page and a subject page, the "ask another question" control's href resolves to the base (`.` / the mount root, not an absolute URL) and the D41 Home link is unchanged — no navigation control is rewritten to a `https://…/srv/wiki/` absolute URL, while subject links on the same page are absolute. *(httptest)*
