# Decision 68 — The autotune driver (`cmd/autotune`): one command from baseline to diff

**Decision.** `cmd/autotune` is the workbench's single front door: one command that prepares the workspace, measures the baseline, drives the unattended ralph loop, narrates progress, and finalizes with evidence and a diff — the operator types one line and reads one diff. It is a dev tool (built locally, never shipped) and, like `cmd/eval-extract`, outside the service; it imports **no agentkit** (all model work happens in its two children: the `eval-extract` binary and ralph), so D63's confinement grep is untouched.

```
autotune <step> [-c key=value ...] [--resume | --from <file>] [-- <ralph args verbatim>]
```

- **Step dispatch.** `<step>` names the pipeline stage being tuned. The layout (`autotune/<step>/`, `tmp/autotune/<step>/`, `eval/<step>/`) is generic from day one, but this release implements **extract only**: any other step exits non-zero naming the step and the supported set. No step argument is the same error.
- **Step-model config: free-form, agentrepl's `-c` convention (research §13).** The run's `-c key=value` set **is** the complete call specification — the driver is an experiment bench, and it inherits nothing from the committed `eval/extract/config.json`'s `eval` block (that block is production's reference, consulted only at manual-adoption time). Accepted keys: `provider`, `model`, `auth`, `auth_file`, `temperature`, `thinking`, `max_tokens`, `max_parse_retries`. `provider` and `model` are **required** — their absence exits non-zero naming the missing key; `auth` defaults to `key`; every other key is optional and, when not passed, is **absent from the spec and never sent** (the backend's own defaults apply — D66). An unknown key, or any attempt to touch `embedding.*`/`weights.*`, exits non-zero naming the key — the scoring side is never operator-variable per run.
- **The resolved config is a file.** At startup the driver writes `tmp/autotune/<step>/config.json`: the `eval` block verbatim from the `-c` set, the `embedding` and `weights` blocks verbatim from the committed config (scoring stays fixed). Everything downstream — the baseline, every loop-turn `run`, the holdout run — points at that one file, so the operator's spec governs all calls without threading flags through the agent, and the file doubles as the workspace's **config stamp**.
- **Workspace lifecycle.** A fresh run (no `--resume`) deletes `tmp/autotune/<step>/` entirely, writes the resolved config, copies the committed `eval/<step>/prompt.txt` over `autotune/<step>/prompt.txt` (with `--from <file>`: that file instead), builds the runner (`go build -o tmp/autotune/<step>/bin/eval-extract ./cmd/eval-extract`), and produces `baseline.json` (`run -repeat 3`, dev split) via that binary, installing the start prompt + baseline as `best/`. `--resume` keeps everything and skips baseline — but first compares the requested resolved `eval` block against the workspace config stamp and exits non-zero on any difference, naming both configs ("workspace was tuned under X, you asked for Y"); lineages under different step models never mix. `--resume` with `--from` is a flag error.
- **The loop, wrapped.** The driver execs `ralph <passthrough args> eval/<step>/improve.md` from `wiki/` — everything after `--` goes to ralph **verbatim** (harness, improver `-c` config, `--max-*` budgets; research §13), and an empty passthrough means ralph's own defaults. ralph's stdout/stderr stream through to the operator unmodified. SIGINT/SIGTERM are forwarded to the child; the driver itself dies only after finalizing.
- **Diff on every accept.** While ralph runs, the driver watches `autotune/<step>/prompt.txt`; each time its content changes it prints a clearly-marked unified diff of `eval/<step>/prompt.txt` → `autotune/<step>/prompt.txt` (the `git diff --no-index` form). The loop agent performs the copy on accept (D67); the driver's watcher makes the improvement visible without trusting the agent to narrate.
- **Finalizer — always runs, one shape.** When ralph exits — DONE, budget exhausted, killed by the forwarded signal, or crashed — the driver finalizes: if `best/prompt.txt` differs from the run's **starting** prompt (an accept happened, this run or a resumed one) and no `holdout-scorecard.json` exists yet, it runs holdout **once** on `best/prompt.txt` via the workspace runner, writes `summary.md` (dev best vs baseline + epsilon, attempt count, holdout composite, an explicit **overfit** call-out when the dev win doesn't hold up on holdout), and prints the final diff. If nothing was ever accepted it prints "no improvement" with the attempt count and evidence path, runs **no** holdout, and leaves `autotune/<step>/prompt.txt` equal to its start. Exit code: 0 either way when the run finalized cleanly; non-zero only for driver errors (bad flags, unprovisionable workspace, ralph unstartable).
- **Adoption stays outside.** The driver never writes to any committed path; `git status` after any run is clean (D67's boundary). The printed diff is a proposal; adoption is the operator copying the winner over `eval/<step>/prompt.txt` through the normal channel.
- **Seams for proof.** The driver's effects run behind two injected seams: an **executor** (starts/waits child processes: the `go build`, the `eval-extract` invocations, ralph) and the filesystem rooted at `wiki/`. Tests drive the CLI with a scripted executor (recording argv, faking exits, mutating the watched file mid-"run") and a temp tree — no real ralph, no network. The real-contract end of every child it execs is already owned elsewhere: `eval-extract`'s behavior by D66's ids (including its live smoke), ralph's by its own repository.

**Rejected.**
- **Reimplementing the improver loop in-process over agentkit** — rebuilds ralph's budget enforcement (iterations/time/spend/tokens with pricing), retry-on-bad-turn, and harness abstraction for no gain; wrapping keeps all of that upstream and the driver dumb.
- **An `improver.*` `-c` namespace translated into ralph flags** — a translation layer that would trail ralph's real surface forever; the verbatim `--` passthrough exposes all of ralph (any harness, any budget) at zero maintenance.
- **A tracked workspace prompt for plain-`git diff` visibility** — a committed file the loop writes makes every run dirty the tree and adds an adoption-time bookkeeping step; the untracked copy plus a printed `--no-index` diff shows the same patch with a clean tree.
- **Env-var config overrides (`EVAL_EXTRACT_MODEL=…`)** — ambient state the scorecard can't prove was in force; the resolved config file is explicit, stamped, and checkable.
- **Inheriting the committed production pins as a base layer under `-c` overrides** (the previous form of this decision) — the pins describe *one* provider's call, and dragging them under a different provider's spec injects parameters the operator never asked for (a pinned `temperature: 0` is a hard 400 on the ChatGPT subscription backend). The bench tests exactly the specification given, nothing more; production faithfulness is the operator's choice of `-c` values, not an inheritance mechanism.
- **Per-step-model workspace subdirectories** — two concurrent lineages of one step is not a real workflow; the config stamp + hard `--resume` mismatch error prevents mixing at lower complexity.

**Verification.**
- R-EYQR-OABI — Step dispatch: `autotune extract` (scripted executor) proceeds; `autotune compile`, an unknown step, and a missing step each exit non-zero naming the offending/missing step and that `extract` is the supported set; `--resume` combined with `--from` is a flag error.
- R-EZYO-2227 — Fresh-run provisioning: on a temp tree, a fresh run deletes pre-existing `tmp/autotune/extract/` content, writes the resolved config, seeds `autotune/extract/prompt.txt` byte-identical to `eval/extract/prompt.txt` (with `--from f`: to `f`), and invokes — in order, via the executor — the runner build, then a baseline `run` with `-repeat 3`, `-split dev`, `-config tmp/autotune/extract/config.json`, installing prompt + baseline under `best/`.
- R-XGGI-JK9U — Free-form config resolution: `-c provider=openai -c model=X` yields a workspace config whose `eval` block carries exactly `provider`, `model`, and default `auth=key` — no `temperature`, `thinking`, or `max_tokens` key present — with `embedding`/`weights` byte-identical to the committed file; `-c temperature=0.5` adds exactly that key; omitting `provider` or `model` exits non-zero naming the missing key; `-c embedding.model=…`, `-c weights.claim=…`, and an unknown key each exit non-zero naming the rejected key, writing nothing.
- R-F16K-FTSW — Resume guard: `--resume` with a workspace stamp matching the requested `eval` block proceeds without re-running baseline (executor records no build/baseline calls); a differing stamp (e.g. model X vs requested Y) exits non-zero naming both, mutating nothing.
- R-F3MD-7DAA — The wrapped exec: with passthrough `-- --harness agentkit -c model=Z --max-time 2h`, the executor receives exactly `ralph --harness agentkit -c model=Z --max-time 2h eval/extract/improve.md`; with no `--`, exactly `ralph eval/extract/improve.md`.
- R-F4U9-L50Z — Diff on accept: when the scripted "ralph" mutates `autotune/extract/prompt.txt` mid-run, the driver's output contains a unified diff from `eval/extract/prompt.txt` to the new content before the run ends; a run whose file never changes prints no diff.
- R-F625-YWRO — Finalizer with a winner: when the child exits (each of: success, non-zero, signaled) with `best/prompt.txt` differing from the start prompt and no holdout scorecard present, the driver invokes exactly one holdout `run` (`-split holdout`, workspace config) via the executor, writes `summary.md` naming dev best, baseline, epsilon, holdout composite, and an overfit verdict when holdout ≤ baseline best, and prints the final diff; a second `--resume` finalization with `holdout-scorecard.json` already present runs no second holdout.
- R-F7A2-COID — Finalizer without a winner: when the child exits with `best/prompt.txt` equal to the start prompt, the driver runs no holdout, prints a "no improvement" line with the attempt count, leaves `autotune/extract/prompt.txt` unchanged, and exits 0.
