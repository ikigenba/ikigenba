# Decision 44 — The ask result page (`GET /{$}` with `?q=…`) + the `MentionsIn` seam

**Decision.** When the root route (D42) sees a non-empty `q`, it calls `Asker.Ask(ctx, owner, q)` (D9) and renders `answer.tmpl`. `owner` is the nginx-supplied `X-Owner-Email` header (the session-gated request carries it, D47); `ask` is owner-scoped only for call recording — the answer content is the same read-only synthesis D9 already produces. The web adds **no** answering logic; it renders D9's `Answer` and decorates it with a mention footer.

**Layout, top to bottom (`answer.tmpl`):**
- **"Ask another question"** — a link to `.` (resolves to `<base>`, the reset home D43). Present on every answer page, `Found` or not.
- **Answer prose** — `Answer.Text` as a paragraph. On `Found:true` this is the cited synthesis; on `Found:false` it is D9's fixed honest-empty sentence. **Both render identically** — the honest-empty text is just the body; there is no separate "no results" branch or styling.
- **Outbound footer** — the distinct subjects whose name or alias occurs in the answer prose, each a subject link `<a href="{{.Href}}">{{.Name}}</a>`. Omitted whole when empty (e.g. honest-empty, whose sentence names no subject). Outbound only — an answer is ephemeral, nothing points *to* it, so there is no inbound section (product decision).

## The `MentionsIn` seam

The outbound footer is **every subject/alias occurrence in the prose** (the product call), not the citations — so it is `Mentions(answerText, allSubjectKeys)`, the pure D12 matcher. D12 loads `SubjectKeys` (subjects ∪ D25 aliases) only inside `PageWithLinks`; this Decision exposes that over arbitrary text as a new method on `wiki.Service`, reusing the exact D12 matcher with no new matching logic:

```go
// MentionsIn loads the current SubjectKeys (all subjects ∪ their D25 alias keys)
// and returns every subject whose name or an alias key occurs as a whole,
// hyphen/edge-bounded run (D12 Mentions) in text — rendered as web Refs
// (Href = "subject/<type>/<slug>", Name = canonical display name). A match via an
// alias key resolves to the CANONICAL subject (D12), so a folded short name in the
// answer links to the survivor. Read-only; no write path.
func (s *Service) MentionsIn(ctx context.Context, text string) ([]Ref, error)
```

The web handler's ask path injects this as a second capability beside `Asker` (a `Mentioner` seam, `MentionsIn(ctx, text) ([]Ref, error)`), and the `cmd/wiki/main.go` adapter maps `wiki.Ref{Path,Name}` → web `Ref{Href:"subject/"+Path, Name}`. Targets are de-duplicated and deterministically ordered (by path), matching D12's footer discipline.

**Why not the citations.** `Answer.Citations` (D9) are only the pages the synthesis drew on; the product requires linking *every* subject/alias the prose names, cited or not. `MentionsIn(answerText)` is that set and reuses D12 exactly, so the answer footer and a page footer share one matcher and one alias semantics.

**Read-only.** `Ask` and `MentionsIn` only read; the ask page mutates nothing (D9 `R-5X56-NF4O` holds; `MentionsIn` has no write path).

**Rejected.**
- **Footer from `Answer.Citations`.** Links only cited subjects, contradicting the product's "every occurrence of any subject/alias." Rejected.
- **A distinct honest-empty page/branch.** D9 already returns a usable `Answer.Text` for the empty case; rendering it as the ordinary body keeps one template and one code path, and the omitted-when-empty footer falls out naturally. Rejected.
- **Re-implementing mention detection in the web layer.** Would fork D12's alias-aware matcher and risk drift. `MentionsIn` reuses it. Rejected.
- **Threading `owner` into the answer rendering.** The answer is identity-independent content; `owner` is only `ask`'s call-recording scope. The template reads no identity. Rejected.
