# Decision 49 вЂ” Token-based CSS for the rendered markdown element set

**Decision.** D48 renders compiled prose to HTML whose element set is wider than
the read surface previously emitted (`h3`вЂ“`h6`, `ol`, `li`, `code`, `pre`,
`blockquote`, GFM `table`/`thead`/`tbody`/`tr`/`th`/`td`, `strong`/`em`/`del`,
plus inline `a`). These must be styled **with the suite's existing design tokens**
so a rendered page is visually consistent with the rest of the suite вЂ” no new
stylesheet, no CSS framework, no ad-hoc values (the product non-goal: "no bespoke
styling system; reuse the shared tokens").

**Scoping container.** The rendered prose is wrapped in a single `.prose`
container in both templates вЂ” `answer.tmpl` and `subject.tmpl` (D44/D45):

```html
<div class="prose">{{ md .Subject.Body }}</div>   <!-- subject.tmpl -->
<div class="prose">{{ md .Answer.Text }}</div>     <!-- answer.tmpl -->
```

The body is **not** wrapped in `<p>` anymore вЂ” `md` (D48) emits its own block
elements (a `<p>` cannot legally contain `<h2>`/`<ul>`/`<table>`). Scoping the
markdown CSS to `.prose` keeps it off the footer `<nav>`/`<ul>` and the search
`<form>` вЂ” necessary because the ask answer's mention/cite footers sit **inside**
the answer `<article>` (D44), so a bare global `ul`/`a` rule would bleed between
body lists and footer link lists.

**Styling.** `layout.tmpl`'s embedded `<style>` block (the single inline
stylesheet, alongside the embedded `tokens.css` that defines the variables) gains
`.prose`-scoped rules for the markdown element set, every value drawn from the
existing tokens вЂ” `var(--color-text)`, `var(--color-text-muted)`,
`var(--color-border)`, `var(--color-accent)`, the `var(--space-*)` scale, the
`var(--text-*)` type ramp, `var(--font-mono)` (for `code`/`pre`),
`var(--border-width)` (table/blockquote rules). No new color, size, or font
literal is introduced. The obsolete `p { white-space: pre-wrap }` rule (which
existed only to preserve newlines in the prior raw-text body) is dropped, since
the body now arrives as structured HTML.

**Boundary.** D49 owns only presentation (the container + CSS); *what* renders is
D48, *which* prose and the footers are D44/D45. tokens.css itself (the variable
definitions) is unchanged вЂ” this Decision only consumes those variables.

**Rejected.**
- **Styling bare element selectors globally** (`h2 { вЂ¦ }`, `ul { вЂ¦ }`). Bleeds
  into the footer link lists and the search form, which are not markdown prose.
  Scope to `.prose`. Rejected.
- **A separate markdown stylesheet / CSS framework** (e.g. a `github-markdown`
  sheet). Adds a second styling system and off-brand visuals, violating the
  product's "reuse the shared tokens, nothing new." Rejected.
- **Inline `style=` attributes on emitted elements.** bluemonday (D48) strips
  `style` attributes by policy, and inline styles can't reference tokens cleanly;
  unmaintainable. Rejected.

inline `<style>`, so both claims are checkable from the response body).
