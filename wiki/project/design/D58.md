# Decision 58 — Inline first-occurrence subject links: the positional matcher + markdown-safe linkifier (`internal/wiki`)

**Problem.** Today a page's / answer's outbound subject links live only in a **footer** (D12 `RenderFooter`, D44/D45 footers). The prose itself carries only whatever literal markdown links the LLM wrote. An agent (or a human) reading the prose has to cross-reference the footer to find where a named subject is navigable. We want the **first occurrence of each referenced subject, inside the prose, rendered as an inline link** to that subject's page — the link text the author's own grammatical surface form, the href the subject's URL — on every prose-bearing surface (MCP `ask`, MCP `page`, the web answer page, the web subject page). The footers **stay** (a purely additive change).

**Decision.** Add a **read-time, pure, markdown-string linkifier** in `internal/wiki` that wraps the first eligible occurrence of each subject in a body of markdown prose as a markdown link, and a `Service` loader that feeds it the current corpus. It reuses D12's matching semantics exactly — exact **normalized whole-run** match against a subject's identity key **or any merge alias key** (D25), resolving via an alias to the **canonical** subject — and adds the one capability D12 lacks: the **raw byte span** of that first surface occurrence, so the surface text can be wrapped in place. It emits markdown (not HTML), so **one** linkifier feeds all four surfaces: MCP returns the linkified markdown directly; the web feeds it through the existing D48 `md` render func, which turns the injected `[…](…)` into a sanitized `<a>`.

```go
// FirstMention is the earliest eligible surface occurrence of one canonical
// subject in a body: the byte span [Start,End) in the ORIGINAL (un-normalized)
// body, and the canonical subject it resolves to. "Eligible" = a whole,
// hyphen/edge-bounded run (D12) matching one of the subject's Keys, lying
// entirely OUTSIDE a skip-region (inline code span, fenced code block, or an
// existing markdown link/autolink). A match inside a skip-region is not
// eligible and does not consume the subject's first occurrence — the next
// eligible run (if any) is used instead.
type FirstMention struct {
    Start, End int     // byte offsets into the original body; body[Start:End] is the surface run
    Subject    Subject // the CANONICAL subject (alias runs resolve to the survivor)
}

// LinkFirstMentions wraps the first eligible occurrence of each subject in
// `others` as a markdown link [ body[Start:End] ]( base + Path(canonical) ),
// preserving the surface run verbatim as the link text (author grammar/casing).
// One link per canonical subject (its earliest eligible run); later occurrences
// stay plain text. Overlapping candidate runs across DIFFERENT subjects resolve
// leftmost-start, longest-run on a tie; a run overlapping an already-placed link
// is skipped and its subject falls to its next eligible run (or goes unlinked).
// The subject whose ID == excludeID is never linked (self-exclusion; "" links
// all). base is the absolute front-door subject base (D56/D59),
// e.g. "https://acct.ikigenba.com/srv/wiki/subject/". Pure and deterministic —
// no map-iteration dependence — table-tested directly.
func LinkFirstMentions(body string, others []SubjectKeys, base, excludeID string) string
```

**The `LinkifyMentions` service loader.** Parallel to D44's `MentionsIn`, a `Service` method loads the current `SubjectKeys` (all subjects ∪ their D25 alias keys, via `AliasStore.ListAll` — the exact set D12/D44 build) and calls the pure linkifier. Every surface reaches inline linking only through this one seam:

```go
// LinkifyMentions loads the current SubjectKeys and returns `text` with the
// first eligible occurrence of each subject wrapped as an inline markdown link
// to its absolute front-door URL (base + canonical Path). excludeID's subject is
// never linked (the page's own subject on a subject page; "" on ask, which has
// no own subject). Read-only; no write path.
func (s *Service) LinkifyMentions(ctx context.Context, text, base, excludeID string) (string, error)
```

**Skip-regions (markdown safety).** Because the linkifier injects into a raw, LLM-authored markdown string, a match is **ineligible** when it lies inside any of: a **fenced code block** (``` ``` ``` / `~~~`), an **inline code span** (`` `…` ``), or an **existing link/autolink** (`[…](…)`, `<scheme:…>`, or a bare URL run). Wrapping there would corrupt the markdown or nest a link. The linkifier scans the body once to mark these spans, then only considers runs outside them; a subject whose sole occurrence is inside a skip-region gets **no** inline link (its footer entry is unaffected). Everything else — headings, emphasis, list items, table cells, blockquotes — is eligible.

**Surface composition (where each surface calls it).**
- **MCP `ask`** (D10) — `handleAskCall` runs `LinkifyMentions(ctx, answer.Text, h.pageBase, "")` and returns the linkified markdown as `answer`; the separate `citations` array (D56) is unchanged.
- **MCP `page`** (D10) — the handler linkifies the prose (`LinkifyMentions(ctx, body, h.pageBase, subjectID)`) **then** appends the D12 footer (`RenderFooter(linkified, mentions, mentionedBy)`); the footer's own links are never re-linkified.
- **Web answer / subject** (D44/D45) — the handler linkifies the prose (subject page excludes its own subject) with the **web** absolute base (D59) before the template routes it through `md`.

`base` is identical in shape everywhere — `AuthServer + Mount + "subject/"` (D56/D82/D84 for MCP; D59 derives the same for web) — so `base + Path` is a dumb concatenation on every surface.

**Additive, not a replacement.** The footers (D12 outbound + inbound, D44 outbound, D45 outbound + inbound) are unchanged; inline links are layered into the prose on top, so a subject can appear both inline (at first occurrence) and in a footer. D12's `Mentions`/`RenderFooter` and D44's `MentionsIn` keep their exact behavior and ids.

**Rejected.**
- **Two linkifiers (a goldmark AST transform for the web + a string injector for MCP).** The AST path is more robust for HTML, but MCP must return a markdown *string*, so it needs string injection anyway; forking into two implementations re-creates exactly the drift D12/D44 were built to avoid. One string linkifier, markdown-structure-aware enough to skip code/link regions, feeds both. Rejected.
- **Linking every occurrence, not just the first.** Turns dense prose into a wall of repeated links; the product asks for the *first* occurrence. Rejected.
- **Rendering the link text as the canonical `Name` instead of the surface run.** Breaks the sentence's grammar ("Vasari's fresco" → the canonical "Giorgio Vasari" mid-sentence). The footer already carries the canonical name; inline preserves the author's form. Rejected.
- **Extending D12 `Mentions` to also return offsets.** `Mentions` is a settled pure matcher with existing ids and callers (D44, D46); widening its contract to carry positions burdens every caller. A sibling positional matcher reuses its *rules* without changing its signature. Rejected.
- **Materializing inline links at ingest/compile time.** Same drift argument as D12's read-time projection — a stored link goes stale on rename/new-subject/merge. Read-time is always correct. Rejected.

**Verification.** Pure ids are table-tested directly against `LinkFirstMentions`; the loader gets a real temp SQLite (per the Testing strategy — link projection runs on the real engine).

- R-82BY-EKDH — **First occurrence only, one link per subject.** *(pure, table-tested)* For a body naming subject `W` (path `entity/giorgio-vasari`) three times, `LinkFirstMentions` wraps **only the first** run as `[…](base + "entity/giorgio-vasari")` and leaves the 2nd and 3rd occurrences as plain text — exactly one inline link for `W`.
- R-83JU-SC46 — **Link text is the surface run, not the canonical name.** *(pure, table-tested)* For body `"Vasari's fresco"` where `vasari` is one of `W`'s Keys, the result is `"[Vasari](base + entity/giorgio-vasari)'s fresco"` — the wrapped text is the verbatim surface run `Vasari` (the possessive `'s` stays outside the link), never the canonical `Giorgio Vasari`.
- R-84RR-63UV — **Alias run resolves to the canonical URL.** *(pure, table-tested)* When the matched run is an **alias** surface form (`Vasari`, an alias key of survivor `W`), the injected href is the **canonical** `base + "entity/giorgio-vasari"` — never a `base + entity/vasari` built from the alias surface, and never the folded-away path.
- R-877J-XNC9 — **Skip-regions are not linked and do not consume the first occurrence.** *(pure, table-tested)* A subject name appearing first inside an inline code span (`` `Vasari` ``), a fenced code block, or an existing link (`[Vasari](x)`) is **not** wrapped there; if a later **eligible** run exists it is linked instead, and if the skip-region run is the *only* occurrence the subject gets **no** inline link (the surrounding code/link markdown is left byte-for-byte intact).
- R-88FG-BF2Y — **Deterministic overlap resolution.** *(pure, table-tested)* When two different subjects' earliest eligible runs overlap, the linkifier links the **leftmost-start** run (longest run on an equal start) and skips the overlapping one, which then falls to its next non-overlapping eligible run (or none); the output is identical regardless of the input slice order (no map-iteration dependence).
- R-89NC-P6TN — **Self-exclusion.** *(pure, table-tested)* With `excludeID` set to subject `S`'s id, an occurrence of `S`'s own name in `body` is **never** wrapped, while other subjects in the same body still link; with `excludeID == ""` every subject including `S` is eligible.
- R-8AV9-2YKC — **Whole-run boundary in the positional matcher.** *(pure, table-tested)* The positional matcher links only a whole, hyphen/edge-bounded run: a subject key occurring as a substring inside a larger token (`cat` within `category`) yields **no** inline link, matching D12's whole-run rule (R-ZY11-SUQS) on the new positional code path.
- R-8C35-GQB1 — **`LinkifyMentions` is alias-aware and composes the absolute base, end to end.** *(real temp SQLite)* With subjects loaded and an alias row `vasari → W` installed in a real DB, `LinkifyMentions(ctx, "Vasari painted it.", "https://acct.ikigenba.com/srv/wiki/subject/", "")` returns `"[Vasari](https://acct.ikigenba.com/srv/wiki/subject/entity/giorgio-vasari) painted it."` — proving it consults `AliasStore.ListAll` (a folded short name links canonically) and concatenates the absolute front-door base; with **no** alias row the same text is returned unchanged. *(composes with D25/D26 alias creation.)*
