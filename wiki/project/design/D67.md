# Decision 67 — The improvement loop: ralph-driven prompt tuning over the workbench

**Decision.** Prompt tuning is an **unattended ralph loop** over the D64–D66 workbench: `eval/extract/improve.md` is a committed, single-prompt ralph loop the operator launches from `wiki/` (`ralph eval/extract/improve.md`, improver model chosen at launch via ralph's `--harness`/`-c model=…` — deliberately unpinned here and fully separate from the eval model pinned in `config.json`). Each turn runs in a fresh context; all state lives in the disposable workspace. This decision is **structural**: it fixes the loop's protocol and the committed assets; every checkable behavior it leans on is owned by D64–D66 (config pins, scorer, `Accept`/`Epsilon`, runner exit codes).

The protocol `improve.md` encodes:

- **Workspace** — `tmp/autotune/extract/` (gitignored, D64): `best/prompt.txt` + `best/scorecard.json` (the incumbent), `baseline.json` (the committed prompt's `-repeat 3` scorecard with epsilon), `candidates/NNN-prompt.txt` + scorecards, `history.md` (one line per attempt: candidate, composite, verdict).
- **Bootstrap** — if the workspace is empty: build `bin/eval-extract`, produce `baseline.json` from the committed `eval/extract/prompt.txt` with `-repeat 3`, install that prompt as `best/`.
- **One turn** — read `history.md` and the incumbent's scorecard (especially per-case misses and spurious claims); write **one** revised candidate prompt under `candidates/`; `run` it against the **dev** split; `compare` against best + epsilon; on accept (exit 0) promote it to `best/`, on reject keep the incumbent; append the history line either way.
- **Stop** — after **5 consecutive rejects** (read from `history.md`) or the operator's ralph budget: run `best/prompt.txt` once against the **holdout** split, write `holdout-scorecard.json` and a final summary into the workspace, and declare `DONE`. Holdout runs exactly once, on the final winner only — a winner that beats baseline on dev but not holdout is reported as overfit, not adopted.
- **Hard boundaries, stated in the prompt and enforced outside it** — the loop writes **only** under `tmp/autotune/extract/`. It never edits `eval/extract/` (prompt, config, gold, improve.md itself), `internal/`, or `cmd/`. Enforcement is mechanical, not politeness: everything it could corrupt is committed, so `git status` after a run shows a clean tree, and the winner's score must **reproduce** — the operator re-runs `eval-extract run` on the claimed winner from the clean checkout before adopting; a score that doesn't reproduce (beyond epsilon) rejects the run.
- **Adoption is manual** — copying `best/prompt.txt` over `eval/extract/prompt.txt` is an operator decision made through the normal spec/commit channel (D64), never something the loop does.

`eval/extract/README.md` is the operator guide: building the runner, producing a baseline, launching the loop, reading the workspace, the reproduce-then-adopt step, and how the gold corpus grows (operator-reviewed, D64).

**Rejected.**
- **A hand-rolled driver script calling the improver model via API** — rebuilds ralph's loop/fresh-context/retry machinery for no gain; ralph already runs any harness agentkit supports and takes per-launch model config.
- **Accumulating one long improver context across turns** — each attempt should reason from scorecard evidence, not from its own chat history of failed theories; fresh context per turn is the feature.
- **Letting the loop adopt winners into `eval/extract/prompt.txt`** — collapses the tune/adopt boundary the product promises; adoption stays a deliberate human change.
- **Running holdout every iteration** — quietly turns holdout into a second dev set and destroys its meaning as a generalization check.

**Verification.** Pure structure and protocol — no ids of its own. Its proof is the behavioral ids it composes: the pins and workspace (R-KICZ-E6WA, R-KJKV-RYMZ, R-KKSS-5QDO), the scorer and acceptance arithmetic (D65's ids, especially R-KWZR-ZFSM), and the runner's run/repeat/compare/exit contracts (D66's ids). The committed assets themselves (`improve.md`, `README.md`, the gitignore line) are checked structurally by their build phase.
