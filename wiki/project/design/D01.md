# Decision 1 — Dependency on the prompts service (unified inference; no provider dependency anywhere)

**Decision.** wiki performs **no direct provider inference**. Its inference dependency is the suite's **prompts service**, reached over the loopback plumbing tier: `POST /complete` for chat-shaped one-shots and `POST /embed` for embeddings (research §12 documents the exact API footprint). The production binary (`cmd/wiki`) carries **no** `github.com/ikigenba/agentkit` package in its dependency closure, and — with the in-repo tuning workbench retired (D70) — the **module as a whole** requires no agentkit and no provider SDK: prompt tuning now runs through an external standalone tool over committed data folders (D71), so no Go code in this module calls a provider.

- **Address resolution:** the composition root resolves prompts' base URL with `registry.BaseURL("prompts")` (the in-repo `registry` replace-sibling wiki already depends on) and injects it as a plain string into the D5 client. No env var, no hardcoded port, no second source of truth.
- **Keys stay out of the Go code:** `ANTHROPIC_API_KEY` and `OPENAI_API_KEY` are read by **no Go source in the module**; provider credentials for production inference live with prompts alone. The tune-folder **scorers** (D72) — standalone scripts under `autotune/`, never compiled into anything — read `OPENAI_API_KEY` from the environment (supplied by the repo `.envrc`) for their embedding and judge calls; that is script data-tooling, not the service, and it is the only place the key's name may appear outside tests.
- **Prod build:** `bin/ship` (`GOWORK=off`) resolves the module graph without a workspace; with no agentkit require, a production-shaped build carries no provider code at all.
- **Failure doctrine:** prompts down or erroring ⇒ wiki's call fails loudly to its caller (job error, ask error). No wiki-side fallback provider, no queueing, no transport-level retry (D5).

**Rejected.**
- **Keeping agentkit beside the prompts client in the service** (dual back ends behind D5) — leaves a second inference path and a keyed provider in the running service, defeating the suite decision that prompts is the sole inference service and the sole spend ledger.
- **Keeping agentkit in the module for a dev workbench** (the previous form of this decision) — the workbench that justified the require is retired (D70); tuning moved to the external `autotune` tool precisely so no target repo carries tuning machinery. A require with no importer is dead weight.
- **A wiki env var for prompts' base URL** — the registry library is the suite's authoritative service→port table; an env override would be a second, drift-prone source. Tests inject a `httptest` URL through the constructor instead.
- **Calling prompts through MCP** — `/mcp` is the product surface for agents; sibling daemons use the loopback plumbing endpoints precisely so inference consumers need no MCP handshake on a hot path (prompts' own design).

**Verification.**
- R-KDHD-V3XI — With `GOWORK=off` (production-shaped resolution, no workspace in scope), `go list -deps ./cmd/wiki` succeeds and its output contains no `ikigenba/agentkit` package — the production binary's dependency closure is agentkit-free.
- R-A3D7-NTLV — The strings `ANTHROPIC_API_KEY` and `OPENAI_API_KEY` appear in **no** non-test `.go` source file anywhere in the module — no Go code reads a provider key (the tune-folder scripts under `autotune/` are not Go source and are outside this assertion).
