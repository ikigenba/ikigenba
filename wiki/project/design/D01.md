# Decision 1 — Dependency on the prompts service (unified inference; agentkit confined to the dev workbench)

**Decision.** The wiki **service** performs **no direct provider inference**. Its inference dependency is the suite's **prompts service**, reached over the loopback plumbing tier: `POST /complete` for chat-shaped one-shots and `POST /embed` for embeddings (research §12 documents the exact API footprint). The production binary (`cmd/wiki`) carries **no** `github.com/ikigenba/agentkit` package in its dependency closure and reads **no** provider key.

The module as a whole, however, **does** require agentkit (pinned to a release tag, currently `v0.7.0`): the development-time eval workbench (D64–D67) calls providers directly because it runs locally with no prompts service up. The boundary is the **binary**, not the module: agentkit is a dev-tool dependency, imported only by the workbench packages (D63 owns the source-tree proof), never reachable from `cmd/wiki`.

- **Address resolution:** the composition root resolves prompts' base URL with `registry.BaseURL("prompts")` (the in-repo `registry` replace-sibling wiki already depends on) and injects it as a plain string into the D5 client. No env var, no hardcoded port, no second source of truth.
- **Keys stay out of the service:** `ANTHROPIC_API_KEY` and `OPENAI_API_KEY` are read nowhere in the service — not in `cmd/wiki` nor any package it depends on; provider credentials for production inference live with prompts alone. The **sole** exception in the module is the eval runner's composition root (`cmd/eval-extract`, D66), which reads them from the environment at dev time for its direct provider calls.
- **Prod build:** `bin/ship` (`GOWORK=off`) resolves the module graph without a workspace — the agentkit require is a real tagged version, so a production-shaped build still succeeds; the shipped binary simply never links agentkit.
- **Failure doctrine:** prompts down or erroring ⇒ wiki's call fails loudly to its caller (job error, ask error). No wiki-side fallback provider, no queueing, no transport-level retry (D5).

**Rejected.**
- **Keeping agentkit beside the prompts client in the service** (dual back ends behind D5) — leaves a second inference path and a keyed provider in the running service, defeating the suite decision that prompts is the sole inference service and the sole spend ledger. The workbench does not reopen this: it is not the service, ships nowhere, and touches none of the service's call paths.
- **Banning agentkit from the module entirely** (the previous form of this decision) — forces the dev workbench either out of the module (where it cannot reuse `internal/extract`'s envelope assembly and validation, so its scores silently drift from what production does) or out of existence. A go.mod line is cheaper than a drifting eval.
- **A wiki env var for prompts' base URL** — the registry library is the suite's authoritative service→port table; an env override would be a second, drift-prone source. Tests inject a `httptest` URL through the constructor instead.
- **Calling prompts through MCP** — `/mcp` is the product surface for agents; sibling daemons use the loopback plumbing endpoints precisely so inference consumers need no MCP handshake on a hot path (prompts' own design).

**Verification.**
- R-KDHD-V3XI — With `GOWORK=off` (production-shaped resolution, no workspace in scope), `go list -deps ./cmd/wiki` succeeds and its output contains no `ikigenba/agentkit` package — the production binary's dependency closure is agentkit-free even though the module requires it.
- R-KEPA-8VO7 — The strings `ANTHROPIC_API_KEY` and `OPENAI_API_KEY` appear in no non-test source file outside `cmd/eval-extract/` — the service reads no provider key; only the eval runner's composition root does.
