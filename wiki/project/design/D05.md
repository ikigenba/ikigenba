# Decision 5 — The LLM seam (`internal/llm`): a prompts `/complete` client + json-mode helper

**Decision.** All chat inference goes through one seam, `internal/llm`, which is now a thin HTTP client for the prompts service's loopback `POST /complete` (D1; API footprint in research §12). Structured JSON stays wiki's responsibility — prompts executes and accounts; wiki owns prompt text, parsing, validation, and corrective retries — so the `JSON[T]` helper, `ExtractJSON` carve, and per-stage `CallSite` shape survive the transport swap. No agentkit type appears anywhere in the package.

```go
package llm

// Client posts completions to the prompts service. Built once at the
// composition root from registry.BaseURL("prompts"); concurrency-safe.
type Client struct {
    baseURL string
    http    *http.Client   // no fixed timeout — calls carry only their ctx deadline (D1 doctrine)
}
func New(baseURL string) *Client

// Config is /complete's generation-config vocabulary, carried VERBATIM —
// wiki invents no vocabulary of its own; validation is prompts' 400 (fail loud).
type Config struct {
    Model       string   `json:"model"`
    Provider    string   `json:"provider,omitempty"`
    Temperature *float64 `json:"temperature,omitempty"`
    MaxTokens   int      `json:"max_tokens,omitempty"`
    Effort      string   `json:"effort,omitempty"`   // e.g. "low" (ask sites)
    Thinking    *bool    `json:"thinking,omitempty"` // false = reasoning explicitly off (extract/compile)
}

// CallSite is the per-stage call config (D19 owns the values and env overrides).
type CallSite struct {
    Stage           string // completes the D62 name as "wiki.<stage>": "extract"|"compile"|"ask-subject"|"ask-synthesis"
    System          string
    Config          Config
    MaxParseRetries int
}

// Attribution is the D62 accounting envelope, threaded per call chain.
type Attribution struct {
    Origin  string // D62 rule: "user:<email>" | "trigger:<source>" | "service:wiki"
    GroupID string // job id for pipeline chains; the per-ask correlation id for asks
}

// JSON runs one structured generation via POST /complete and returns a validated T.
// Request: {origin, name: "wiki."+site.Stage, group_id, attempt, model, provider,
// config, system, messages}. messages is the full stateless replay: the first
// attempt sends [{user: userText}]; a corrective retry re-sends the original user
// turn, the model's bad reply as an assistant turn, and the corrective user turn,
// with attempt incremented — each retry is its own /complete call on the record.
// Flow: 200 → response text → ExtractJSON → json.Unmarshal → validate; on
// unmarshal/validate failure re-prompt up to MaxParseRetries then error (never a
// silent zero T). A 400 is a configuration/validation bug: returned immediately
// as an error, never retried. A 502/5xx or transport error is returned
// immediately (fail loud; no transport retry — retry policy lives in prompts'
// config vocabulary if a site ever wants it). Truncation (usage.Output at the
// MaxTokens ceiling, D18) returns ErrTruncated immediately WITHOUT re-prompting.
func JSON[T any](ctx context.Context, c *Client, site CallSite, attr Attribution, userText string, validate func(*T) error) (T, error)
```

- **`ExtractJSON`** (unchanged) carves the outermost JSON value out of a possibly-decorated reply: trim, pick whichever of the first `{` or first `[` opens earliest, return the span to the matching last `}`/`]`. Fences, prose preambles, trailing commentary, stray backticks all fall outside the span. No-op on bare JSON. Still the **only** JSON-from-a-decorated-reply implementation; `ask` (D9) parses its final answers through it.
- **Stateless by construction** — every call posts a complete `messages` array; nothing is held between calls, so no history can bleed between stages (the property the old fresh-Conversation rule guaranteed is now structural).
- **`Converse` is retired** — its tool-bearing path had no production caller; `/complete` is tool-less and `JSON` is the seam's whole chat surface (no orphaned surface, D63).
- **`validate`** stays caller-supplied (extract D6 / compile D7 / ask D9), so retry loops and domain rules stay co-located without `internal/llm` knowing the domain.
- **Usage on the response** (`usage`, `cost_usd`) is read for the D18 truncation check and otherwise ignored: accounting is prompts' job (its `calls` table), and wiki records nothing (D63).
- **Testing:** the falsifying substrate is a `httptest.Server` playing prompts — scripting 200/400/502, canned bodies, capturing request JSON. No provider mock, no network.

**Rejected.**
- **A forced-tool / server-side structured-output contract** — prompts deliberately leaves parsing and corrective retries with the caller (its D29 rejection); wiki's validators are domain logic coupled to wiki's types.
- **Sending only the corrective turn on retries** — `/complete` is stateless; the replay must carry the full user/assistant/user history or the model loses the context of what it got wrong.
- **Retrying transport errors in wiki** — fail-loud doctrine (D1); a flaky prompts must surface, not be papered over.
- **Keeping the agentkit Conversation back end beside the HTTP client** — rejected in D1.

(Regression guard for the fenced-extraction ingest failure.)
- R-J9YL-P6K0 — `JSON` returns the validated `T` for a well-formed (including fenced) response, with the caller's `validate` applied.
- R-JCEE-GQ1E — `JSON` retries on unmarshal/validate failure up to `MaxParseRetries` (bad-then-good succeeds) and otherwise returns an error rather than a silent zero `T` (always-bad errors).
- R-0X4N-U0XB — The `/complete` request JSON faithfully carries the call: `name` is `wiki.<stage>`, `model`/`provider`/`temperature`/`max_tokens`/`effort`/`thinking` come verbatim from the `CallSite`'s Config, `system` from the site, and the final message role is `user` (captured via the httptest prompts).
- R-0ZKG-LKEP — A corrective retry posts a **new** `/complete` call whose `messages` replay the original user turn, the bad assistant reply, and the corrective user turn, with `attempt` incremented to 2; the first attempt carried `attempt` 1 and a single user turn (statelessness + on-the-record retries).
- R-10SC-ZC5E — A `400` from prompts returns an error naming prompts' `{"error":…}` body immediately, consumes **no** parse retry, and makes no further HTTP call (a config bug fails loudly, never silently degrades).
- R-1209-D3W3 — A `502` (provider failure) or a transport error (connection refused) returns an error immediately with no wiki-side retry; the error reaches `JSON`'s caller.
