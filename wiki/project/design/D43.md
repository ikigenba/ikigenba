# Decision 43 — The home / reset page (`GET /{$}`, no `q`)

**Decision.** The state served at the bare mount root with no query — and the target every "ask another question" control returns to (D44/D45 link to `.`, i.e. the base). It is rendered by `home.tmpl` filling the D42 layout, and reads **only** the orphan list (D46); it touches no `Asker`, no `PageFinder`, no identity (nginx already gated the request). The page is otherwise a pure function of service/version/mount.

This is the page D39 mounts at `GET /{$}`; D39's gating, embedded-Carbon, exact-root, ungated-in-process, and name+version behaviors (`R-LAND-*`) continue to hold for it — the name+version now lives in the **footer**, not a hero.

**Three parts, top to bottom:**

```html
<!-- search form: GET to the base, single text input named q.
     Submitting navigates to /srv/wiki/?q=<question> → the ask result (D44). -->
<form action="" method="get" role="search">
  <input type="text" name="q" aria-label="Ask the wiki" autofocus>
  <button type="submit">Ask</button>
</form>

<!-- orphan index: the OrphanLister Refs (D46), each a subject link.
     OMITTED ENTIRELY when the list is empty. -->
{{if .Orphans}}
<nav aria-label="Unreferenced subjects">
  <ul>{{range .Orphans}}<li><a href="{{.Href}}">{{.Name}}</a></li>{{end}}</ul>
</nav>
{{end}}

<!-- footer: service name + version (mono label) — preserves D39's promise. -->
```

- The **search form** is `method="get"` with `action=""`, so it resolves against `<base>` (the mount root) and submits as `/srv/wiki/?q=…`. The input is named **`q`** — the exact key D42's root dispatch (R-WFPZ-2LTM) splits on. No JS.
- The **orphan index** lists the injected `OrphanLister.Orphans(ctx)` result (D46) — subjects with zero inbound mentions, the entry points pure link-following cannot reach. Each `Ref` renders `<a href="{{.Href}}">{{.Name}}</a>` where `Href` is already the mount-relative `subject/<type>/<slug>` tail (D42). The list is **omitted whole** (no `<nav>`, no header) when empty — an empty wiki, or one where every subject is referenced, shows just the search box and footer. No empty-state copy.
- The **footer** carries the service name + version.

**Empty-wiki fall-through.** With nothing ingested, `Orphans` is empty → home is search box + footer. Submitting any question routes to D44, which returns the honest "nothing here" (D9). No special empty-state handling is needed anywhere.

**Data dependency.** The home handler calls `OrphanLister.Orphans(ctx)` and nothing else. The orphan computation's cost/correctness is D46's concern; here it is a single seam call whose result is rendered or omitted.

**Rejected.**
- **An explicit empty-state message** ("nothing ingested yet" / "everything is connected") when the orphan list is empty. The omitted-section behavior is unambiguous and avoids copy that would need to distinguish "empty wiki" from "fully-referenced wiki" — two states the home page has no reason to tell apart. Rejected per the product call.
- **Listing all subjects on the home page** (not just orphans). The reachable subjects are reachable by following links; an all-subjects index would duplicate navigation and bury the unreachable ones that actually need an entry point. Orphans-only is the product decision. Rejected.
- **A JS-driven search box (XHR/fetch the answer).** The surface is plain GET navigation (back button, bookmarkable, no-JS), consistent with Carbon's no-JS-required posture. Rejected.

**Verification.** httptest against the D42 handler with a recording/stub `OrphanLister`; no DB/LLM/identity.

- R-OMRY-L9O8 — **Search form submits the query as `?q=` via GET.** The home body (`GET /`, no `q`) contains a `<form method="get">` whose action resolves to the base and whose single text input is named exactly `q` — so a submitted question navigates to the root with `?q=<question>` (the key D42 R-WFPZ-2LTM dispatches on). A form using `method="post"` or an input named anything but `q` fails this.
- R-ONZU-Z1EX — **Orphan list renders the seam's Refs as subject links.** Given an `OrphanLister` stub returning two Refs (e.g. `{Href:"subject/entity/acme-corp", Name:"Acme Corp"}`, `{Href:"subject/concept/widget", Name:"Widget"}`), the home page renders both as `<a href="subject/entity/acme-corp">Acme Corp</a>` / `<a href="subject/concept/widget">Widget</a>` in the returned order — proving the home page calls `Orphans` and emits each as a mount-relative subject link.
- R-OP7R-CT5M — **Empty orphan list omits the section.** With an `OrphanLister` stub returning an empty slice, the home page renders the search form **and** contains **no** orphan `<nav>`/list markup (no header, no `<ul>`) — the section is omitted whole, not rendered empty.
