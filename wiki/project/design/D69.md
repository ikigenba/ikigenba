# Decision 69 — The analysis eval workbench: tuning the ask question-analysis prompt

**Decision.** The autotune workbench (D64–D68) gains its second step: **`analysis`**, the ask pipeline's question-analysis call (`internal/ask.Analyze` — question in, `wiki.QueryAnalysis` out: `sub_queries` capped at four, `keywords`, `aliases`). The extract workbench's shape transfers wholesale; this decision owns only what is genuinely new — the analysis prompt as data, the analysis gold corpus and its scorer, the sibling runner, and the driver's second step. Where a mechanism is unnamed here, D64–D68's statement of it governs with `extract` replaced by `analysis`.

```
eval/analysis/
  prompt.txt      the adopted analysis prompt — the one production runs
  prompt.go       package analysisprompt: //go:embed prompt.txt → var Instructions string
  config.json     the pinned reference config (below)
  improve.md      the ralph tuning-loop prompt (D67 protocol, analysis paths)
  README.md       operator guide (D67 shape, analysis specifics)
  gold/
    dev/<case>/question.txt + gold.json
    holdout/<case>/question.txt + gold.json
```

- **Prompt as data.** `eval/analysis/prompt.go` (package `analysisprompt`) embeds `prompt.txt`; `internal/ask` exports `var DefaultAnalysisInstructions = analysisprompt.Instructions` and two production-faithful seams mirroring D66's: `ask.RenderAnalysis(instructions, question string) string` — the exact prompt assembly `Analyze` sends (instructions + the trimmed question) — and `ask.NormalizeAnalysis(*wiki.QueryAnalysis)` — the exact production normalization (trim, case-insensitive dedupe, sub_queries capped at 4). The initial `prompt.txt` content is today's `analysisPrompt` instruction text, moved verbatim; production behavior is unchanged by the move.
- **`config.json` — the reference.** Same three-block shape as D64's, loaded by the same fail-loudly discipline. The `eval` block mirrors the production ask-subject call site (D19: model `claude-sonnet-4-6`, max_tokens 16384) with `max_parse_retries: 2`; production's reasoning-effort knob (`low`) has no bench vocabulary (D68's accepted keys carry `thinking`, not effort levels), so the reference block simply omits it — the divergence is noted in `README.md`, and per D68 nothing is inherited from this file at tuning time anyway. The `embedding` block is byte-identical in meaning to extract's (`text-embedding-3-small`, 1536, threshold 0.80, margin 0.03 — the same pinned scoring function, still never operator-variable). `weights` names the three lists: `{"sub_queries": 0.50, "keywords": 0.30, "aliases": 0.20}`, summing to 1 — sub_queries weighted highest because retrieval quality rides on them.
- **The gold corpus — one universe.** One case = `question.txt` (the operator-voice question fed to `Analyze`) + `gold.json`:

```json
{"difficulty": "easy|medium|hard",
 "gold": {"sub_queries": ["..."], "keywords": ["..."], "aliases": ["..."]}}
```

  Gold questions are authored **against the extract gold corpus's fictional universe** (D64) — the documents are the shared fixture, so every question has a verifiable right decomposition grounded in known text. The split follows the document split: a `dev/` question draws only on `eval/extract/gold/dev/` content, a `holdout/` question only on `eval/extract/gold/holdout/` content — no dev question leaks holdout material. Like D64: the build loop seeds **one dev and one holdout case** (drafted from named extract-corpus documents, operator-reviewed after landing); growth toward a working corpus (~15 dev / ~8 holdout) happens by the operator-review channel, outside any build phase. A case never moves from holdout to dev.
- **The scorer — the D65 machinery over string lists.** `internal/eval` gains `AnalysisConfig`/`LoadAnalysisConfig`, `AnalysisGoldCase`/`LoadAnalysisGold`, and `ScoreAnalysisCase(ctx, gold, got wiki.QueryAnalysis, embed EmbedFunc, cfg) (AnalysisCaseScore, error)`. Each of the three lists is scored independently by **exactly D65's claim-alignment rules** — greedy best-first embedding pairing, threshold, margin over best alternative, digit-token agreement, one match per string — yielding per-list precision/recall/F1; `composite = w_subq·F1_subq + w_kw·F1_kw + w_alias·F1_alias`. D65's honest-empty carve-out applies **per list**: gold list empty and output list empty ⇒ that list scores 1.0; empty gold with any output ⇒ 0. Everything else is shared, not reimplemented: the `EmbedFunc` seam, the disk cache, `Epsilon`, `Accept`, deterministic marshal (sorted keys, fixed precision, no timestamps, config + prompt-SHA echoed).
- **The runner — `cmd/eval-analysis`, a sibling not a fork.** Same CLI surface as D66 verbatim (`run`/`compare`, same flags, same split/repeat semantics, same stderr progress contract, same atomic-write failure doctrine, same exit codes). The agentkit plumbing D66 built — provider construction over the five providers, `EVAL_` alias precedence, key/sub auth, the retry-on-invalid loop — is **factored into `internal/eval`** (already inside D63's agentkit confinement) and consumed by both runners; `cmd/eval-extract` keeps its behavior bit-for-bit (its D66 ids continue to pass unchanged). The analysis call path uses `ask.RenderAnalysis` + JSON-parse into `wiki.QueryAnalysis` + `ask.NormalizeAnalysis`; a response that fails to parse re-prompts up to `max_parse_retries`. Its live smoke (build tag `eval_live`, mirroring D66's) performs one real chat analysis of a one-sentence question and one real embedding call.
- **The driver — step two of N.** `cmd/autotune`'s supported-step set becomes `{extract, analysis}` (D68's dispatch bullet, rewritten there); the fresh-run runner build target is per-step (`go build -o tmp/autotune/<step>/bin/eval-<step> ./cmd/eval-<step>`). Everything else in D68 — free-form `-c` config, workspace lifecycle, resume stamp, alias injection, watcher, finalizer — is already step-generic and applies as written. The D67 loop protocol applies with analysis paths; `eval/analysis/improve.md` encodes it, differing from extract's only in paths and in describing the analysis artifact (three lists, not subjects/claims).
- **Ordering on record.** The tuning-workbench roadmap is analysis → ask-synthesis → compile; this decision builds analysis only. Synthesis and compile get their own decisions when their turn comes (synthesis needs a scorer design for cited prose; compile needs page-level gold).

**Rejected.**
- **Exact-match scoring for keywords/aliases (hybrid scorer)** — two scoring mechanisms to maintain, and exact match punishes legitimate variation (singular/plural, reordering) exactly as D65 rejected for claims; one embedding mechanism with the digit guard covers all three lists.
- **A fresh fictional universe for analysis gold** — a second universe doubles the fixture-authoring burden and forfeits the cross-step property that a question's right answer is checkable against a committed document; the shared universe also feeds the later synthesis and compile workbenches.
- **Generalizing `cmd/eval-extract` into one multi-step binary** — a step flag that switches gold format, call path, and scorer inside one binary couples the steps' release cadence and makes D66's extract ids assert against a moving target; two thin composition roots over one shared library is the same reuse without the coupling.
- **Scoring analysis by downstream retrieval hit-rate** — the honest end-to-end metric, but it needs an indexed corpus and makes the score depend on retrieval internals that move independently; list-vs-gold keeps the workbench's determinism contract. Revisit if tuned prompts stop translating into better retrieval.

**Verification.**
- R-BICU-BZG6 — `ask.DefaultAnalysisInstructions` is byte-identical to the contents of `eval/analysis/prompt.txt` (the test reads the file and compares), so editing the file is editing the production analysis prompt.
- R-BJKQ-PR6V — Envelope identity: the prompt the production `Analyze` path sends (captured via a fake D5 client) is byte-identical to `ask.RenderAnalysis(DefaultAnalysisInstructions, question)` for the same question, and production's normalization of a raw `QueryAnalysis` (whitespace, duplicate case-variants, >4 sub_queries) equals `ask.NormalizeAnalysis` applied to the same value.
- R-BKSN-3IXK — `LoadAnalysisConfig` parses `eval/analysis/config.json` with every pinned value intact (model `claude-sonnet-4-6`, max_tokens 16384, max_parse_retries 2; the embedding pin with threshold 0.80 and margin 0.03; weights sub_queries 0.50 / keywords 0.30 / aliases 0.20 summing to 1); a config missing a required field or with weights not summing to 1 fails loudly naming the field.
- R-BM0J-HAO9 — `LoadAnalysisGold` walks a corpus root and returns every case under `dev/` and `holdout/`; a case directory missing `question.txt`, or whose `gold.json` is malformed (bad JSON, missing `gold` object), fails loudly naming the case.
- R-BN8F-V2EY — List alignment accepts a pairing that clears the threshold, clears the margin over its best alternative, and has agreeing digit tokens — verified per list with an injected fake `EmbedFunc` returning constructed vectors.
- R-BOGC-8U5N — List alignment rejects each way independently: below threshold; within margin of a competing alternative; non-empty disjoint digit-token sets despite high similarity — each yields no match for the affected strings.
- R-BPO8-MLWC — One-to-one and rollup: with one output string similar to two gold strings, it matches only the better one and the other is a miss; on a hand-computed fixture the per-list F1s and `composite = 0.50·F1_subq + 0.30·F1_kw + 0.20·F1_alias` come out exactly.
- R-BQW5-0DN1 — Honest empty per list: an empty gold list with an empty output list scores that list 1.0; an empty gold list with any output string scores it 0; a case whose three gold lists are all empty and whose output is all empty earns composite 1.0.
- R-BTBX-RX4F — `eval-analysis run` end-to-end against a fake chat provider and fake `EmbedFunc`: given a prompt file, a gold dir, and the pinned config, it scores every dev case, writes a scorecard whose numbers match `ScoreAnalysisCase` for the same inputs, and exits 0; a response that fails to parse once then succeeds is re-prompted and the case still scores.
- R-BUJU-5OV4 — Failure exits: an unreadable prompt file, a malformed gold case, and a chat call that fails past its retries each exit non-zero naming the cause, and no scorecard file is left at `-out` (atomic temp-file + rename).
- R-BVRQ-JGLT — Progress: a `run` over a multi-case corpus (fake providers) writes one per-call progress line to stderr for every case×repeat, and neither stdout nor the `-out` scorecard contains progress text.
- R-BWZM-X8CI — Shared auth in the sibling: for `eval-analysis`, `EVAL_OPENAI_API_KEY` wins over `OPENAI_API_KEY` for chat (provider `openai`, `auth=key`) and embedding construction, a missing key failure names both variable names, `auth=sub` with provider `openai` and a valid auth-file fixture builds the subscription provider, and `auth=sub` with a non-openai provider fails naming the provider — all before any call.
- R-BY7J-B037 — Live smoke (`-tags eval_live`, real keys): one real agentkit chat call returns an analysis for a one-sentence question that parses into a `QueryAnalysis` with ≥1 sub_query after normalization, and one real embedding call returns a non-empty vector of the configured dimensions.
- R-BZFF-ORTW — Driver step two: `autotune analysis` (scripted executor) provisions `tmp/autotune/analysis/`, seeds `autotune/analysis/prompt.txt` from `eval/analysis/prompt.txt`, and invokes the runner build as `go build -o tmp/autotune/analysis/bin/eval-analysis ./cmd/eval-analysis` followed by a baseline `run` against the analysis workspace config — while `autotune extract` still builds `./cmd/eval-extract`.
