# Decision 64 — The eval workspace: the extract prompt as data, the pinned config, and the gold corpus

**Decision.** The extract prompt and everything needed to measure it live as **data under `eval/extract/`**, committed with the code. The running service compiles the prompt in from that file, so "the prompt in the code" and "the standalone prompt file" are one artifact; the improvement loop (D67) only ever writes *candidates* elsewhere, and adopting a winner is a deliberate, manual change to this committed file.

```
eval/extract/
  prompt.txt      the adopted extract prompt — the one production runs
  prompt.go       package extractprompt: //go:embed prompt.txt → var Instructions string
  config.json     the pinned eval configuration (below)
  improve.md      the ralph tuning-loop prompt (D67)
  README.md       operator guide: running the eval, the loop, adoption (D67)
  gold/
    dev/<case>/document.txt + gold.json       what the loop tunes against
    holdout/<case>/document.txt + gold.json   final-winner confirmation only (D67)
```

- **Prompt as data.** `eval/extract/prompt.go` is a tiny package (`extractprompt`) that embeds `prompt.txt` and exports it as `Instructions`. `internal/extract`'s exported `DefaultPromptInstructions` becomes a `var` initialized from `extractprompt.Instructions` — same exported name, same production call path (D6's prompt content rules are unchanged; this decision owns only where the text lives). Go's embed rule (files must sit in the embedding package's directory) is why the package sits beside the file.
- **`config.json` — the pins.** One typed, load-fails-loudly config (loader in `internal/eval`, D65):

```json
{
  "eval":      {"provider": "anthropic", "model": "claude-sonnet-4-6", "temperature": 0,
                "thinking": false, "max_tokens": 16384, "max_parse_retries": 2},
  "embedding": {"provider": "openai", "model": "text-embedding-3-small", "dimensions": 1536,
                "threshold": 0.80, "margin": 0.03},
  "weights":   {"subject": 0.35, "claim": 0.50, "field": 0.15}
}
```

  The `eval` block mirrors the production extract call site (D6/D19) — the whole point is measuring the prompt under the call production makes, and a run with no overrides measures exactly these pins. `threshold`/`margin` are the claim-alignment knobs (D65), calibrated empirically and updated by committing a config change; `weights` must sum to 1. The `eval` block additionally accepts optional `auth` (`key`, the default when absent, or `sub`) and `auth_file` fields (D66); the committed file states neither. Only the `eval` block is ever overridable — the runner (D66) via flags for experiments, the driver (D68) via `-c` into a resolved workspace config; `embedding` and `weights` are never overridden, because changing them silently redefines the score.
- **The gold corpus.** One case = one directory: `document.txt` (the source text fed to extract) + `gold.json` (the hand-approved truth). Gold shape extends the extraction shape with per-subject aliases:

```json
{"difficulty": "easy|medium|hard",
 "header": {"source": "...", "title": "...", "tags": ["..."], "received_at": "RFC3339"},
 "gold": [{"type": "entity|event|concept", "kind": "...", "name": "...",
           "aliases": ["..."], "occurred_at": "", "claims": ["..."]}]}
```

  The corpus is **operator-reviewed data**: cases are drafted interactively (agent drafts, operator corrects and approves) and committed; the build loop only seeds the initial cases (recovered from the retired harness's gold, `git show 7b99268c^:wiki/testdata/eval/extract/...`) — one into `dev/`, one into `holdout/`. Growth toward the working corpus (~15 dev / ~8 holdout) happens by the same review channel, outside any build phase. The dev/holdout split is fixed at authoring time; a case never moves from holdout to dev.
- **The disposable workspaces.** Everything a tuning run writes lives in two untracked areas (relative to `wiki/`, where the driver runs): `tmp/autotune/<step>/` holds the run **evidence** — resolved config, baseline, candidates, scorecards, history, holdout scorecard, summary, the embedding cache, the built runner binary — and `autotune/<step>/prompt.txt` is the **visible incumbent copy**: seeded from the committed prompt at run start, overwritten on every accepted candidate, and the file the driver diffs against `eval/<step>/prompt.txt` (D68). `wiki/.gitignore` gains `/tmp/` and `/autotune/`. Nothing under either is ever committed.

**Rejected.**
- **Keeping the prompt a Go string constant with a parallel eval copy** — two statements of the truth that drift; the embed makes them one artifact while keeping the binary static.
- **Loading the prompt from disk at runtime** — the service is one static binary with no runtime file dependencies; embed preserves that.
- **Gold under `testdata/`** — `testdata` is for incidental test fixtures; the corpus is a curated, operator-reviewed asset the whole workbench depends on, and it lives beside the prompt it measures.
- **Auto-generating gold in a build phase** — unreviewed gold encodes a model's extraction opinions as truth; the corpus is only as good as its review.

**Verification.**
- R-KICZ-E6WA — `extract.DefaultPromptInstructions` is byte-identical to the contents of `eval/extract/prompt.txt` (the test reads the file and compares), so editing the file is editing the production prompt.
- R-KJKV-RYMZ — The `internal/eval` config loader parses `eval/extract/config.json` into the typed config with every pinned value above intact (model `claude-sonnet-4-6`, temperature 0, thinking off, max_tokens 16384; embedding `text-embedding-3-small` with its dimensions, threshold, and margin; weights summing to 1); a config missing a required field or with weights not summing to 1 fails loudly with an error naming the field.
- R-KKSS-5QDO — The gold loader walks a corpus root and returns every case under `dev/` and `holdout/` (the seeded cases load successfully); a case directory missing `document.txt`, or whose `gold.json` is malformed (bad JSON, unknown type, empty name, subject with zero claims), fails loudly with an error naming the case.
