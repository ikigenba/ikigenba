# Decision 18 â€” Output-token budget & honest truncation handling

**Decision.** Every call site carries an explicit, generous output budget, the seam **detects truncation** and surfaces a **distinct reason**, and truncation is **terminal for the call** so no retry re-truncates. (History: with no budget, generations ran at a provider-adapter default of 4096; a large extract overran it, the reply stopped mid-JSON, `ExtractJSON` (D5) carved an incomplete value, and the job died as `unexpected end of JSON input` after burning every re-prompt identically.) Extends the D5 seam and the D6/D7 call-site defaults; the atomic no-partial-write failure path is D4 (R-MB7F).

## (a) `MaxTokens` on the call site, faithfully applied

The ceiling lives on the D5 `Config` (`max_tokens`), carried verbatim in the `/complete` request, so the configured ceiling reaches prompts (and through it the provider) instead of any silent adapter default. A production JSON call site **never leaves `MaxTokens` at 0**; the default call sites below set it explicitly.

## (b) Truncation detected at the seam

`/complete` returns the call's aggregate usage; detection lives in the D5 client after a `200`. When the response's reported **output-token usage meets or exceeds the site's `MaxTokens`** (and `MaxTokens > 0`), the response is a **truncation**, not a normal reply: `JSON` returns the distinct sentinel `llm.ErrTruncated` (wrapping stage/usage) instead of the truncated text. (The attempt itself is on prompts' record like every call â€” wiki records nothing, D63.) When the response carries **no** usage, detection cannot fire and the call falls through to the existing parse/retry path unchanged.

## (c) Truncation is terminal â€” no futile re-prompt

`JSON`'s retry loop treats `ErrTruncated` as **non-retriable**: returned immediately without consuming a `MaxParseRetries` iteration, because a re-prompt under the same ceiling truncates identically. (A normal unmarshal/validate failure still retries, D5 R-JCEE.) The error propagates to integrate (D14), whose failure path ends the job `failed` with this reason and writes **zero** subjects/claims/pages (D4 R-MB7F).

## (d) The budgets

Each stage owns its ceiling in its `DefaultCallSite` (D6 extract, D7 compile; the ask sites per D19), generously above realistic output and tunable via manifest knobs:

- **extract** â€” sized well above the largest realistic per-document extraction, e.g. **16384**. Contract is "generously above realistic extract output," not the exact integer.
- **compile** â€” sized above the 12,000-char page cap (~3â€“4 K tokens) plus structural overhead, e.g. **16384**.
- **ask** â€” both ask call sites cap `MaxTokens` at **16384**, uniform with the other stages.

These are headroom, not a guarantee any document fits: a pathologically large source can still exceed the budget, and when it does it fails **honestly** (b/c). Chunking oversized sources remains a deferred follow-on.

**Rejected.**
- **Just raising the budget without detecting truncation.** A bigger ceiling pushes the failure out but never removes it. Detection + a clear reason is the durable fix.
- **Inferring truncation from the parse error text** (`unexpected end of JSON input`). Brittle â€” couples wiki to the encoding/json wording and confuses a malformed-but-complete reply with a truncated one. Usage-at-ceiling is precise and provider-grounded.
- **Retrying a truncation with a larger budget automatically.** Doubles cost and latency and still has a ceiling; failing fast with a clear reason is simpler. (Deferred with chunking.)

**Verification.**
- R-MSKH-GPX5 â€” `JSON` carries the call site's `MaxTokens` verbatim as the `/complete` request's `max_tokens` (captured via the httptest prompts), instead of omitting it for a downstream default.
- R-MTSD-UHNU â€” When the `/complete` response reports output usage at/above the call site's `MaxTokens` with a truncated body, `JSON` returns a distinct truncation error (not a generic JSON-parse error).
- R-MV0A-89EJ â€” A truncation is terminal for the call: `JSON` returns immediately without re-prompting, even with `MaxParseRetries > 0` (exactly one HTTP call is made), distinguishing it from a retriable unmarshal/validate failure.
- R-MW86-M158 â€” `extract.DefaultCallSite` and `compile.DefaultCallSite` each carry `MaxTokens >= 16384` (a test that passes at 4096 does not cover this), and the extractor/compiler the composition root builds are constructed from them, so the production stages run at that budget.
