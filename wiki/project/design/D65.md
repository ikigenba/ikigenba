# Decision 65 — The deterministic scorer (`internal/eval`): rules for subjects and fields, pinned embeddings for claims

**Decision.** `internal/eval` scores one extraction against one gold case and rolls case scores into a scorecard, **deterministically**: same extraction + same gold + same cached vectors ⇒ the same bytes out. No LLM judges anything, anywhere in scoring. The only model in the path is the pinned embedding model (D64 config), used strictly as a frozen text→vector function.

```go
package eval

type Config struct{ Eval EvalCall; Embedding Embedding; Weights Weights }
func LoadConfig(path string) (Config, error)          // R-KJKV-RYMZ (D64)

type GoldCase struct{ Name, Difficulty string; Header extract.DocumentHeader; Document string; Gold []GoldSubject }
type GoldSubject struct{ Type, Kind, Name string; Aliases []string; OccurredAt string; Claims []string }
func LoadGold(root string) (dev, holdout []GoldCase, err error)   // R-KKSS-5QDO (D64)

// EmbedFunc is the seam to the embedding provider; the cache wraps it, tests fake it.
type EmbedFunc func(ctx context.Context, texts []string) ([][]float32, error)
func NewDiskCache(dir, model string, next EmbedFunc) EmbedFunc

func ScoreCase(ctx context.Context, gold GoldCase, got []extract.ExtractedSubject, embed EmbedFunc, cfg Config) (CaseScore, error)
func Aggregate(cases []CaseScore, cfg Config) Scorecard
func (s Scorecard) MarshalDeterministic() ([]byte, error)

func Epsilon(composites []float64) float64            // max − min over baseline runs
func Accept(candidate, best, epsilon float64) bool    // candidate > best + epsilon, strictly
```

- **Step 1 — subject pairing (pure rules).** An extracted subject matches a gold subject when their `Type` is equal and the extracted `Name` normalizes equal to the gold `Name` or any gold `Alias`. Normalization: Unicode-lowercase, trim, collapse internal whitespace, strip punctuation. Each subject pairs at most once. Unmatched gold subjects are **misses**; unmatched extracted subjects are **spurious**.
- **Step 2 — field checks (pure rules).** On each matched pair, `Kind` (normalized equality) and `OccurredAt` (exact string equality; empty-vs-empty counts as correct) are each right or wrong. Field accuracy = correct fields / (2 × matched pairs); 0 when nothing matched.
- **Step 3 — claim alignment (pinned embeddings), scoped within each matched pair.** All gold claims and extracted claims of the pair are embedded (via the `EmbedFunc`, one deterministic vector per sentence). Cosine similarity is computed for every gold×extracted combination; candidate pairs are taken greedily best-first, each claim usable once. A pairing counts as a **match** only if all three hold:
  1. similarity ≥ `threshold`;
  2. it exceeds each participant's best *alternative* pairing by ≥ `margin` (near-duplicate claims about the same subject must not cross-match);
  3. the two sentences' **digit-token sets are not disjoint when both are non-empty** (digit tokens = maximal runs of `0-9`; "$40 million, March 2025" ⇒ {40, 2025}) — a claim about $40 million never matches one about 1998, while a pair where either side carries no digits is unconstrained by this rule.
  Unmatched gold claims are misses; unmatched extracted claims are spurious. Claims of unmatched subjects are all misses/spurious respectively.
- **Step 4 — rollup (arithmetic).** Per case: subject precision/recall/F1 (from matched/spurious/missed subjects), claim precision/recall/F1 (from step 3 across all pairs), field accuracy, and `composite = w_subject·F1_subj + w_claim·F1_claim + w_field·fieldAcc`. The scorecard aggregates: per-case rows plus the mean composite over cases (equal case weight) — the number the loop compares. F1 with an empty denominator is 0 — **except honest agreement on emptiness**: when a component's gold side and extracted side are *both* empty (no gold subjects and no extracted subjects; a matched pair set with no claims on either side; zero field checks because zero matched pairs on an empty-gold case), that component scores **1.0**, so a case whose gold is empty and whose extraction is empty earns composite 1.0, while an empty-gold case with any extracted subject still earns 0 subject-F1 (all spurious). Without this carve-out an "honest empty" case could never score above 0 and the corpus could not reward declining to invent.
- **Determinism mechanics.** Iteration orders are explicitly sorted (cases by name, subjects and claims by input order, candidate pairs by (similarity desc, gold index, extracted index)); `MarshalDeterministic` emits sorted keys and fixed-precision floats; the scorecard carries **no timestamps**. It records the config it ran under (eval call config, embedding pin, threshold/margin/weights) and the SHA-256 of the prompt text that produced the extraction.
- **The embedding cache.** `NewDiskCache` wraps an `EmbedFunc` with an on-disk cache keyed by `(model, SHA-256(text))`, one vector per file under the given dir (the loop points it at `tmp/`, D64). Gold-claim vectors are computed once per corpus, ever; a fully-cached scoring run makes zero provider calls, which is also what makes historical scores immune to provider-side model drift.
- **Acceptance arithmetic lives here, not in prose.** `Epsilon` (the noise floor: max−min of the baseline prompt's composites over N repeat runs) and `Accept` (strictly greater than best + epsilon; ties and within-noise wobble reject) are library functions so the runner's `compare` (D66) and the loop (D67) share one implementation.

**Rejected.**
- **An LLM judge for claim equivalence** — non-deterministic, and pathological as an optimization target (the loop learns to please the judge). The old harness's judge is exactly what this design retires for good.
- **Pure string rules for claims** — punishes correct paraphrase, so the loop optimizes toward parroting gold wording; embeddings tolerate rewording while staying reproducible.
- **Hungarian assignment** — optimal one-to-one assignment, but claim sets are tens not thousands and the margin rule already rejects ambiguous pairings; greedy is simpler to implement and debug, and is equally deterministic. Revisit only on observed assignment instability.
- **Cross-document claim alignment** — aligning claims only within an already-matched subject keeps pairings small and unambiguous; whole-document alignment invites spurious cross-subject matches.

**Verification.**
- R-KM0O-JI4D — Subject pairing: an extracted subject whose name normalizes equal to a gold alias (case/whitespace/punctuation differing) matches that gold subject; a gold subject with no counterpart is counted a miss and an extracted subject with no counterpart is counted spurious; a name-equal pair with differing `Type` does not match.
- R-KN8K-X9V2 — Field checks on a matched pair: a `Kind` differing only in case still counts correct; a wrong `OccurredAt` counts wrong; field accuracy equals correct/(2×matched) on a hand-computed fixture and 0 when no subject matched.
- R-KOGH-B1LR — Claim alignment accepts a pairing that clears the threshold, clears the margin over its best alternative, and has agreeing digit tokens — verified with an injected fake `EmbedFunc` returning constructed vectors.
- R-KPOD-OTCG — Claim alignment rejects each way independently: a pair below threshold; a pair within `margin` of a competing alternative; and a pair whose two sides carry non-empty, disjoint digit-token sets despite high similarity — each scenario yields no match for the affected claims.
- R-KQWA-2L35 — One-to-one: with one extracted claim similar to two gold claims, it matches only the better one and the other gold claim is a miss — no claim participates in two matched pairs.
- R-KTC2-U4KJ — Rollup arithmetic: on a fixture with known matched/missed/spurious counts, subject and claim precision/recall/F1, field accuracy, and the weighted composite equal the hand-computed values; an empty extraction against non-empty gold yields recall 0, F1 0, not NaN.
- R-ESN9-RFM1 — Honest emptiness: a case with empty gold and an empty extraction scores composite exactly 1.0; the same empty-gold case with one extracted subject scores subject-F1 0 (spurious), and a non-empty-gold case with an empty extraction still scores 0 — the carve-out rewards agreement on emptiness only, never emptiness itself.
- R-KUJZ-7WB8 — Determinism: scoring the same extraction against the same gold twice (cached vectors) yields byte-identical `MarshalDeterministic` output, including across differing map-iteration orders.
- R-KVRV-LO1X — The disk cache: a second scoring run over the same texts performs **zero** calls on the wrapped `EmbedFunc` (counting fake), and a changed model name or changed text misses the cache.
- R-KWZR-ZFSM — Acceptance: `Epsilon` returns max−min over a slice of baseline composites; `Accept` is true only strictly above best+epsilon — equal to best+epsilon, and anything below, reject.
