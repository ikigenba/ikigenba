# Decision 62 — Origin and correlation attribution on every inference call

**Decision.** Every call wiki makes to prompts carries two accounting fields from prompts' grammar (research §12): an **origin** naming the causing identity and a **group_id** correlating the causal chain. The rule, uniformly: *origin is the causing identity when known; the correlation id is minted once at the initial user action and propagated verbatim* (the suite correlation-id standard, `docs/correlation-ids.md`).

**Origin** (`user:<email>` | `trigger:<source>` | `service:wiki`):

- **Ask path** (MCP `ask` tool or the web `?q=` surface): `user:<email>` from the request identity the chassis already provides (MCP `Identity` / the session-gated web request's owner). Applies to `wiki.embed-query`, `wiki.ask-subject`, `wiki.ask-synthesis`.
- **Job-driven pipeline** (`wiki.extract`, `wiki.compile`, `wiki.embed-page`, merge's LLM work): `user:<job.owner>` — the `jobs.owner` column already records who enqueued the ingest; carried from the job row into every call the worker makes for it. When `owner` is empty (legacy rows, internally-generated work): `service:wiki`.
- **Event-triggered work** (none exists today; e.g. a future dropbox ingest-folder consumer): `trigger:<source>` per the grammar — stated here so the case lands correctly when built.

**Correlation (`group_id`)** — a suite ULID minted at the initial user action, per the standard:

- **Job chains**: the ingest job id **is** the correlation id (the durable-root reuse rule — the job row is the user action's durable record). Every call the worker makes for a job carries `group_id` = the job id, verbatim. This restores the old per-job `llm_calls` filter through prompts' `calls` tool.
- **Asks**: no durable root exists, so the ask handler mints one fresh ULID per ask (via `internal/ids`, which grows a `New()` mint following the standard's shape) and threads it through the whole fan-out — the query embed, every per-subject call, and the synthesis call — so one question's full cost and call chain is one `group_id` query away.

**Threading.** Origin and group_id travel as the D5 `Attribution` value, passed explicitly down the call chain (worker → extract/compile/embed-page; ask handler → retrieve/ask sites). No context-smuggling: the old `WithJobID` context key retires with the recorder (D63); attribution is a visible parameter because stages now *must* supply it.

**Rejected.**
- **`service:wiki` for everything** — simplest, but discards causing-identity attribution prompts' `usage` can group by, which the owner explicitly wants tied back to the initial user action.
- **Context-key threading (the old `WithJobID` pattern)** — invisible parameters invite a stage to silently drop attribution; an explicit `Attribution` argument makes the compiler enforce it.
- **A second wiki-side correlation store** — the id lives in prompts' rows; wiki holds it only in flight.

**Verification.**
- R-16VU-W6UV — An ask driven with identity `alice@example.com` posts **all** its `/complete` and `/embed` calls with `origin` `user:alice@example.com` (captured via the httptest prompts through the real ask path).
- R-183R-9YLK — A worker-processed job whose row carries `owner` `bob@example.com` posts its extract, compile, and embed-page calls with `origin` `user:bob@example.com`; the same job with an empty `owner` posts `origin` `service:wiki`.
- R-19BN-NQC9 — One ask mints exactly one fresh correlation id (26-char Crockford-base32 ULID shape) and every call of that ask's fan-out (embed-query, each ask-subject, ask-synthesis) carries it as `group_id`; a second ask mints a different id.
- R-1AJK-1I2Y — Every call the worker makes for a job carries `group_id` equal to that job's id, verbatim, across extract, compile, and embed-page.
