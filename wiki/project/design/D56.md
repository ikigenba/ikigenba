# Decision 56 — `ask` MCP citations are fully-qualified front-door URLs

**Problem.** The MCP `ask` tool returns `{found, answer, citations:[{path,title}]}`, where `path` is the bare public subject path (`entity/tsr`) — not even mount-prefixed. An agent that reached the wiki through the front door (nginx) cannot open that: it is a relative reference with no origin and no `/srv/wiki/` mount. The links an agent gets back do not resolve. We want each `ask` citation to carry a **fully-qualified URL** to the cited page — the exact URL nginx would serve — and to be correct whether the wiki runs in production (`https://<account>.ikigenba.com/srv/wiki/…`) or a local deployment (`http://localhost:8080/srv/wiki/…`).

**Decision.** The change is contained to the **MCP `ask` result builder** (`internal/mcp`). The `ask` internal pipeline is untouched: `ask.Asker` still returns `ask.Answer{Found, Text, Citations []ask.Citation{Path, Title}}` with `Path` the bare relative `type/slug` (D9, D11). The MCP layer composes the URL when it shapes the tool result.

- **Front-door page base.** The front-door origin is already derived once, at startup, by the appkit config layer as `Config.AuthServer` — `https://<account>.ikigenba.com` in production (composed from `IKIGENBA_DOMAIN`) and `http://localhost:8080` in local dev — and exposed on the runtime as `rt.AuthServer()`. The wiki mount is the existing `wiki.Mount` constant (`"/srv/wiki/"`). A cited page is served by the **read surface's subject route** (D45), `GET /subject/{type}/{slug}`, so the front-door page URL for a bare citation path `type/slug` is `<origin>` + `/srv/wiki/` + `subject/` + `type/slug`. The page base bakes that `subject/` route segment in:

  ```go
  // in NewHandler(rt *appkit.Router, opts ...Option):
  pageBase := strings.TrimRight(rt.AuthServer(), "/") + wiki.Mount + "subject/" // e.g. "https://acct.ikigenba.com/srv/wiki/subject/"
  ```

  `AuthServer` carries no trailing slash and `wiki.Mount` has a leading and trailing slash, so the join yields exactly one separator; the `TrimRight` is belt-and-suspenders against an override that adds one. The `"subject/"` literal mirrors the read surface's own citation-href composition (`internal/web` builds `"subject/" + citation.Path` against the same route), so the two agree on the route by construction. The base is captured into the `Handler` used by `Tools(...)` (a new unexported field `pageBase`, set by `NewHandler` before it builds the tool table — no new public `Option`, because the value comes from `rt`, not a caller).

- **URL composition, per citation.** `askToolResult` gains the page base and emits `url` in place of `path`:

  ```go
  func askToolResult(answer any, pageBase string) map[string]any {
      found, text, sourceCitations := answerFields(answer)
      citations := make([]map[string]string, 0, sourceCitations.Len())
      for i := 0; i < sourceCitations.Len(); i++ {
          c := indirect(sourceCitations.Index(i))
          citations = append(citations, map[string]string{
              "url":   pageBase + stringField(c, "Path"), // e.g. "https://acct.ikigenba.com/srv/wiki/subject/entity/tsr"
              "title": stringField(c, "Title"),
          })
      }
      return map[string]any{"found": found, "answer": text, "citations": citations}
  }
  ```

  `handleAskCall` passes `h.pageBase`. The relative `Path` is read from the citation and never emitted on its own — the result field is `url`, and `path` is gone from the `ask` result. The `title` is unchanged (the subject's readable name). The `subject/` route segment lives entirely in `pageBase`, so `askToolResult` stays a dumb `base + path` concatenation.

- **Empty / honest-empty is untouched.** URL composition runs only inside the per-citation loop, so an empty `Citations` stays an empty `citations`, and the `found:false` honest-empty result (D9) is byte-for-byte as before. No new failure mode: `pageBase` is a plain prefix string, always set.

**Scope containment (why this seam and not the source).** Changing `ask.Citation.Path` at the source would leak absolute URLs into the **human web read surface** (D44), whose ask page renders the same citations as in-browser relative hrefs (`"subject/" + citation.Path`) under nginx (correct there — a browser resolves them against the current origin). This Decision therefore changes **only** the MCP tool result an agent consumes; `ask.Answer` / `ask.Citation`, `internal/ask`, and `internal/web` are unchanged, and the web ask page keeps its relative hrefs.

**Downstream doc truth-up.** D10's `ask` result row (`citations:[{path,title}]`) and its "paths out … `ask` citations all carry `type/slug` paths" bullet are rewritten in place to reflect the `url` field (the derived-from-path, no-internal-id guarantee holds — the URL is built from the public path and carries no ULID). D10 pins no id on the `ask` citation shape, so no id is retired.

**Rejected.**
- **Reconstruct the origin per-request from `Host` / `X-Forwarded-*`.** Adds a trust surface (services deliberately trust only the headers nginx sets, and `Host` is not among them today) and a front-door config we do not guarantee, for no gain on a single-tenant box. `AuthServer` is the canonical per-box front-door origin, already the value used to compose `RESOURCE_ID`/`AUTH_SERVER`.
- **Keep `path` and add `url` alongside.** The relative `path` has no value to an agent behind the front door and invites ambiguity about which to open. One unambiguous field.
- **Change `ask.Citation.Path` at the source.** Leaks absolute URLs into the web surface (D44), which needs relative hrefs. The MCP-result seam keeps the change where it belongs.
- **Share the `subject/` route segment as an exported constant across `internal/web` and `internal/mcp`.** Would unify the two literals and prevent drift if the route ever moves, but it forces this Decision — whose whole premise is "change only the MCP result builder; `internal/web` is untouched" — to reach into `internal/web` to consume the constant, enlarging the seam for a route that has never moved. A plain literal mirroring `web.go`'s existing style is the minimal in-scope fix; unifying is the right move only at the moment the route actually changes.

**Verification.**
- R-Y7OR-PH1I — The MCP `ask` result's citations carry a `url` field and **no** `path` field: for an `ask.Answer` whose citation `Path` is `entity/tsr`, the tool result's citation is `{url, title}` (title preserved), and no `path` key appears on any citation.
- R-HOJB-ZR3T — **Prod-vs-local correctness, with the `subject/` route segment (falsifiable across both origins):** with the handler built over `rt.AuthServer() == "https://acct.ikigenba.com"`, a citation for path `entity/tsr` yields `url == "https://acct.ikigenba.com/srv/wiki/subject/entity/tsr"`; with `rt.AuthServer() == "http://localhost:8080"` the same citation yields `url == "http://localhost:8080/srv/wiki/subject/entity/tsr"` — the URL includes the `/srv/wiki/subject/` route prefix in full (a URL missing the `subject/` segment fails), with exactly one `/` between each part (no `//`, no missing mount, no missing route segment).
- R-YA4K-H0IW — **Scope containment:** `ask.Asker.Ask` still returns `ask.Answer` citations whose `Path` is the bare relative `type/slug` (no origin, no mount, no `subject/`) — the absolute-URL composition lives only in the MCP result builder, and the `internal/web` ask page continues to render relative hrefs unchanged.
