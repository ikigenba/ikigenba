# Decision 9 — Chassis-owned standard tools: `health` and `reflection`

**Problem.** Every service's MCP surface carries the same two non-domain tools
— `health` (the health envelope, same bytes as the HTTP `/health` route) and
`reflection` (the event-plane self-description over the registry and
subscriptions) — re-implemented per service from the same Router accessors. They
are pure chassis: a change to the envelope or the reflection schema is a
twelve-edit change today.

**Decision.** `appkit/mcp.New` (D8) registers `health` and `reflection`
automatically from its `Options`; services declare only domain tools. The
implementations replicate the per-service semantics exactly:

- **`health`** — no input; returns `server.Envelope(version, service, details)`
  as a JSON text result, where `details` comes from `Options.Health` (nil →
  `{}`), so the MCP tool and the HTTP `/health` route render from the single
  envelope builder they already share.
- **`reflection`** — optional `event_type` string input.
  - Without `event_type`: returns the event-graph index — `publishes` rendered
    from `Options.Publishes()` when non-nil (the dynamic-producer preference,
    e.g. cron) else `Options.Events`, and `subscribes` rendered from
    `Options.Subscriptions` (nil → empty) — the same index shape the services
    emit today (`outbox.Registry.Index()` + the subscription list).
  - With `event_type`: returns that type's detail schema
    (`Registry.Detail`); an unknown type returns an `isError` result whose
    message names the known types (the services'
    `reflectionUnknownTypeError` behavior).
- Both appear in `tools/list` with the descriptions the services use today, so
  a converted service's listed surface is unchanged.
- The names are **reserved**: D8's `New` rejects a declared Tool named `health`
  or `reflection`, so chassis and service can never silently disagree about who
  answers.

**Rejected.**
- **Leaving health/reflection service-side (plumbing-only hoist)** — keeps two
  copy-pasted tools in every service and re-opens drift on the reflection
  schema; the inputs (Envelope, Registry, Subscriptions) are all chassis-held
  already.
- **A services-may-override seam** — no service has ever legitimately varied
  either tool; an override seam is a hypothetical-future flag, rejected on
  principle (complexity only when forced).
- **Folding reflection into health** — different axes (liveness vs event
  graph), different consumers; the suite's tool vocabulary already treats them
  as distinct.

**Verification** (through the D8 `ServeHTTP` seam, with real `outbox.Registry`
and `consumer.Subscription` values).
- R-ML2U-CBQM — with zero declared tools, `tools/list` still lists `health` and
  `reflection`; `tools/call health` returns a result embedding the
  `server.Envelope` shape — `status`, `service`, `version` matching Options,
  and `details` from the injected reporter (a reporter returning
  `{"queue": 3}` is visible in the result).
- R-MMAQ-Q3HB — `tools/call reflection` (no arguments) returns `publishes`
  listing the registry's event types and `subscribes` listing the injected
  subscriptions; with `Options.Publishes` set, its returned registry is used
  instead of `Options.Events` (a type present only in the live provider
  appears; one present only in Events does not).
- R-MNIN-3V80 — `tools/call reflection` with `event_type` naming a registered
  type returns that type's detail schema (its payload fields appear).
- R-MOQJ-HMYP — `tools/call reflection` with an unknown `event_type` returns an
  `isError` result whose message contains the known type names — not a crash,
  not an empty success.
