# Decision 10 — Chassis-owned consumer loops: `Spec.Consumers`

**Problem.** The event-plane consumer *engine* is already shared
(`eventplane/consumer.Run` owns the SSE client, the `feed_offset` cursor, and
the reconnect/stall semantics), but every consuming service still hand-writes
the composition-root wiring around it: a ~30-line `runConsumer` per upstream
that resolves a feed-URL env var, assembles `consumer.Config`, logs the start
line, calls `Run`, and wraps the error — plus the `var rt *appkit.Router`
capture dance that smuggles the Router out of `Handlers` so those closures can
reach `rt.DB()`/`rt.Logger()`. Worse, the *declaration* of the consumer role is
already split across three Spec fields that must manually agree: `Consumes`
(manifest `CONSUMES=`), `Subscriptions` (reflection), and the actual `Workers`
loops. notify's env-name drift (`CRM_FEED_URL` vs the scripts/prompts
`<APP>_<SRC>_FEED_URL` convention) is the existing proof that copyable
boilerplate rots.

**Decision.** The consumer role becomes a single declared table the chassis
wires end to end:

```go
// appkit (root package)

// Consumer declares one upstream this service consumes. Source and
// Subscriptions are static (readable by the one-shot manifest verb and the
// reflection tool without a Router); Handler is a factory the chassis calls
// with the finished Router after Handlers (and Producer) have run.
type Consumer struct {
    Source        string                  // upstream app name, e.g. "crm"
    Subscriptions []consumer.Subscription // what this loop listens for
    Handler       func(rt *Router) consumer.Handler
}

// Spec gains:
Consumers []Consumer
```

For each entry, at serve time the chassis:

1. resolves the feed URL as `<APP>_<SRC>_FEED_URL`, defaulting to
   `registry.BaseURL(source) + "/feed"`, and the first-subscription choice as
   `<APP>_<SRC>_FROM`, defaulting to `"tail"` — pure `getenv` resolution in
   `appkit/config`, table-shaped like the existing DB/WWW path resolution (D5);
2. builds `consumer.Config{FeedURL, From, DB: rt.DB(), Source,
   ConsumerID: spec.App, Logger: rt.Logger()}` — `ConsumerID` is always the app
   name (all three existing consumers already do exactly this);
3. calls `entry.Handler(rt)` strictly after `Handlers` (and `Producer`) have
   completed, so the factory sees the same fully-built Router;
4. launches `consumer.Run` on the serve context as an internal worker with the
   exact `Workers` lifecycle semantics (a returning worker cancels serve; the
   first error is the exit error), logging the start
   (`"starting <app> <source> consumer"` with `feed_url`/`from` — never a
   secret) and wrapping any escape as
   `event-plane consumer (<source>): <err>`.

**Derivations and the legacy fields.** When `Consumers` is non-empty, the
manifest verb emits `CONSUMES=` from the table's sources in declaration order,
and the reflection tool's `subscribes` list is the concatenation of the
entries' `Subscriptions` in declaration order. The legacy `Spec.Consumes` and
`Spec.Subscriptions` fields remain for unconverted services (strict
additivity), but a Spec that sets `Consumers` together with either legacy field
fails loudly at startup — merging semantics would reopen the drift this seam
closes. `Spec.Workers` survives unchanged as the general background-task seam;
it is not deprecated and may coexist with `Consumers`.

**Module wiring.** `appkit/go.mod` gains `require registry v0.0.0` plus the
committed `replace registry => ../registry`, mirroring the existing
`eventplane` replace-sibling. **Known consequence (outside `appkit/`, not built
by any appkit phase):** a dependency's `replace` directives do not apply
transitively, so — exactly as the `eventplane` require already forced — every
module that requires appkit must mirror the replace. notify, prompts, scripts,
and sites already carry `replace registry => ../registry`; the remaining nine
(crm, cron, dashboard, dropbox, github, gmail, ledger, webhooks, wiki) need the
one-line addition or their `GOWORK=off` builds break. That mechanical sweep is
an operator step recorded in `project/plan/plan.md` § Operator steps, per this
workspace's D4 precedent.

**Rejected.**
- **An eventplane helper the app still wires into `Workers`** — folds the
  Config assembly but keeps the rt-capture dance and the three-field manual
  agreement; the drift risk survives.
- **A whole-table provider `func(rt *Router) []Consumer`** — the one-shot
  `manifest` verb must emit `CONSUMES=` without opening a DB or building a
  Router, so sources must be static in the Spec literal.
- **Plain handler values instead of factories** — forces apps to build handler
  dependencies without the Router (a second logger, a second config
  resolution), re-creating the duplication the seam removes.
- **A per-entry default-feed-URL field instead of the registry dependency** —
  one line of copyable boilerplate per upstream survives per service; notify's
  env-name drift shows exactly how that rots. registry is a static leaf module
  appkit already conceptually sits above (every app resolves its own
  `Spec.Port` from it).
- **Allowing `Consumers` to mix with the legacy fields** — flexibility with no
  customer; the manual-agreement drift returns through the side door.
- **Fail-loud (no default) feed URLs** — prod sets no feed env vars today; this
  would break every consumer deploy until units grow env lines, for nothing.

**Verification** (config resolution table-tested over injected `getenv` maps;
the loop end-to-end against a real SSE feed served by `httptest` over a real
`t.TempDir()` SQLite DB — the substrate that can falsify cursor and delivery
claims).

- R-4199-A0U9 — the manifest emitted for a Spec with
  `Consumers: [{Source:"a"},{Source:"b"}]` and no legacy `Consumes` contains
  exactly `CONSUMES=a,b`; a Spec with neither field emits no `CONSUMES` line.
- R-42H5-NSKY — the reflection tool for a Spec whose two `Consumers` entries
  carry distinct subscription lists reports `subscribes` equal to their
  concatenation in declaration order, with no legacy `Spec.Subscriptions` set.
- R-44WY-FC2C — a Spec setting `Consumers` together with legacy `Consumes`
  (and, separately, with legacy `Subscriptions`) fails loudly at startup with
  an error naming the conflict, rather than merging or silently preferring one.
- R-464U-T3T1 — with `<APP>_<SRC>_FEED_URL` unset, the resolved feed URL for
  app `A` consuming source `S` equals `registry.BaseURL(S) + "/feed"` (the
  loopback address from the registry table, not a hardcoded literal).
- R-47CR-6VJQ — with `<APP>_<SRC>_FEED_URL` set to a sentinel URL, the
  resolved feed URL is that sentinel — the override wins over the registry
  default.
- R-48KN-KNAF — the resolved `From` for app `A`/source `S` is `"tail"` when
  `<APP>_<SRC>_FROM` is unset and the env value verbatim when set.
- R-49SJ-YF14 — a serve run whose Spec declares **two** consumers against two
  real `httptest` SSE feeds delivers each feed's events to that entry's handler
  (built by a factory that observably received the post-`Handlers` Router) and
  advances **two independent** `feed_offset` cursor rows in the real DB, each
  keyed by its `Source` with `consumer_id` equal to `Spec.App` — a shared or
  swapped cursor row, or a factory invoked before `Handlers`, fails this.
