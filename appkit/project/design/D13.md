# Decision 13 — Identity keys on `X-Owner-Id`; the four owner headers ride the request

**Problem.** The suite is converting from email-keyed to id-keyed ownership
(`docs/owner-id-design.md`): the dashboard mints a stable opaque owner id
(`identities.id`) and nginx forwards it fail-closed on every authenticated
request as `X-Owner-Id`, beside `X-Owner-Email`, `X-Owner-Name`, and
`X-Owner-Picture`. Email is about to become user-mutable (GitHub alternate
login), so it can no longer serve as the identity key. The chassis's
`server.Identity` today carries only `OwnerEmail`/`ClientID` and the identity
gate refuses on a missing `X-Owner-Email` — services literally cannot key on
the id, and the gate anchors trust to the mutable field.

**Decision.** `server.Identity` carries all four owner headers plus the client
id; the identity gate keys on `X-Owner-Id` **alone**, as a hard flip — there is
no interim state where an email-only request is admitted:

```go
package server // appkit/server

// Identity is the authenticated caller, as told to us authoritatively by
// nginx. OwnerID is the stable key: all scoping, lookups, and authorization
// compare it and only it. OwnerEmail/OwnerName/OwnerPicture are display data,
// never used for logic.
type Identity struct {
    OwnerID      string // X-Owner-Id
    OwnerEmail   string // X-Owner-Email
    OwnerName    string // X-Owner-Name
    OwnerPicture string // X-Owner-Picture
    ClientID     string // X-Client-Id
}
```

- `requireIdentityHeaders` refuses exactly when `X-Owner-Id` is empty, with
  the unchanged refusal shape: `401`, the `WWW-Authenticate` bearer challenge
  pointing at the PRM document, and the `unauthorized` JSON body. The other
  three owner headers and `X-Client-Id` are copied into `Identity` verbatim
  and are never gated on — an id-only request (no email) is admitted.
- The trust rationale is unchanged: nginx is the sole ingress and sets the
  `X-Owner-*` headers only after a successful auth_request, the server binds
  loopback, and loopback machine callers assert the headers themselves.
- Downstream in-chassis consumers of the identity (D8's transport hand-off,
  D9's health envelope) carry the widened shape; their contracts are stated in
  their own Decisions.

**Rejected.**
- **Gating on both `X-Owner-Id` and `X-Owner-Email`** — re-anchors trust to
  the mutable field for no gain; the dashboard always sends both today, but
  the id is the contract and the email is display.
- **An interim accept-either gate (email fallback)** — the id header is
  already live fail-closed on every surface in production; a fallback window
  would protect nothing while weakening the gate and adding a removal phase.
- **Only `OwnerID` + `OwnerEmail` on `Identity` (drop Name/Picture)** — the
  headers are already on every request; a service rendering display data would
  need a second lookup the suite deliberately doesn't have (no resolve
  endpoint, per the suite design).

**Verification** (via `httptest` through the real gated route table —
`server.New` + `RequireIdentity` — the substrate that can falsify the gate).
- R-DDVL-DPVB — a request without `X-Owner-Id` is refused `401` with the
  `WWW-Authenticate` bearer challenge and the inner handler **not** invoked,
  **even when the request carries `X-Owner-Email`** — a build still gating on
  email fails this.
- R-DF3H-RHM0 — a request carrying `X-Owner-Id` but **no** `X-Owner-Email`
  reaches the inner handler — the discriminating property against the old
  email-keyed gate.
- R-DGBE-59CP — the `Identity` visible to the inner handler (via
  `IdentityFrom`) carries the request's `X-Owner-Id`, `X-Owner-Email`,
  `X-Owner-Name`, `X-Owner-Picture`, and `X-Client-Id` header values verbatim
  in the corresponding fields.
