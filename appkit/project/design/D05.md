# Decision 5 — WWW-root resolution in `appkit/config`

**Problem.** Services are moving their Go-served web assets (page templates +
`static/` CSS/fonts) out of `//go:embed` and onto disk. The chassis needs one
authoritative answer to "where is this service's web-asset root?" that works in
all three situations the existing DB-path composition already handles: the
deployed box, local dev, and an explicit override.

**Decision.** `appkit/config` resolves a per-app **www root** exactly the way it
resolves the DB path — composed from `IKIGENBA_ROOT` with a dev fallback and a
per-app env override that wins:

```go
// Config gains one field.
type Config struct {
    // …existing fields…
    WWWPath string // web-asset root (env <APP>_WWW_PATH)
}
```

Resolution, in a new `composeWWWPath(getenv, up, app)` alongside
`composeDataPaths`:

1. `IKIGENBA_ROOT` set (the box) → `<root>/<app>/share/current/www` — i.e.
   `/opt/<app>/share/current/www`, inside the shipped, versioned, atomically
   swapped `share` tier. Web assets are **release artifacts**: they deploy and
   roll back with the binary, are never written at runtime, and are not part of
   the backed-up `state/`. (`state/www` remains what it is today: the sites
   service's nginx-served user content — a different thing this Decision does
   not touch.)
2. `IKIGENBA_ROOT` unset (dev) → `./share/www` — correct when the process runs
   from the service root (`go run ./cmd/<app>` in `<app>/`, or the ralph loop),
   and pointing at the same files `bin/ship` will bundle.
3. `<APP>_WWW_PATH` set → wins verbatim over either composition (the seam
   `bin/start` uses to point a repo-root-launched dev binary at
   `$repo/<app>/share/www`).

`Resolve` always composes `WWWPath` (it is just a string; nothing is checked at
resolve time). Whether the path must *exist* is the chassis-integration
decision's concern (D7): only a service that declares `Spec.WWW` has its root
validated, so unconverted services are untouched by a path that resolves to
nothing.

**Rejected.**
- **`state/www` as the root** — puts release-owned UI files in the backed-up,
  runtime-mutable state tree: every deploy would either clobber state or drift
  from the shipped version, and restore semantics would be ambiguous. The
  `share` tier already has exactly the right lifecycle (shipped, versioned,
  atomic swap, rollback).
- **Embedding with an extract-on-boot step** — reintroduces the embed machinery
  the suite is removing, plus a write path into the release tree.
- **A `Spec` field carrying the path** — the path is environment, not identity;
  it belongs with the other env-composed paths in `config`, overridable the
  same way.
- **Validating existence in `Resolve`** — would break every unconverted service
  and every non-serve verb; existence is a serve-time concern gated on
  `Spec.WWW` (D7).

**Verification.**
- R-LWOU-OWWQ — with `IKIGENBA_ROOT=/opt` (and the domain/data-path env the
  existing tests set), `Resolve("crm", …)` returns
  `WWWPath == "/opt/crm/share/current/www"` — the shipped `share/current` tier,
  not a `state/` path.
- R-LXWR-2ONF — with `IKIGENBA_ROOT` unset, `Resolve("crm", …)` returns
  `WWWPath == "./share/www"`.
- R-LZ4N-GGE4 — with `CRM_WWW_PATH=/somewhere/else` set, `Resolve("crm", …)`
  returns `WWWPath == "/somewhere/else"` regardless of `IKIGENBA_ROOT` — the
  explicit override beats both compositions.
